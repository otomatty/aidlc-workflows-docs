v2.2.0でルーティングが変更されました: キーワードの明確な一致は一致したスコープ名を示す1行確認（例: bugfix）を出し、一致がない・リッチな記述の場合はコンポーズ提案（下記）にルーティングされます。旧来の静的feature既定確認は廃止。検出後は必ずスコープ名とステージ数を示す確認プロンプトが出ます。プロジェクト単位の既定スコープは AWS_AIDLC_DEFAULT_SCOPE で設定できます。

アダプティブコンポーザー（v2.2.0）

9つのストックスコープに加えて、コンポーザーエージェントがタスクに合わせたカスタムのEXECUTE/SKIPステージグリッドを合成できます。

  • /aidlc compose "<task>" — ストックスコープが一致する場合でもコンポーザーを強制起動。--new-scope でカスタムスコープ合成を強制
  • /aidlc compose --report <path> — スキャンレポート（SonarQube等）をトリアージし、コンパクトな「修正して出荷」グリッドに構成。ストック一致（bugfix / security-patch）ならファイル書き込みなしでルーティング
  • in-flight recompose — 実行中に未着手ステージのEXECUTE/SKIPを反転（recompose --skip/--add）。監査ロック下で厳格検証され、完了済み・カーソル後方・walking-skeletonゲートアンカーの反転は拒否。チャットでの「このステージはスキップできる?」もコンダクターが認識して同じゲートに通す
  • 合成スコープは keywords: [] で書き出され、一回限りのプランが将来のキーワードルーティングを書き換えません。提案は常に人間の承認ゲート（approve/edit/reject）を通ります

深度とテスト戦略

深度（3レベル）

Minimal / Standard / Comprehensive。各ステージが生成するアーティファクトの詳細度を制御。スコープごとに既定値を持ち、--depth で上書き可能。

テスト戦略（3レベル）

Minimal / Standard / Comprehensive。生成されるテストの量を深度とは独立に制御。--test-strategy で指定。workshopスコープの既定はMinimal。

リーンスコープ（poc / bugfix / refactor / infra / security-patch）が上流プロデューサーステージをスキップする場合の入力欠落は、v2.1.7以降 consumes_absent（expected: true/false 注釈付き）として設計上の欠落と事故の欠落が区別され、各ステージ本文は「(if exists)」の明示的フォールバック付きで入力を読みます。詳細はアーキテクチャを参照。
