# 運用フェーズ — ステージリファレンス（4.1〜4.7）

<a id="phase-overview"></a>
## フェーズの概要

運用フェーズは、AI-DLC ライフサイクルの全 5 フェーズにおける第 5 フェーズです。構築フェーズで構築・テストされたソフトウェアを、デプロイ、監視、インシデントへの備え、パフォーマンス検証、継続的な最適化へと進めます。パイプライン設定、環境プロビジョニング、デプロイ実行、可観測性、インシデント対応、パフォーマンス検証、フィードバック収集にまたがる 7 つのステージ（4.1 から 4.7）で構成されます。

運用の全 7 ステージは **条件付き** です。スコープと実行計画に応じて実行されます。たとえば、最小実用製品、概念実証、バグ修正、リファクタリングのスコープでは運用フェーズ全体をスキップします。インフラストラクチャおよびセキュリティパッチのスコープでは、一部のステージ（デプロイと環境のステージ）を実行します。

すべてのステージは **インライン** で実行されます（運用フェーズにサブエージェントはありません）。すべてのステージで、承認ゲート、質問形式、完了メッセージ、状態追跡について `stage-protocol.md` に従います。

| 4.4   | 可観測性の設定      | 条件付き | 監視、ダッシュボード、アラーム、トレーシングの設定が必要な場合に実行する    | 運用担当    | （なし）              | インライン                           |

| 4.7   | フィードバックと最適化  | 条件付き | 継続的な運用監視と最適化が必要な場合に実行する    | 運用担当    | AWS プラットフォーム担当  | インライン                           |

<a id="multi-agent-stages"></a>
### 複数エージェントのステージ

3 つの運用ステージには複数のエージェントが関与します。

- **4.2 環境プロビジョニング**: AWS プラットフォーム担当（主担当）+ 開発セキュリティ運用担当（セキュリティ態勢の検証）+ コンプライアンス担当（データレジデンシー、規制統制）
- **4.3 デプロイメント実行**: パイプライン・デプロイ担当（主担当）+ 開発担当（データベースマイグレーション）
- **4.7 フィードバックと最適化**: 運用担当（主担当）+ AWS プラットフォーム担当（コスト最適化、ドリフト検出）

すべての場合で、コンダクターはまず主担当エージェントを呼び出し、その出力をコンテキストとして支援エージェントを呼び出します。すべての委譲はコンダクターが実行し、エージェント同士が相互に呼び出すことはありません。

## ステージ 4.7: 継続的なフィードバックと最適化

<a id="metadata-6"></a>
### メタデータ

| プロパティ          | 値                                                                                             |
|-------------------|---------------------------------------------------------------------------------------------------|
| ステージ             | 4.7                                                                                               |
| フェーズ             | 運用                                                                                         |
| 実行         | 条件付き（単発のデプロイではスキップ）                                                        |
| 主担当エージェント        | 運用担当                                                                                  |
| 支援エージェント    | AWS プラットフォーム担当（コスト最適化、ドリフト検出）                                           |
| 入力            | 運用フェーズの全成果物、本番監視データ                                         |

<a id="purpose-6"></a>
### 目的

SLO 準拠レビュー、コスト最適化分析、インフラストラクチャのドリフト検出、運用上の洞察の収集を行います。これは AI-DLC ワークフロー全体の**最終ステージ**です。

<a id="outputs-6"></a>
### 出力

| 成果物                              | 説明                                                |
|---------------------------------------|------------------------------------------------------------|
| SLO レポート                         | SLO 準拠レポート、エラーバジェット消費率              |
| コスト分析                      | AWS コストエクスプローラー分析、最適化の推奨事項   |
| ドリフトレポート                       | AWS 設定ドリフト検出レポート、信頼できるアドバイザーレビュー  |
| フィードバックループ                      | 運用上の洞察、改善提案、次のアイデア創出サイクルへの入力 |
| フィードバック最適化質問書    | 回答を含む確認質問                          |

<a id="approval-gate----three-option-unique"></a>
### 承認ゲート — 3 選択肢（固有）

ステージ 4.7 には**固有の 3 選択肢の承認ゲート**があります。

1. **承認** — ワークフロー完了。AI-DLC ライフサイクル全体が終了します。
2. **変更を要求** — 修正フィードバックを提供します。
3. **新しいアイデア創出サイクルを開始** — フィードバックループの洞察を新しいステージ 1.1 にフィードバックします。

これは、AI-DLC ライフサイクルの循環的な性質を表しています。

<a id="phase-summary"></a>
## フェーズの概要

**デプロイメントステージ（4.1〜4.3）:**
- 4.1 デプロイメントパイプライン — CD パイプライン設定、デプロイメント戦略、ロールバックランブック
- 4.2 環境プロビジョニング — セキュリティ態勢レビューを伴う AWS 環境のプロビジョニングと検証
- 4.3 デプロイメント実行 — 成果物のデプロイ、スモークテスト、ヘルスチェック、データベースマイグレーション

**運用準備ステージ（4.4〜4.6）:**
- 4.4 可観測性の設定 — ダッシュボード、アラーム、SLO、ログクエリ、トレーシング、異常検出
- 4.5 インシデント対応 — ランブック、インシデント計画、エスカレーションマトリクス
- 4.6 パフォーマンス検証 — 負荷テスト、NFR 目標の検証、キャパシティプランニング

**継続的改善（4.7）:**
- 4.7 フィードバックと最適化 — SLO 準拠、コスト分析、ドリフト検出、フィードバックループ

**スコープの適用範囲:**
- エンタープライズ / 機能 / ワークショップ: 全 7 ステージ
- インフラストラクチャ: ステージ 4.1〜4.4（デプロイメントパイプライン、環境プロビジョニング、デプロイメント実行、可観測性の設定）
- セキュリティパッチ: ステージ 4.1、4.3（デプロイメントパイプライン、デプロイメント実行）
- 最小実用製品 / 概念実証 / バグ修正 / リファクタリング: 運用フェーズ全体をスキップ
