<a id="scopes"></a>
# スコープ

スコープは、フレームワークの 32 ステージのうち、ある種類の作業に対して *どれを* 実行し、どれを外すかを決める調整つまみです。バグ修正には市場調査もデプロイパイプラインも不要ですが、規制対象のエンタープライズ機能にはその両方が必要です。利用者に毎回ステージを手で選ばせる代わりに、AI-DLC には名前付きの 9 スコープが用意されています。各スコープはステージ全体に対して厳選した EXECUTE / SKIP の判定であり、既定の深度とテスト戦略も組み合わせて持っています。スコープを選べば、残りが連鎖的に決まります。

ハーネスエンジニアにとって、スコープは完全にデータです。他の基本要素と同様にファイルとして著述します。構成は 2 つの部分に分かれます。1 つは `core/scopes/aidlc-<name>.md` ファイル（識別情報。名前、深度、キーワード、説明を持つ）、もう 1 つは各ステージの所属タグ（フロントマターの `scopes:` 一覧で、そのステージがどのスコープで走るかを宣言する）です。スコープを追加したり調整したりするのに TypeScript は不要です。この章では、スコープが何でできているか、チーム用スコープをどう追加するか、既存スコープをどう調整するか、そしてツールが何を検査し、何を人に任せるのかを説明します。

9 つのスコープの一覧全体、用途、そして利用者が見る経路選択表は、ユーザーガイドの [スコープ、深度、テスト戦略](../guide/05-scopes-and-depth.md) を参照してください。この章は、同じデータの著述側です。

---

<a id="what-a-scope-is-made-of"></a>
## スコープは何でできているか

スコープは 2 か所に著述されます。この分割こそが要点です。スコープの *識別情報* は専用ファイルにあり、*所属関係*（どのステージがそのスコープで走るか）はステージ側へ転置して置かれます。

**1. スコープファイル — `core/scopes/aidlc-<name>.md`。** `core/sensors/` と同じく、スコープごとに 1 ファイルです。`feature` スコープのフロントマターは次のようになります。

```yaml
---
name: feature
depth: Standard
keywords: []
description: Default for new features, practical depth
---

# feature scope

Prose intent: why these stages, why skip those.
```

フロントマターのフィールドは、1 つの必須フィールドと 3 つの任意設定に分かれます。

| フィールド | 必須 | 役割 |
|-------|----------|--------------|
| `name` | はい | スコープ名。ファイル名の拡張子を除いた部分と一致する必要があります（`aidlc-feature.md` → `feature`）。 |
| `depth` | はい | 既定の詳細度。`Minimal`、`Standard`、`Comprehensive` のいずれかです。 |
| `testStrategy` | いいえ | 深度と独立にテスト量を上書きします。既定では `depth` に一致します。 |
| `keywords` | いいえ | `/aidlc <freeform text>` の自動検出に使う自然言語の引き金。空の一覧なら対象外です。 |
| `description` | いいえ | `/aidlc --help` に表示される 1 行の説明です。（`SKILL.md` のコンパイル済みスコープ表には `Scope` / `Depth` / `TestStrategy` / `EXECUTE` / `Total` だけが出て、説明は出ません。） |

**2. 所属タグ — 各ステージの `scopes:` フロントマター。** ステージは、自分がどのスコープで走るかを `core/aidlc-common/stages/<phase>/<slug>.md` のフロントマターに書きます。

```yaml
scopes:
  - enterprise
  - feature
  - mvp
```

あるステージがスコープ名を書いていれば、そのスコープでは `EXECUTE` です。書いていなければ `SKIP` です。ビルド工程の `bun .claude/tools/aidlc-graph.ts compile` は、各ステージの `scopes:` 一覧をコンパイル済みの EXECUTE / SKIP 表、つまり `.claude/tools/data/scope-grid.json` へ *転置* します。これは `stage-graph.json` と同じく純粋な転置であり、`compile --check` によって差分監視されます。実行時に読むのはこの表であり、手で編集してはいけません。3 つの初期化ステージはすべてのスコープを必ず名指しします（常に走るからです）。

ここで特に理解しておく価値がある判断点は、`depth` と `testStrategy` の関係です。深度は各ステージが出力する成果物の詳細量を制御し、テスト戦略は生成するテストの量を制御します。両者は意図的に独立しています。多くのスコープは `testStrategy` を省略し、`depth` をそのまま継承します。つまり標準深度のスコープは標準量でテストします。出荷済みスコープでこの関係を崩す例が `workshop` です。`"depth": "Standard"`（参加者が学べるよう成果物は十分に出す）でありつつ、`"testStrategy": "Minimal"`（セッションを速く保つため高速なナイキストテストにする）です。こういう分割を望むスコープでは両方を宣言します。各段階の意味はユーザーガイドの [3 つの深度](../guide/05-scopes-and-depth.md#the-3-depth-levels) と [3 つのテスト戦略](../guide/05-scopes-and-depth.md#the-3-test-strategy-levels) を参照してください。

フィールドごとの網羅的な契約、`keywords` が単語境界で照合されること、自由形式の呼び出しが曖昧なときにアルファベット順でどう同順位を解決するかまで含む正式仕様は、開発者リファレンスの [コントリビューション § スコープの追加](../reference/11-contributing.md#adding-a-scope) にあります。この章は判断の要点をまとめたものであり、そちらが正式仕様です。

<a id="how-scopes-relate-to-stages"></a>
## スコープとステージの関係

スコープとステージは、互いを反対側から指し示します。両方向を同時に意識すると理解しやすくなります。

**ステージ**は自分自身の識別情報、つまりフェーズ、主担当エージェント、受け取り・生成する成果物、そして現在は `scopes:` 一覧によってどのスコープで走るかを宣言します。**スコープ**は自分自身の識別情報、つまり名前、深度、キーワード、説明を、自分の `.md` ファイルに宣言します。スコープファイルの中にステージごとの所属関係は書きません。所属関係はステージ側にあります。両者を結び付けるのはスコープ名です。新しいステージを追加したら（[ステージの追加](02-adding-a-stage.md) を参照）、スコープへの所属は *そのステージ側* に書きます。`scopes:` 一覧に、走らせたいすべてのスコープ名を入れるのです。何も書かれていないステージは、どこでも `SKIP` です。コンパイル時の転置により、それらのステージ別一覧が表に変換されるので、所属関係は 1 回だけステージ上に著述します。9 個の別々のスコープブロックに再宣言する必要はありません。

この分離は、このガイド全体の基礎であるデータとコードの線引きと同じです（[ハーネスエンジニアガイド](00-overview.md) を参照）。スコープファイルは *識別情報* に関するデータ、ステージの `scopes:` 一覧は *所属関係* に関するデータ、コンパイル済みの表はその転置です。

<a id="adding-a-team-scope"></a>
## チーム用スコープを追加する

たとえばチームが `hotfix` スコープを欲しいとします。`bugfix` よりさらに軽量で、緊急の本番修正に対して回帰テストとデプロイだけは欲しいが、ほかはいらない、という想定です。必要な変更は、新しいスコープファイル、実行したい各ステージへの `scopes:` タグ、そして再コンパイルです。以下の規律に従ってください。確認手順と完全なコマンド行は [コントリビューション § スコープの追加](../reference/11-contributing.md#adding-a-scope) にあります。

<a id="steps"></a>
### 手順

1. **`core/scopes/aidlc-hotfix.md` を置く。** もっとも近い既存スコープである `aidlc-bugfix.md` をコピーし、フロントマターを編集します。`name: hotfix` を設定し、`depth` を選び、自由形式の自動検出に使いたい `keywords`（`[hotfix, urgent]` など）があれば加え、ヘルプ文用の `description` を書き、`depth` と異なるときだけ `testStrategy` を追加します。本文には意図を短い文章で書きます。

2. **`hotfix` で走らせたいステージにタグを付ける。** `EXECUTE` にしたい各ステージ（`core/aidlc-common/stages/<phase>/` の下）で、そのフロントマターの `scopes:` 一覧に `hotfix` を加えます。タグを付けなければそのスコープでは `SKIP` です。3 つの初期化ステージは必ず含めてください（常に走るため）。

3. **再コンパイルする。** `bun .claude/tools/aidlc-graph.ts compile` を実行し、タグを `scope-grid.json` に転置します。その後、`SKILL.md` のコンパイル済みスコープ表を再生成します（`bun .claude/tools/aidlc-utility.ts scope-table` を実行し、`<!-- BEGIN: compiled ... -->` / `<!-- END: compiled ... -->` マーカーの間に貼り付けます）。続けて `bun .claude/tools/aidlc-graph.ts compile --check` と `bun .claude/tools/aidlc-utility.ts scope-table --check` を走らせ、差分がないことを確認します（終了コード 0）。

4. **スコープが解決され、受け入れられることを確認する。** `/aidlc --doctor` を実行します。次に、新しいスコープで初期化したときに状態ファイルの `Scope:` 行が正しいか、環境変数による既定値とワークフロー途中の `--scope` 変更の両方で受け入れられるかを確認します。

5. **キーワード推論を確認する**（`keywords` を書いた場合のみ）。引き金のいずれかを含む自由形式の語句が新しいスコープを検出し、`feature` にフォールバックしないことを確かめます。

6. **スコープ対応のドキュメントを更新し、経路選択テストを追加する。** ドキュメントの中にはスコープを手で列挙している箇所が複数あります。ユーザーガイドのスコープリファレンスと経路選択表、カスタマイズ章の有効値一覧、オーケストレーターリファレンスのスコープとステージの対応です。これらを同じ変更で更新してください。もし新しいスコープが既存スコープにない SKIP パターンを持つなら、既存のスコープ別テストを手本にワークフローテストも追加します。

7. **（任意）入力できるランナーを生成する。** スコープファイルが配置された瞬間から、そのスコープは `/aidlc --scope <name>` で完全に使えます。ランナーは必須ではありません。もし 1 語のコマンド（`/aidlc-hotfix`）が欲しければ、ランナースキルを生成します。`bun .claude/tools/aidlc-runner-gen.ts scopes --all` は、すべてのスコープファイルに対する `skills/aidlc-<scope>/SKILL.md` を出力します（あるいは生成器の `FIRST_BATCH` を編集して、厳選した一部だけを出荷しても構いません）。各ランナーは、およそ 6 行のシェルで `aidlc-orchestrate next --scope <name>` を `done` まで駆動するだけです。ランナーは、すでに実行可能なスコープをパッケージ化したものであり、スコープファイル自体が定義です。`hooks:` ブロックは持ちません。決定論的な中核処理（監査、センサー、実行時コンパイル、状態検証）はプロジェクト全体に対して `settings.json` へ登録されているため、すべてのランナーが自動継承します。スコープファイルを追加または改名したら、そのたびに生成器（または `scopes --check`）を再実行してください。

<a id="what-validates-automatically"></a>
### 自動で検証されること

この実装では、有効なスコープ一覧は実行時に `.claude/scopes/*.md` の存在から `validScopes()` が導出します。そのため、ファイルが配置された瞬間に多くのことが揃います。

- スコープは、`init`、`--scope` の変更、環境変数の既定値解決、`doctor` のすべてで一斉に有効になります。どれも同じ補助関数を参照しているため、個別のコード編集は不要です。
- エラーメッセージには、新しいスコープがアルファベット順で自動的に現れます。
- `keywords` を与えていれば、自由形式の `/aidlc <text>` はフロントマター一覧が埋まった瞬間からそれを自動検出します。必要なのは `SKILL.md` の文章編集ではなく表の再生成だけです。
- 転置の差分ガード（`compile --check`）は、ステージの `scopes:` タグを編集したのに表を再コンパイルしていない場合にビルドを失敗させます。

<a id="what-does-not-validate-automatically"></a>
### 自動では検証されないこと

- **入力ミスしたスコープ名を持つ `scopes:` タグも解析自体は通る。** あるステージのフロントマターに `hotfix` ではなく `hotfx` と書いてもコンパイルは問題なく通ります。できるのは、誰も問い合わせない表の列を 1 つ作るだけです。もっとも、`validScopes()` は `.md` ファイルから導出されるため、ファイルのないスコープは呼び出し時に拒否されます。しかしステージ側の入力ミスタグは、そのステージを本来のスコープから静かに落としてしまいます。安全柵として頼れるのは `/aidlc --doctor` とスコープ別テストです。
- **コンパイル済みスコープ表はずれ得る。** ステージの `scopes:` を編集したのに手順 3 の再コンパイルと表の再生成を飛ばすと、エンジンは古い表を読み続けます。`--check` フラグ（テストスイートでも実行される）がこれを捕まえますが、それは実行した場合だけです。
- **スコープ別のフェーズ順序網羅性。** 出荷済みのフェーズ順序テストは、既知のスコープ名の固定一覧を反復します。新しいスコープは、その一覧に加えるまでテスト対象になりません。同じ変更で忘れず追加してください。
- **手で保守するドキュメント。** ドキュメントを検索して直してくれるものはありません。スコープリファレンス、経路選択表、カスタマイズの有効値一覧は文章です。スコープファイルと歩調を合わせるのは自分です。

<a id="tuning-an-existing-scope"></a>
## 既存スコープを調整する

調整は小さな編集ですが、変更先はスコープファイルよりもステージ側であることが多い、という点が重要です。よくある変更は 2 つです。

- **あるステージを入れる / 外す。** ステージの `scopes:` 一覧にスコープ名を追加または削除します。たとえばチームが「最初の 1 版でも監視は必ず仕込む」と決めたので、`observability-setup` に `mvp` を足す、といった場合です。タグを 1 つ変えたら、`compile` とスコープ表を再生成し、`--doctor` を実行します。
- **既定の深度やテスト戦略を変える。** スコープの `core/scopes/aidlc-<name>.md` フロントマターにある `depth` を変える、あるいは `testStrategy` を追加・削除して、そのスコープが何を生成するかを再調整します。各スコープは自分の既定値を持っているため、この変更はそのスコープを選ぶすべてのワークフローへ連鎖します。もちろん利用者は個別の実行ごとに `--depth` や `--test-strategy` で上書きできますが、スコープの値がチーム全体の基準です。

いずれの場合も、上の手順 3 と同じ再コンパイルと診断の組が適用されます。編集は小さくても、確認内容は同じです。

1 つ、階層化に関する注意があります。出荷済みスコープの調整は、フレームワークが出荷するファイル自体を直接編集することです。ステージの `scopes:` タグか、出荷済みの `core/scopes/aidlc-*.md` を変えることになります。フォークとして既定値を変えたいなら、それは妥当です。ただし、`aidlc-` 系統のファイルを変えているので、フレームワーク更新時に差分の調整が必要になるかもしれません。チーム固有の動作を、全員が依存する既定値に触れずに表したいなら、出荷済みの 9 スコープに並べてまったく新しいスコープファイルを追加する方が明快です。
