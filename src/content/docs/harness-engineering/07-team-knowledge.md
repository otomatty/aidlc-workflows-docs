# チームナレッジ

ナレッジとは、エージェントが作業前に読む分野文脈です。あなたのコーディング標準、アーキテクチャ上の好み、ドメイン用語集、そしてチームで定着したパターンが含まれます。ハーネスを形作る要素の中で、ナレッジだけはフレームワークが強制する制約ではなく、エージェントが読むファイルを追加することで調整します。この章では、その文脈をエージェントへ与えるための実際の作業手順、つまりファイルの置き場所、どのエージェントがそれを見るか、そしてナレッジにすべきかルールにすべきかという判断を説明します。

[ルールと学習ループ](05-rules-and-the-loop.md) を読んでいるなら、その違いを常に意識してください。ルールはフレームワークが強制する恒常的な判断であり、ナレッジはエージェントが作業中に重み付けして参照する参考資料です。どちらもエージェントの振る舞いを形作りますが、属する平面も読み込まれ方も異なります。

<a id="two-tiers-framework-knowledge-and-yours"></a>
## 2 層: フレームワークのナレッジと、あなたのナレッジ

AI-DLC のナレッジは 2 層に分かれており、あなたが編集できるのはそのうち 1 つだけです。

**第 1 層: 方法論ナレッジ** は、この実装とともに `.claude/knowledge/` の下へ出荷されます。各エージェントがステージを実行するために使う方法論リファレンス、たとえば `aidlc-architect-agent/architecture-guide.md`、`aidlc-developer-agent/code-generation-guide.md`、および `aidlc-shared/` の横断資料を持ちます。**ここは触らないでください。** これらのファイルはフレームワーク更新のたびに上書きされます。そこへ何か足しても、次にチームが新しいバージョンを取り込んだ時点で消えます。

**第 2 層: チームナレッジ** はあなたのものです。これは `aidlc/knowledge/`（`aidlc/spaces/<space>/knowledge/` の省略表記）というスペース単位の場所にあり、スペースの `memory/`、`codekb/`、`intents/` と同じ階層に置かれます。したがって、1 つのインテントの記録に閉じ込められず、そのスペース内のすべてのインテントを通じて蓄積されます。ここには会社固有の標準、方針、慣習を置きます。フレームワークが上書きすることはありません。エンジンは最初の `/aidlc` で空の `aidlc/knowledge/` ディレクトリを 1 つ作るだけで、その中身はあなたに任せます。あなたが実際に中身を入れるのはこのディレクトリです。（フレームワークが *強制すべき* 恒常的プラクティス、つまりエージェントが参考資料として重み付けするものではない内容は、代わりに `aidlc/spaces/<space>/memory/` にあるスペースのメモリー層へ置きます。）

この 2 層分割は、このガイド全体を支えるデータとコードの境界線を、ナレッジへ当てはめたものです。フレームワークは自分の方法論を持ち、あなたは自分の文脈を持ち、更新時には一方だけを差し替えて他方へ触れずに済みます。両層の完全なディレクトリ形状は [ナレッジシステム → 2 層アーキテクチャ](../reference/10-knowledge-system.md#two-tier-architecture) を参照してください。

<a id="team-wide-versus-agent-specific-placement"></a>
## チーム全体向けとエージェント固有の配置先

第 2 層は慣習としてエージェントの配置に従います。スペース単位の `aidlc/knowledge/` の下に `aidlc-shared/` ディレクトリを 1 つ、さらにエージェントごとに 1 ディレクトリを置きます。どこへファイルを置くかで、どのエージェントが読み込むかが決まります。

| 置き場所 | 読み込む者 | 用途 |
|-----------|-----------|------------|
| `aidlc/knowledge/aidlc-shared/` | **すべての** エージェントが、すべてのステージで読む | 命名規則、コミット形式、プロジェクトのドメイン用語集のような横断標準 |
| `aidlc/knowledge/<agent>-agent/` | **そのエージェントだけ** が、かつそれが主担当のときだけ読む | アーキテクト向けのアーキテクチャパターン、DevSecOps 向けのセキュリティ方針など、1 つの役割だけに必要な分野文脈 |

ディレクトリ名はエージェントのスラグと完全一致しなければなりません。`architect/` ではなく `aidlc-architect-agent/` です。ディレクトリ名のタイポは、ファイルが黙って無視されるもっとも多い原因です。フレームワークはエージェント自身のディレクトリを名前で探し、何も見付からなければエラーなしでそのまま先へ進みます。エンジンはこれらのサブディレクトリを作りません。`aidlc/knowledge/` は起動時には空で（後述）、必要なディレクトリを正しいスラグであなたが自分で作ります。

ある標準が本当に 11 エージェントすべてへ当てはまるときだけ `aidlc-shared/` を使ってください。アーキテクトにだけ意味があるパターンなら、`aidlc-architect-agent/` に置くべきです。そうすればアーキテクチャステージに文脈を足しつつ、ほかのエージェント全員の視界を濁さずに済みます。ユーザーガイドの [会社標準を追加する実例](../guide/08-knowledge.md) には、ディレクトリ作成、執筆、検証まで含む一連の手順があります。最初の 1 ファイルを書く前に一度読んでおく価値があります。

各ディレクトリが何のためのものかというエージェント別の一覧は [ナレッジシステム → チームナレッジの追加](../reference/10-knowledge-system.md#adding-team-knowledge) を参照してください。

<a id="how-an-agent-loads-knowledge"></a>
## エージェントはどうナレッジを読み込むか

ナレッジファイルは登録したり、どこかへ配線したりしません。正しいディレクトリに存在していること自体が登録です。ステージ開始時、コンダクターは固定の 6 ステップで文脈を読み込み、そのうちあなたの第 2 層ファイルは手順 4 と 5 で入ります。

1. ルール — 解決済みの `aidlc/spaces/<space>/memory/` チェーン（最初に読み込む）
2. 第 1 層の共有方法論 — `.claude/knowledge/aidlc-shared/`
3. 第 1 層のエージェント別方法論 — `.claude/knowledge/<agent>-agent/`
4. **第 2 層のチーム共有** — `aidlc/knowledge/aidlc-shared/`
5. **第 2 層のチーム別エージェント固有** — `aidlc/knowledge/<agent>-agent/`
6. 先行ステージの成果物 — 現在のステージが `consumes` で宣言している出力

手順 4 と 5 は、そのディレクトリが存在し、かつ中にファイルがあるときだけ実行されます。だからチームナレッジが何もないプロジェクトでは、そこは単に飛ばされます。読み込みは各ステージ開始時に毎回行われるので、ファイル編集の効果は次の `/aidlc` 実行から現れます。消すべきキャッシュも、再起動も不要です。削除も同じくらい直接的で、ファイルを消せば、その後の実行では見えなくなります。同期すべき登録簿はありません。

ここで特に腹落ちさせておきたい帰結が 1 つあります。エージェントはナレッジファイルを **文字どおりに、かつ同じ重みで** 読みます。古くなったファイルや矛盾したファイルは、積極的にエージェントを誤誘導します。最新のファイルと同じ権威を持ってしまうからです。第 2 層ツリーは、定期的な刈り込みが必要なコードと同じように扱ってください。レトロの中で短い見直しを定期的に入れるだけでも、十分に正直な状態を保てます。

完全な手順契約、優先順位規則、シーケンス図は [ナレッジシステム → 6 段階の読み込み順](../reference/10-knowledge-system.md#6-step-knowledge-loading-order) にあります。

<a id="knowledge-or-a-rule"></a>
## ナレッジか、ルールか

もっとも多いハーネスエンジニアの間違いは、ルールにすべきものをナレッジへ置くか、その逆をすることです。両者は置き換え可能ではなく、上の読み込み順序がその理由を示しています。ルールは、フレームワークが実行前にコンパイルする厳密加算チェーンとして先に解決されます。ナレッジは、エージェントがステージ中に重み付けして使う参考資料です。

実用的な判定方法があります。**その指示に違反したとき人間のレビュー担当者がステージ出力を差し戻すなら、それはルールです。** レビュー時の背景文脈として使うだけなら、ナレッジです。

| ナレッジを選ぶとき | ルールを選ぶとき |
|---------------------------|------------------------|
| エージェントに参照してほしい参考資料を渡す | エージェントが従わなければならない振る舞い上の判断を述べる |
| 「これが私たちの使うパターンです」 | 「X をしてはいけない」 / 「常に Y をする」 |
| 内容が情報的で文脈依存 | 内容が規範的で譲れない |
| 長文の文章、表、図でもよい | 短く、命令形で、1 行ずつがよい |
| 例: API Gateway の標準、ドメイン用語集 | 例: 「PII をログへ出してはいけない」「すべてのデータアクセスはリポジトリ層を通す」 |

たとえば、チームが API をどう設計するかを説明する文書はナレッジです。`aidlc/knowledge/aidlc-architect-agent/` に置きます。一方、「すべてのアーキテクチャ判断は最低 2 つの代替案を記録しなければならない」のような譲れない項目はルールであり、スペースのメモリー層（`aidlc/spaces/<space>/memory/`）に置きます。そこではフレームワークがエージェントをその規則に従わせます。層チェーンに沿ってルールを著述し、修正を学習ループで昇格させる方法は [ルールと学習ループ](05-rules-and-the-loop.md) を参照してください。ユーザーガイドの [ナレッジとルールの比較表](../guide/08-knowledge.md) には、さらに多くの例があります。

<a id="where-the-tier-2-tree-comes-from"></a>
## 第 2 層ツリーはどこから来るか

これはチームが作ります。最初の `/aidlc` でエンジンが作るのは、空の `aidlc/knowledge/` ディレクトリ 1 つだけです。そこで止まり、ツリーのひな型生成も、エージェントごとのサブディレクトリ作成も、README の初期配置も行いません。第 2 層レイアウト（`aidlc-shared/` とエージェントごとの 1 ディレクトリ）は、ペルソナが探す慣習であって、エンジンが書く構造ではありません。内容があるディレクトリだけを自分で作ります。読み込み側は各エージェント自身のディレクトリを名前でたどるので、期待されるスラグ（`aidlc-architect-agent/` であって `architect/` ではない）で正確に作成してください。名前にタイポがあるとエラーなしで黙って読み飛ばされます。

ディレクトリ内の命名規則はありません。任意の `.md` ファイルが読み込まれます。`architecture.md` のような曖昧な名前ではなく、`api-gateway-standards.md` のような説明的で 1 トピック 1 ファイルの名前にすると、四半期ごとの整理がかなり楽になります。ディレクトリ用の README を出発点として欲しければ、第 1 層には手でコピーできる任意テンプレートが出荷されています。詳しくは [ナレッジシステム → テンプレートシステム](../reference/10-knowledge-system.md#template-system) を参照してください。

この章とガイド全体との境界について、1 つ補足があります。ここであなたが中身を入れるエージェントディレクトリは、エージェントがペルソナファイルで宣言しているものと同じです。[エージェントの追加](03-adding-an-agent.md) を行うと、その新しいエージェントの第 2 層ナレッジディレクトリは `aidlc/knowledge/<new-agent-slug>/` になります。これはチームが作るディレクトリであり、同じ手順 4 と 5 で読み込まれます。[概要](00-overview.md) のメンタルモデルはここでも有効です。ステージがエージェントを指名し、エージェントがナレッジを読み、あなたはその全体をコードではなくデータの編集で形作ります。

<a id="spaces-knowledge-for-more-than-one-team"></a>
## スペース: 複数チーム向けのナレッジ

ここまでの前提は 1 チームです。**複数チームが 1 つのプロジェクトを共有する** 場合、AI-DLC は各チームのやり方、ナレッジ、記録を、それぞれ独立した **スペース**、すなわち `aidlc/spaces/<name>/`（`memory/`、`knowledge/`、`codekb/`、`intents/` を持つ同形ディレクトリ）に保持します。この章で使ってきた `aidlc/knowledge/` という省略表記は、実際には `aidlc/spaces/<active-space>/knowledge/` のことです。1 チームだけならアクティブなスペースは常に `default` なので、この違いが表に出ません（利用者向けの説明は [ユーザーガイドのスペースとインテントの章](../guide/03-spaces-and-intents.md) にあり、ここはそのハーネスエンジニア版です）。

これが、あなたが著述するナレッジとルールに対して意味することは次のとおりです。

- **チームナレッジはスペースごとです。** `aidlc/knowledge/aidlc-<agent>-agent/` に配置するファイルは 1 つのスペースの中に住みます。第 2 のチームには、自分たち用の空の `knowledge/` ツリーが別に用意されます。あなたのファイルが相手へ漏れることはなく、相手のファイルがあなたのエージェント文脈を薄めることもありません。
- **メソッド層もスペースごとです。** `aidlc/spaces/<space>/memory/` のルール（`org.md` → `team.md` → `project.md`）はアクティブなスペースの中で解決されます。新しいスペースはフレームワークの既定状態から用意されます。つまり `org.md` はコピーされ、空の `team.md` と `project.md` が新規作成されるので、新しいチームはフレームワーク既定値から始めて、自分たち自身のプラクティスを育てます。他チームのものを継承はしません。
- **スペースは `core/` で著述しません。** スペースは実行時のチームデータであり、インストール済みプロジェクトで `/aidlc space-create <name>` を使って作ります。これは、チームナレッジとフレームワークソースを分けるのと同じ「データでありコードではない」という境界線です。複数チーム対応のために `core/` へ何か足したり、何かを再生成したりする必要はありません。その機能はエンジンに出荷済みです。

この章のメンタルモデルは、各スペースの内部でもそのまま変わりません。ステージがエージェントを指名し、エージェントがナレッジを読み、あなたはデータを編集してその全体を形作ります。スペースは、*誰の* データかを分けるだけです。だから 2 つのチームが 1 つのプロジェクトで AI-DLC を使っても、文脈、プラクティス、記録が衝突しません。
