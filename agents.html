エージェント

14エージェント（ドメインエキスパート11 + レビュアー2 + コンポーザー1） ｜ 最終更新: 2026-07-09（最新コミット 6f597ce）

AI-DLCはコンダクターがステージ実行中に体現する11のドメインエキスパートペルソナと、v2.0.0で追加された2つのレビュアーペルソナ、v2.2.0で追加されたコンポーザーエージェントを使います。インラインステージ（出荷グラフのすべてのマルチエージェントステージ）では、コンダクターが各エージェントを自身のコンテキスト内でペルソナとして採用します。Task によるサブエージェント派遣は mode: subagent ステージ（reverse-engineering、code-generation）に限られます。どちらの場合も委譲は必ずコンダクターが行い、エージェント同士が直接呼び合うことはありません。

高判断力が要る8エージェント（product / design / architect / developer / quality / devsecops / compliance / aws-platform）は opus モデルで、テンプレート的な出力が中心の3エージェント（delivery / pipeline-deploy / operations）は sonnet で動作します。v2.1.5以降、ティア選択されるモデルには1Mコンテキスト版がピン留めされます。v2.2.15で、エージェントfrontmatterのキーが不活性な modelOverride: からClaude Codeが実際に読む model: にリネームされ、委譲5エージェント（architecture-reviewer / product-lead / delivery / pipeline-deploy / operations）のSonnetピンがClaude Codeで実際に効くようになりました。

レビュアーペルソナ（v2.0.0で追加）

ステージのfrontmatterに reviewer: を宣言すると、ステージ本文の実行後・人間の承認ゲート前に、レビュアーが助言的品質ゲートとしてレビューループ（既定最大2回）を回します。レビュアーが自動承認することはなく、最終判断は常に人間です。

v2.1.8で、レビュアー機構が全ハーネスで確実に動作するよう修正されました: Codexオーケストレーターに§12aレビュアーステップが追加され（従来 reviewer: は黙って無視されていた）、Kiroではレビュアーが subagent.trustedAgents に登録されて信頼プロンプトで停止しなくなり、Kiroレビュアーの書き込みは aidlc/spaces/** にキャップされました（ビルダーより広い無制限 fs_write を持っていた問題の修正）。さらにv2.2.16で、アーキテクチャレビュアーの読み取りスコープが「レビュー対象アーティファクト + ステージが consumes: する共有インセプション契約」に限定され、ユニット単位レビューのコストがユニット数に比例して増大する問題が解消されました（兄弟ユニットのファイルは設計が明示的に統合ポイントとして名指しする場合のみスポットチェック）。

コンポーザーエージェント（v2.2.0で追加）

aidlc-composer-agent

コンダクターがコンポーズ要求（/aidlc compose、コンポーズ提案の受諾、実行中の再構成）で派遣する14体目のエージェント。タスク記述とワークスペーススキャン（detect --json）を読み、SKIPごとの根拠付きでEXECUTE/SKIPステージグリッドを提案します。人間が承認するまで一切書き込まないpropose-only設計で、承認後にスコープファイル（keywords: []）とscope-gridエントリの2ファイルを書き出します。敵対的なkeep-bias（安易にステージを削らない）を持ち、提案は validate-grid で決定論的に検証されます。Kiroではスコープレジストリの2パスのみに fs_write を許可する専用サンドボックス設定を持ちます（コンダクター自身はそこに書けない）。

ナレッジのロード

各エージェントは方法論ナレッジを .claude/knowledge/<agent-name>/ から、チームナレッジをスペースレベルの aidlc/knowledge/<agent-name>/（チームが作成した場合）からロードします。出荷された13のエージェントファイルはアップグレードで上書きされるため直接編集せず、チーム標準はスペースレベルのナレッジに追加します。新しいエージェントは .claude/agents/<slug>.md に必須frontmatter付きで置けばユーザー所有ファイルとして追加できます。

情報の流れ（要約）

コンダクター（SKILL.md）が全エージェントへ委譲します。主要なアーティファクトフロー: product が要件・ストーリー・スコープを architect へ → architect がアーキテクチャ・ユニット仕様を developer へ → developer がコードを quality へ → quality がテスト結果を developer へ返す。aws-platform がインフラを provision して pipeline-deploy へ → pipeline-deploy がデプロイして operations へ。そして operations の運用知見が product へ還流し、ループが閉じます。compliance と devsecops は横から制約・セキュリティレビューを供給します。
