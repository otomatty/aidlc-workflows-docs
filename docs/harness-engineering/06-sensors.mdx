title: センサー
description: ステージの出力に対して自動で走る決定論的かつ助言的な検査と、その作成・結び付け方法を説明します。
---
センサーは、エージェントがステージの出力を書き出したときに自動で発火する、決定論的で助言的な検査です。ルールがエージェントの読む文章（[ルールと学習ループ](/harness-engineering/rules-and-the-loop)）であるのに対し、センサーは実行されるコードです。つまり、ルールの順方向側に対する制御ループのフィードバック側です。ルールが「ユーザーストーリーは前提・操作・結果の形式に従う」と述べ、センサーが、たった今エージェントが書いたファイルに必要な見出しがあるかをバイト単位で検証します。

この章では、ハーネスエンジニアが実際にセンサーで行う作業を説明します。出荷済みの 4 つを理解し、新しいマニフェストを作成し、それを実行させたいステージへ結び付けます。フィールドごとの完全な契約は開発者リファレンスの [センサーシステム](/reference/sensor-system) にあります。この章ではそれを言い換えて繰り返すのではなく、スキーマ上の判断が必要になるたびに参照先を示します。

---

<a id="what-a-sensor-is"></a>
## センサーとは何か

センサーマニフェストは、`core/sensors/` の下に置く、YAML フロントマター付きの Markdown ファイルです。フロントマターは純粋な **機能記述子** であり、その検査が何で、どう呼び出すかを述べます。どのステージが使うかは何も書きません。その結び付けはステージ側にあり、そこがこの章の中心的な考え方であり、マニフェストとステージが疎結合に保たれる理由です。

実行時の振る舞いを定める性質は 2 つあり、何かを著述する前にどちらも腹落ちさせておく価値があります。

- **ステージ中の `Write` と `Edit` で発火する。** エージェントが出力ファイルを書いたり編集したりすると、`PostToolUse` フックが、その時点で有効なステージにどのセンサーが適用されるかを調べ、一致するものを順に実行します。ワークフロー中に手でセンサーを起動することはありません。すべてのファイル書き込みに同乗して走ります（デバッグ用にマニフェストの `command:` を人が直接実行することはできます。詳細はリファレンスを参照してください。ただしワークフロー経路ではフックが呼び出します）。
- **助言的であり、決してブロックしない。** このリリースにおけるセンサー結果は、ゲートではなくテレメトリーです。失敗したセンサーは監査行を 1 件出し、何が足りないかを正確に示す詳細ファイルも作りますが、ステージの承認ゲートやワークフロー自体は止めません。あなたはそのシグナルを見て、どう扱うかを決めます。`default_severity` は現状 `advisory` に固定されており、`blocking` は将来のリリース向けに予約されています。詳しくは後述の [`default_severity`](#judgment-calls-matches-and-default_severity) を参照してください。

発火のたびに、インテントの `audit/` シャードへ 1 行残ります。ログを検索するときはイベント名の大小文字が重要です。センサー開始時は **`SENSOR_FIRED`**、通過時は **`SENSOR_PASSED`**、欠落を見付けたときは **`SENSOR_FAILED`** です。失敗した行は、インテントの記録ディレクトリ内にある `<record>/.aidlc-sensors/<stage-slug>/` 配下の詳細ファイルへリンクし、具体的な欠落内容、たとえば足りない見出し、参照されていない上流成果物、リントエラーを示します。これが実行中にどう見えるかの利用者向けの説明は、ユーザーガイドの [ルールと学習ループ](/guide/rules-and-the-learning-loop) にあります。

<a id="the-four-sensors-that-ship"></a>
## 出荷される 4 つのセンサー

`.claude/sensors/` の下には、`aidlc-` 接頭辞付きで 4 つのマニフェストが出荷されます。

| マニフェスト | 発火対象 | 検査内容 |
|----------|----------|--------|
| `aidlc-required-sections.md` | 記録ディレクトリの Markdown 出力 | 出力に必要な H2 見出しがあるかを検査する、汎用的な内容形状チェック |
| `aidlc-upstream-coverage.md` | 記録ディレクトリの Markdown 出力 | 文章が、そのステージが受け取ると宣言した各上流成果物を参照しているか |
| `aidlc-linter.md` | `.ts` / `.js` のコード出力 | 設定済みリンター（既定では ESLint）を包む |
| `aidlc-type-check.md` | `.ts` / `.tsx` のコード出力 | 設定済み型チェッカー（既定では `tsc`）を包む |

4 つすべては `matches:` グロブで絞り込まれます（詳細は後述）。最初の 2 つの文書形状チェックは成果物ツリーに限定されており、出荷済みマニフェストでは `**/{aidlc-docs,intents}/**` を使います。これはインテントごとの記録ツリーで、移行前のプロジェクト向けに旧 `aidlc-docs/` 側も残しています。後半 2 つのコード品質チェックは、それぞれの言語グロブ（`**/*.{ts,js}`、`**/*.{ts,tsx}`）に限定されます。独自センサーを書く前に、まず `aidlc-required-sections.md` を最初から最後まで読んでください。4 つの中でもっとも小さく、フロントマターと本文を含む全体像が見えます。

<a id="how-binding-works-pull-authoring"></a>
## 結び付けの仕組み: 利用側からの指定

マニフェストには **ステージを指定するフィールドがありません**。`applies_to:` は存在せず、フレームワークが意図的に取り除いています。どのセンサーをその出力に対して発火させるかは、ステージ自身が自分のフロントマターで名指しします。

```yaml
# core/aidlc-common/stages/construction/code-generation.md
---
slug: code-generation
phase: construction
sensors:
  - linter
  - type-check
---
```

これは **利用側からの指定** であり、ハーネスモデルにおけるほかの結び付けと同じ向きです。つまり、利用側（ステージ）が機能（センサー）を指名し、その逆はありません。`sensors:` の一覧には、`aidlc-linter` ではなく `linter` のように接頭辞なしの ID を入れます。これは、その ID がマニフェストのフロントマターにある `id:` フィールドと一致し、その `id:` が `aidlc-` 接頭辞を除いたファイル名の語幹と一致するためです。

この方式の利点は参照の局所性です。ステージファイルを開けば、そのステージの実行時にどの検査が走るかがその場で分かります。どこかのマニフェストがこのステージを対象だと主張していないかを、全ファイルから探し回る必要はありません。この `sensors:` 区画自体は [ステージの構造](/harness-engineering/anatomy-of-a-stage) でステージ側から説明しています。ここではセンサー側から見ています。

ワークフロー開始時にコンパイル解決器が `.claude/sensors/` を走査し、すべてのマニフェストを ID で索引付けし、各ステージについて宣言された各インポートを引き当てます。ID に対応するマニフェストがなければ、ステージ実行時に黙って失敗するのではなく、コンパイル時にはっきりエラーにします。解決済みのステージ別ビューはステージグラフノードへ焼き込まれ、フックは発火時にそこから読み取ります。ここで覚えておくべき帰結が 1 つあります。ワークフロー途中でマニフェストを編集しても、その実行中の実行回に対して何が発火するかは **変わりません**。そのコンパイル時点のスナップショットが、次のワークフロー開始まで保持されるからです。完全な解決処理の仕組みは [ステージがセンサーをインポートする仕組み](/reference/sensor-system#how-stages-import-sensors) にあります。

<a id="authoring-a-new-sensor"></a>
## 新しいセンサーを著述する

センサーの追加は 2 回の書き込みです。まずマニフェスト、次に結び付けです。

**1. `core/sensors/aidlc-<id>.md` にマニフェストを置きます。** ファイル名の語幹（`aidlc-` 接頭辞を除いたもの）は、フロントマターの `id:` と一致しなければなりません。フロントマターは短く、必須 5 項目と少数の任意項目です。

| フィールド | 必須 | 内容 |
|-------|----------|------------|
| `id` | はい | ケバブケース。`aidlc-` を除いたファイル名の語幹と一致 |
| `kind` | はい | 現在受け入れられる値は `deterministic` のみ |
| `command` | はい | ディスパッチャーが実行する正規の呼び出し接頭辞 |
| `default_severity` | はい | 現在受け入れられる値は `advisory` のみ |
| `description` | はい | 1 行の人間向け説明 |
| `category` | いいえ | 自由形式ラベル（出荷済みマニフェストでは `document-shape`、`code-quality` を使用） |
| `matches` | いいえ | どのファイル書き込みでこのセンサーを発火させるかを絞るグロブ |

`command:` は完全な引数列ではなく **接頭辞** です。ディスパッチャーが発火時に実行時コンテキストを後ろへ付け足します。常に `--stage <slug>` を追加し、その後にセンサーの入力形状に合うファイルフラグを付けます。文書センサーでは `--output-path <path>`、コードセンサー（`linter`、`type-check`）では `--file-path <path>` です。こうしてマニフェストは純粋な機能記述子のままであり、発火ごとのフラグを埋め込みません。ディスパッチャーが組み立てる正確な呼び出しは [`command:` の呼び出し契約](/reference/sensor-system#command-invocation-contract) に記載されています。完全なスキーマ、すなわち `input_schema`、`output_schema`、`timeout_seconds`、未知キーに対する前方互換性の方針は [センサーマニフェストのスキーマ](/reference/sensor-system#sensor-manifest-schema) を参照してください。

**2. ステージの `sensors:` 一覧へ ID を追加して結び付けます。** ディレクトリに置かれているだけのマニフェストは何も起こしません。対象にしたいステージを開き、フロントマターの `sensors:` 一覧へ接頭辞なしの ID を加えると、次のコンパイルから有効になります。複数ステージで走らせたいなら、それぞれに ID を追加します。厳密加算であり、上書き層について考える必要はありません。あるステージで止めたいなら、そのステージから ID を削除します。マニフェスト自体は変わらず、変わるのはインポート一覧だけです。

`aidlc-` のファイル名接頭辞は、独自センサーを含めてすべてのセンサーで必須です。コンパイル解決器（`aidlc-graph.ts` の `loadSensors`）は、`SENSOR_FILE_REGEX = /^aidlc-([a-z][a-z0-9-]*)\.md$/` でマニフェストを検出しており、この接頭辞がないファイルは黙って読み飛ばすため、発見も結び付けもされません。センサー名は `core/sensors/aidlc-<id>.md`、`id: <id>` としてください。ファイル名から ID への規則は、接頭辞の後ろの語幹に適用されます。

<a id="judgment-calls-matches-and-default_severity"></a>
## 判断が必要な箇所: `matches` と `default_severity`

マニフェストの大半は機械的です。実際の著述判断を伴うのは 2 つのフィールドです。

**`matches` — どの形のファイルを解析するべきか。** このグロブは発火フィルターであり、事実上必須です。フックは、書き込まれたパスがこれに一致するときだけセンサーを発火させ、**`matches` がない** エントリはまったく発火しません。コード品質センサーはコード用グロブに設定されています。`aidlc-linter.md` は `**/*.{ts,js}`、`aidlc-type-check.md` は `**/*.{ts,tsx}` を使うため、コード書き込み時だけ発火して文章では静かです。文書形状センサーは成果物ツリーに限定するので、ステージが書く任意の Markdown 成果物に対して発火します。自分の検査が意味を持つファイル形状を決め、それを過不足なく覆う最も狭いグロブを書いてください。空の `matches: ""` は解析時に拒否されます。また、グロブを省略するとセンサーは一切走らないため、「全部で発火する」モードはありません。対象の形を明示しなければなりません。フックは発火時に、書き込まれたパスをこのグロブと照合します。完全な挙動は [`matches` フィルター](/reference/sensor-system#matches-filter) にあります。

**`default_severity` — 現時点では助言のみ。** 現在受け入れられる値は `advisory` だけなので、自由に選ぶというより固定の選択です。それでも名前を出す価値があるのは、これが、あなたが引き受ける契約を定義するからです。センサーは知らせるものであって、強制しません。ゲートを止める `blocking` は将来のリリース向けに予約されています。それまでは、どのセンサーも人が読む第二の意見であり、壁にはなりません。予約値に関する方針は [`default_severity`](/reference/sensor-system#default_severity) にあります。

<a id="when-the-learning-loop-installs-a-sensor-for-you"></a>
## 学習ループがセンサーを代わりに導入してくれる場合

センサーを常に手で作成するとは限りません。§13 の学習ゲートは、ワークフロー中にセンサー提案を確認したとき、それを導入できます。つまり、ある決定論的な検査をそのステージ出力に走らせるべきだと判断してゲートでそれを選ぶと、フレームワークが、手で行うのと同じ 2 回の書き込みによる導入を行います。あなたのプロジェクトの `.claude/sensors/aidlc-<id>.md` に **プロジェクト層** のマニフェストをひな型として作り（出荷済みフレームワーク配布物には決して書かず）、発生元ステージの `sensors:` インポート一覧へ新しい ID を不可分に追記します。このゲート確認経路は **`SENSOR_PROPOSED`** の監査行を出すため、どの結び付けも黙って導入されることはありません。ループとその `SENSOR_PROPOSED` 行については [ルールと学習ループ](/harness-engineering/rules-and-the-loop) で扱い、利用者向けの手順はユーザーガイドの [ルールと学習ループ](/guide/rules-and-the-learning-loop) にあります。

この章で説明した手で著述する経路と、ループが導入する経路は、どちらも同じ成果物、すなわちマニフェスト 1 つとインポート一覧の 1 項目を生みます。違うのは、誰がきっかけを作るかだけです。あなたが直接ファイルを編集するのか、それともゲートがワークフロー途中の修正を拾うのかです。
