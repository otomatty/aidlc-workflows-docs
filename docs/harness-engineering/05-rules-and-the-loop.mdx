ルールは、チームがすべてのワークフローに持ち込む恒常的な判断です。つまり、毎回の実行の冒頭で言い直していた「いつもこうする」を永続化したものです。ハーネスエンジニアは 2 つの方法でそれを著述します。ルールファイルを直接編集するか、ワークフロー中の修正を**学習ループ**に永続的なルールへ昇格させてもらうかです。この章は、ユーザーガイドの [ルールと学習ループ](/guide/rules-and-the-learning-loop) 章に対する著述側の手引きです。ユーザーガイド側はループの概念と ANZ の銀行業務例を最初から最後まで説明します。ここでは運用に焦点を当てます。何をしたいときにどのファイルを編集するか、ルールを積み重ねたとき厳密加算モデルがどう振る舞うか、そして一度限りの修正を恒常的なルールにするためにループをどう運用するかを説明します。

ルールは制御ループの順方向側です。エージェントが作業を始める *前に* 読む文章です。[センサー](/harness-engineering/sensors) はフィードバック側であり、書き込みの *後に* 発火する決定論的な検査です。ルールは対となるセンサーを指名でき、この 2 つの側面は組み合わせて使うように設計されています。

<a id="the-five-layers-and-which-file-you-edit"></a>
## 5 層と、どのファイルを編集するか

ルールは `core/memory/` の下に Markdown ファイルとして置かれます。これはスペースメモリー層の著述ソースであり、スコープごとに 1 ファイルです。`scope:` フロントマターフィールドはありません。この実装ではファイル名からスコープを導出するため、どのファイルを選ぶか自体が、どのスコープに著述するかを意味します。

| 適用したい対象 | 編集するファイル | スコープ |
|---|---|---|
| 組織内のすべてのプロジェクト | `org.md` | 組織（フレームワークの既定値） |
| チームが実行するすべてのプロジェクト | `team.md` | チーム |
| この 1 つのプロジェクト | `project.md` | プロジェクト |
| 1 つのフェーズに属するすべてのステージ | `phases/<phase>.md` | フェーズ |

4 つのフェーズファイルは `phases/ideation.md`、`phases/inception.md`、`phases/construction.md`、`phases/operation.md` です（初期化は起動時専用なのでルールファイルを出荷しません）。5 つ目の層、つまりステージ別ルール（`aidlc-stage-<slug>.md`）は将来のリリース用に予約されており、まだ著述できません。

どのファイルを使うかでは、2 つの判断が必要です。

- **`org.md` はフレームワークが出荷します。** トランクベース開発、歩く骨格の方針、スコープごとのテスト方針など、すべてのプロジェクトが継承する既定値を持ちます。上流として扱ってください。多くのハーネスエンジニアはここを編集せず、チームまたはプロジェクトに著述します。
- **プロジェクトスコープは、あらゆることを書く場所ではなく、永続的な差異のための場所です。** `project.md` を使うのは、このプロジェクトがチーム全体のプラクティスから安定して外れる場合だけにしてください。たとえば、チームはスカッシュを使うがこのモノレポだけリベースを使う、古いプロジェクトなのでテストの最低基準を下げる、などです。そのルールがチームのすべてのプロジェクトに役立つなら、置き場所は `team.md` です。

各ファイルは、`## Way of Working`、`## Testing Posture`、`## Deployment`、`## Code Style` などの話題別見出しの下に置く平文です。ルールを追加するとは、適切な見出しの下に箇条書きを 1 つ足すことです。ファイル名とスコープの対応表、および解決処理の正式な契約は [ルールシステム § ファイル名から導出するスコープ](/reference/rule-system#filename-derived-scope) と [§ 配置](/reference/rule-system#layout) を参照してください。

<a id="strict-additive-layers-stack-none-silently-overrides"></a>
## 厳密加算: 層は積み重なり、どれも黙って上書きされない

チェーンは、各ワークフローの開始時に 5 層を通って解決されます。

```
org → team → project → phase → stage
```

モデルは**厳密加算**です。適用可能なすべてのルールが同時にエージェントのコンテキストに現れます。実行時に落とされたり抑制されたりするものはありません。組織の既定値、チームのプラクティス、プロジェクトの特殊化が連結され、さらにステージがフロントマターに `phase:` をすでに宣言しているため、対応するフェーズルールが付きます（ステージが自分でほかのものをインポートするのと同じ、利用側から著述元を引く方向です）。解決済みチェーンはワークフロー開始時に各ステージノードへ 1 回組み込まれ、実行時には二度とチェーンをたどり直しません。詳しくは [ルールシステム § 厳密加算の実行時モデル](/reference/rule-system#strict-additive-runtime-model) と [§ 5 層の継承](/reference/rule-system#five-layer-inheritance) を参照してください。

著者にとってこのモデルが意味することは、`overrides:` ブロックも `enforcement:` キーワードも不要だということです。狭い層が広い層に静かに勝つことはありません。代わりに、あなたが書いたすべての層がエージェントの視野に同時に存在します。したがって著述の仕方も変わります。上位層を抑制する切り替えを探すのではなく、そのスコープで適用すべきルールを肯定的に書き、それが上位層と一緒に積み重なることを前提にします。

<a id="conflicts-are-rejected-when-you-write-not-resolved-at-runtime"></a>
### 競合は実行時に解決されるのではなく、書き込む時点で拒否される

実行時に何も上書きされない以上、広いスコープのルールと *矛盾する* ルールは解決器が扱えない問題になります。フレームワークはそれを実行時ではなく**書き込み時**に防ぎます。ある `## Heading` の下にチームスコープのルールを追加しようとすると、受け入れゲートが `org.md` の同じ見出しと提案文を比較し、矛盾を見付けると書き込みを止め、**修正**、**省略**、**上位へ報告**（組織ルールの所有者へ上げる）の 3 つの選択肢を出します。プロジェクト階層の書き込みは組織に対してだけ検査されます。チームとプロジェクトの違いは妥当なプロジェクトの特殊化であり、方針違反ではないからです。

この検査は、フレームワークが所有する 2 つの受け入れゲート、つまりプラクティス検出の確認ゲートと、後述する学習ゲートの両方で走ります。したがってルールが解決器に届く時点では、すでに競合検査を通過しています。その後の状態を見せる読み取り専用の `/aidlc --doctor` 行も 2 つあります。`rule-drift`（ルールのずれ）行は、チームまたはプロジェクトの内容が、記載済みの組織見出しと重なっている箇所を示し（人が確認すべき矛盾候補）、`paired-coverage`（対の網羅率）行は、センサーを名指ししているルールのうち実際に解決されるものがいくつあるかを報告します。どちらも助言的であり、終了コードは変えません。詳しくは [ルールシステム § ルールのずれの検出](/reference/rule-system#rule-drift-detection) を参照してください。

<a id="operating-the-learning-loop"></a>
## 学習ループを運用する

ルールを著述するもう 1 つの方法は、ファイルを自分で開かないことです。ワークフロー中にエージェントを修正し、その修正をゲートで確認すると、フレームワークが代わりにルールとして書いてくれます。これが学習ループです。ほとんどのステージ実行では何も追加されず、それが健全です。ループが発火するのは、そのステージ中に起きた何かを「残す価値がある」と判断したときだけです。

仕組みは [ユーザーガイドの章](/guide/rules-and-the-learning-loop) にあります。ここではハーネスエンジニアの視点から、ループが何を生成し、どこに着地するのかを見ます。

1. **ステージ中に記録が付けられる。** コンダクター（有効なステージを動かしている稼働中の `/aidlc` セッション）は、観察ログを `<record>/<phase>/<stage>/memory.md` に付けます（インテントの記録ディレクトリ、`aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` の下です）。項目は `Interpretations`、`Deviations`、`Tradeoffs`、`Open questions` の 4 見出しの下に入ります。自動作成・自動保守されるので、手で編集してはいけません。このループで言語モデルが担う *唯一の* 仕事は記録を書くことです。ステージ終了後の集計、提示、振り分け、書き込みは、決定論的なツールか、ゲートでのあなたの明示的な選択が行います。
2. **ゲートが候補を見せる。** 承認ゲートの前に学習ゲートが `memory.md` を読み、空でない記録行を一字一句そのまま候補として表示します。さらに自由記述の "anything to add for next time?" 欄があり、そこに観察事項を入力して 4 見出しのどれに属するかを選べます。
3. **何を残すかをあなたが確認し、フレームワークが書き込む。** 確認済みの学習事項は日付付きの項目として残ります。ファイルパスを選ぶ必要はありません。書き込み先は見出しによって決まります。

<a id="where-a-kept-learning-lands"></a>
### 残した学習事項はどこへ着地するか

確認済みの学習事項は、そのままプラクティスです。プラクティス検出が確認するのと同じスペースメモリーファイル、`aidlc/spaces/<space>/memory/team.md` と `memory/project.md` に着地します。別の `*-learnings.md` 保存先はありません。ゲートは話題別の見出しに基づいて振り分けます。

| ゲートでの見出し | 着地先 | 監査イベント |
|---|---|---|
| `Interpretation / Deviation / Tradeoff`（解釈 / 差異 / トレードオフ） | `aidlc/spaces/<space>/memory/project.md`（既定） | `RULE_LEARNED` |
| 同じものを 1 クリックで昇格 | `aidlc/spaces/<space>/memory/team.md` | `RULE_LEARNED` |
| `Open question`（未解決の疑問） | どこにも書かれない。調査項目は昇格されない | — |

既定のスコープは**プロジェクト**です。もっとも狭いスコープです。学びがこのプロジェクトを超えて役立つ場合は、1 クリックの "promote to team" 操作により `memory/team.md` へ広げられます。組織へ広げる経路はありません。組織のプラクティスはフレームワーク出荷時または組織によって著述され、別の手続きを通るべきものだからです。既定を狭くしておくことで、1 つのプロジェクトだけの予想外の出来事が、誤って組織全体のルールになるのを防ぎます。学習ループとプラクティス検出は、異なるライフサイクルを通じて同じファイルに書き込みます。ループは日付付き項目を 1 件ずつ追記し、プラクティス検出は区画全体を確認します。解決器が並べる順序は、明快な整数チェーンの組織 → チーム → プロジェクト → フェーズです。

<a id="when-the-learning-is-a-check-not-a-rule-sensor_proposed"></a>
### 学習事項がルールではなく検査の場合（`SENSOR_PROPOSED`）

残す価値があるものが、*繰り返し手で確認している検査* である場合もあります。ステージの出力を見て、毎回同じ欠落を目視で確かめているような場合です。そうした学習事項をゲートで確認すると、フレームワークはそれをルールではなくセンサーの結び付けとして扱います。2 つの書き込みを不可分に行い、`.claude/sensors/` の下にセンサーマニフェストのひな型を作り、発生元ステージの `sensors:` インポート一覧に新しいセンサー ID を追記します。そして `SENSOR_PROPOSED` の監査行を残します。これが、フレームワークのリリース以外でステージファイルに許される唯一の正式な編集です。増えるのはインポート一覧だけで、ステージ本文には触れません。その後は [センサー](/harness-engineering/sensors) に従ってマニフェストを手で著述します。ループは結び付けのひな型を作り、検査自体はあなたが著述します。

<a id="learnings-apply-on-the-next-workflow-not-the-one-in-flight"></a>
### 学習事項が効くのは現在の実行ではなく、次のワークフロー

ゲートで捉えた学習事項は、現在の実行で残っているルールを**変えません**。このワークフローでは、あなたはすでに会話の中でエージェントを修正済みだからです。ルールは次回のためにあります。新しい行はディスクに書かれますが、実行中のワークフローは開始時にコンパイルした見方を保持し続けます。次に `/aidlc` を走らせたとき、コンパイルが再実行され、ディレクトリ走査が新しいファイルを拾い、最初のステージからルールが効き始めます。

これは手作業での著述でも同じです。ワークフローの途中で `team.md` を編集しても、進行中の実行には遡って影響しません。ルールが効くのは次のコンパイル境界からです。すぐ効かせたいなら、そのワークフローを完了または再起動して、コンパイル処理に編集内容を読み直させる必要があります。
