エージェントはフレームワークにおける「誰が」です。分野、ツールの許可リスト、階層を持つペルソナです。出荷済みの 11 エージェントは、プロダクト、デザイン、デリバリー、アーキテクチャ、AWS プラットフォーム、コンプライアンス、DevSecOps、開発、品質、パイプラインデプロイ、運用を扱います。チームにフレームワークがまだ扱っていない分野（たとえばデータガバナンスのレビュー担当者やモバイルの専門家）が必要になったら、`core/agents/` に Markdown ファイルを 1 つ置くだけでペルソナを追加できます。TypeScript は不要です。

この章ではワークフローを説明します。ペルソナファイルとは何か、フロントマターのどこで判断が必要か、そして *見える* ことと *有効である* ことは別だ、という 2 段構えの事実です。フィールドごとの契約全文は開発者リファレンスに譲り、利用者の立場からこれらのエージェントが何者かについては [ユーザーガイド — エージェント](/guide/agents-overview) を参照してください。

---

<a id="what-a-persona-file-is-and-where-it-lives"></a>
## ペルソナファイルとは何か、どこに置くのか

各エージェントは `core/agents/<slug>-agent.md` という 1 つの平坦なファイルです。上部に YAML フロントマター、下部に Markdown の本文があります。出荷済みファイルにはすべて `aidlc-` 接頭辞が付いています（`aidlc-architect-agent.md`、`aidlc-developer-agent.md` など）。あなたが追加するファイルはあなたのものであり、この接頭辞は必須ではありません。出荷済みの 11 エージェントはフレームワークのファイルとして扱ってください。更新時に上書きされるため、既存エージェントが *何を知っているか* を変えたいなら、エージェントファイルを編集するのではなくチームナレッジで上書きします（[チームナレッジ](/harness-engineering/team-knowledge) を参照）。本当に新しいペルソナを作るとは、別のファイルを新規作成し、それをあなたが所有する、という動きです。そうすれば更新をまたいでも残ります。

フレームワークが解析するのはフロントマターです。本文は、エージェントが有効化されたときに自分自身について読む文章です。責務、担当するステージ、ナレッジの読み込み方、作業原則などを書きます。機械が読むのはフロントマターだけであり、本文はエージェント自身に役割を示すためのものです。出荷済みファイルの構造に合わせて書いてください。

実際のエージェントからフロントマターを抜き出すと、たとえば `core/agents/aidlc-architect-agent.md` は次のようになっています。

```yaml
---
name: aidlc-architect-agent
display_name: Architect Agent
examples:
  - tech-stack.md
  - infrastructure-preferences.md
description: >
  Solutions architect responsible for application design, domain modelling,
  NFR patterns, and component decomposition.
disallowedTools: Task
tier: judgment
---
```

<a id="the-frontmatter-contract-and-the-calls-you-make"></a>
## フロントマターの契約と、そこで行う判断

フィールドごとの完全なスキーマはリファレンスにあります。ここでは、実際に著述時に判断する点だけを取り上げます。

**`name` はファイル名の拡張子を除いた部分と一致していなければなりません。** `aidlc-data-governance-agent.md` というファイルなら、`name: aidlc-data-governance-agent` を宣言します。パーサーはこれをキーにしているので、ここがずれると「存在するのに解決されないエージェント」が生まれます。

**エージェントは既定でセッションの全ツールを継承します。** 出荷済みの 11 エージェントはどれも `tools:` の許可リストを宣言していないため、それぞれがセッションで利用可能なすべてのツール、`Read`、`Edit`、`Write`、`Glob`、`Grep`、`AskUserQuestion`、`Bash`、`WebSearch`、さらに継承された MCP ツールまで使えます。ペルソナを絞り込みたいなら、任意の `tools:` 許可リストを追加し、そのペルソナに許可したいツールだけを列挙します。`tools:` を書くと、そのペルソナはそこに書いたツール *だけ* に狭まり、完全修飾した `mcp__<server>__<tool>` ID を一緒に書かない限り、継承された MCP ツールはすべて除かれます（MCP の継承に関する注意は後述）。分野上、本当に利用範囲を狭めたい場合にだけ使ってください。多くのペルソナは完全継承のままが最善です。

**MCP サーバーはエージェントごとに許可するものではなく、継承されます。** プロジェクトルートの `.mcp.json` に宣言された 5 つの MCP サーバーはセッションに用意され、すべてのエージェントが自動で継承します。エージェントごとに著述する許可はありません。特定サーバーをペルソナから遠ざけたいときは、そのサーバーを含まない完全修飾した `mcp__<server>__<tool>` 一覧に `tools:` の許可リストを絞ってください（`mcp__<server>` という単独のトークンは何もせず、サーバー単位の許可にはなりません）。この継承と制限のモデルは `t110` の登録整合性テストで検証されています（[テスト](/reference/testing) を参照）。

**`disallowedTools` には `Task` を含める必要があります。** これは任意ではありません。エージェントは委譲された作業者として動きます。エンジンの `run-stage` ディレクティブが `mode: subagent` を持っているとき、`Task` を呼び出すのはコンダクター（稼働中の `/aidlc` セッション）です。`Task` をエージェントに許すと、エージェント自身がサブエージェントを起動でき、フレームワークが防ごうとしている連鎖的な委譲が起こります。出荷済みエージェントはすべて `Task` を禁止しており、あなたのエージェントも同様でなければなりません。

**`tier` は作業の種類を表し、パッケージャーがそれをハーネスごとのモデルと推論量のキーへ射影します。** 中核エージェントのフロントマターには、生の `model:` や `effort:` を著述しません。これらは `dist/<harness>/` における射影の出力であり、`core/tools/aidlc-tiers.ts` の階層表から導出されます。曖昧な意図を解釈し、密な文脈の下でアーキテクチャ上のバランスをはかるような、多数の制約を伴う推論が後続にも効くペルソナなら `judgment` を選びます。`judgment` エージェントはセッションのモデルと推論量の両方をそのまま継承するため、黙って下位へ落とされません。明示的な基準に照らして新しい入力を判断するレビュー型ペルソナには `balanced` を選びます。出力の大半が既存の型に従い、方法論がエージェントのナレッジファイルにすでに記述されている場合、たとえばデリバリー計画、CI/CD の YAML、運用手順書のひな型作成のような場合にのみ `templated` を選びます。`templated` は推論量を下げる唯一の階層です。迷ったら `judgment` を使ってください。射影表（とプロジェクトの `tier_cap`）で後から費用を下げることはできますが、低すぎる階層で著述されたペルソナは黙って推論不足になります。詳細な射影表と上限の上書きは [エージェントシステム](/reference/agent-system) を参照してください。

さらに 2 つ、表示向けのフィールドがあります。`display_name` はステータスラインが表示する、人が読めるラベルです（アーキテクトは "Architect Agent" と表示されます）。`examples` は、エージェントから例への対応表に出す推奨ナレッジファイル名の一覧です。これは *利用者に見せる提案* であり、実行時に読み込むものでもエンジンがディスクへ書くものでもありません。

必須・任意フィールドの正確な表と共有設定の対応表は [エージェントシステム: フロントマターの契約](/reference/agent-system#frontmatter-contract) を参照してください。

<a id="visible-is-not-active-the-two-step-truth"></a>
## 見えることと有効であることは別物: 2 段階の真実

ここが最重要ポイントです。ファイルを置くとエージェントは *見える* ようになりますが、ステージに結び付けて初めて *有効* になります。どちらか片方だけでは足りません。両方必要です。

- **検出によって見えるようになります。** `.claude/tools/aidlc-lib.ts` の `loadAgents()` は、次の呼び出しで `.claude/agents/` 内のすべての `.md` ファイルを読み、メタデータ対応表を導出します。コード編集も登録手順も不要で、ファイルの存在自体が登録です。この時点でステータスラインはその表示名を描画でき、チームはスペース単位の `aidlc/knowledge/<slug>-agent/` ディレクトリに標準を置けるようになります。
- **ステージへの結び付けによって有効になります。** ステージは、自分のフロントマターの `lead_agent` / `support_agents` フィールドにスラッグを書くことで、主担当エージェントと支援エージェントを指名します（コンパイル後は `.claude/tools/data/stage-graph.json` に入ります）。どこかのステージがそのスラッグを参照するまでは、どの `run-stage` ディレクティブにもそのペルソナは現れないため、コンダクターが委譲することはありません。

これはフレームワークの基本的な非対称性をなぞっています。ステージがエージェントを指名し、エージェントがステージを指名することはありません。だからエージェントファイル単体では、意図的に休眠状態です。新しいペルソナを本当に動かしたければ、それを使うべきステージを編集してください。結び付けの仕組みは [ステージの追加](/harness-engineering/adding-a-stage) にあります。

各エージェントはナレッジディレクトリとも対になります。`core/knowledge/aidlc-<slug>-agent/` がフレームワークの方法論用、スペース単位の `aidlc/knowledge/<slug>-agent/` がチームの上書き用です。スペース単位の `aidlc/knowledge/` ディレクトリは自由形式で、起動時には空です。チームは内容が必要になったときにエージェントごとのサブディレクトリを作ります。エンジンがひな型を作ることはありません。この 2 層のナレッジワークフローは [チームナレッジ](/harness-engineering/team-knowledge) で扱います。

<a id="the-steps"></a>
## 手順

リファレンスの手順に合わせて、ワークフローを最初から最後までまとめます。

1. **エージェントファイルを作る** — `core/agents/<slug>-agent.md` に、必須のフロントマターを入れます。`name`、`display_name`、`examples`、`description`、`disallowedTools`（`Task` を含む）、`tier` です。任意の `tools:` 許可リストでペルソナを狭めることもできます。省略すればセッションの全ツールを継承します。本文は、出荷済みファイルの構造（中核的な責務、担当ステージ、協働、ナレッジの読み込み、主要原則）に合わせて書きます。
2. **ナレッジファイルを追加する** — `core/knowledge/aidlc-<slug>-agent/` の下に、そのペルソナが有効化時に読み込む方法論を置きます。
3. **ステージに結び付ける** — そのペルソナが主導または支援すべき各ステージファイル（`core/aidlc-common/stages/<phase>/<slug>.md`）のフロントマター `lead_agent` / `support_agents` にスラッグを追加し、その後 `bun .claude/tools/aidlc-graph.ts compile` を実行して `stage-graph.json` を再生成します。`stage-graph.json` はビルド成果物であり、手で編集してはいけません。次のコンパイルで上書きされます（[ステージの追加](/harness-engineering/adding-a-stage#4-compile-so-stage-graphjson-regenerates) を参照）。これが有効にする工程です。
4. **チームナレッジのディレクトリを文書化する** — チームが標準を置くのは、スペース単位の `aidlc/knowledge/<slug>-agent/` です。エンジンはこのディレクトリを作りません。チームが内容を持ったときに自分で作ります（スペースの `aidlc/knowledge/` 自体は自由形式で起動時には空です）。
5. **手で保守するドキュメント表を更新する** — フェーズ参加対応表とエージェントから例への対応表は自動再生成されません（後述の「自動では検証されないこと」を参照）。エージェントを追加した変更と同時に更新してください。

検出、インテント作成、ステータスラインの確認コマンドまで含めた完全な手順は [コントリビューション: エージェントの追加](/reference/contributing#adding-an-agent) にあります。新規追加ではなく、既存エージェントのツール、階層、ステージ割り当てを変えたいなら [エージェントシステム: エージェントの変更方法](/reference/agent-system#how-to-modify-an-agent) を参照してください。

<a id="what-validates-automatically"></a>
### 自動で検証されること

- `loadAgents()` は、次の呼び出しで `.claude/agents/` にある新しい `.md` ファイルを自動検出します。コード編集も登録も不要です。
- `name` または `display_name` が欠けていると、パーサーはファイル名と欠落フィールド名を含むエラーを投げます。
- エージェントはスラッグのアルファベット順で返されるため、検出順はどのプラットフォームでも同じです。
- インテント作成は空のスペース単位 `aidlc/knowledge/` ディレクトリを作りますが、エージェントごとのサブディレクトリや README は初期配置しません。
- ステータスラインは、導出されたメタデータから表示名を描画します。

<a id="what-does-not-validate-automatically"></a>
### 自動では検証されないこと

- **ステージグラフへの参加。** `stage-graph.json` はエージェントをスラッグで参照します。エージェントを追加しても、そこへ接続しなければ存在するだけで決して実行されません。検出と有効化は別の工程です。
- **ナレッジファイルの存在。** `examples` はエージェントから例への対応表に出す推奨ファイル名であり、何かを作成したり検証したりするものではありません。実際の内容は、スペース単位のナレッジディレクトリである `aidlc/knowledge/<slug>-agent/` の下に置きます。
- **手で保守するドキュメント表。** [エージェントシステム](/reference/agent-system#phase-participation) のフェーズ参加対応表と、ナレッジ README ひな型のエージェントから例への対応表は手で編集します。エージェントを追加する変更と同時に更新してください。
- **エージェントファイルの本文。** 解析されるのはフロントマターだけで、本文は有効化時にエージェント自身が読みます。出荷済み 11 ファイルの構造に合わせて、丁寧に書く必要があります。
