> [ハーネスエンジニアガイド](/harness-engineering/overview) の一部です。前提章: [ステージの構造](/harness-engineering/anatomy-of-a-stage)。設計リファレンス（仕組み、インストール時の根拠、ハイブリッド配布モデル、実装済みの現状）は [プラグイン機構](/reference/plugin-mechanism) を参照してください。

**AIDLC プラグイン**（**プラグイン**）とは、再利用可能で任意導入の AIDLC コントリビューション群です。新しいステージ、エージェント、スコープ、メソッド / ルール（メモリ層）、センサー、方法論ナレッジ、そして既存コアステージに対する加算的変更を含みます。プラグインは独自のディレクトリにパッケージ化され、独自のリポジトリから公開され、利用者が選んだプラグイン集合の上から利用者のインストールへ **コンポーズ** されます。プラグインは `core/` を決して編集しません。すべてのプラグインを無効にしたインストールは、素のコアとバイト単位で一致します。

ファーストパーティープラグイン（AIDLC チームが出すもの）とサードパーティープラグイン（それ以外の誰かが出すもの）は、**機械的には同一** です。構造も継ぎ目もコンポーザーも保証も同じで、違うのは出どころ、つまりプラグインがどのリポジトリにあり、誰がレビューしたかだけです。

この章では `test-pro` プラグインを最初から最後までたどります。あなた自身のプラグインは、その形をそのまま写してください。

<a id="when-to-write-a-plugin-vs-a-plain-stagerule"></a>
## プラグインを書くべきときと、単なるステージ / ルールで済むとき

- **ステージ / エージェント / ルール**（[2〜6 章](/harness-engineering/overview)）は、みんなが受け取るフレームワークの恒久的な一部です。
- **プラグイン** は _任意導入で所有主体が明確_ なものです。独自のリポジトリで出荷し、オプトインのスコープ（および / または `when:` 述語）のときだけ有効になり、利用者が自分のインストールへコンポーズするかどうかを選びます。すべてのプロジェクトが欲しがるわけではないドメインパック（完全な運用フェーズ、コンプライアンスプラグイン、テスト用プラグインなど）に使ってください。

<a id="1-the-directory--manifest"></a>
## 1. ディレクトリとマニフェスト

プラグインは、宣言的なマニフェストとコア形のサブツリーを持つ 1 つのディレクトリ（そして 1 つのバージョン管理リポジトリ）です。

```text
test-pro/
  .aidlc-plugin/plugin.json                          # the manifest
  stages/construction/test-pro-integration.md        # NEW stages
  stages/operation/test-pro-full-suite.md
  contributions/construction/nfr-requirements.md      # MODIFY existing core stages (§3)
  contributions/construction/nfr-design.md
  contributions/construction/build-and-test.md
  contributions/operation/performance-validation.md
  sensors/aidlc-coverage-threshold.md                 # NEW sensor manifests
  sensors/aidlc-requirement-coverage.md
  tools/aidlc-sensor-coverage-threshold.ts            # the sensor scripts
  tools/aidlc-sensor-requirement-coverage.ts
  scopes/test-pro-validation.md                       # NEW plugin scope
  agents/test-pro-metrics-agent.md                    # NEW support persona
  knowledge/test-pro-metrics-agent/methodology.md     # plugin methodology knowledge
```

`.aidlc-plugin/plugin.json` は **宣言的** なマニフェストです。最上位は共通のプラグインマニフェスト構造をなぞっており、マーケットプレイスやホストツール群が一覧化、版管理、信頼判定をできるようになっています。AIDLC 固有の設定は、ネストされた `aidlc` ブロックに入ります。

```jsonc
{
  "name": "test-pro",                 // == dir name; "core", "aidlc", and "aidlc-*" are reserved
  "version": "0.1.0",                 // semver; checked by dependents
  "description": "Full-featured testing plugin — unit/branch coverage, functional, integration, regression, edge, and API positive+negative.",
  "author": { "name": "AWS AIDLC" },
  "dependencies": ["core"],           // other plugins, e.g. ["compliance@^1.2.0"]
  "aidlc": {
    "contributes": {                  // which subtrees this plugin ships
      "stages": "stages/",            // NEW stage files
      "overlays": "contributions/",   // CONTRIBUTION files (§3 — modify existing)
      "agents": "agents/",            // NEW personas
      "scopes": "scopes/",            // NEW scope identities
      "knowledge": "knowledge/",      // methodology knowledge for agents
      "sensors": "sensors/",          // sensor manifests
      "tools": "tools/"               // sensor scripts (so a sensor can run)
    }
  }
}
```

`contributes` のキーはコアサブツリー（`stages`、`agents`、`scopes`、`memory`、`sensors`、`knowledge`、`tools`）へ対応し、コンポーズ時にコアと並べてマージされます。`tools` は CLI スクリプトをハーネスの `tools/` ディレクトリに着地させるので、プラグインが **実行可能なセンサー**（`sensors/` のマニフェストと `tools/` のスクリプトの組）を出荷できます。`memory` は `rules/` ディレクトリではなくデフォルトスペースのメソッドシードへマージされます（そのディレクトリはもう読まれません。§4 を参照）。`overlays` は特別で、**コピーされません**。マージ（§3）が消費するステージごとのコントリビューションを置く場所です。

プラグインが実際に使うキーだけを出荷してください。`test-pro` はサポートエージェント、プラグインスコープ、エージェント別の方法論ナレッジを出荷しますが、リードには引き続き `aidlc-quality-agent` を再利用します。

> **番号範囲は取りません。** ステージ番号は表示専用なので、プラグインはマニフェストで番号範囲を確保しません。§2 を参照してください。

<a id="2-add-a-new-stage"></a>
## 2. 新しいステージを追加する

プラグインステージは通常のステージファイル（[ステージの構造](/harness-engineering/anatomy-of-a-stage) を参照）ですが、追加ルールが 2 つだけあります。

- `plugin:` フィールドで、あなたのプラグイン名を指定すること。
- `produces:` する成果物には、必ず `<plugin>-` 接頭辞を付けること（例: `test-pro-integration-test-results`）。

`bundle:` は改名前の所有権キーであり、修正方法を示すエラーとともに拒否されます（`plugin:` と書いてください）。この語は、将来あり得る「プラグインの集合（collection of plugins）」概念のために予約されています。

ステージの **識別子はスラッグ** です。重要な箇所、つまりエッジ、ジャンプ、解決のすべてでスラッグが使われます。`number:` は **表示ヒント** でしかありません。ステータス出力や `SKILL.md` のステージ表で並び順を決めるだけで、グラフ上の位置はスラッグベースの `requires_stage` エッジが決めます。したがって、見た目に自然な `number:`（`test-pro-integration` なら `build-and-test` の `3.6` の後として `3.85` など）を書きますが、それを挿入してもコアを振り直すことはなく、範囲を主張することもありません。

スコープへステージを載せるには `scopes:` を使います（それ以外では SKIP です）。必要なら `when:` 述語も宣言できます。`test-pro-full-suite` は、本来はその上流プロデューサーがプランに載っているときだけ走らせたいステージです。

```yaml
scopes:
  - enterprise
when:
  producer-in-plan: test-pro-regression-suite
```

> **`when:` は解析されますが、まだ評価されません。** スキーマは述語を検証し、パーサーも読み取りますが、今日のエンジンにはそれを実際に消費する箇所がありません。そのため `when:` を持つステージも、宣言した `scopes:` では無条件に EXECUTE です。前方互換性のために書くのは構いませんが、実際の挙動のゲートは今のところ `scopes:` に置いてください。

スコープメンバーシップと `when:` 述語については [スコープ](/harness-engineering/scopes) を参照してください。

<a id="3-modify-an-existing-core-stage-a-contribution"></a>
## 3. 既存のコアステージを変更する（コントリビューション）

ここがコントリビューションの継ぎ目です。既存のコアステージを **編集せずに**、加算的に変えます。コントリビューションは `<plugin>/contributions/<phase>/<slug>.md` に置きます。以下は `test-pro` が `nfr-requirements` に加えるコントリビューションです。

```markdown
---
target: nfr-requirements      # the existing core stage you're enriching
plugin: test-pro
adds:                         # STRUCTURAL — set-unioned into the stage node
  produces:
    - test-pro-testability-requirements   # <plugin>- prefixed
  required_sections:
    - "Testability Requirements"          # machine-enforced
    - "Coverage Targets"
fragments:                    # PROSE — spliced into the stage body
  - anchor: after-step:6
    order: 100
---

## fragment: after-step:6

### Step 6b (test-pro): Capture testability NFRs

…prose the agent will see, appended after the target stage's Step 6…
```

何を追加できるかを整理します（すべて加算的であり、**上書きも削除もありません**。設計上そうなっています）。「ステータス」は、コンポーズフックが今日マージするものと、設計済みだが延期されているものを示します（18 章の §5 / §8 と対応。実装するか、延期だと明示するかであり、黙って何もしない空振りは許しません）。

- `adds.produces` / `adds.consumes` / `adds.sensors` — ✅ 対象ステージのソースフロントマターへ和集合として統合されます。
- `adds.required_sections` — ✅ ステージの `required_sections` へマージされます。ただし **今日のところ機械的には強制されません**。フィールドは書き込まれ、バリデーションも通りますが、コンパイル済みグラフノードまでは届かず、出荷済み `required-sections` センサーはテンプレートから期待値を導くので、欠落セクションが理由でステージが失敗することはまだありません。今のところは宣言的な意図として扱ってください。
- `adds.requires_stage` / `adds.scopes` — ⏳ **延期中**。コントリビューションで宣言はできますが、コンポーズはマージせずドロップログに記録します。現時点では DAG / スコープエッジにはなりません。これで挙動をゲートできると期待しないでください。
- `fragments` — ✅ ステージ本文に差し込まれる文章ブロックです。各フラグメントの本文は、コントリビューションファイル内の `## fragment: <anchor>` ブロックです。

<a id="fragment-anchors"></a>
### フラグメントアンカー

| アンカー | フラグメントの挿入位置 | ステータス |
| ------------------ | ------------------------------------------------------------------ | ------ |
| `after-step:<n>`   | `### Step <n>` の直後（次の `###` / `##` の前） | ✅ |
| `before-step:<n>`  | `### Step <n>` の直前 | ✅ |
| `end-of-steps`     | `## Steps` ブロックの末尾 | ✅ |
| `in:<Compartment>` | 指定した `## <Compartment>` ブロックの末尾（例: `in:Sensors`） | ✅ |
| `after-questions`  | 質問生成ステップの後 | ⏳ 未実装 — `locateAnchor` に分岐がなく、「不明なアンカー」として棄却されます。`after-step:<n>` を使ってください。 |

フラグメントは `(order, plugin)` で決定論的に整列されます。同じ `(plugin, anchor, order)` の衝突が起きた場合、同一ファイル内でも今回の実行で別ファイル間でも、**ログ付きで棄却** され、後勝ちにはなりません。2 つの *異なる* プラグインが同じステージへコントリビューションした場合は、構造的追加は和集合として統合され、フラグメントはこの同じ順序規則で交互に並びます。つまり本当にマージされます。

差し込まれた各フラグメントは、コンテンツハッシュを持つセンチネルコメントで包まれます（`{/* plugin:<plugin>:<anchor>:<order>:<hash> */} … {/* /plugin:… */}`）。再コンポーズが冪等であり、更新時に新しいフラグメントが古いブロックを置き換えられるのは、この仕組みのおかげです。ここから 2 つの作成ルールが導かれます。

- **フラグメント本文の中にセンチネルに見える行を書かないこと。** `{/* /plugin:… */}` に一致する行が本文内にあると、ブロック終端と誤認され、更新時の差し込み処理を壊します。
- **プレリリース版ビルドからの更新に注意すること。** このブランチの *レビュー用ビルド*（センチネルにハッシュが入る前）からコンポーズされたインストールには、古いハッシュなしマーカーが残っています。更新時にそれを認識できないため、同じブロックが 2 回差し込まれます。影響するのは PR ブランチのインストールだけです。クリーンな基底から再コンポーズするか、旧ブロックを 1 回だけ手で消してください。

<a id="4-packaging-the-other-primitives"></a>
## 4. ほかのプリミティブをパッケージ化する

`test-pro` はステージ、コントリビューション、センサー、サポートエージェント、スコープ、方法論ナレッジを出荷します。より豊かなプラグインは、後からメソッド / ルールも追加できます。メモリの投影は延期されたままです（18 章 §8 のステータスを参照）。

- **エージェント。** `agents/<plugin>-<role>-agent.md` を `plugin:` 付きで置きます。プラグイン接頭辞がコアのファイル名接頭辞 `aidlc-` を置き換え、ファイル名の語幹はフロントマターの `name` と一致しなければなりません（例: `agents/test-pro-metrics-agent.md` は `name: test-pro-metrics-agent` を持ちます）。コンポーズ後に自動で検出され、プラグインのステージから `lead_agent` / `support_agents` として指名できます。同じパスで内容が異なる衝突が起きても上書きされず、コンポーズがドロップログに記録します。[エージェントの追加](/harness-engineering/adding-an-agent) を参照してください。
- **センサー。** `sensors/aidlc-<id>.md` のマニフェスト **と** `tools/` 配下のスクリプトを両方出荷します。両方必要です。マニフェストだけでも検出はされますが、実行するにはスクリプトが `tools/` に存在しなければなりません。自分のステージへは `sensors:` で結び付け、コアステージへはコントリビューションの `adds.sensors` で結び付けます。[センサー](/harness-engineering/sensors) を参照してください。
- **メソッド / ルール。** *（⏳ 延期中。）* `contributes.memory` で `memory/` サブツリー、すなわち `memory/phases/<phase>.md`（または `memory/{org,team,project}.md`）を出荷します。設計上は **デフォルトスペースのメソッドシード**（`aidlc/spaces/default/memory/`）へマージされますが、パッケージャー / コンポーズフックはまだ `memory/` を投影もマージもしません。`rules/` ディレクトリを出荷してはいけません。そのパスはもう読まれません（ルール層はスペースごとのメモリへ移動しました）。[ルールとループ](/harness-engineering/rules-and-the-loop) を参照してください。
- **ナレッジ。** エージェント別の **方法論** ナレッジを `knowledge/<agent-slug>/` に出荷します。これはフレームワーク出荷側の `<harness>/knowledge/` ツリーへ投影され、そのエージェントがステージを主導またはサポートするときに読み込まれます。ただし **ドメイン / チームナレッジ**（`aidlc/spaces/<space>/knowledge/`）は初期化時に空の利用者ランタイム状態であり、プラグインからは出荷しません。[チームナレッジ](/harness-engineering/team-knowledge) を参照してください。
- **スコープ。** スコープの **識別子** は、`scopes/<plugin>-<name>.md` に出荷する 1 ファイルです。プラグイン接頭辞がコアのファイル名接頭辞 `aidlc-` を置き換え、ファイル名の語幹はフロントマターの `name` と一致しなければなりません（例: `scopes/test-pro-validation.md` は `name: test-pro-validation` を持ちます）。プラグインが作成したステージのメンバーシップは、それらのステージの `scopes:` フロントマターのリストです。コントリビューションの `adds.scopes`（§3）はまだ延期中で、マージされずドロップログに記録されるため、既存のコアステージへ自分のスコープを加える手段としては当てにしないでください。[スコープ](/harness-engineering/scopes) を参照してください。

<a id="5-distribution--install"></a>
## 5. 配布とインストール

パッケージャーは、あなたのプラグインを **実際のホストプラグイン** として出力します（ハーネスごとに 1 つの投影先。`.claude-plugin/plugin.json`、`.codex-plugin/plugin.json`、そしてキーロ向けフォルダー投影）。その出力を、セマンティックバージョンのタグと `marketplace.json` を持つバージョン管理リポジトリとして公開し、チームはホストのネイティブコマンドでインストールします。

<a id="claude--codex-host-store"></a>
### クロード / コデックス（ホストストア）

```bash
# teams run these in their host CLI:
/plugin marketplace add <your-org>/<your-plugin-repo>    # Claude
/plugin install test-pro@<marketplace>                   # Claude

codex plugin marketplace add <your-org>/<your-plugin-repo>   # Codex
codex plugin add test-pro@<marketplace>                      # Codex
```

出力されたプラグインに同梱される **セッション開始フック** が自動コンポーズを実行します。選択されたすべてのプラグインのサブツリーとコントリビューションをマージし、統合結果を検証し、ステージグラフとスコープグリッドをコンパイルし、その結果をプロジェクトへ投影します。オーケストレーターはそのコンパイル済みグラフだけを頼りに振り分けるため、プラグインステージはコンポーズされた瞬間から実行対象になります。本文やスキルファイルを手で編集する必要はありません。

<a id="kiro-no-store--folder-drop-then-run-the-composer-explicitly"></a>
### キーロ（ストアなし。フォルダー配置の後、コンポーザーを明示実行する）

```bash
# git pull your plugin repo, copy the Kiro projection into the project:
cp -r dist/plugins/<name>/kiro/. <project>/
# run the composer explicitly (the one working invocation today):
AIDLC_PLUGIN_ROOT="<plugin-root>" AIDLC_PROJECT_DIR="<project>" \
  AIDLC_HARNESS_DIR=.kiro bun "<plugin-root>/hooks/compose.ts"
# open in Kiro IDE or kiro-cli chat → /aidlc
```

> **まだ接続されていません。** 初回入力で自動的にコンポーザーを起動する `.kiro.hook` と、`aidlc plugin compose` のラッパー CLI は設計済みですが未実装です。出力される `.kiro.hook` は今のところ不活性で、キーロ CLI は `.kiro.hook` を読まず、`${PLUGIN_ROOT}` を設定するキーロホストも存在しません。実装されるまでは、上の明示的な `bun compose.ts` 呼び出しを使ってください。延期項目の全一覧は 18 章 §8「ステータス」を参照してください。

<a id="trust"></a>
### 信頼

信頼は **ホストネイティブ** です。プラグイン側で何かを作る必要はありません。
- クロード: 組織管理者が `strictKnownMarketplaces` を設定します（管理対象で、利用側からは上書き不可）。
- コデックス: プラグインごとに 1 回の信頼確認プロンプトがあります。内容ハッシュ固定です。
- キーロ: 該当なし（フォルダー配置であり、ホスト側のゲートなし）。

> **具体例** — `plugin.json`、`marketplace.json`、`managed-settings.json`（組織の信頼設定）、`aidlc.lock.json` — は [`examples/test-pro/`](https://github.com/awslabs/aidlc-workflows/blob/main/docs/reference/examples/test-pro/) にあります。プラットフォームチーム向けの全体例は [プラグイン機構 §8](/reference/plugin-mechanism) も参照してください。

<a id="rules-of-the-road"></a>
## 守るべきルール

- **番号は表示専用です。** 自然な `number:` は書きますが、範囲は主張しません。ステージを挿入してもコアは番号を振り直しません。
- **成果物の名前空間分離。** 生成するすべての成果物には `<plugin>-` 接頭辞を付けます。コアや別プラグインの成果物と衝突してはいけません。
- **プリミティブ名は一意です。** プラグインのスコープ / エージェント / センサーは、コアや別プラグインと衝突してはいけません。衝突した場合はコンポーズエラーになり、出どころも表示されます。（メソッドファイルはメモリシードへファイル単位で加算マージされます。）
- **依存関係** *(⏳ 延期中).* `dependencies` は、本来は `name@^x.y.z` 制約を依存先プラグインの `version` に対して解決し、循環を拒否するための設計ですが、**現時点では何もこのフィールドを読みません**。宣言しても効果はありません（18 章 §8 のステータスを参照）。
- **追加専用です。** コントリビューションができるのは追加だけです。コアステージのフィールド、エージェント、本文を上書きしたり削除したりはできません。（上流挙動そのものを *変えたい* のであれば、それはプラグインではなくフレームワーク設計の判断です。）
