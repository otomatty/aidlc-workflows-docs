> [AI-DLC ドキュメント](/) の一部 · [ユーザーガイド](/guide/introduction) · **ハーネスエンジニアガイド** · [開発者リファレンス](/reference/overview)

AI-DLC は方法論であり、この実装はそれを、利用中のハーネスですぐ動く形で提供します。Claude Code、Kiro CLI、Kiro IDE、Codex CLI、opencode で、14 のエージェント（11 の分野別エキスパート、2 つのレビュアー、そしてコンポーザー）、32 のステージ、9 のスコープ、そして一連のルールとセンサーをそのまま使えます。このガイドは、その方法論を**作り変えたい**人のためのものです。実行するステージを変える、フレームワークが扱っていない分野向けのエージェントを加える、スコープを引き締める、恒常的なルールを教える、あるいはステージに決定論的な検査を結び付ける、といった作業を扱います。

それらはすべて**コードを書かずに**行えます。

---

<a id="three-readers-three-guides"></a>
## 3 種類の読者と 3 つのガイド

AI-DLC のドキュメントは、話題別ではなく、あなたが何をしたいかに応じて分かれています。

| ガイド | あなたは… | 変更するもの… |
|-------|----------|-------------|
| [ユーザーガイド](/guide/introduction) | AI-DLC *を使って* ソフトウェアを作る人 | `.claude/` の中身は何も変えず、`/aidlc` を実行し、ゲートで回答し、成果物をレビューする |
| **ハーネスエンジニアガイド**（このガイド） | チーム向けに AI-DLC の振る舞い *そのもの* を形作る人 | フレームワークが読む**データ**: ステージ、エージェント、スコープ、ルール、センサー、ナレッジ |
| [開発者リファレンス](/reference/overview) | AI-DLC *自体* を変更する人 | そのデータを読む**コード**: オーケストレーター、フック、CLI ツール、コンパイル処理、テストスイート |

このガイドと開発者リファレンスの境界線は、**データとコードの違い**です。ハーネスエンジニアが触るものはすべて、YAML フロントマター付きの Markdown ファイルか JSON 設定であり、フレームワークが実行時に読み込む宣言的なデータです。ステージを追加する、エージェントを追加する、スコープを定義する。これらに *TypeScript の編集を不要にする* こと自体がフレームワークの設計原則です。変更のために `.ts`、つまりオーケストレーター、フック、ツールを編集する必要が出た瞬間、それは開発者リファレンスの領域です。

<a id="the-mental-model-stages-are-what-agents-are-who"></a>
## メンタルモデル: ステージは「何を」、エージェントは「誰が」

フレームワークの大半は 2 つの基本要素で成り立っており、それを取り違えないことがこの仕事の中心です。

- **ステージ**は作業の単位、つまり *何をするか* です。どの成果物を受け取り、どれを生成し、どのエージェントが主導し、どう実行されるかを宣言します。ステージはワークフローグラフのノードです。
- **エージェント**はペルソナ、つまり *誰がその作業を担うか* です。分野の専門知識、ツールの許可リスト、モデルを持ちます。エージェントはステージ *の中に* 読み込まれます。

ステージは主担当エージェントを指名しますが、エージェント側がステージを指名することはありません。この非対称性は意図的なものです。これにより、作業の割り当て変更はステージを編集するだけで済み、担当者を書き換える必要がありません。また、担当者の追加もエージェントファイルを置くだけで済み、ステージがそれを使うと決めるまでワークフローを乱しません。

作業をステージに沿って進める仕組みは 2 つあり、ハーネスエンジニアはその両方が読む**データ**を形作ります。決定論的な**エンジン**（`core/tools/aidlc-orchestrate.ts`。サブコマンドは `next`、`report`、`park` のちょうど 3 つ）は、`aidlc-state.md` とコンパイル済みの `stage-graph.json` を読み、次に何を走らせるかを決め、型付きディレクティブを 1 つ出力します。**コンダクター**（`skills/aidlc/SKILL.md`）は、それぞれのディレクティブを実行に移す薄い転送ループです。経路選択はエンジンにあり、あなたが編集するステージファイル、スコープ、ルールが、それを操舵する入力になります。

ハーネスエンジニアが設定するその他の要素は、すべてこの 2 つにぶら下がります。

- **スコープ**は、ある種類の作業に対して *どの* ステージを走らせるかを決めます（バグ修正では 32 ステージ中 7 つ、エンタープライズ機能ではすべて）。
- **ルール**は、すべてのワークフローに持ち込まれる恒常的な判断、つまりチームの「いつもこうする」です。
- **センサー**はステージに結び付けられた決定論的な検査であり、あらゆるファイル書き込みのたびに走る助言的な第二の見解です。
- **ナレッジ**は、エージェントが作業前に読み込む分野固有の文脈です。

<a id="what-you-can-change-without-code"></a>
## コードを書かずに変えられること

これらはすべて `core/` に著述します。ここは人が著述するハーネス中立なソースであり、その後にハーネスごとのツリーを再生成します（下の [ビルドモデル](#the-build-model-author-in-core-regenerate-the-harnesses) を参照）。

| 変更内容 | 著述場所 | 章 |
|--------|-------|---------|
| ステージの動作を編集する | `core/aidlc-common/stages/<phase>/<slug>.md` | [ステージの構造](/harness-engineering/anatomy-of-a-stage) |
| まったく新しいステージを追加する | 適切なフェーズディレクトリの新規ファイル + グラフへの接続 | [ステージの追加](/harness-engineering/adding-a-stage) |
| エージェントを追加または変更する | `core/agents/<name>-agent.md` | [エージェントの追加](/harness-engineering/adding-an-agent) |
| スコープを定義する | `core/scopes/aidlc-<name>.md` + 各ステージの `scopes:` タグ | [スコープ](/harness-engineering/scopes) |
| 恒常的なルールを教える | `core/memory/{team,project}.md` | [ルールと学習ループ](/harness-engineering/rules-and-the-loop) |
| 決定論的な検査を結び付ける | `core/sensors/` 配下のセンサーマニフェスト + ステージの `sensors:` インポート | [センサー](/harness-engineering/sensors) |
| チームの分野別ナレッジを追加する | `aidlc/knowledge/<agent>-agent/`（実行時のスペース単位ナレッジディレクトリ） | [チームナレッジ](/harness-engineering/team-knowledge) |
| コンストラクションとスウォームの方針を形作る | `core/memory/` + `units-generation` ステージ | [コンストラクションとスウォーム](/harness-engineering/construction-and-swarm) |

各章は *どう行うか* を説明し、完全なスキーマについては [開発者リファレンス](/reference/overview) へリンクします。リファレンスが正式な契約であり、このガイドは実務向けの解説です。

1 行だけ例外があります。**チームの分野別ナレッジ**は、あなた自身のプロジェクトで実行時に、スペース単位（`aidlc/knowledge/`。スペースの `memory/`、`codekb/`、`intents/` と同じ階層）へ追加する文脈です。これは `core/` の一部ではなく、フレームワークが上書きすることもありません。上のそれ以外は、すべて `core/` に著述するフレームワークのソースです。

<a id="naming-rules-and-where-they-are-enforced"></a>
## 命名規則とその強制箇所

ステージのファイル名（拡張子を除いた部分）は、フロントマターの `slug` と一致していなければなりません。`aidlc-graph compile` は、この不一致とステージスラッグの重複をハードエラーとして拒否します。センサーのファイル名 / id の検査はコンパイル時のハードエラーです。スコープとエージェントで宣言された名前が重複するとローダーエラーとなり、両方のファイルがエラーに示されます。スコープ / エージェントのファイル名と名前のずれは、著述者がファイル名を変更するか `name` を修正できるよう、`/aidlc --doctor` が助言として報告します。

<a id="the-build-model-author-in-core-regenerate-the-harnesses"></a>
## ビルドモデル: `core/` に著述し、ハーネスを再生成する

ハーネスエンジニアが著述するものはすべて **`core/`** にあります。ここが人の手で著述する、ハーネス中立な唯一の正本です（ステージは `core/aidlc-common/stages/`、エージェントは `core/agents/`、そのほかスコープ、ルール、センサー、ナレッジ、ツール、フック）。実際に実行するハーネスごとの `dist/<harness>/` ツリー（`dist/claude/.claude/`、`dist/kiro/.kiro/`、`dist/kiro-ide/.kiro/`、`dist/codex/`、`dist/opencode/`）は、`core/` と薄い `harness/<name>/` 層から**生成**されます。さらに**差分監視**も行われ、そこを手で編集すると CI が拒否します。手順は常に次のとおりです。

```bash
# 1. edit the source in core/ (never dist/)
$EDITOR core/aidlc-common/stages/inception/my-stage.md

# 2. regenerate every harness tree from core/ + harness/
bun scripts/package.ts

# 3. confirm no drift (the CI guard; run before committing)
bun scripts/package.ts --check
```

以降の章で `bun .claude/tools/aidlc-graph.ts compile`（または別のツール）を実行するように書かれている場合、そのコマンドは *インストール済み* のツリー、つまりあなたのプロジェクトの `.claude/`（または `.kiro/` / `.codex/`）に対して動き、実行時用にグラフを再コンパイルします。そこが著述場所ではありません。**著述は `core/` で行い、ツールはハーネスディレクトリで実行する**、ということです。この分離、すなわち著述されたソースと生成された実行環境の違いが、このガイド全体で常に意識すべき点です。完全なビルド契約は [新しいハーネスへの移植](/harness-engineering/porting-to-a-new-harness) と、開発者リファレンスの [アーキテクチャ § ソースと配布物](/reference/architecture#source-vs-distribution-one-core-many-harnesses) を参照してください。

<a id="when-you-cross-into-the-developer-reference"></a>
## どこから開発者リファレンスの領域に入るか

変更対象がフレームワークのデータではなくコードになったら、[開発者リファレンス](/reference/overview) を参照してください。

- オーケストレーターの経路選択や状態機械
  （[オーケストレーター](/reference/orchestrator)、
  [状態機械](/reference/state-machine)）。エンジン、コンダクター、ディレクティブ、ランナー、スコープ形状、スウォームの正式な契約については
  [スキルシステム](/reference/skill-system) を参照
- フックや CLI ツール（[フックとツール](/reference/hooks-and-tools)）
- ステージグラフのコンパイル処理や監査イベントの分類体系
- テストスイート（[テスト](/reference/testing)）

ステージやエージェントを追加することはワークフローグラフには *触れます* が、それを読むコード自体は変えません。だからこのガイドの範囲です。グラフのコンパイル方法を変える、あるいは新しい監査イベントを追加するのはコード変更なので、そちらの領域です。

<a id="how-this-guide-is-organized"></a>
## このガイドの構成

初めて読むときは順番に進めてください。

1. **[ステージの構造](/harness-engineering/anatomy-of-a-stage)** — ステージファイルの形式、フロントマターの契約、本文の 3 区画、グラフのコンパイル方法。何かを変える前に、まず理解すべき最重要事項です。
2. **[ステージの追加](/harness-engineering/adding-a-stage)** — 一連の手順。ファイルを著述し、依存関係の辺をつなぎ、コンパイルし、スコープに現れることを確認します。
3. **[エージェントの追加](/harness-engineering/adding-an-agent)** — ペルソナを著述し、主導または支援するステージに結び付けます。
4. **[スコープ](/harness-engineering/scopes)** — スコープとステージの対応を定義し、調整します。
5. **[ルールと学習ループ](/harness-engineering/rules-and-the-loop)** — レイヤーチェーンに沿ってルールを著述し、修正内容を学習ループでルールへ昇格させます。
6. **[センサー](/harness-engineering/sensors)** — 決定論的な検査を著述し、ステージに結び付けます。
7. **[チームナレッジ](/harness-engineering/team-knowledge)** — エージェントにチームの分野別文脈を与えます。
8. **[コンストラクションとスウォーム](/harness-engineering/construction-and-swarm)** — チームのコンストラクション自律性方針をルール層で設定し、`units-generation` を通じてユニットごとのボルト群をどこまで並列実行できるかを形作ります。
9. **[新しいハーネスへの移植](/harness-engineering/porting-to-a-new-harness)** — `harness/<name>/` ディレクトリ 1 つとマニフェスト行 1 つで別の CLI ハーネスを追加します。`core/` の編集は不要です。マニフェスト契約、フックアダプター、`emit.ts` を扱います。
10. **[プラグインの著述](/harness-engineering/authoring-a-plugin)** — `plugins/<name>/` に、再利用可能で任意導入の **AIDLC プラグイン**をパッケージ化します。新しいステージ、エージェント、スコープ、センサーと、既存の中核ステージへの追加的な拡張を含み、ハーネスごとに実際のホストプラグインとして出力されます。設計は開発者リファレンスの単独章（[18 仕組み](/reference/plugin-mechanism)）にあります。
