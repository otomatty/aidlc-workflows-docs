---
title: ステージの追加
description: 新しいステージファイルを著述し、依存関係の辺を結び、スコープへ載せるまでの手順を説明します。
---
ステージはワークフローグラフのノードです。どの成果物を受け取り、どれを生成し、どのエージェントが主導し、どう実行されるかを宣言する作業単位です。ステージの追加は、ハーネスエンジニアが行う変更の中でもっとも構造的なものです。方法論そのものに、新しい工程を導入することになるからです。この章では、その一連の手順を説明します。フェーズを選び、ファイルを著述し、依存関係の辺を結び、グラフをコンパイルし、新しいステージが期待どおりの場所に着地したことを確認します。

まず [ステージの構造](/harness-engineering/anatomy-of-a-stage) を読んでください。そこでは、この手順が前提にしているファイル形式、つまりフロントマターの契約と 3 区画の本文を説明しています。ここでは *ワークフロー* に焦点を当てます。各フィールドを 1 つずつ列挙するのではなく、どこで判断が必要か、何をどう検証するかを扱います。完全な契約は、各工程から [ステージ定義](/reference/stage-definition) へリンクしています。

この進め方は、開発者リファレンスにある [スコープの追加](/reference/contributing#adding-a-scope) と [エージェントの追加](/reference/contributing#adding-an-agent) の手順と対になっています。つまり、番号付きの手順を示し、その後で *自動的に検証されること* と *自分で確認しなければならないこと* を明確に分けます。

---

<a id="before-you-start-is-a-new-stage-the-right-move"></a>
## 始める前に: 本当に新しいステージが必要か？

新しいステージが居場所を得るのは、既存のどのステージも生成していない成果物を生成するとき、あるいは既存ステージのどれも扱っていない方法で成果物を受け取るときです。もし変えたいのが *既存ステージの動作* だけなら、手順の文言を直す、主担当エージェントを付け替える、センサーを足す、といった変更はそのステージファイルを直接編集すれば足ります。新しいノードは不要です。既存ステージをその場で編集する方法は [ステージの構造](/harness-engineering/anatomy-of-a-stage) を参照してください。

ステージの追加は**データ作業**です。YAML フロントマター付きの Markdown ファイルを著述し、JSON 成果物を再コンパイルします。TypeScript の変更は不要です。ここで意識すべき境界は 1 つだけです。もしグラフの *コンパイラー自体* に新しい動作が必要なら（まだ理解していないフロントマターキー、新しい探索規則など）、それはハーネス側の作業ではなく開発者リファレンスの関心事です。仕様がすでに認識する予約キー（`when`、`on_failure`、`blocks_on` など）は [ステージ定義](/reference/stage-definition) に列挙されています。そこにないものへ手を伸ばすなら、それはコード側の変更です。

<a id="steps"></a>
## 手順

<a id="1-decide-which-phase-the-stage-belongs-to"></a>
### 1. そのステージが属するフェーズを決める

ステージファイルは `core/aidlc-common/stages/<phase>/<slug>.md` の下に置かれます。フェーズはディレクトリ名で表し、5 つあります。

```
core/aidlc-common/stages/
├── initialization/
├── ideation/
├── inception/
├── construction/
└── operation/
```

どのフェーズに置くかは見た目の問題ではありません。そのステージに、コンパイル時にどの `phases/<phase>.md` ルール層が付くかを決めます。コンストラクションフェーズのステージはコンストラクションのフェーズルールを継承し、インセプションのステージはインセプションのルールを継承します。（初期化にはフェーズルールファイルがありません。）ファイルを置きやすい場所ではなく、その作業がライフサイクルのどこで起こるのかに従ってフェーズを選んでください。

<a id="2-create-the-stage-file-with-required-frontmatter"></a>
### 2. 必須フロントマターを持つステージファイルを作る

選んだフェーズディレクトリに新しい `<slug>.md` を置きます。スラッグはファイル名の拡張子を除いた部分であり、それ以外のあらゆる場所でステージの識別子になります。`requires_stage` の辺、スコープ対応、監査ログでも使われます。慎重に決めてください。後で名前を変えると影響が広がります。

フロントマターはグラフの辺と実行契約を宣言します。構造を担うフィールドは次のとおりです。

| フィールド | 役割 |
|-------|--------------|
| `requires_stage` | 依存関係の辺。どのステージがこのステージに先行しなければならないか |
| `consumes` | このステージが読む成果物。各項目に `required` 真偽値がある |
| `produces` | このステージが書く成果物（そのまま順方向の辺になる） |
| `lead_agent` | このステージに責任を持つペルソナ |
| `support_agents` | 主担当の後にコンダクターが読み込む任意の観点 |
| `mode` | `inline`、`subagent`、または予約済みの `agent-team` |
| `for_each` | 任意。実体ごとの反復を駆動する成果物名 |

本文は `## Steps` から始まります。主担当エージェントが従う命令形の文章です。その後ろに `## Sensors` と `## Learn` の区画が続きます。フィールド一覧の完全版、型、制約については [フィールドリファレンス — 使用条件](/reference/stage-definition#field-reference--when-to-use) を参照してください。

<a id="3-wire-the-dependency-edges-so-the-graph-places-it"></a>
### 3. グラフがそのステージを正しい位置に置けるよう依存関係の辺を結ぶ

ここで初めて、新しいステージがワークフローに差し込まれます。接続を担うフィールドは 3 つあり、それらは互いに整合していなければなりません。

- **`requires_stage`** は辺を表します。役割は 2 種類です。*意味的なデータ依存関係*（「成果物 X を受け取り、それはステージ Y が生成する」ので `Y` を足す）と、*表示順の辺*（同じフェーズの 2 ステージにデータ依存関係はないが、固定順で実行したい場合）です。コンパイラーのアルファベット順による同順位解決は、明示的な配置の代用品ではなく、最後の安全網に過ぎません。順序の意図があるなら辺を自分で書いてください。
- **`consumes`** は、そのステージが読む成果物を列挙します。各項目は `required` 真偽値を持ちますが、その意味は *有効な計画に対して* 解釈されます。`required: true` は「このスコープで生成元ステージが走るなら、この受け取り条件を満たす必要がある」であって、「生成元は常に走る」ではありません。既存開発でのみ必要な受け取り条件には `conditional_on: brownfield` を付けます。無条件の受け取りではフィールド自体を省略します（`always` という値はありません）。
- **`produces`** は順方向の辺の一覧です。下流ステージが「成果物 Z を生成するのは誰か」と尋ねたとき、グラフは `producersOf()` を通して答えます。つまり `produces: [Z]` を宣言したステージが上流に結ばれます。

この 3 つが整合していれば、コンパイラーがステージを自動的に正しい位置に置きます。位置合わせのために `stage-graph.json` を手で編集することはありません。`requires_stage`、`consumes[].required`、`consumes[].conditional_on`、`for_each` の細かな意味（集約が宣言ではなく推論されることも含む）は [フィールドリファレンス — 使用条件](/reference/stage-definition#field-reference--when-to-use) を参照してください。

<a id="4-compile-so-stage-graphjson-regenerates"></a>
### 4. ハーネスを再生成して `stage-graph.json` を再コンパイルする

今 `core/` の下に著述した YAML が正本です。パッケージャーを実行して、`core/` からすべての `dist/<harness>/` ツリーを再生成します。これにより新しいステージファイルがコピーされ、グラフも再コンパイルされます。

```bash
bun scripts/package.ts            # regenerate every harness from core/ + harness/
bun scripts/package.ts --check    # the CI drift guard — run before committing
```

実行時に読むのはコンパイル済み成果物、つまり `<harness-dir>/tools/data/stage-graph.json` です（たとえばインストール済みの Claude Code ツリーなら `.claude/tools/data/stage-graph.json`）。これはパッケージャーが YAML から生成します。すでにインストール済みのツリーを相手に反復したいなら、そのツリーだけを直接再コンパイルすることもできます。

```bash
bun .claude/tools/aidlc-graph.ts compile
```

どちらの場合でも、著述手順は一方向です。`core/` の YAML を編集し、パッケージャー（あるいはインストール済みツリーに対する `compile`）を実行すると JSON が更新され、実行時ローダー（`loadStageGraph()`）はその新しいノードをそのまま読み込みます。`stage-graph.json` を手で編集してはいけません。ビルド成果物であり、次のコンパイルで上書きされます。処理図と CI の差分ガードの全体像は [著述手順](/reference/stage-definition#authoring-flow) を参照してください。

<a id="5-verify-it-appears--and-in-which-scopes"></a>
### 5. そのステージが現れたこと、そしてどのスコープに入るかを確認する

新しいノードがコンパイルされ、どこで走るのかを確認します。

```bash
# Topological order of the full graph — your slug should appear
bun .claude/tools/aidlc-graph.ts topo

# Who produces / consumes your stage's artifacts
bun .claude/tools/aidlc-graph.ts producers <artifact>
bun .claude/tools/aidlc-graph.ts consumers <artifact>

# The stages on a given scope's path — does your stage run for this scope?
bun .claude/tools/aidlc-graph.ts scope <scope-name>

# Dependency sanity for a scope
bun .claude/tools/aidlc-graph.ts validate-scope <scope-name>
```

新しいステージは、**どのスコープにも自動では入りません**。スコープへの所属はステージ自身に置かれています。フロントマターの `scopes:` 一覧が、そのステージを実行するすべてのスコープを列挙します。何のスコープも書かれていないステージは、どこでも `SKIP` です。したがってステージを追加した後は、どのスコープでそれを走らせるかを決め、ステージの `scopes:` 一覧に各スコープ名を加える必要があります。そのうえで再コンパイルすると、転置結果が更新されて `scope-grid.json` に反映されます。詳しくは [スコープ](/harness-engineering/scopes) を参照してください。ここが意図的な接続点です。ステージ本文を著述するとそのステージは *存在* し、`scopes:` タグを付けると *実行* されます。

<a id="to-add-a-stage-write-a-stage-file--the-runner-is-opt-in-sugar"></a>
## ステージを追加するとはステージファイルを書くこと。ランナーは任意の簡便機能

この章の見出しそのものが拡張性の契約でもあります。**ステージを追加するには、ステージファイルを書く。** 構造的に必須なのはそれだけです。ステージがグラフにコンパイルされれば（上の手順 2〜4）、それだけで単体実行可能になり、スキルや登録処理は不要です。

```bash
bun .claude/tools/aidlc-orchestrate.ts next --stage <your-slug> --single
```

エンジンの `--single` モードは、その 1 ステージだけを隔離して実行します。解決済みの受け取り・生成パス、主担当エージェント、ルール、センサーを持つ単一の `run-stage` ディレクティブを出力し、コンダクターがそれを実行し、合成 ID の `STAGE_STARTED` / `STAGE_COMPLETED` の組を監査ログに残します。`--single` 実行は意図的に隔離されています。**主ワークフローの `Current Stage` には決して触れません。** ツールは単体実行から主ワークフローを進めることを拒否するので、ステージを 1 つだけ単独実行しても、進行中のワークフローが脱線することはありません。

<a id="the-runner-skill-is-optional-packaging"></a>
### ランナースキルは任意のパッケージ

出荷される単体実行可能なステージには、それぞれ `skills/aidlc-<slug>/SKILL.md` に薄いランナースキルも付きます。これにより `/aidlc-<slug>`（例: `/aidlc-application-design`）として入力できます。これらは **`--single` フラグの上に載った任意の簡便機能**に過ぎません。およそ 6 行のシェルで `next --stage <slug> --single` を駆動するだけです。手書きではなく、生成器がコンパイル済みステージのスラッグごとに出力するため、ランナーの集合がステージの集合から手作業でずれることはありません。（3 つの起動用初期化ステージにはステージ別ランナーがありません。単独の `--single` に意味がなく、初期化フェーズ全体が `/aidlc-init` コマンドだからです。これはエンジンのインテント作成処理をパッケージ化しています。）ステージを追加または削除した後は、ランナーを再生成します。

```bash
# Regenerate every runner dir from the compiled stage list
bun .claude/tools/aidlc-runner-gen.ts write

# CI drift guard: exits 1 if the runner set != the compiled stage set
bun .claude/tools/aidlc-runner-gen.ts check
```

ランナーは **`hooks:` ブロックを持ちません。** 決定論的な中核処理（監査、センサー、実行時グラフのコンパイル、状態検証）はプロジェクト全体に対して `settings.json` へ登録されているため、すべてのランナーが自動で継承します。ランナーごとに複製するものはありません。また、ランナーはコンダクターのペルソナを自前で読み込みません。エンジンが最初の `run-stage` ディレクティブに組み込んで届けるからです。ランナー本文は、自分が何をするかと、駆動する 1 コマンドを書くだけです。

ランナースキルをすべて消しても、すべてのステージは `/aidlc --stage <slug> --single` で実行できます。ランナーは、すでに単体実行可能なステージをパッケージ化しているだけであり、定義そのものではありません。著述方法は一貫して「ステージファイルを書く」です。

これらのランナーの正式な契約、つまりエンジン、コンダクター、`run-stage` ディレクティブがコンパイル済みステージを入力可能な `/aidlc-<slug>` スキルに変える仕組みについては、開発者リファレンスの [スキルシステム §4（スキルとランナー）](/reference/skill-system) を参照してください。

<a id="what-validates-automatically-vs-what-you-must-check-yourself"></a>
## 自動で検証されることと、自分で確認すべきこと

<a id="what-validates-automatically"></a>
### 自動で検証されること

- **グラフ上の配置。** `compile` すれば、そのステージの辺（`requires_stage`、`consumes`、`produces`）はグラフに解決されます。トポロジカル順、生成元・利用先の検索、循環検出は、追加したノードを自動で含めます。
- **コンパイル時のフィールド検証。** コンパイラーはグラフを構築するときにフロントマターを検証します。著述上の誤りは実行時に静かに壊れるのではなく、`compile` 時に明確に失敗します。`lead_agent` と `support_agents` は `loadAgents()` により実際の `.claude/agents/*.md` ファイルに照合されます。固定のエージェント列挙型はありません。対応するファイルがないエージェント名をステージが指しているとコンパイルは失敗します（`lead_agent "<name>" has no matching .claude/agents/*.md`）。つまり入力ミスのまま実行時に 404 となるグラフは出荷できません。なお予約済みの `orchestrator` スラッグ（起動用初期化ステージで使うコンダクター自身）は例外で、エージェントファイルを持ちません。
- **CI の差分ガード。** `bun .claude/tools/aidlc-graph.ts compile --check` は、ツリーに差分がなければ `0`、ステージの YAML を編集したのに JSON を再コンパイルしていなければ `1` で終了します。CI もこれを実行するので、コンパイルし忘れた変更はマージ前に止まります。
- **フェーズルールの付与。** ステージはディレクトリによって自分のフェーズを宣言しているため、対応する `phases/<phase>.md` ルール層はコンパイル時に自動で付きます。自分で辺を結ぶ必要はありません。

<a id="what-you-must-check-yourself"></a>
### 自分で確認すべきこと

- **スコープへの参加。** コンパイラーはステージをグラフに置きますが、**どのスコープで走るかは決めません**。ステージ自身の `scopes:` フロントマター一覧に各スコープ名を加え、さらに再コンパイルして `scope-grid.json` の転置結果を更新するまでは、そのステージはグラフには存在してもどこでも走りません。必要な各スコープで `aidlc-graph.ts scope <scope-name>` を実行して確認してください。
- **本文。** パーサーが読むのはフロントマターだけです。`## Steps` の本文は主担当エージェントがステージ実行時に読みます。ほかのステージファイルの構造に合わせて、明確に書いてください。パーサーは指示の曖昧さや欠落を検知しません。
- **文章と辺の整合性。** `requires_stage` と `## Steps` の文章はずれ得ます。パーサーが見るのは辺だけで、文章は見ません。もし手順に「インテント文を読め」と書くなら、その生成元ステージが本当に `requires_stage` に入っていなければなりません。ここは自分で同期を保つ必要があります。
- **各スコープにおける `required` の意味。** `required: true` の受け取り条件でも、生成元を省略するスコープでは意味を失います。これは不具合ではなく正当な動作ですが、その場合にステージ本文が生成元の不在を穏当に扱えるか（「利用できる場合」という代替表現など）は自分で確認する必要があります。
- **ドキュメント。** 新しいステージにより件数や表が変わり、ドキュメントがそれを手で列挙しているなら、そのドキュメントも同じ変更で更新してください。ドキュメント方針に従います。

<a id="the-boundary-case-data-work-vs-code-work"></a>
## 境界事例: データ作業とコード作業

ステージファイルを著述してグラフを再コンパイルするところまでは、完全にハーネスエンジニアのデータ作業です。Markdown、YAML、JSON であり、`.ts` は不要です。ここで黙って越えてはいけない一線は、「そのステージを動かすためにグラフの *コンパイラー* の動作を変えなければならないかどうか」です。たとえば認識していないフロントマターキー、新しい辺の種類、新しい探索規則が必要なら、それはデータを読む側のコードの変更であり、ここではなく [開発者リファレンス](/reference/stage-definition) の領域です。仕様に予約キーの名前空間があるのは、将来の構造拡張を場当たり的にせず、予測可能な形で導入するためです。利用側がまだ実装されていないキーはスキーマが拒否します。コンパイラーがまだ理解していないキーを欲しくなったら、そこで止まってください。それはフレームワーク変更であり、検証規律は [エージェントの追加](/reference/contributing#adding-an-agent) と同種の手順に従いますが、実装先はコードです。
