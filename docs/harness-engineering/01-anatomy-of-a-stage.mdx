---
ステージは AI-DLC ワークフローの原子です。つまり、1 つの工程で *何が起こるか* を表します。ハーネスエンジニアが行う他のすべての変更はステージファイルの上に成り立つので、この章は最初に読むべきものです。読み終えるころには、どのステージの `.md` ファイルを開いても、各部分を誰が読むのかが分かり、ファイルを編集することで変えられるフィールドと、フレームワークのコードに固定されている動作とを区別できるようになっているはずです。

ステージファイルは、YAML フロントマターを持つ 1 つの Markdown ファイルです。フロントマターはグラフのノード、すなわちステージが受け取る成果物、生成する成果物、主担当エージェント、実行方法を宣言します。本文はエージェントが従う文章です。両者が 1 ファイルに共存しているのは意図的であり、その理由をこの章全体で解きほぐしていきます。

---

<a id="two-readers-one-file"></a>
## 2 種類の読み手と 1 つのファイル

ステージファイルには 2 種類の読み手がいて、その読み取り範囲は重なりません。

- **パーサー**は YAML フロントマターだけを読みます。構造化されたグラフ項目、つまり辺、主担当エージェント、実行モードを生成し、本文は完全に無視します。
- ステージを実行する**エージェント**は本文だけを読みます。文章で示された手順に従って成果物を作り、フロントマターには触れません。

両方を 1 つのファイルに入れておくと、ファイルを開いたときにグラフの辺と実行手順を並べて見られます。レビュー担当者は 1 つのステージを読むだけで、何を生成するのか、そしてそれを生み出す指示が何かを同時に確認できます。これをグラフ用の YAML ファイルとエージェント用の文章ファイルに分けると、その場で両方を見渡せなくなります。ステージをレビューしやすくする性質は、まさにそこにあります。

この分離は、「今どこを編集しているのか」を理解する鍵でもあります。フロントマターのフィールドを変えれば、変わるのは *グラフ*（依存関係の辺、主導するエージェント、実行モード）です。本文を変えれば、変わるのは *作業*（エージェントが実際に何をするか）です。この 2 つは独立しています。

実際のステージは `core/aidlc-common/stages/<phase>/<slug>.md` に著述されます。たとえば `stages/inception/application-design.md` です。この章を読みながら 1 つ開いてみてください。終わるころには、その構造が自然に読めるようになっています。

<a id="the-frontmatter-at-a-glance"></a>
## フロントマターの概要

フロントマターは、YAML キーを並べた平坦なブロックです。`slug` や `phase` のように機械的なものもあり、それらは隣のステージからコピーすれば済みます。実際に判断が必要で、著述や編集の前に理解しておく価値があるのは次のフィールドです。

| フィールド | 何を決めるか | 必要な判断 |
|-------|-----------------|-------------------|
| `requires_stage` | 依存関係と実行順の辺 | このステージは本当に別のステージの出力を受け取るのか、それとも単にその後で走ればよいだけか？ |
| `consumes` \| このステージが読む成果物 \| 成果物ごとに `required` か、また既存開発か新規開発かに応じて `conditional_on` が必要か？ |
| `produces` | このステージが書く成果物 | これが他のステージから見える順方向の辺になります。 |
| `lead_agent` / `support_agents` | 誰がこのステージを実行するか | 1 つのペルソナが責任を持ち、支援役が別の観点を加えます。 |
| `mode` \| どう実行されるか \| `inline`（コンダクターのコンテキスト内）か `subagent`（新しいコンテキストへ委譲）か。 |
| `for_each` | 反復実行するかどうか | 各実体ごとに 1 回実行するための成果物名を指定します。 |

特に誤解しやすい判断点をいくつか挙げます。

- **`consumes[].required` は全体ではなく有効な計画に対して効きます。** `required: true` は「*この* ワークフローで生成元ステージが走るなら、この受け取り条件を満たす必要がある」という意味であり、「生成元が常に走る」という意味ではありません。スコープは意図的に上流ステージを省略するため、一律の全体要件にすると、それらのスコープが構造的に無効になってしまいます。ステージ本文は入力がない場合も穏当に扱います（「生成されている場合」のような文章です）。
- **`consumes[].conditional_on` は既存開発と新規開発の分岐を表します。** `conditional_on: brownfield` と付いた受け取り条件は、ワークフローが既存開発のときだけ必須です。無条件の受け取りではフィールド自体を省略します。`always` という値はありません。
- **`mode` は処理の振り分け方法です。** `inline` は短いステージをコンダクター自身のコンテキストで走らせます。`subagent` は、長いステージ（コンストラクションのコード生成など）を新しいコンテキストに委譲し、主コンテキストの容量を圧迫しないようにします。複数のエージェントが 1 ステージに関わることは `support_agents` で表し、`mode` では表しません。コンダクターは主担当を先に起動し、その後に支援役を順番に読み込みます。
- **`for_each` は反復対象の成果物を指名します。** ユニットごとに 1 回ずつ走る 5 つのコンストラクションステージは `for_each: unit-of-work` を宣言します。それ以外のステージはこのフィールドを省略し、1 回だけ実行されます。集約は宣言するものではなく、グラフから推論されます。
- **`lead_agent` と `support_agents` は `core/agents/*.md` に対して検証されます。** 固定の一覧はありません。エージェントを追加するとは、そのディレクトリにファイルを置くことです（[エージェントの追加](/harness-engineering/adding-an-agent) を参照）。

ここで述べているのは方向付けであって契約全文ではありません。フィールド一覧全体、型、制約、そして将来 AI-DLC が追加する予約名前空間のフィールドまで含めた完全版は、開発者リファレンスの [フィールドリファレンス — 使用条件](/reference/stage-definition#field-reference--when-to-use) を読んでください。

<a id="the-three-compartment-body"></a>
## 3 区画の本文

フロントマターの下では、本文は常にこの順番の 3 区画で構成されます。`## Steps`、`## Sensors`、`## Learn` です。`application-design.md` を見ると、3 つすべてに内容が入っているのが分かります。

- **`## Steps`** は、エージェントが従う命令形の文章です。ペルソナを読み込み、先行する文脈を確認し、質問ファイルを作り、成果物を生成し、承認ゲートを提示する。そうしたステージ固有の作業はここにあります。*グラフに触れずにステージの動作を変えたい* とき、もっとも頻繁に編集する区画です。
- **`## Sensors`** は、そのステージの出力に結び付けられた決定論的な検査を記述します。`application-design.md` では、`required-sections` と `upstream-coverage` がステージの Markdown 成果物に対して発火し、それぞれ何を検証するのかが説明されています。結び付け自体はフロントマター側の `sensors:` 一覧で行われます。この区画は、その結び付けが何をするのかを人間向けに説明したものです。センサー全体の話は [センサー](/harness-engineering/sensors) で扱います。
- **`## Learn`** は、学習ループの手順を記述します。ステージの実行中にエージェントが付ける `memory.md` の記録と、承認ゲートで保持した観察事項がプラクティスやセンサーへどう振り分けられるかです。重要なのは、この手順が書き込む先は *スペースのメモリー層*（`aidlc/spaces/<space>/memory/`）とハーネスのセンサー設定（`.claude/sensors/`）であり、ステージファイル自体に書き戻すことはない、という点です。

この 3 区画は前もって宣言されており、`v0.5.0` で追加されたセンサーと学習の結び付けも、本文構造を作り直すことなく素直に収まりました。本文モデルの完全版と、各区画に何を含めてよいかは [3 区画の本文モデル](/reference/stage-definition#three-compartment-body-model) を参照してください。

1 つ、しっかり意識しておきたい境界があります。ステージファイルはフレームワークの成果物であり、その形は不変です。本文の `## Steps` / `## Sensors` / `## Learn` 構造はワークフローによって書き換えられません。ワークフロー中に許可されている唯一の編集は、学習ループがフロントマターの `sensors:` インポート一覧に新しいセンサー ID を追記することです。それ以外でステージファイルを変えるのは、ハーネスエンジニアが意図的に行う編集だけです。

<a id="consumes-and-produces-from-the-graph"></a>
## グラフから見た受け取りと生成

依存関係グラフはどこかに直接書かれているわけではありません。各ステージが宣言する `consumes` と `produces` の組み合わせから立ち上がります。ステージの `produces` 一覧は、そのステージから伸びる順方向の辺です。別のステージが「`scope-document` を生成するのは誰か」と尋ねると、その答えはそれを宣言したステージです。ステージの `consumes` 一覧は逆方向の辺です。それらの成果物が存在するまで、そのステージは実行できません。

`requires_stage` は依存関係を明示し、さらに純粋な実行順も表します。役割は 2 つあります。

1. **意味的なデータ依存関係**。「成果物 X を受け取り、それはステージ Y が生成する」ので、`requires_stage` に Y を入れる。
2. **表示順の辺**。同じフェーズ内でデータ依存関係はないが、実行順だけは固定したい 2 ステージの間に張る辺。明示的に弱い辺を書いておけば、アルファベット順による同順位解決に依存せず、安定した表示順にできます。

新しいステージを著述したり、既存ステージを移動したりするとき、ワークフローへの組み込みを担うのはこの 3 つの一覧です。ここを正しくすると、そのステージは正しいスコープに、正しい順序で、必要な入力が満たされた状態で現れます。まったく新しいステージを実際に接続する手順は [ステージの追加](/harness-engineering/adding-a-stage) で順を追って説明します。

<a id="stage-graphjson-is-compiled--never-hand-edit-it"></a>
## `stage-graph.json` はコンパイルされるものであり、手で編集してはいけない

この実装では、グラフは `dist/claude/.claude/tools/data/stage-graph.json` にある JSON ファイルから実行されます。このファイルは**ビルド成果物**であり、ソースファイルではありません。正本は各ステージファイルにある YAML フロントマターであり、JSON はそれをコンパイルした結果です。

流れは次のとおりです。

```
edit stage .md YAML  →  compile  →  stage-graph.json  →  runtime reads it
```

ステージのフロントマターを編集したら、`stage-graph.json` を再生成するためにコンパイルを実行し、その結果を一緒にコミットしてください。CI は差分検査を走らせ、JSON が YAML からコンパイルされる内容と一致しなければマージを失敗させます。つまり、コンパイルのし忘れは出荷前に検知されます。

したがって守るべきルールは 1 つです。**`stage-graph.json` を手で編集しないこと。** YAML を編集し、再コンパイルしてください。手で編集した内容は次のコンパイルで上書きされるか、差分検査に引っ掛かります。コンパイルコマンドと差分ガードは [著述手順](/reference/stage-definition#authoring-flow) に記載されています。

<a id="what-a-stage-contains-versus-what-it-does"></a>
## ステージが「含むもの」と「行うこと」

この章が扱っているのは、ステージファイルが *何を含むか*、つまり形式です。これとは別に、ステージが実行時に *何をするか* を定める対となる契約があります。承認ゲート、質問の流れ、状態のチェックボックス、完了メッセージといった動作契約であり、分野に関係なくすべてのステージで共通です。それが [ステージプロトコル](/reference/stage-protocol) です。ステージ本文に「承認ゲートについては `stage-protocol.md` に従うこと」と書かれていたら、それが指している契約はこれです。

この 2 つは頭の中で分けておいてください。ステージの *定義*（この章と [リファレンス 15](/reference/stage-definition)）はファイル形式です。ステージの *プロトコル*（[リファレンス 04](/reference/stage-protocol)）は、すべてのステージ作業を取り囲む実行時の動作です。あなたが編集するのは定義であり、プロトコルは基本的にそのままです。

フロントマターと本文がどう組み合わさるかを示す完全な注釈付き例は [実例](/reference/stage-definition#worked-example) を参照してください。
