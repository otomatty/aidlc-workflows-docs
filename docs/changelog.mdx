## 2.5.7
- 日付: 2026-07-23（コミット ccf284b、7/23検知）
- 種別: fix

**Kiro IDE の agent-stop フック回帰を修正**: 転送ループのリマインダーと `SESSION_ENDED` が一度も実行されない回帰を修正した。Kiro IDE はエージェント停止時に空の `USER_PROMPT` を渡すため、アダプターの入口ガードが stdin 待ちに入り（IDE は stdin を開くが何も書かない）、agent-stop フックが黙って無効化されていた。プロンプト送信時とツール使用後のフックは影響なし（これらのイベントでは IDE が `USER_PROMPT` を埋める）。アダプターはいかなるイベントでも stdin を読まなくなり（コンテキストは `USER_PROMPT` 環境変数のみで到着し、stdin の値はもともと消費されていなかった）、同じハングを抱えていたディスパッチャー経路（`aidlc adapter kiro-ide <target>`）も stdin を読まない形で同様に修正された。壊れた版と修正版の両方で Kiro IDE 上の実機検証済み。**アップグレード: `dist/<harness>/` シェルの再コピーが必要（Kiro IDE では `.kiro/hooks/aidlc-kiro-adapter.ts` と `.kiro/tools/aidlc.ts` に修正が入る）。**

同日、スウォームレフェリーの E2E テストカバレッジも復旧している（コミット 27eefb0）。

**§12a レビュアーステップの全完了経路でのエンジン強制**: レビュアーを持つステージは、あらゆる完了経路（対話・ユニット単位・ユニット主導・自律 Construction）で、エンジンが強制する監査可能な検証境界を持つようになった。`aidlc-log.ts review` がディスパッチ前に `REVIEW_REQUESTED` を、`READY`/`NOT-READY` 判定後に `REVIEW_COMPLETED` を記録し（監査イベントは 72 種類から 74 種類に増加）、隔離ステージ実行はメインワークフローと別タグで記録される。`approve`・`advance`・`finalize`・`complete-workflow` は、宣言されたレビュアーによる新鮮な判定が無いレビュアー付きステージの完了を拒否する。却下・改訂・ワークフロー再開始・関連ジャンプ・宣言済み成果物のその後の書き込みは古い受領記録を無効化する。ユニット単位の強制では該当ユニットごとに 1 件のレビューを要求し、成果物が変わったユニットのみを無効化、キャッシュ済みユニットデータを正の `unit-of-work-dependency.md` に照らして検証し、古いキャッシュを自己修復する。自律 Code Generation は #568 のオーナー・コラボレーター・検証者契約を維持し、収束した各スウォームユニットは追加の人間プロンプト無しに Bolt ワークツリー内でレビューされてから確定する。人間のチェックポイントが増えるわけではない。**アップグレード: `dist/<harness>/` シェルの再コピーが必要。**

## 2.5.0
- 日付: 2026-07-17（コミット 3f16023、7/21検知）
- 種別: feat

**3ロールアンサンブル**（North Star Goal 1）: ステージの `mode` が「本体実行中に誰が誰と話すか」という通信トポロジーになり、サポートエージェントが指揮者の演じ分けの声ではなく、独立ディスパッチされる実在のコラボレーターになった。`inline`・`subagent` に加えて2つの新モード: `pipeline`（チェーン: リンクが宣言順に成果物を共同執筆し、各リンクは上流の仕事をすべて見る。最終リンクが完成させる）と `mob`（境界付きラウンドのメッシュ: 全サポートエージェントがリードのドラフトに対し相互に盲目な並列コントリビューションを行い、未解決の異議はトリアージされる — 判断系はステージ途中で人間へ構造化質問として浮上、知識系の争点は confirm-or-maintain の1ラウンド。維持された異議は承認ゲートで逐語引用される）。`subagent` はハブ&スポークに強化。全ディスパッチされたコラボレーターは自分のコントリビューションファイル（`<stage>/contributions/<agent-slug>.md`）を書き、リードのみが `produces[]` 成果物を編集する。このファイルは決定論的な完了証拠を兼ね、宣言済みコラボレーターのファイルが欠落・不正なままでは mob / subagent+supports ステージの承認をエンジンが拒否する（`AIDLC_DISABLE_ENSEMBLE_EVIDENCE=1` が回避ハッチ）。自律 Construction では途中停止せず異議を記録に残す。§12a レビュアーループは不変で、NOT-READY はリード単独を再起動する。出荷ステージの変更: `user-stories` が mob に（Product Manager リード + Design/Developer/Quality）、`reverse-engineering` が pipeline に（developer がスキャン、architect が統合・執筆 — 挙動変更ゼロ）、`practices-discovery` が subagent ハブ&スポークのショーケースに。出荷トポロジーは 28 inline / 2 subagent / 1 pipeline / 1 mob。Kiro CLI/IDE の委譲ロスターは全14ペルソナで完結（ハーネスごとに9つの新規エージェント設定、conductor の trustedAgents、コントリビューションファイル用のスコープ付き書き込み許可）。ステージのライフサイクル遷移は report 所有になり、`aidlc-state.ts` の直接ライフサイクル変更はエンジン所有権エラーで失敗する（スクリプトは `aidlc-orchestrate.ts report` へ移行が必要）。`aidlc-graph compile` はユニット別 Construction ビルドステージが非 `subagent` モードのとき stderr に advisory を出す。**アップグレード: `dist/<harness>/` シェルの再コピーが必要。Kiro CLI/IDE では `.kiro/agents/`（9つの新規委譲先エージェント設定と更新済み conductor `trustedAgents`）も再コピーすること。**
