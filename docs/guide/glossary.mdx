---
title: 用語集
description: AI-DLC ドキュメント全体で一貫して使う主要用語の定義です。
---
AI-DLC 用語の正規定義です。ユーザーガイドと開発者リファレンスの全体で、これらの用語を一貫して使います。

---

| 用語 | 定義 |
|------|-----------|
| **エージェント（Agent）** | オーケストレーターが各ステージで有効にする 11 個の分野別専門家ペルソナの 1 つです（例: aidlc-product-agent、aidlc-architect-agent）。それぞれが専門的な視点と知識を提供します。 |
| **承認ゲート（Approval gate）** | 各ステージの終わりにある対話型の確認地点です。作業の承認、変更依頼、または 3 回の改訂後の現状受け入れを選べます。初期化ステージには承認ゲートがありません。 |
| **自律モード（Autonomy mode）** | ワーキングスケルトン後の段階選択プロンプトを経て、`aidlc-state.md` の `Construction Autonomy Mode` に記録される設定です。`autonomous` では以後の Bolt をゲートなしで実行し、`gated` では Bolt ごとに承認を求めます。選択前の既定値は `unset` です。 |
| **実行単位（Bolt）** | 構築フェーズにおける実行単位です。1 つの作業ユニット（または依存関係で結ばれた少数の作業ユニット群）について、ステージ 3.1〜3.5 を 1 周します。ステージ 3.6（ビルドとテスト）と 3.7（CI パイプライン）は Bolt ごとではなく、すべての Bolt が完了した後に 1 回だけ実行します。構築フェーズ最初の Bolt がワーキングスケルトンです。関連: [並列バッチ]、[ワーキングスケルトン]、[段階選択プロンプト]。 |
| **成果物（Artifact）** | ステージが生成し、インテントの記録ディレクトリ（`aidlc/spaces/<space>/intents/<YYMMDD>-<label>/`）に保存する、版管理された Markdown 文書です。例: `requirements.md`、`code-summary.md`、`initiative-brief.md`。 |
| **監査証跡（Audit trail）** | インテントの記録ディレクトリにある `audit/` 配下の追記専用イベントログです。クローンごとのシャード（`<host>-<clone>.md`）として書き込み、読み取り側が時刻順にまとめます。インテントから本番環境までを完全に追跡できるよう、ISO 形式の時刻付きで 68 種類のイベントを記録します。 |
| **CLI ツール（CLI tool）** | この実装が必要とする外部コマンドラインユーティリティです。実行時の前提ツールは `bun` だけです。Claude Code の組み込みツールとは区別します。 |
| **Claude Code ツール（Claude Code tool）** | Read、Write、Edit、Bash、Glob、Grep、Task、AskUserQuestion など、Claude Code に組み込まれた機能です。エージェントは既定でセッション全体のツール群を継承し、必要に応じて `tools:` 許可リストで絞り込めます。同梱時に設定される唯一の制限は `disallowedTools: Task` です。 |
| **Codex（Codex）** | OpenAI の Codex CLI ハーネスです。現在提供する AI-DLC ハーネス配布物の 1 つであり、`core/` と `harness/codex/` から `dist/codex/` に生成されます。呼び出しには `$aidlc` を使います。詳細は [Codex CLI での AI-DLC](/guide/harnesses/codex-cli) を参照してください。 |
| **コマンド（Command）** | 利用者が AI-DLC を呼び出すための入力です。`/aidlc` に続けてスコープ、フラグ、または自由形式の説明を入力します。内部では `/aidlc` が Claude Code のスキルに対応します。 |
| **コンパクション（Compaction）** | コンテキストウィンドウがいっぱいになったときに、Claude Code が以前の会話内容を自動で要約する処理です。この実装は `aidlc-state.md` と `.aidlc-recovery.md` により、コンパクションをまたいで状態を保持します。 |
| **コンダクター（Conductor）** | `/aidlc` セッション本体（`SKILL.md`）です。軽量な転送ループを動かし、**エンジン**に次の動作を問い合わせ、ステージ実行・質問・群処理の展開などを実施して結果を報告し、この手順を繰り返します。経路選択ではなく実行品質を担います。詳細は [エンジンとスキルシステム](/reference/skill-system) を参照してください。 |
| **制御ループ（Control loop）** | 作業前に適用する常設の判断である**ルール**と、出力に対して起動する決定論的な検査である**センサー**を、前向き制御とフィードバックとして組み合わせ、ステージを導いて検証する仕組みです。CLI 配布物を指す**ハーネス**とは別概念です。以前は両方を「harness」と呼んでいました。 |
| **コア（Core）** | `core/` にある、ハーネスに依存しない手書きの唯一の正本です。エンジン、ステージ、エージェント、ルール、スコープ、センサー、ナレッジ、フック、セッションスキルを含みます。すべてのハーネス配布物はここから生成されます。編集対象はここであり、`dist/` ではありません。 |
| **深さ（Depth）** | 各ステージが生成する詳細さを制御する 3 段階（Minimal、Standard、Comprehensive）の 1 つです。スコープごとに既定の深さがあり、任意の承認ゲートで上書きできます。詳細は [スコープ、深さ、テスト戦略](/guide/scopes-and-depth) を参照してください。 |
| **指示（Directive）** | **エンジン**が `next` ごとに出力する型付きの命令です（例: `run-stage`、`ask`、`print`、`done`、`invoke-swarm`）。これにより、**コンダクター**が次に実行すべきことが厳密に決まります。詳細は [エンジンとスキルシステム](/reference/skill-system) を参照してください。 |
| **配布物（Distribution）** | 1 つのハーネス向けに生成・コミットされ、差分監視される `dist/<harness>/` ツリーです（例: `dist/claude/`、`dist/kiro/`、`dist/codex/`）。利用者はこれを自分のプロジェクトへコピーし、保守担当者が手作業で編集することはありません。**パッケージャー**が**コア**から生成します。 |
| **エンジン（Engine）** | ステージ間の経路選択を担う決定論的なオーケストレーションツールです（`aidlc-orchestrate.ts` の `next` / `report` サブコマンド）。スコープ解決、ステージ順序、移動、再開、ゲート状態を管理し、**コンダクター**が実行する型付きの**指示**を出力します。詳細は [エンジンとスキルシステム](/reference/skill-system) を参照してください。 |
| **外部ツール（External tool）** | ステージが利用する第三者製のツールまたはサービスです（例: AWS CLI、Maven、npm）。Claude Code ツールとは区別します。 |
| **ガードレール（Guardrail）** | ルールファイル内の本文セクション（`## Forbidden`、`## Mandated`、フェーズルールのガードレール見出し）で表す規範的な行動制約です。入れ物はルールであり、「ガードレール」はその中の規範的内容を指します。**ルール**も参照してください。 |
| **ハーネス（Harness）** | AI-DLC コアを、対応するコマンドラインエージェント環境へ投影した CLI 配布物です。対応対象は追加できます（現在は Claude Code、Kiro CLI、Codex CLI）。*注: このリポジトリでは「harness」に 4 つの意味があります:* (1) **利用者向け文書における標準的な CLI 配布物**、(2) 旧称としてのルールとセンサーの**制御ループ**、(3) `harness/<name>/` のソース側ディレクトリ、(4) `tests/harness/` のテスト補助ディレクトリ。利用者向け文書で単に「ハーネス」と呼ぶ場合は 1 の意味です。 |
| **フック（Hook）** | イベントに応じて Claude Code が自動実行する TypeScript スクリプトです。この実装では 13 個のフックを `settings.json` にプロジェクト全体向けとして登録します。ワークフローの背骨（監査記録、センサー起動、実行時グラフ生成、ステータスライン同期、コンパクション時の状態検証、サブエージェント追跡、ターン終了時のループ強制）に加え、セッションライフサイクル、利用者ターンの記録、状態遷移とレビュー担当者の読み取り範囲のガード、ステータスラインコマンドを担います。ワークフローが動作していないときは各フックが自ら処理を省略します。 |
| **インライン実行（Inline execution）** | オーケストレーターがエージェントのペルソナを読み込み、会話内で直接ステージを実行する既定の実行方式です。利用者とのリアルタイム対話に対応します。 |
| **インテント（Intent）** | スペースの `intents.json` レジストリ内の 1 行（`{uuid, slug, dirName, scope, repos, status}`）として追跡する作業単位で、専用の**記録ディレクトリ**を `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` に持ちます。`<YYMMDD>` は記録を時系列で並べる短い UTC 日付接頭辞（例: `260624` = 2026-06-24）、`<label>` は依頼内容を短く表したケバブケースの名前です。同日・同名の衝突時は `-2`、`-3`、… を付けます。正規かつ衝突しない識別子はディレクトリ末尾ではなく、レジストリ行に保存する時系列 UUIDv7 です。最初の `/aidlc` でエンジンが最初のインテントを自動作成し、`active-intent` ポインターが現在のものを選びます。関連: **スペース**、**記録ディレクトリ**。 |
| **Kiro（Kiro）** | Kiro ハーネスです。現在提供する AI-DLC ハーネス配布物の 1 つで、`core/` から `dist/kiro/`（CLI）と `dist/kiro-ide/`（IDE）へ生成されます。AI-DLC の方法論は Kiro のエージェントリソース用グロブを通じて `aidlc/spaces/<active-space>/memory/` から読み込まれます。呼び出しには `/aidlc` を使います。詳細は [Kiro IDE での AI-DLC](/guide/harnesses/kiro-ide) と [Kiro CLI での AI-DLC](/guide/harnesses/kiro-cli) を参照してください。 |
| **ナレッジ（Knowledge）** | エージェントがステージ開始時に読み込む参照資料です。`.claude/knowledge/` にフレームワークと同梱される方法論ナレッジと、`aidlc/spaces/<space>/knowledge/` に置いてスペース内の全インテントで共有するチームナレッジの二層で構成します。後者は自由形式で、初期作成時は空です。 |
| **段階選択プロンプト（Ladder prompt）** | ワーキングスケルトンの Bolt の終わりに一度だけ表示し、以後を自律実行するか、Bolt ごとにゲートを設けるかを選ぶプロンプトです。選択結果は自律モードとして記録され、残りのすべての Bolt に適用されます。 |
| **学習ループ（Learning loop）** | ステージ内の修正や気付きを、継続利用できる実践事項とセンサーへ変換する v0.5.0 の仕組みです。オーケストレーターは観察事項を `memory.md` に記録し、承認ゲートで提示します。利用者が残す項目を確認すると、確定した学びを `aidlc/spaces/<active-space>/memory/project.md` に実践事項として書き込むか（ワンクリックで `team.md` へ昇格可能）、新しいセンサーの雛形を作り、次のワークフローから適用します。詳細は [ルールと学習ループ](/guide/rules-and-the-learning-loop) を参照してください。 |
| **ライフサイクル（Lifecycle）** | AI-Driven Development Life Cycle、すなわち AI-DLC 方法論全体です。この方法論を 1 回実行することをワークフローと呼びます。 |
| **マニフェスト（Manifest）** | 各ハーネスの `harness/<name>/manifest.ts` です。**コア**をそのハーネスの**配布物**へ投影する方法（ディレクトリ対応、ルール名変更、手書きファイル、任意の `emit` プラグイン）を定義する宣言的な契約です。新しいハーネスを追加する作業の大半は、1 つのマニフェストを記述することです。 |
| **MCP サーバー（MCP server）** | プロジェクトルートの `.mcp.json` に宣言し、Claude Code セッションへ提供する外部ツールサーバーです。この実装は `context7` と 4 つの AWS サーバー（`aws-mcp`、`aws-pricing`、`aws-iac`、`aws-serverless`）を同梱します。各エージェントはセッションの全 MCP サーバーを継承します。エージェント単位の付与機能はなく、利用を制限する場合は `tools:` 許可リストを特定の `mcp__<server>__<tool>` 識別子に絞ります。資格情報がないサーバーは利用できないだけで、ワークフローを妨げません。詳細は [ハーネス基本要素の対応 — MCP サーバー](/reference/claude-features#mcp-servers) と [はじめに](/guide/getting-started#mcp-servers-optional) を参照してください。 |
| **ステージメモリ（memory.md）** | `<record>/<phase>/<stage>/memory.md` にあるステージごとの観察日誌です。ステージ開始時に自動作成され、オーケストレーターが管理するため、手動では編集しません。解釈、逸脱、トレードオフ、未解決の質問を記録し、承認ゲートで学習ループが入力として読み込みます。 |
| **モブ実行（Mob execution）** | 制限付きラウンドで実行するディスパッチ型メッシュ（`mode: mob`）です。主担当がドラフトを作成し、互いにブラインドな協働者が並行でコントリビューションファイルを書き、主担当が統合します。未解決の判断は人間へ委ねられることがあります。ユーザーストーリー（2.4）が出荷されているモブです。 |
| **マルチリポジトリインテント（Multi-repo intent）** | 複数の同階層コードリポジトリにまたがるインテントです。対象リポジトリは作成時に、`--repos a,b` による明示指定か、ワークスペースルート直下で `.git` を持つ全ディレクトリの自動検出によって取得し、`intents.json` の行に `repos` として保存します。構築フェーズでは各 Git 操作を `--repo <name>` で特定のリポジトリへ固定します。記録済みリポジトリがないインテントは従来の単一リポジトリ方式で、Git はプロジェクトディレクトリで動作します。[成果物リファレンス](/guide/artifacts-reference) を参照してください。 |
| **オーケストレーター（Orchestrator）** | ワークフローの駆動方法全体を指す総称です。次の動作を決める決定論的な**エンジン**と、それを実行する**コンダクター**（`SKILL.md`）を組み合わせたものです。呼び出しには `/aidlc` を使います。詳細は [エンジンとスキルシステム](/reference/skill-system) を参照してください。 |
| **パッケージャー（Packager）** | `scripts/package.ts` です。**コア**と各**マニフェスト**から、すべての `dist/<harness>/` **配布物**を再生成します。`bun scripts/package.ts` ですべてを生成し、`--check` は CI でバイト単位の差分検査を実行します。 |
| **並列バッチ（Parallel batch）** | 依存関係が満たされ、互いに依存しない複数の Bolt をまとめたものです。オーケストレーターが並列実行し、バッチ末尾の 1 つの承認ゲートで含まれる全 Bolt をまとめて扱います。 |
| **パイプライン実行（Pipeline execution）** | 宣言した順序でリンクを実行するディスパッチ型チェーン（`mode: pipeline`）です。各リンクは上流のすべての作業を参照でき、最後のリンクが成果物を完成させます。リバースエンジニアリング（2.1）が出荷されているパイプラインです。 |
| **フェーズ（Phase）** | ライフサイクルを構成する 5 つの大区分です。初期化（0）、アイデア創出（1）、構想（2）、構築（3）、運用（4）からなり、各フェーズは 3〜8 個のステージを持ちます（初期化 3、アイデア創出 7、構想 8、構築 7、運用 7）。 |
| **フェーズ境界検証（Phase boundary verification）** | フェーズ遷移時に自動実行する追跡可能性の検査です。下流ステージが処理を積み重ねる前に、欠落したリンク、孤立した成果物、不整合を検出します。 |
| **プレーン（Plane）** | フレームワークが分離する 3 つの関心領域です。ネットワーク設計の用語を借りています。**制御プレーン**はステージ定義・ルール・センサーなど、コンパイル時に解決する実行内容の構造を扱います。**データプレーン**は実際のステージ実行、Bolt、監査情報を扱います。**管理プレーン**は `/aidlc --doctor`、監査照会、`CLAUDE.md` を扱います。利用者向けの導入は [ルールと学習ループ](/guide/rules-and-the-learning-loop)、完全なモデルは `docs/reference/02-plane-architecture.md` を参照してください。 |
| **記録ディレクトリ（Record dir）** | 1 つのインテントの成果物、ステージごとの `memory.md` 日誌、`aidlc-state.md`、`audit/` シャードを保持する専用ディレクトリです。場所は `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` で、文中では `<record>/` と表します。インテントごとに 1 つあり、`active-intent` ポインターが現在のものを選びます。関連: **スペース**、**インテント**。 |
| **復旧用目印（Recovery breadcrumb）** | `PreCompact` フックが書き込む隠しファイル `.aidlc-recovery.md` です。最後に検証したステージと時刻を保持し、コンパクション後の状態破損を検出するために使います。 |
| **レビュー担当者（Reviewer）** | 品質ゲートを担当するエージェントです。要件・ストーリー・モックアップを扱う `aidlc-product-lead-agent`、または技術設計を扱う `aidlc-architecture-reviewer-agent` を指します。ステージが `reviewer:` フィールドを宣言している場合、本文処理が成果物を生成した後に別のサブエージェントとして起動します。主要成果物へ `## Review` 判定（READY / NOT-READY）を追記し、NOT-READY の場合は作成担当を `reviewer_max_iterations`（既定 2）まで再実行します。未解決の指摘は人間の承認ゲートへ提示し、最終判断は必ず人間が行います。詳細は [エージェント](/guide/agents-overview) を参照してください。 |
| **ルール（Rule）** | ワークスペースルートのアクティブスペースのメモリ層（`aidlc/spaces/<active-space>/memory/`）に一度記述し、各ハーネス固有の読み込み機構を通じて対象ステージのコンテキストへ取り込む永続的な行動規則です。適用順は組織 → チーム → プロジェクト → フェーズ → ステージの厳密な加算式 5 層で、上位層を上書きせず、適用対象をすべて追加します。ルールは**制御ループ**の前向き制御側であり、決定論的な検証のためセンサーと組み合わせることがあります。詳細は [ルールと学習ループ](/guide/rules-and-the-learning-loop) を参照してください。 |
| **実行時グラフ（Runtime graph）** | インテントの記録ディレクトリにある、ワークフローごとの `runtime-graph.json` 成果物です。構造上のステージグラフをデータプレーン側に写したもので、承認ゲートごとに監査ログから生成します。実行したステージ、分岐した Bolt、起動したセンサー、`memory.md` の項目数を記録し、`/aidlc --doctor` と学習ループが照会可能な実行状況として読み込みます。 |
| **スコープ（Scope）** | enterprise、feature、mvp、poc、bugfix、refactor、infra、security-patch、workshop の 9 個の名前付き構成の 1 つで、実行するステージと深さを決めます。自由形式のインテントから自動検出することもできます。 |
| **センサー（Sensor）** | `.claude/sensors/` 内のマニフェストで定義する決定論的な検証です（例: `aidlc-linter.md`、`aidlc-type-check.md`）。`PostToolUse` フックがステージ出力への Write / Edit を検知すると起動し、助言用の `SENSOR_*` 監査行を記録します。ワークフローは停止しません。各ステージは `sensors:` フロントマター一覧で起動対象を宣言します。センサーは**制御ループ**のフィードバック側、ルールは前向き制御側です。詳細は [ルールと学習ループ](/guide/rules-and-the-learning-loop) を参照してください。 |
| **テスト戦略（Test strategy）** | テスト量を制御する 3 段階（Minimal、Standard、Comprehensive）の 1 つです。生成するテスト数と種類を決めます。深さとは独立しており、スコープが独自の既定値を宣言しない限り、深さと同じ段階を使います（例: workshop の既定値は Minimal）。詳細は [スコープ、深さ、テスト戦略](/guide/scopes-and-depth#the-3-test-strategy-levels) を参照してください。 |
| **セッション（Session）** | `/aidlc` を実行する 1 回の Claude Code 会話です。再開機構により、1 つのワークフローが複数のセッションにまたがることもあります。 |
| **スキル（Skill）** | YAML フロントマターを持つ Markdown ファイルでスラッシュコマンドを登録する、Claude Code の基本要素です。AI-DLC のオーケストレーターは `/aidlc` スキルとして実装されています。利用者向け文書では、可能な限り「スキル」ではなく「コマンド」を使います。 |
| **スペース（Space）** | `aidlc/spaces/<space>/` にあるチーム単位のワークスペースです。`memory/`、`knowledge/`、インテント記録（`intents/`）を保持します。Git 管理外の `aidlc/active-space` ポインターが現在のスペースを選び、既定値は `default` です。単一チームの利用者は通常 `spaces/default/` だけを使います。関連: **インテント**、**ナレッジ**。 |
| **ステージ（Stage）** | ライフサイクルを構成する 32 個の個別手順の 1 つです。各ステージには主担当エージェント、定義済みの入力と出力、共通のステージ手順があります。番号はフェーズごとに付けます（例: 1.1、2.4、3.5）。 |
| **状態ファイル（State file）** | インテントごとの永続的なワークフロー状態を保持する `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/aidlc-state.md` です。6 種類のチェックボックス（`[ ]` / `[-]` / `[?]` / `[R]` / `[x]` / `[S]`）を使い、ステージ進捗、スコープ、ワークスペース情報、セッション再開情報を追跡します。 |
| **サブエージェント実行（Subagent execution）** | コンダクターがハーネスのディスパッチツールを通じて別のエージェントコンテキストを起動する、ディスパッチ型ハブ（`mode: subagent`）です。コード生成（3.5）は単一エージェントによる集中実行です。プラクティスの発見（2.2）はハブアンドスポークで、主担当のドラフト、互いにブラインドな 3 つの支援コントリビューション、人間へのインタビュー、主担当による統合の順に進みます。 |
| **作業ユニット（Unit of work）** | ステージ 2.7（ユニット生成）で分解する、独立して実装可能な解決策の一部分です。1 個以上の作業ユニットを 1 つの Bolt にまとめ、構築フェーズで実行します。 |
| **ワーキングスケルトン（Walking skeleton）** | 構築フェーズ最初の Bolt です。すべての統合点を通る最小のエンドツーエンド構成で、残りの構築へ進む前に全体像を確認できるよう、必ずゲート付きかつ対話形式で実行します。承認直後に段階選択プロンプトが起動します。 |
| **ユーティリティコマンド（Utility command）** | `/aidlc` に渡す、ワークフローを開始しないフラグです。`--status`、`--doctor`、`--version`、`--stage`、`--phase`、`--scope` などがあり、完全なワークフローを実行せず特定の操作だけを行います。 |
| **ワークフロー（Workflow）** | `/aidlc` の呼び出しからステージ完了までを含む、AI-DLC ライフサイクルの 1 回のエンドツーエンド実行です。feature や bugfix など、特定のタスクにスコープを絞ります。 |
