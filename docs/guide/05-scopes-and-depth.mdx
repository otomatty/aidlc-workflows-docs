---
title: スコープ、深度、テスト戦略
description: 実行するステージ、成果物の詳細度、生成するテストの量を、スコープ・深度・テスト戦略がどう制御するかを説明します。
---
スコープは **どのステージを実行するか** を制御します。深度は各ステージが **どれだけ詳細な成果物を出すか** を制御します。テスト戦略は **どれだけ多くのテストを生成するか** を制御します。これら 3 つが組み合わさることで、企業向けの包括的な機能開発から素早いバグ修正まで、タスクに合わせてライフサイクルを適応させます。

---

<a id="the-9-core-scopes"></a>
## 9 つのコアスコープ

コアは名前を持つ 9 つのスコープを同梱しています。各スコープは、実行するステージの集合と既定の深度レベルを定義します。プラグインのインストールでスコープを追加でき、`bun .claude/tools/aidlc-utility.ts select-plugins <names>` を使えば、どのプラグインスコープを可視にするかをインストール単位で絞り込めます。`plugins` の選択でコアが無効化されている場合（`aidlc` が省略されている場合）、コアのスコープファイルはインストールされたまま残りますが、コアが再有効化されるまで有効なランタイムスコープにはなりません。初期化ステージは、有効化されているすべてのスコープで引き続き実行されます。

<a id="enterprise"></a>
### enterprise

**使用場面:** 完全な監査証跡、コンプライアンスレビュー、本番品質の運用が必要な、規制対象の企業向け機能を構築するとき。

- **ステージ:** 全 32
- **既定の深度:** Comprehensive
- **含まれるもの:** コンプライアンス、セキュリティ、運用の全ステージ

<a id="feature"></a>
### feature

**使用場面:** 任意のサイズの新機能を構築するとき。AI-DLC がより具体的な一致を判断できない場合の既定スコープです。

- **ステージ:** 全 32
- **既定の深度:** Standard
- **含まれるもの:** すべてのステージ、標準的な成果物の詳細度

<a id="mvp"></a>
### mvp

**使用場面:** 新規開発（グリーンフィールド）で実用最小限の製品を構築するとき。後半の運用ステージは省略しつつ、設計とコンストラクションは完全に保持します。

- **ステージ:** 32 のうち 22
- **既定の深度:** Standard
- **省略:** 7 つすべての運用ステージ（デプロイパイプライン、環境のプロビジョニング、デプロイの実行、オブザーバビリティ、インシデント対応、パフォーマンス検証、フィードバック）に加え、アイデア創出の市場調査、チーム編成、承認と引き継ぎ（10 省略、22 実行）

<a id="poc"></a>
### poc

**使用場面:** 実現可能性を素早く証明したいとき。アイデア創出とインセプションの多くを省略し、素早くコードに到達することを重視します。

- **ステージ:** 32 のうち 8
- **既定の深度:** Minimal
- **省略:** 市場調査、実現可能性、チーム編成、モックアップ、ユーザーストーリー、および大半の運用ステージ

<a id="bugfix"></a>
### bugfix

**使用場面:** 特定のバグを修正するとき。インテントの取り込みからコード生成とテストまで、簡略化された経路を取ります。

- **ステージ:** 32 のうち 7
- **既定の深度:** Minimal
- **省略:** 市場調査、実現可能性、チーム編成、モックアップ、大半の設計・アーキテクチャステージ、すべての運用ステージ

<a id="refactor"></a>
### refactor

**使用場面:** 機能は変えずに既存コードを整理・再構成するとき。

- **ステージ:** 32 のうち 8
- **既定の深度:** Minimal
- **省略:** bugfix に近く、コード分析、設計、実装に集中

<a id="infra"></a>
### infra

**使用場面:** インフラの変更（新しい環境、CDK/CloudFormation の更新、コスト最適化）を行うとき。

- **ステージ:** 32 のうち 13
- **既定の深度:** Standard
- **省略:** ユーザー向けのステージ（ストーリー、モックアップ、ユーザーフロー）。アーキテクチャ、インフラ、デプロイに集中

<a id="security-patch"></a>
### security-patch

**使用場面:** CVE やセキュリティ脆弱性に対応するとき。セキュリティに関係するステージに絞った高速な経路を通ります。

- **ステージ:** 32 のうち 10
- **既定の深度:** Minimal
- **省略:** 市場調査、チーム編成、モックアップ、セキュリティに関係しない設計ステージ

<a id="workshop"></a>
### workshop

**使用場面:** AI-DLC のワークショップやトレーニングセッションを実施するとき。プロジェクトはファシリテーターにより事前に決められ、参加者はモブとしてインセプション、コンストラクション、運用を進めます。

- **ステージ:** 32 のうち 25
- **既定の深度:** Standard
- **既定のテスト戦略:** Minimal（ナイキストモデル）。ワークショップのペースを速く保つためです
- **省略:** すべてのアイデア創出ステージ（1.1-1.7）。プロジェクトのスコープは事前に決まっています

複数開発者向けの手動レシピと担当宣言（claim）の仕組みは [ワークショップモード](/guide/workshop-mode) を参照してください。

<a id="scope-routing-table"></a>
## スコープのルーティング表 (Scope Routing Table)

権威あるデータは `.claude/scopes/aidlc-<name>.md` ファイル（スコープの識別情報）とプラグインのスコープファイル、そして各ステージの `scopes:` フロントマター（所属）にあり、それが `.claude/tools/data/scope-grid.json` へコンパイルされます。コンパイル済みグリッドには、現在のプラグイン選択で有効化されているスコープのみが含まれます。コンパイル済みの実際の表は `bun .claude/tools/aidlc-utility.ts scope-table`、利用者向けの 1 行説明は `bun .claude/tools/aidlc-utility.ts help` で確認できます。

| スコープ | EXECUTE / 合計 | 深度 | テスト戦略 | 用途 |
|-------|-----------------|-------|---------------|----------|
| `enterprise` | 32 / 32 | Comprehensive | Comprehensive | 規制対象の企業向け機能、完全な監査証跡 |
| `feature` | 32 / 32 | Standard | Standard | 新機能の既定 |
| `mvp` | 22 / 32 | Standard | Standard | グリーンフィールド、後半の運用を省略 |
| `poc` | 8 / 32 | Minimal | Minimal | 実現可能性を素早く証明 |
| `bugfix` | 7 / 32 | Minimal | Minimal | 特定のバグを修正 |
| `refactor` | 8 / 32 | Minimal | Minimal | 既存コードの整理 |
| `infra` | 13 / 32 | Standard | Standard | インフラの変更 |
| `security-patch` | 10 / 32 | Minimal | Minimal | CVE 対応 |
| `workshop` | 25 / 32 | Standard | **Minimal** | AI-DLC のワークショップまたはトレーニングセッション |
| （自動検出） | 可変 | 可変 | 可変 | 自由記述のインテントから AI が決定 |

スコープ間のセレモニー（手続きの重さ）の差は桁違いです。`poc` は 8 ステージ・5 つの承認ゲートで動きますが、`feature` は 32 ステージすべてと 29 のゲートを持ち、コンストラクションでは 5 つの設計ステージが作業単位ごとにファンアウトします。そのためスコープ確認の行では、ワークフロー開始前に何へ同意しているのかが正確に分かるよう、コンパイル済みグリッドから得た正確な数値、つまりステージ数、承認ゲート数、作業単位ごとのファンアウトが必ず表示されます。推定値は使いません。

> **プロジェクトごとの既定スコープ:** チームは `.claude/settings.json` に `AWS_AIDLC_DEFAULT_SCOPE` を設定することで、プロジェクトの既定スコープを事前に固定できます。ワークショップのように、参加者全員がフラグを覚えなくても `workshop` で始まってほしい場合に便利です。詳しくは [カスタマイズ § プロジェクトごとの既定スコープ](/guide/customization#per-project-default-scope) を参照してください。

<a id="auto-detection-from-freeform-intent"></a>
## 自由記述のインテントからの自動検出

スコープを明示的に指定する必要はありません。何をしたいかを説明すれば、オーケストレーターがキーワードから適切なスコープを検出します。

```
/aidlc Build a REST API for inventory management
```

エンジンは、あなたのインテントをキーワードパターンと照合して解析します。

| キーワード | 検出されるスコープ |
|----------|---------------|
| "fix", "bug", "broken" | `bugfix` |
| "refactor", "clean up", "simplify" | `refactor` |
| "infrastructure", "deploy", "infra" | `infra` |
| "security", "CVE", "vulnerability", "patch" | `security-patch` |
| "proof of concept", "prototype", "poc", "spike" | `poc` |
| "mvp", "minimum viable" | `mvp` |
| "workshop", "lab", "training" | `workshop` |
| それ以外すべて | コアが有効な場合は `feature`。それ以外の場合は、曖昧でないときに限り、唯一有効なプラグインの最初のスコープ |

**曖昧さの解消ルール:** 入力にスコープキーワードと、5 語を超えるプロジェクト説明の両方が含まれている場合、その一致は偶発的なものとみなされ、代わりにコンポーズ提案が発火します。これにより、たとえば "Fix the infrastructure monitoring dashboard" のような入力が、より適切に仕立てた計画を出すべき場面で `infra` に機械的に振り分けられるのを防ぎます。

明確なキーワード一致の後には、一致したスコープと、それが伴うセレモニーの規模が、コンパイル済みグリッドから引いた 1 行の確認として表示されます。

```
Starting a "bugfix" workflow for: "fix login bug" - 7 of 32 stages, 4 approval gates, 1 stage repeats per unit of work in Construction. Confirm to proceed,
name a different scope, or say "compose" for a tailored plan.
```

そのまま続行するには承認し、開始前に軌道修正したければ、別のスコープ（または `compose`）で返答してください。

<a id="the-adaptive-composer"></a>
## 適応型コンポーザー (The Adaptive Composer)

標準のスコープが明確に合わない場合（長い散文、キーワード一致なし、または長い説明の中にキーワードが埋もれている場合）、`/aidlc` は無言で `feature` を既定にせず、仕立てた計画を **コンポーズ（COMPOSE）** する提案を出します。これを強制することもできます。

```
/aidlc compose "harden the deployment pipeline and add observability"
/aidlc-compose "same thing, as a typeable shortcut"
/aidlc compose --report sonar.json     # compose from a scan report
/aidlc --new-scope "..."               # force a custom scope even on a stock match
```

コンポーザーエージェントは、タスクとワークスペース走査の結果（ブラウンフィールド / グリーンフィールド、使用言語）を読み、適切な EXECUTE/SKIP グリッドを、スキップする各ステージの理由付きで提案します。あなたはゲートでそれを承認、編集、却下できます。明示的な承認があるまでは何も書き込まれず、ワークフローも開始されません。承認すると次のいずれかになります。

- 提案が標準のスコープと一致していた場合、そのスコープでワークフローが直接誕生します（コードレベルの検出結果を多く含むスキャンレポートは、よく `bugfix` や `security-patch` に振り分けられます）
- カスタムグリッドの場合、コンポーザーは実際のスコープ（`scopes/aidlc-<name>.md` と `scope-grid.json` のエントリ）を著述し、同じターンの中でそのスコープでワークフローを誕生させます。コンポーズされたスコープは以後、標準のスコープと同じように解決され（`/aidlc --scope <name>`）、グラフの再コンパイル後も保持されます。`aidlc-graph.ts compile` は、コンポーズ済みのグリッドエントリを再生成後の `scope-grid.json` に折り込み、ステージのフロントマターだけからグリッドを再構築し直すわけではありません

**キーワードの健全性:** コンポーズされたスコープは `keywords: []` で出荷されるため、1 回限りの計画が将来のキーワード自動検出に参加することはありません。将来のプロンプトにも反応させるかどうかは、ゲートで明示的に確認される質問であり、副作用ではありません。

**実行中の再構成 (In-flight recompose):** ワークフローの途中でも、`/aidlc compose` は実行中ワークフローの PENDING ステージを組み替える提案を出せます。不要になったものをスキップし、やはり必要だと気づいた保留中のステージを追加し直す形です。切り替えは保留中かつカーソルより先のステージにのみ適用され、完了済みと実行中のステージは凍結されます。残るステージが必要な入力を失わないよう検証され、監査ロックの下で決定論的な `recompose` 動詞により適用され、`RECOMPOSED` 監査イベントが記録されます。コンストラクションの最初の EXECUTE ステージ（ウォーキングスケルトンのゲートの起点）は切り替えられません。

文字どおりの動詞を使う必要はありません。ワークフローの途中で "can we skip market research? we already know this market" のように会話すると、それも再編成の要求として認識され、同じゲートと同じ `recompose` 動詞に振り分けられます。ステージ名を自分で挙げた場合（"drop market-research and team-formation" など）は、コンダクターがコンポーザーエージェントをディスパッチせずに直接ゲートを提示することもあります。その場合でも承認ゲートと検証は同一です。Kiro と Codex では、確実で文書化された経路として、文字どおりの `/aidlc compose "<request>"` を使うのが推奨です。

<a id="the-3-depth-levels"></a>
## 3 つの深度レベル

深度は、各ステージが生成する成果物の詳細度を制御します。スコープが既定の深度を定めますが、上書きできます。

| 深度 | 成果物の詳細度 | 使用場面 |
|-------|----------------|-------------|
| **Minimal** | 必須の核となる内容のみ。短い文書、主要な意思決定、最小限の補足分析。 | 素早い修正、パッチ、概念実証 |
| **Standard** | バランスのよい詳細度。完全な要件、根拠付きのアーキテクチャ決定、十分なテスト計画。 | ほとんどの機能開発と MVP |
| **Comprehensive** | 企業向けの完全な詳細。網羅的な要件、コンプライアンスマトリクス、詳細な NFR 仕様、完全な監査ドキュメント。 | 規制対象の機能、企業向けデプロイ |

<a id="how-depth-affects-stages"></a>
### 深度がステージに与える影響

各ステージで、エージェントは有効な深度に応じて出力を調整します。

- **Minimal:** 1〜2 ページの成果物、主要な意思決定のみ、任意セクションは省略
- **Standard:** 完全な成果物、必須セクションをすべて含み、根拠は簡潔
- **Comprehensive:** 拡張された成果物、任意セクションも含み、詳細な根拠付けとコンプライアンスの相互参照を付ける

<a id="overriding-depth"></a>
### 深度を上書きする

深度は 3 つの地点で変更できます。

1. **`--depth` CLI フラグを使う**。呼び出し時に上書きします:
   ```
   /aidlc --depth comprehensive
   /aidlc --scope bugfix --depth standard
   /aidlc --stage code-generation --depth minimal
   ```
2. **スコープ確認時**。検出されたスコープをオーケストレーターが確認するとき、単に承認する代わりに `--depth <level>` と返答します
3. **任意の承認ゲート**。フィードバックの一部として別の深度レベルを要求します

各セッションの最初の完了メッセージでは、次のことが再確認されます。

```
**Project depth**: Standard — depth adapts artifact detail.
**Test strategy**: Standard — test strategy controls test volume.
You can request different depth or test strategy at any approval gate.
```

<a id="specifying-scope-directly"></a>
## スコープを直接指定する

<a id="explicit-scope"></a>
### スコープを明示する

```
/aidlc feature
/aidlc bugfix
/aidlc enterprise
```

<a id="scope-with-description"></a>
### スコープと説明を一緒に指定する

```
/aidlc bugfix Fix the login timeout issue
/aidlc poc Build a quick prototype for the search feature
```

<a id="override-scope-with-utility-command"></a>
### ユーティリティコマンドでスコープを上書きする

```
/aidlc --scope bugfix
/aidlc --scope enterprise --stage code-generation
```

`--scope` フラグは、ジャンプ操作のために `--stage`、`--phase`、`--depth` と組み合わせて使えます。

<a id="override-depth"></a>
### 深度を上書きする

```
/aidlc --depth minimal
/aidlc --scope bugfix --depth comprehensive
/aidlc --scope enterprise --depth standard --stage code-generation
```

`--depth` フラグは、スコープの既定の深度レベルを上書きします。有効な値は `minimal`、`standard`、`comprehensive` です（大文字小文字は区別しません）。

<a id="override-test-strategy"></a>
### テスト戦略を上書きする

```
/aidlc --test-strategy minimal
/aidlc --depth standard --test-strategy minimal
```

`--test-strategy` フラグは、深度とは独立にテスト戦略を上書きします。完全な説明は下の [3 つのテスト戦略レベル](#the-3-test-strategy-levels) を参照してください。

<a id="the-3-test-strategy-levels"></a>
## 3 つのテスト戦略レベル

テスト戦略は **どれだけ多くのテストを生成するか** と **どのテスト種別を含めるか** を制御します。これは深度とは独立です。深度が制御するのは成果物の詳細度（文書、図、質問）であり、テスト戦略が制御するのはテストの量のみです。この分離により、速度を優先したいときに、Standard の深度の完全なワークフローを Minimal のテストで実行できます。

<a id="minimal-nyquist-model"></a>
### 最小 (Minimal) — ナイキストモデル

信号処理のナイキストレート、すなわち信号を再構成するのに必要な最小のサンプリング周波数に着想を得ています。Minimal のテスト戦略は、すべての要件を検証するために必要最小限のテストを生成します。多くも少なくもありません。

- **特定された要件ごとに 1 テスト**（コンポーネント駆動ではなく要件駆動）
- **ハッピーパスの下限:** 対応する要件がなくても、各コンポーネントに最低 1 つはハッピーパスの単体テストを付ける
- **単体テストのみ**。統合テスト、E2E テスト、パフォーマンステスト、セキュリティテストは省略
- 一般的なプロジェクトで **合計約 5〜15 テスト**
- 厳格なルールではなく緩やかな指針。セーフティクリティカルな文脈であれば、エージェントはこれを超えてもよい

**適した場面:** ワークショップ、トレーニングセッション、概念実証、素早いバグ修正。つまり、完全なテストスイートに投資するよりも、正しさの確認を素早く済ませたい文脈です。

<a id="standard-per-component-model"></a>
### 標準 (Standard) — コンポーネント単位モデル

コンポーネント間の境界を検証する、バランスの取れたテストカバレッジです。

- **コンポーネントごとに 5〜8 テスト**
- **単体テスト + 統合テスト**（コンポーネント間の主要な境界）
- E2E、パフォーマンス、セキュリティのテストは、NFR 要件が明示的に要求する場合のみ含める
- **テストピラミッドの比率:** 約 75% 単体 / 20% 統合 / 5% E2E
- 緩やかな指針

**適した場面:** ほとんどの機能開発と MVP。テストに過剰投資せず、十分なカバレッジを得たい場合です。

<a id="comprehensive-full-coverage-model"></a>
### 包括的 (Comprehensive) — フルカバレッジモデル

すべてのテスト種別をまたぐ、徹底したテストカバレッジです。

- **コンポーネントごとに 10〜15 テスト**
- **すべてのテスト種別:** 単体 + 統合 + E2E + パフォーマンス（NFR がある場合）+ セキュリティ（NFR がある場合）
- **テストピラミッドの比率** は全種別を横断して適用
- 緩やかな指針

**適した場面:** 企業向けの機能、規制対象のシステム、またはテストカバレッジの監査証跡が必要なあらゆる文脈です。

<a id="how-test-strategy-defaults-work"></a>
### テスト戦略の既定値の仕組み

ほとんどのスコープでは、テスト戦略は **深度レベル** と同じ値になります。深度が Standard なら、テスト戦略も Standard です。ただし、一部のスコープは独自の既定値を宣言します。

| スコープ | 深度 | テスト戦略 | 異なる理由 |
|-------|-------|---------------|----------------|
| `workshop` | Standard | **Minimal** | 学習のために成果物は完全にしつつ、ペースを維持するため、テストは高速なナイキストモデルにする |

その他すべてのスコープでは、テスト戦略は深度を継承します。必要なら常に `--test-strategy` で上書きできます。

<a id="overriding-test-strategy"></a>
### テスト戦略を上書きする

テスト戦略は 3 つの地点で変更できます。

1. **`--test-strategy` CLI フラグを使う**。呼び出し時に上書きします:
   ```
   /aidlc --test-strategy minimal
   /aidlc --depth standard --test-strategy minimal
   /aidlc --scope bugfix --test-strategy comprehensive
   ```
2. **ワークフローの途中**。実行中のワークフローのテスト戦略を変更します:
   ```
   /aidlc --test-strategy comprehensive
   ```
3. **任意の承認ゲート**。フィードバックの一部として別のテスト戦略を要求します

<a id="common-depth-test-strategy-combinations"></a>
### よくある深度とテスト戦略の組み合わせ

| 深度 | テスト戦略 | 効果 | 使用場面 |
|-------|--------------|--------|-------------|
| Standard | Standard | 完全な成果物、バランスの取れたテスト | ほとんどの機能開発（既定） |
| Standard | Minimal | 完全な成果物、ナイキストモデルのテスト | ワークショップ、時間制限のあるセッション |
| Minimal | Minimal | 軽量な成果物、軽量なテスト | 素早いバグ修正、パッチ |
| Comprehensive | Comprehensive | すべて完全 | 規制対象の企業向け機能 |
| Comprehensive | Standard | 完全な成果物、バランスの取れたテスト | 現実的なテスト方針を取る企業向け開発 |
| Minimal | Comprehensive | 軽量な成果物、徹底したテスト | 確信が必要なクリティカルなバグ修正 |

<a id="choosing-the-right-scope"></a>
## 適切なスコープを選ぶ

| 状況 | 推奨スコープ |
|-----------|------------------|
| 本番アプリケーション向けの新機能 | `feature` |
| グリーンフィールドの製品をゼロから作る | `mvp` または `feature` |
| アプローチの素早い検証 | `poc` |
| 既知のバグを修正 | `bugfix` |
| 振る舞いを変えないコード整理 | `refactor` |
| 新しい AWS 環境や CDK の変更 | `infra` |
| CVE またはセキュリティ脆弱性への対応 | `security-patch` |
| コンプライアンスが必要な規制対象の機能 | `enterprise` |
| AI-DLC のワークショップまたはトレーニングラボ | `workshop` |

迷ったら `feature` から始めてください。32 のステージすべてを含み、各承認ゲートで個別のステージをスキップできます。
