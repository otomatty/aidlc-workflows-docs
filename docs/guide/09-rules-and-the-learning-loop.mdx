ルールは、エージェントがあなたのプロジェクトでどのように働くかを形作る、永続的な文章による指示です。学習ループは、そのルールが書かれる仕組みです。ワークフロー中にエージェントを修正し、その修正をステージの承認ゲートで確認すると、フレームワークがそれを保存するので、同じ修正を二度行う必要がなくなります。

この章は利用者向けの概説です。ルールがどこにあるか、5 つの層がどう積み上がるか、学習ループがどのように修正を取り込むか、そしてセンサーがルールに並ぶ決定論的なセカンドオピニオンをどう提供するかを扱います。スキーマレベルの仕組みについては、それぞれの箇所で開発者リファレンスへのリンクを示しています。

<a id="rules-at-a-glance"></a>
## ルールの概要

ルールは、アクティブスペースのメモリ層の Markdown ファイル、つまり `aidlc/spaces/<active-space>/memory/` にあります。これはワークスペースルートにある手で編集可能な単一セットで、各ハーネスがそれぞれのネイティブな取り込み機構（Claude の `@` インポートスタブ、Kiro のリソースグロブ、Codex の `AIDLC_RULES_DIR`）を通じて読み込みます。各ファイルは、そのスコープに応じて名前が付いています。

```
aidlc/spaces/<active-space>/memory/
├── org.md                 # framework + organization-wide defaults
├── team.md                # your team's affirmed practices
├── project.md             # this project's specialization
└── phases/
    ├── ideation.md
    ├── inception.md
    ├── construction.md
    └── operation.md
```

ファイルの中に `scope:` フィールドはありません。スコープはファイル名から決まります。`org.md` はフレームワークの既定値（トランクベース開発、テスト方針、ウォーキングスケルトン方針）を持ちます。`team.md` にはチームが確認済みのプラクティスが入ります。`project.md` にはこの 1 つのプロジェクトに固有の内容が入ります。4 つの `phases/<phase>.md` ファイルは、そのフェーズのすべてのステージに適用されるルールを持ちます。たとえばインセプション（inception）フェーズのルールでは、すべてのアーキテクチャ決定に少なくとも 2 つの代替案を記録することを要求します。

各ファイルは、話題ごとの見出し（`## Way of Working`、`## Testing Posture`、`## Deployment`、`## Code Style` など）の下に置かれた平易な文章です。あなたが読めて、手で編集でき、フレームワークは学習ループを通じてそこへ書き込みます。

スキーマ、ファイル名とスコープの対応表、解決器の仕組みは、開発者リファレンスの [ルールシステム](/reference/rule-system) に記載されています。

<a id="the-five-layer-chain"></a>
## 5 層チェーン

ルールは、すべてのワークフローの開始時に 5 層のチェーンを通じて解決されます。

```
org → team → project → phase → stage
```

このモデルは **厳密な加算型（strict-additive）** です。適用可能なすべてのルールがエージェントのコンテキストに現れます。実行時に何かが黙って落ちたり上書きされたりすることはありません。組織の既定値、チームのプラクティス、プロジェクトの特化内容が連結され、ステージがすでに自分のフェーズを宣言しているので、対応するフェーズルールがそのまま付加されます。（5 つ目の層であるステージ単位のルールは、将来のリリース向けに予約されています。）

これは以前のバージョンからの意図的な変更です。`overrides:` ブロックも `enforcement:` キーワードも、もはやありません。適用されるすべての層が同時に存在し、競合は実行時に届く前、つまりルールが書き込まれる時点で捕捉されます（後述の [書き込み時の競合検査](#admission-time-conflict-checks) を参照）。

このチェーンはワークフロー開始時に、フレームワークがステージ定義、ルール、センサーを 1 つのグラフへコンパイルするときに **一度だけ** 解決されます。ワークフロー全体を通じてエージェントが読むのは解決済みのビューであり、実行中にチェーンをたどり直すことはありません。このコンパイル境界は、プレーンモデルが説明しているものと同じです。章末の [プレーン: 全体への当てはめ方](#planes-how-it-fits-together) を参照してください。

<a id="the-learning-loop"></a>
## 学習ループ

学習ループは、一度きりの修正を永続的なルールに変える仕組みです。ほとんどのステージ実行では何も追加されません。そしてそれは健全です。このループが動くのは、ステージ中に何かが表面化し、それを保持する価値があるとあなたが判断したときだけです。

このループには、ユーザーから見える 4 つの瞬間があります。ステージ実行中にエージェントが日誌を付け、ゲートが候補を提示し、あなたが残すものを確認し、フレームワークが保持された項目を次回用に書き込みます。

<a id="the-memorymd-diary"></a>
### `memory.md` の日誌

ステージ実行中、フレームワークは `<record>/<phase>/<stage>/memory.md` に観察ログを継続的に残します。これはインテントの記録ディレクトリ、すなわち `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` の下です。ステージ開始時に自動作成され、手で編集してはいけません。記入項目は 4 つの標準見出しの下に入ります。

- **解釈（Interpretations）** — ステージの文章が曖昧だったためにエージェントが行った解釈
- **逸脱（Deviations）** — エージェントが意図的にステージの文章から外れた箇所と、その理由
- **トレードオフ（Tradeoffs）** — 検討した選択肢と、その選択理由
- **未解決の質問（Open questions）** — 次回実行前に確認すべきこと、またはまだ不確かな文脈

各記入項目にはタイムスタンプが付きます。このループの中で言語モデルが担う仕事はこれだけです。すなわち、ステージの間に観察内容を `memory.md` へ書くことです。ステージ完了後の処理、つまり記入項目の集計、その提示、適切なファイルへの振り分け、実際の書き込みは、すべて決定論的なツールか、ゲートにおけるあなたの明示的な選択によって行われます。

<a id="the-gate-ritual"></a>
### ゲートの儀式

各承認ゲートの前に、フレームワークは学習ゲートを実行します（プロトコルでは §13 の儀式と呼ばれます）。2 つの情報源から候補を集め、1 つの確認リストとして提示します。

1. **エージェントの日誌を、そのまま提示する。** 決定論的なツールが `memory.md` を読み、4 つの見出しの下にある空でない各行を、出典の見出しを添えて候補として出力します。言い換えも、「重要そうなもの」への絞り込みもありません。書かれた行がそのまま見えます。
2. **必ず「次回のために追加することはありますか？」（"Anything to add for next time?"）と尋ねる自由記述チャネル。** あなたは観察内容を入力し、それが 4 つの見出しのどれに属するかを選びます。分類として求められるのは、その見出しの選択だけです。

残したい候補にチェックを付けます。もしそのステージの `memory.md` が空なら、フレームワークは日誌が付けられたかどうかの証明をあなたに求めません。静かにその事実を記録して先へ進みます。

<a id="where-a-kept-learning-goes"></a>
### 保持された learning はどこへ行くか

あなたがファイルパスを選ぶことはありません。書き込み先は見出しによって決まります。

- 解釈、逸脱、トレードオフはプラクティスとして `aidlc/spaces/<active-space>/memory/project.md` に入ります（確認された学びは *プラクティスそのもの* です）。話題ごとの見出しの下に配置されます。
- 未解決の質問は昇格しません。これらは調査項目であり、導入すべきルールではないからです。

既定のスコープは **project** です。教訓が 1 つのプロジェクトを超えて当てはまる場合には、「チームへ昇格（promote to team）」のワンクリック操作で、保持した学びを `memory/project.md` から `memory/team.md` へ広げられます。組織（org）まで広げる経路はありません。組織ルールはフレームワークが出荷するか、組織側で別のプロセスにより著述されるので、学習ループが org スコープへ書くことはありません。最も狭いスコープを既定にすることで、1 つのプロジェクトでの驚きが誤って組織全体のルールにならないようにしています。

確認された学びは *プラクティスそのもの* です。`aidlc/spaces/<active-space>/memory/project.md` や `team.md` といった、プラクティス発見（practices-discovery）が確認するのと同じアクティブスペースのメモリファイルに入ります。別個の継続的な `*-learnings.md` という置き場はありません。これら 2 つの経路が異なるのはライフサイクルだけです。プラクティス発見はセクション全体を決定論的に確認しますが、学習ループはゲートを通じて、日付と話題見出しの付いた記入項目を 1 つずつ追記します。

保持された学びが **センサーの結び付け** でありルールではない場合、つまりステージ出力に対して新しい決定論的検査を走らせたい場合、フレームワークは 2 つのファイル書き込みを原子的に行います。センサーマニフェストの雛形を作り、新しいセンサーの id を発生元ステージのインポート一覧に追記します。日誌、ゲートでの確認、そこから生じるファイル書き込みはそれぞれ監査行（`RULE_LEARNED` または `SENSOR_PROPOSED`）を残すため、どんなルールも黙って導入されることはありません。

<a id="admission-time-conflict-checks"></a>
### 書き込み時の競合検査（admission-time conflict checks）

保持された学びがディスクに書かれる前に、フレームワークは `memory/org.md` に対してセクション単位の検査を行います。提案された記入項目が同じ見出し下の組織ルールと矛盾していれば、ゲートは止まり、衝突する組織ルールの文をその場で引用します。あなたは、その項目を修正するか、スキップするか、組織ルールの管理者へエスカレーションするかを選びます。矛盾したルールがそのまま書き込まれることはありません。そのため実行時の解決器は単純なままでよく、競合検査を通過したルールだけを見れば済みます。

同じセクション単位の検査は、プラクティス発見の確認ゲートも守ります。そして組織の方針が **後から** 変わり、すでにディスクにある team または project のルールと衝突した場合は、`/aidlc --doctor` がそのずれを要求に応じて通知します。どのファイルの、どのセクションが、どの組織ルールの文と衝突しているかを名指しするので、チームは対応できます。この doctor の検査は助言的であり、ブロックはしません。doctor の 2 つの助言行については [CLI コマンド](/guide/cli-commands) と [トラブルシューティング](/guide/troubleshooting) で説明しています。

<a id="applies-next-workflow-not-mid-run"></a>
### 今の実行中ではなく、次のワークフローに適用される

あるゲートで記録された学びは、現在のワークフローの残りのルールを **変更しません**。今回の実行については、あなたはすでに会話の中でエージェントを修正しています。ルールは次回のためのものです。新しい行はディスクにありますが、進行中のワークフローは開始時にコンパイルしたビューを使い続けます。

次にワークフローを開始するとき、そのコンパイルが新しいファイルを読み、最初のステージからそのルールが適用されます。これはルーターが BGP から得る安定性と同じ性質です。経路はパケットが飛んでいる途中で再計算されず、AI-DLC もワークフローの途中で再コンパイルしません。その見返りが予測可能性です。ワークフローの前半であなたが承認したゲートは、安定したルールの集合に対して行われたものであり、フレームワークは進行中の実行の足元を後から変えません。

<a id="worked-example-the-anz-banking-project"></a>
## 実例: ANZ の銀行プロジェクト

具体的な例で、ループが端から端までどう動くかを見てみましょう。

Sam が ANZ の銀行プロジェクトで `/aidlc feature` を実行します。ワークフローは `requirements-analysis` に到達します。Sam は関係者向けメモに「the transaction shouldn't duplicate on retry」（リトライ時にトランザクションが重複してはならない）と書きました。意味していたのは銀行取引、つまり処理対象の支払いです。ところがプロダクトエージェントは "transaction" をデータベーストランザクションと読み取り、ACID セマンティクスの要件と解釈しました。Sam がそれを修正し、エージェントは成果物を更新して先へ進みます。

フレームワークはこれを予測していませんでした。ANZ 固有の用語についてのルールもなく、一般用語とドメイン用語の衝突を捉えるセンサーもありません。エージェントが単純に曖昧さにぶつかったのです。

**1. 日誌に記録される。** フレームワークは `<record>/inception/requirements-analysis/memory.md` の `## Interpretations` の下に次の記入項目を追記します。

```
- 2026-05-21T09:14:32Z — Stakeholder note used "transaction"; interpreted as
  database-transaction. Sam corrected to mean banking-transaction (the payment
  being processed, not the DB write). Worth flagging for ANZ project context.
```

この時点ではルールはまだ導入されていません。これは単なるエージェントの日誌です。

**2. ゲートがそれを提示する。** ステージが完了します。承認ゲートの前に、学習ゲートが `memory.md` を読み、この解釈の行を候補として表示します。さらに「次回のために追加することはありますか？」（"Anything to add for next time?"）と尋ねます。Sam はトランザクション用語の候補にチェックを入れ（解釈 → `memory/project.md` に入ります）、さらに「エージェントが AWS アカウントの用語を既定で使い続けたため、銀行の顧客エンティティは常に『ANZ customer』と呼ぶべき」という自由記述のメモを追加します。Sam はこの追加に「逸脱（Deviations）」の見出しを選びます。これも同じ `memory/project.md` ファイルへ送られます。

**3. 競合検査が走る。** フレームワークは両方の記入項目を組織のプラクティス（`memory/org.md`）と比較します。"ANZ transaction" も "ANZ customer" という用語も組織ルールではカバーされていないので、どちらも通過します。決定論的なツールが出所情報付きで両方の行を `memory/project.md` に書き込み、監査ログにはそれぞれ `RULE_LEARNED` イベントが記録されます。

**4. 現在のワークフローはそのまま続く。** ステージが承認され、ワークフローは `user-stories` へ進みます。新しい行はディスクにありますが、このワークフローのコンパイル済みビューには入りません。今回の実行については、Sam がすでにステージ内でエージェントを修正済みだからです。

**5. 次のワークフローで拾われる。** その日の後で Sam が `/aidlc bugfix` を実行します。ワークフロー開始時のコンパイルがスペースメモリ層をたどり、`memory/project.md` を拾って、すべてのステージのコンテキストに含めます。bugfix ワークフローの最初のステージから、エージェントは "transaction" が支払いを意味し、顧客エンティティが "ANZ customer" であると分かっています。

コストは 1 回だけ支払われました。1 回のゲートでの確認と、1 回のファイル書き込みです。それが、ディレクトリ走査にファイルを 1 つ追加するだけの代価で、以後のすべてのワークフローに効いてきます。

<a id="sensors-the-deterministic-second-opinion"></a>
## センサー: 決定論的なセカンドオピニオン

ルールはエージェントが読む文章です。センサーは、エージェントがステージ出力を書いたときに自動で走る決定論的な検査です。ルールが「ユーザーストーリーは Given/When/Then 形式に従う」と述べるところで、センサーは必要な見出しがファイルに実際に存在するかをバイト単位で検証できます。ルールはステージに向けてエージェントを前へ導き、センサーはエージェントが実際に何を作ったかを後ろ向きに検証して返します。

<a id="how-sensors-fire"></a>
### センサーはどう発火するか

エージェントがステージ中に出力ファイルを書くか編集すると、`PostToolUse` フックが、そのステージに適用されるセンサーを確認し、一致するものをそれぞれ実行します。一致判定はファイルの形状によります。コード品質センサーは `**/*.{ts,js}` を解析すると宣言しているので、TypeScript と JavaScript の書き込みでのみ発火します。あらゆるステージ出力で発火する文書形状センサーは絞り込みを省略します。ワークフロー中にあなたが手でセンサーを呼び出すことはありません。すべての `Write` と `Edit` に伴って走ります。

このリリースにおけるセンサーの結果は **助言的** です。失敗したセンサーは監査行と、何が欠けているかを正確に示す詳細ファイルを生成しますが、ステージの承認ゲートをブロックしたりワークフローを止めたりはしません。シグナルは見えます。どう扱うかはあなたが決めます。

<a id="what-you-see-in-the-audit-log"></a>
### 監査ログで何が見えるか

センサーの活動は、インテントの `audit/` シャードに `Sensor Fired`、`Sensor Passed`、`Sensor Failed` の行として現れます。失敗した行は詳細ファイルへのリンクを持ちます（例: `<record>/.aidlc-sensors/<stage-slug>/required-sections-<timestamp>.md`）。そこには、欠けている見出し、参照されていない上流成果物、リントエラーなど、具体的な不足が列挙されます。監査ログについては [状態管理と監査証跡](/guide/state-and-audit) で扱います。

<a id="the-four-framework-sensors"></a>
### フレームワークに同梱される 4 つのセンサー

フレームワークには 4 つのセンサーが同梱されます。

| センサー | 発火対象 | 検査内容 |
|--------|----------|--------|
| `required-sections` | 記録ディレクトリ内の任意の Markdown 出力 | 出力に必須の H2 見出しが含まれるか（汎用的な内容構造の検査） |
| `upstream-coverage` | 記録ディレクトリ内の任意の Markdown 出力 | ステージの成果物（集合として評価される）が、ステージの `consumes` に宣言された各上流成果物を、スラッグ、ウィキリンク、または生成元ステージのディレクトリパスのいずれかで参照しているか |
| `linter` | `.ts` / `.js` のコード出力 | 設定済みのリンターを実行する（既定は ESLint） |
| `type-check` | `.ts` / `.tsx` のコード出力 | 設定済みの型チェッカーを実行する（既定は `tsc`） |

各ステージは、自分の出力に対してどのセンサーを発火させるかを宣言します。独自のセンサーを追加することもできます。`.claude/sensors/` の下にマニフェストを著述し、その id を実行対象のステージに追加してください。ゲートで確認すれば、学習ループがセンサーを導入してくれることもあります。マニフェストの形式、ステージ別の一覧表、著述の手引きは [センサーシステム](/reference/sensor-system) にあります。自身のプロジェクトへ追加する手順は [カスタマイズ](/guide/customization) を参照してください。

<a id="planes-how-it-fits-together"></a>
## プレーン（Planes）: 全体への当てはめ方

上の内容は、プレーンを意識しなくても使えます。しかし基盤となる設計はネットワーキングから流儀を借りており、それに名前を与えると、「次のワークフローで適用される」という振る舞いが腑に落ちます。

現代的なルーターは仕事を 3 つのプレーンに分けます。AI-DLC も同じ分割を採ります。

- **制御プレーン（Control plane）** — 何を走らせるべきかという *スキーマ*。ステージ定義、ルール、センサーです。ネットワーキングでいえば経路計算です。設定から、何がどこに適用されるかを決めます。制御プレーンは 1 回しか走らないので、遅くても賢くても構いません。
- **データプレーン（Data plane）** — *実際の実行*。ステージの実行、エージェントの呼び出し、インテントの記録ディレクトリにあるファイルです。ネットワーキングでいえばパケット転送です。高速で、繰り返され、参照によって動きます。データプレーンは解決済みの答えを読むだけで、再導出しません。
- **管理プレーン（Management plane）** — *観測と設定* の窓口。`/aidlc --doctor`、監査ログ、`CLAUDE.md` です。ここで設定し、ここで問い合わせます。人間のペースで使うプレーンです。

制御プレーンは、ワークフロー開始時にルールとセンサーを 1 度だけグラフへコンパイルします。データプレーンは、その実行の残りで、事前解決済みの答えをそのグラフから読みます。だからこそワークフローの途中で捉えた学びは、次のコンパイルを待つのです。フレームワークは「トポロジー変更の時点」（ワークフロー開始時）で答えを計算し、「パケットの時点」（各ステージごと）では計算しません。その結果として、再現可能な実行と、再起動後のきれいな復旧が得られます。

利用者としては、たいてい一度に 1 つの水平なスライスだけに触れます。ワークフローを走らせる、学びを捉える、チームのプラクティスをカスタマイズする、起きたことを監査する、といった具合です。各スライスは、その下のプレーンに触れつつも、あなたにそれらを意識させません。完全なモデル、コンパイル境界、復旧特性は [プレーンアーキテクチャ](/reference/plane-architecture) にあり、そこで生じるテレメトリー成果物は [ランタイムグラフ](/reference/runtime-graph) に記載されています。
