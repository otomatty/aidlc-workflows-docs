<a id="1-the-problem"></a>
## 1. 問題

バージョン 0.5.0 では、ルール、センサー、そしてワークフロー実行中の観測を
取り込み、それを次回自動的に発火する場所へ書き込む学習ループが導入されます。
背後のモデルには実際の複雑さがあります。複数種別の
ステージごとの取り込み、ファイル名で結び付く共通既定ルール、
センサーマニフェスト、フックチェーンです。顧客がそれらすべてを
理解しなければフレームワークを使えない状態であってはならず、
またフレームワークもホットパスで検索コストを払うべきではありません。

正しい捉え方は *「設計をより単純にする」* ことではありません。
*「こうしたシステムで既に確立している規律を借りる」* ことです。
現代ネットワーキングの 3 プレーンアーキテクチャは最も近い類比であり、
そこで得られた教訓はほぼそのまま移せます。

<a id="2-the-networking-analog"></a>
## 2. ネットワーキングの類比

現代のルーター（または SDN コントローラー）は、仕事を 3 つのプレーンへ分けます。

- **マネジメントプレーン** -- SNMP、システムログ、監視画面、NETCONF、ジーアールピーシー、CLI。
  設定を受け取り、可観測性を返します。人間の周期
  （1 回設定し、ときどき問い合わせる）。
- **コントロールプレーン** -- BGP、OSPF、IS-IS、経路計算。
  トポロジーが与えられたとき、各パケットがどの経路を通るべきかを
  決定します。トポロジーの周期（数秒から数分）。
- **データプレーン** -- フローテーブル、アクセス制御リスト、ASIC 転送。この
  パケットを今すぐ、ナノ秒単位で、テーブル検索により転送します。回線速度です。

盗む価値のある教訓は 5 つあります。

<a id="filter-evaluation-never-happens-at-packet-rate"></a>
### フィルター評価は決してパケットレートでは走らない

OSPF はトポロジーが変わったときに 1 回だけ最短経路を計算します。その結果は
データプレーンのフローテーブルへインストールされます。その後の各パケットは
テーブル検索だけです。ナノ秒単位、決定論的、フィルターロジックなし。
賢い仕事は *トポロジー変更時* に起こるのであり、*パケット処理時* ではありません。

<a id="plane-interfaces-are-explicit"></a>
### プレーン間インターフェースは明示的

オープンフロー、P4、NETCONF、ジーアールピーシー。コントロールプレーンはデータプレーンの
メモリへ直接手を突っ込みません。代わりに「このフローをインストールせよ」
という構造化メッセージを送ります。データプレーンはそれを確認します。
これにより失敗モードを診断可能にできます。インストール拒否は
黙った見落としではなく、明確なシグナルになります。

<a id="failures-isolate-cleanly"></a>
### 障害はきれいに分離される

コントロールプレーンがクラッシュしても、データプレーンは最後に
インストールされたルールを使って転送を続けます。データプレーンが
失敗した場合、コントロールプレーンはハートビート経由でそれに気付き、
その障害を避けて再ルーティングします。どちらのプレーンも
相手を巻き込んで落とさないことは、荷重を支える特性です。

<a id="telemetry-closes-the-loop"></a>
### テレメトリがループを閉じる

データプレーンはフロー統計、廃棄数、待ち時間を報告します。コントロールプレーンは
そのテレメトリを消費して判断します。輻輳を避けて再ルーティングするか、
サービスをスケールするか、ピアを停止と見なすか。*データプレーンが
報告を返すことで、コントロールプレーンはより賢くなる* のです。

<a id="three-planes-three-cadences"></a>
### 3 つのプレーン、3 つの周期

マネジメントプレーンは人間の周期で動きます。コントロールプレーンは
トポロジーの周期で動きます。データプレーンは回線速度で動きます。
各プレーンは、その仕事が許すなら遅くてもかまいません。

<a id="3-mapping-to-ai-dlc"></a>
## 3. AI-DLC への対応付け

この対応は、聞こえるよりずっと近いものです（以下の `<record>/` はアクティブな意図の
記録ディレクトリ、つまり `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` を指し、
監査証跡は `<record>/audit/` 配下のクローンごとの断片ディレクトリです）。

| ネットワーキング | AI-DLC における対応物 |
|---|---|
| コントロールプレーン（BGP、OSPF、経路計算） | ステージ定義、ルール、センサー -- 何を実行すべきかというスキーマ |
| データプレーン（パケット転送、フローテーブル） | ステージ実行、ボルト、エージェント呼び出し -- 実際の実行 |
| マネジメントプレーン（SNMP、監視画面、CLI） | `/aidlc --doctor`、設計支援機能（将来）、監査照会、`CLAUDE.md` |
| 経路制御プロトコル（BGP / OSPF） | コンパイル: `aidlc-graph.ts compile` がステージのフロントマター、ルール、センサーを読み、各ステージのプル取り込みをソースレジストリーに対して解決し、グラフを出力する |
| ASIC に読み込まれた FIB（転送情報ベース） | `stage-graph.json`。各ステージの `rules_in_context` + `sensors_applicable` がコンパイル時に解決済み |
| オープンフロー / NETCONF（インターフェース） | `stage-graph.json` -- コンパイルとオーケストレーターの間の明示的インターフェース |
| テレメトリー（NetFlow、エスフロー） | 監査ログ、センサー発火、`memory.md` の記録 → インテントの `runtime-graph.json` |
| 反応的なフローのインストール（PACKET_IN） | 学習ループ: データプレーンが観測を報告 → ユーザーが確認 → ルール／センサーへのファイル書き込み。新しいセンサー結合を取り込む場合は、発生元ステージのフロントマター編集も伴う |
| 事前の経路インストール（BGP 広告） | フレームワークのプルリクエスト: いかなるワークフロー実行前でも新しいステージ／ルール／センサーを出荷する |
| トポロジー変更 → 経路を再計算 | ワークフロー開始 → コンパイルが現在のソースファイルを読む。以後の再計算は次のワークフローで発火 |
| 事前インストール済み FIB はパケット転送中も安定 | ワークフロー中の学習ループ書き込みは進行中のコンパイル済みビューへ影響しない。適用は次のワークフロー開始時（§5 を参照） |
| テーブル検索による回線速度の転送 | オーケストレーター + ディスパッチャーはグラフノードから事前解決済みフィールドを読むだけ。ランタイムで解決の走査はしない |

`stage-graph.json` のプレーンラベルとランタイムの読み取りパターンは、
FIB / 経路テーブル / CLI の分離をそのまま写しています。設定はマネジメントプレーンから入り、
コントロールプレーンがそれをコンパイルし、データプレーンが
検索により実行します。

<a id="4-the-compile-boundary"></a>
## 4. コンパイル境界

荷重を支える洞察はこうです。ネットワーキングは、フィルター評価や
継承の走査をパケットレートで実行しません。そこでは仕事が高コストすぎるからです。
コントロールプレーンが 1 回だけ計算し、それを FIB へインストールし、
データプレーンは検索で転送します。AI-DLC も同じパターンに従います。
ワークフロー開始時に 1 回コンパイルし、そのワークフロー全体では解決済みビューを読み、
次のワークフロー開始時に再コンパイルします。

<a id="what-compiles-what-stays-on-disk"></a>
### 何をコンパイルし、何をディスクに残すか

| 状態 | ライフサイクル | ディスク上のソース | コンパイル先 | 読み取り元 |
|---|---|---|---|---|
| ステージ DAG、スコープルーティング、成果物作成 | フレームワークのバージョン管理下（フレームワークのプルリクエストで変更） | ステージのフロントマター（`.claude/aidlc-common/stages/*.md`） \| `stage-graph.json` | オーケストレーター、診断、設計支援 |
| **ルール**（散文、規範的） | 可変。フレームワークのプルリクエストまたは学習ループで書き込み | `aidlc/spaces/<space>/memory/<scope>.md`（ファイル名から導出。組織／チーム／プロジェクトは全ステージに付与） \| ステージごとの `stage-graph.json` の `rules_in_context` | オーケストレーター（解決済みビュー）。Claude Code の自動読み込みは文脈内の散文のためにソースを読む |
| **センサー**（マニフェスト、検証チェック） | 可変。フレームワークのプルリクエストまたは学習ループで書き込み（マニフェストは一度作成し、ステージは ID で取り込む） | `.claude/sensors/aidlc-<id>.md` \| ステージごとの `stage-graph.json` の `sensors_applicable` | ディスパッチャーはステージ開始時に解決済みリストを読む。ツール使用後フックはそこから発火 |
| ワークフロー実行テレメトリー | ワークフローごとに蓄積 | `audit/` の断片、`memory.md`、ボルトの分岐 \| `<record>/runtime-graph.json` | 診断、ゲート儀式、将来のワークフロー横断オブザーバー |
| ステージごとの観測ログ | ステージ実行ごと | `<record>/<phase>/<stage>/memory.md` | （コンパイルなし — 直接読む） | このステージのゲートにおけるゲート儀式 |

コンパイル済み成果物は 2 つあり、どちらもワークフロー単位です。`stage-graph.json` は
解決済みのコントロールプレーンビューを持ちます。すなわち、ステージ DAG、スコープルーティング、
成果物作成に加え、継承が事前解決されたステージごとのルールリストと
センサーリストです。`runtime-graph.json` は、あらゆる遷移区分の監査イベントごとに
再コンパイルされるもので、監査
ログの完全なイベントソースによる走査です。ディスク上のソースファイルは著述面であり、
ランタイムが読むのはコンパイル済みグラフです。

<a id="one-compile-at-workflow-start"></a>
### ワークフロー開始時に 1 回だけコンパイル

コンパイルはステージのフロントマターを読み、`aidlc/spaces/<space>/memory/` と
`.claude/sensors/` を走査し、ファイル名で共通既定ルール
（`org.md`, `team.md`, `project.md` はすべての
ステージに適用）を付与し、そのうえで各ステージのプル取り込みをソース
レジストリーに対して引きます。

- ステージの `phase: <name>` フィールドは、対応する
  `phases/<name>.md` ルールを付与します（ステージごとに 1 ルール）。
  詳細は [ルールシステム](/reference/rule-system) を参照してください。
- ステージの `sensors: [<id>, ...]` リストは、各 ID を
  `.claude/sensors/` に対して解決します。未知の ID は明確にコンパイルを失敗させます。
  ステージが取り込んだものが黙って発火しない、ということは許されません。詳細は
  [センサーシステム](/reference/sensor-system) を参照してください。

コンパイルは、その答えを各
ステージノードへ焼き込んだ `stage-graph.json` を出力します。ワークフローのあいだ、
オーケストレーターとディスパッチャーはその事前解決済みフィールドを読みます。ワークフロー中の
学習ループ書き込みはソースファイルを更新しますが、進行中のコンパイル済み
ビューへは影響しません。ユーザーはすでにそのステージ内でオーケストレーターを修正しており、
ルールは次回のためのものだからです。次のワークフローのコンパイルがそれを拾います。
これは、BGP がパケット転送の途中で経路を再計算しないのと同じ形です。

<a id="locked-and-atomic"></a>
### ロックとアトミック性

コンパイルが初日から対処しなければならない失敗モードは 2 つあります。この
実装では並列エージェント用のボルトごとの作業ツリーが大半の場面で状態を分離しますが、
`data/stage-graph.json` はリポジトリー共有であり、作業ツリーのスコープではありません。さらに、
ユーザーは同じチェックアウトに対して 2 つの端末から `/aidlc` を起動できます。
そのため、コンパイルはいずれにせよ防御が必要です。

- **同時コンパイル** は `data/stage-graph.json` への書き込みを競合させます。
  コンパイルはバージョン 0.4.0 以来 `withAuditLock`（`lib.ts`）の下で走るため、
  同時起動は直列化されます。2 つ目は 1 つ目の完了を待ち、
  その後、新しいソース状態に対して実行されます。
- **書き込み途中のクラッシュ。** 書き込み途中で中断されたコンパイルは、
  利用側が不正な JSON を読む状態を残してはなりません。コンパイルは一時ファイルへ書き、
  出力を検証したのち、POSIX `rename(2)` によりアトミックにリネームします。
  読み取り側が目にするのは前回コンパイルの出力か新しい出力のどちらかであり、
  中途半端に書かれたファイルではありません。

どちらのパターンも、状態と監査の作業のために既にコードベースへ存在しており、
コンパイルはそれを継承します。

<a id="two-graphs-two-lifecycles"></a>
### 2 つのグラフ、2 つのライフサイクル

2 つのコンパイル済み成果物は、利用側も更新周期も異なります。

- **`stage-graph.json`** -- コントロールプレーン。ワークフロー開始時にコンパイルされます。
  ワークフロー全体を通して安定です。これを読むのは
  オーケストレーター（DAG トポロジー、スコープルーティング）とディスパッチャー（ステージごとの
  `sensors_applicable`）です。Claude Code の自動読み込みは
  散文コンテキストのためにソースルールファイルも並行して消費します。
- **`runtime-graph.json`** -- データプレーン。ワークフロー単位成果物であり、
  `<record>/runtime-graph.json` にあります。あらゆる遷移区分の監査イベントごとに
  再コンパイルされます（監査ログの完全なイベントソースによる走査）。
  実行テレメトリを集約します。どのステージが走ったか、どの
  ボルトが分岐したか、どのセンサーが発火したか、どの `memory.md` が存在するかです。これを読むのは
  ゲート儀式（候補の提示）、診断（実行健全性の把握）、将来のワークフロー横断オブザーバーです。詳細は
  [ランタイムグラフ](/reference/runtime-graph) のスキーマ、
  コンパイルのライフサイクル、リカバリーモデルを参照してください。

この 2 つを混ぜると、キャッシュ可能性の境界を壊します。`stage-graph.json` は
ワークフローの寿命全体でキャッシュされるべきものであり、
`runtime-graph.json` はイベントごとに変化します。同じ成果物にすると、
  データプレーンが 1 回センサー発火を報告するたびに、コントロールプレーン全体を
再発行することになります。

<a id="why-rules-and-sensors-flow-through-the-same-compile"></a>
### ルールとセンサーが同じコンパイルを通る理由

コンパイルは、すべてのコントロールプレーン入力に対して対称です。ステージの
フロントマター、ルールファイル、センサーファイルはすべて
ワークフロー開始時に `aidlc-graph.ts` によって読まれ、すべてが
ステージごとの解決済みビューへ寄与します。ネットワーキングも同じです。BGP と OSPF は同じ FIB へ流れ込み、
ACL とポリシーは同じフローテーブルへ流れ込みます。1 つのコンパイル済みビューに、
複数のソースです。

具体的には、各ステージノードは 2 つのフィールドを得ます。

```json
{
  "slug": "requirements-analysis",
  "phase": "inception",
  "sensors": ["required-sections", "upstream-coverage"],
  "rules_in_context": [
    {"path": "aidlc/spaces/default/memory/org.md", "scope": "org"},
    {"path": "aidlc/spaces/default/memory/team.md", "scope": "team"},
    {"path": "aidlc/spaces/default/memory/project.md", "scope": "project"},
    {"path": "aidlc/spaces/default/memory/phases/inception.md", "scope": "phase"}
  ],
  "sensors_applicable": [
    {"id": "required-sections", "path": ".claude/sensors/aidlc-required-sections.md"},
    {"id": "upstream-coverage", "path": ".claude/sensors/aidlc-upstream-coverage.md"}
  ]
}
```

`matches`（解決済みセンサーエントリーに存在する場合）は、センサー側の
能力フィルターです。つまり *「このセンサーはこのグロブに一致するファイルを解析する」*
という意味であり、コンパイルはその値をマニフェストから
ステージの解決済みエントリーへ逐語でスナップショットします。ツール使用後フックは発火時に
マニフェストを再度開くことなく、その値を読みます。

結果は 2 つあります。

- **ディスパッチャーはランタイムで解決の走査をしない。** ステージ突入時に
  ノードから `sensors_applicable` を読むだけです。すでに検索済みであり、
  すでに付与済みです。ツール使用後フックはこの事前解決済みリストから発火します。
- **診断と設計支援は 1 つの解決済みビューを問い合わせる。** 「このステージへ
  どのルールが適用されるか?」「このファイルを編集するとどのセンサーが発火するか?」の両方が、
  ファイルシステムの走査なしで `stage-graph.json` から直接答えられます。

<a id="the-networking-lesson-applied"></a>
### ネットワーキングの教訓を適用する

コントロールプレーン（コンパイル）が、遅く賢い仕事を 1 回だけ行います。その後、
データプレーン（オーケストレーター + ディスパッチャーの読み取り）は解決済み
ビュー上で高速に動きます。ワークフロー開始時のコンパイルは、ステージごとの取り込みを
すべて解決します。つまり、すべての `sensors:` ID の検索、すべての `phase:` の付与、
ファイル名による共通既定ルールの取り込みです。ワークフロー中を通して、
ランタイムはグラフノードの事前解決済みフィールドを読みます。BGP は
パケット転送の途中で経路を再計算しません。AI-DLC もワークフロー途中で
再コンパイルしません。ワークフロー中の学習ループ書き込みはソース
ファイルを更新し、それらがコンパイル済みビューへ入るのは次のワークフロー開始時です。

<a id="5-recovery-as-an-emergent-property"></a>
## 5. 創発的特性としてのリカバリー

ここまで説明してきたデータプレーン、すなわち `runtime-graph.json`、監査ログ、
状態文書、`memory.md`、成果物ツリーには、そのための特別なコストを払わずに
得ている性質があります。要約圧縮後やクリーンな
再起動後に始まった新しいハーネスセッションは、この 5 つのソースをまとめて読めば、
前のセッションがどこまで進んでいたかを再開するのに十分なワークフロー状態を
再構築できます。リカバリーはバージョン 0.5.0 に後付けされた機能ではなく、
この設計のデータ規律から生じる創発的性質です。

<a id="five-sources-one-picture"></a>
### 5 つのソース、1 つの全体像

| ソース | 記録するもの | 読み順 |
|---|---|---|
| 成果物ツリー（`<record>/<phase>/<stage>/*.md`） | 決定そのものの完成形 | 最初 |
| ステージごとの `memory.md` | 意思決定の途中で何に気付いたか | 2 番目 |
| 監査ログ（`<record>/audit/` の断片） | 各決定がいつ起き、誰が承認したか | 3 番目 |
| 状態文書（`<record>/aidlc-state.md`、ステージごとの状態） | いまワークフローのどこにいるか | 4 番目 |
| `runtime-graph.json` | ステージ横断サマリー（所要時間、センサー発火、学習回数） | 5 番目 |

成果物を最初に読むのは、それが実際に
合意された内容の耐久的記録だからです。残り 4 つのソースは旅路についての情報です。
何が検討され、いつ起き、何が保留中で、ステージをまたいで見ると
どんなパターンがあるのか。誰かの
途中作業を引き継ぐ人間も、同じ順番で読みます。出力が先、メモが次、
タイムラインが 3 番目、現在カーソルが 4 番目、最後に要約ビューです。

<a id="what-recovery-reconstructs-cleanly"></a>
### リカバリーがきれいに再構築できるもの

決定、出力、ステージ内コンテキスト、タイムライン、現在位置、
どのボルトがどんな結果で分岐したかです。新しいセッションがこの
5 つのソースを読めば、ワークフローが何を生成したか、ユーザーが
どのような修正を加えたか、どのゲートが承認済みか、何がまだ
保留中か、そして前回の実行がおおむねどのように進んでいたかを把握できます。

<a id="what-recovery-cant-reconstruct"></a>
### リカバリーが再構築できないもの

前のセッションの会話リズムです。たとえばステージ途中での
ユーザーとの質問応答パターン、`memory.md` に確定される前の LLM の作業仮説、
途中で中断されたツール呼び出しなどです。会話バッファは
セッションとともに失われます。これは LLM セッションの本質的性質であり、
設計で修正できる性質ではありません。リカバリーの仕事は、新しいセッションへ
再方向付けして継続できるだけの材料を与えることであって、
前のセッションの正確な精神状態を再現することではありません。

<a id="the-consistency-constraint"></a>
### 整合性制約

5 つのソースは、異なるコードパスにより書かれながらも、
リカバリーが成り立つ程度には整合していなければなりません。正準なのは監査ログです
（追記専用であり、「何が起きたか」の唯一の真実源）。他の 4 つがそれと整合するのは、
それらの書き込みもバージョン 0.4.0 由来の同じ
`withAuditLock` プリミティブでゲートされているからです。新しいセッションがこの
5 つのソースのどれを読むにしても、監査ログと突き合わせることで乖離を検出し、調停できます。

<a id="what-this-means-for-the-framework"></a>
### これがフレームワークに意味すること

フレームワークはすでに、この小さな版を実践しています。現在の
`aidlc-state.md` には `Scope` フィールドがあり、これは意図の
誕生時に書かれ、セッション再開時に読まれるので、ワークフローのスコープは
オーケストレーターが再導出しなくても文脈圧縮を越えて生き残ります。
一般化すると、耐久的な何かを記録するデータプレーンのあらゆる部分が、
状態ファイルだけでなくリカバリー面の一部になります。より深い原理は、
*データプレーンがリカバリーのために構造化されている* ということです。
あらゆる耐久レコードがリカバリー面の一部であるため、再開経路は
既に存在する基底の上に小さく足されるだけであり、後付けではありません。
