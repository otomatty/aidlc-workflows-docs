<a id="phase-overview"></a>
## フェーズの概要

初期化フェーズは、AI-DLC ワークフローの全 5 フェーズの最初のフェーズです。ステージ 0.1 から 0.3 を実行し、**インテントを誕生させます**。つまり、状態ファイル、ディレクトリのスキャフォールディング、ワークスペースの分類、ルーティング設定を含むレコードディレクトリを `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/`（以下では `<record>/` と表記）に作成します。スキャフォールド専用コマンドはありません。ワークスペースシェルは `dist/<harness>/` にあらかじめ組み込まれており、最初の `/aidlc` 実行時（または構築するものを説明したとき）にエンジンが最初のインテントを自動で誕生させます。

このフェーズの 3 ステージは、すべてのスコープで実行されます。条件付きステージはありません。すべてのステージは承認ゲートなしで自動進行します。

ウェルカムメッセージは、セッション開始時に `settings.json` の `companyAnnouncements` エントリを通じて表示されます。これはステージではありません。ステージファイル、監査イベント、チェックボックスはいずれもありません。

3 つのステージはすべて、1 秒を大幅に下回る時間で完了する単一の決定的な `bun .claude/tools/aidlc-utility.ts intent-birth --scope <scope>` 呼び出し内で実行されます。コンダクターは可観測性のためにサイドバーに 3 つのタスク（ワークスペースのスキャフォールディング、ワークスペースの検出、状態の初期化）を作成し、ツールが戻るとすべてを完了としてマークします。

<a id="stage-01--workspace-scaffold"></a>
## ステージ 0.1 — ワークスペースのスキャフォールディング

| フィールド | 値 |
|-------|-------|
| ステージ番号 | 0.1 |
| スラッグ | ワークスペースのスキャフォールディング |
| フェーズ | 初期化 |
| 主担当エージェント | （オーケストレーター） |
| 支援エージェント    | — |
| 実行 | 常に実行 |
| モード | 自動進行（承認ゲートなし） |

<a id="steps"></a>
### 手順
1. 必要に応じて `<record>/` ディレクトリを作成する
2. 全 5 フェーズのステージ成果物ディレクトリと `<record>/verification/` を作成する
3. 空のスペースレベル `aidlc/knowledge/` ディレクトリを作成する（自由形式。エージェントごとのサブディレクトリおよび README は作成しない）
4. インテントの `audit/` シャードディレクトリヘッダーを作成し、`WORKFLOW_STARTED` を出力する
5. `STAGE_STARTED`、`WORKSPACE_SCAFFOLDED`、`STAGE_COMPLETED` イベントを追記する

<a id="inputs"></a>
### 入力
- なし（エントリポイント）

<a id="outputs"></a>
### 出力
- ステージサブディレクトリを含む `<record>/initialization/`、`ideation/`、`inception/`、`construction/`、`operation/`
- `<record>/verification/`
- 空のスペースレベル `aidlc/knowledge/` ディレクトリ（スペースの `intents/` と同階層）
- インテントの `audit/` シャードディレクトリ（ヘッダー、セッション、スキャフォールドイベント）

<a id="notes"></a>
### 注記
- 冪等です。すでに存在するディレクトリとファイルはスキップします
- LLM 経由ではなく、`aidlc-utility intent-birth` 内で実行されます

<a id="stage-02--workspace-detection"></a>
## ステージ 0.2 — ワークスペースの検出

| フィールド | 値 |
|-------|-------|
| ステージ番号 | 0.2 |
| スラッグ | ワークスペースの検出 |
| フェーズ | 初期化 |
| 主担当エージェント | （オーケストレーター — 決定的なルールベーススキャナー） |
| 支援エージェント    | — |
| 実行 | 常に実行 |
| モード | 自動進行（承認ゲートなし） |

<a id="steps-1"></a>
### 手順
1. プロジェクトディレクトリを 1 階層の深さで走査し、存在する場合は既知のソースディレクトリ（`src/`、`app/`、`lib/`、`pages/`、`components/`、`tests/`）も走査する。最上位でシグナルを検出できない場合は、任意の名前の各サブディレクトリについて同じシグナルセットで 1 階層下まで走査する。これにより、コンテナフォルダ（例: `wordbook/`）内にネストされたプロジェクトを、新規プロジェクトと誤分類せずに検出する
2. 拡張子ごとにファイルを数え、主言語と副言語を特定する
3. 既知の設定ファイル名からフレームワークを検出し、`package.json` の依存関係からリアクトを検出する
4. マニフェストとロックファイルからビルドシステムを検出する
5. `.gitmodules`（存在する場合）を読み、宣言されたサブモジュールパスについて初期化済みかをそれぞれ確認する
6. `stages/initialization/workspace-detection.md` のルールを使用して新規プロジェクトと既存プロジェクトを分類する
7. `STAGE_STARTED`、`WORKSPACE_SCANNED`、`STAGE_COMPLETED` イベントを追記する

<a id="inputs-1"></a>
### 入力
- プロジェクトのファイルシステム（読み取り専用スキャン）

<a id="outputs-1"></a>
### 出力
- ワークスペースの分類（新規プロジェクト／既存プロジェクト）
- 技術スタック（言語、フレームワーク、ビルドシステム）
- スキャン結果を記録する `WORKSPACE_SCANNED` 監査イベント

<a id="notes-1"></a>
### 注記
- `aidlc-utility intent-birth` 内で決定的なスキャナーとして実行されます。LLM サブエージェントはディスパッチしません。
- シンボリックリンクは追跡しません（`lstatSync` による循環保護）
- `.claude/`、`<record>/`、`node_modules/`、`.git/`、`dist/`、`build/`、`.next/`、`target/`、`vendor/` を除外します
- `devDependencies` しかない `package.json` はツール／スキャフォールディングとして扱われ、それだけで既存プロジェクトに分類されることはありません
- 少なくとも 1 つのサブモジュールパスエントリを含む解析可能な `.gitmodules` は既存プロジェクトのシグナルです（サブモジュールディレクトリが未初期化でも、リポジトリメタデータがコードを宣言しているためです）。サブモジュールパスが未初期化の場合、スキャンは警告し、`git submodule update --init --recursive` を示します。これは `WORKSPACE_SCANNED` イベント（サブモジュールフィールドと詳細の対処方法）および誕生時の標準出力に表示されるため、コンダクターが伝達できます。言語はスキャン結果のままです

<a id="stage-03--state-initialization"></a>
## ステージ 0.3 — 状態の初期化

| フィールド | 値 |
|-------|-------|
| ステージ番号 | 0.3 |
| スラッグ | 状態の初期化 |
| フェーズ | 初期化 |
| 主担当エージェント | （オーケストレーター） |
| 支援エージェント    | — |
| 実行 | 常に実行 |
| モード | 自動進行（承認ゲートなし） |

<a id="steps-2"></a>
### 手順
1. 状態契約を読み取る
2. スコープマッピング、深さ、テスト戦略を適用する
3. 新規プロジェクトでは `reverse-engineering` を SKIP としてマークする
4. 初期化後の最初のステージを `[-]` に設定した完全な `<record>/aidlc-state.md` を書き込む
5. `STAGE_STARTED`、`WORKSPACE_INITIALISED`、`STAGE_COMPLETED` イベントを追記する

<a id="inputs-2"></a>
### 入力
- ワークスペース検出からのワークスペース分類（同じツール呼び出し）
- スコープ設定（`--scope` フラグまたは既定の `poc`）
- 渡された場合の深さ／テスト戦略のオーバーライド
- `.claude/knowledge/aidlc-shared/state-template.md` の状態契約
- コンパイル済みの `tools/data/stage-graph.json` と `tools/data/scope-grid.json`

<a id="outputs-2"></a>
### 出力
- `<record>/aidlc-state.md`（完全に設定済み）
- `WORKSPACE_INITIALISED` 監査イベント

<a id="notes-2"></a>
### 注記
- 既存プロジェクトはリバースエンジニアリング（ステージ 2.1）へルーティングされます
- 新規プロジェクトは最初の非初期化ステージへルーティングされます（機能／概念実証ではインテントの取り込み、バグ修正／リファクタリングでは要件分析、ワークショップではプラクティスの発見です。ワークショップはアイデア創出の全ステージをスキップし、新規プロジェクトではリバースエンジニアリングが SKIP に格下げされるためです）
- 明示的な誕生パッケージングである `/aidlc-init` から呼び出された場合、オーケストレーターはこのステージの後に停止します
- ワークフロー開始時（`/aidlc <scope>` または構築するものを説明したとき）に呼び出された場合、オーケストレーターは初期化後の最初のステージへ進みます

<a id="re-initialization"></a>
## 再初期化

再初期化フラグはありません。最初のインテントの誕生はインテントごとに一度だけ実行されます。ワークスペースシェル自体はあらかじめ組み込まれており、再スキャフォールディングされることはありません。最初からやり直すには、新しいインテントを誕生させます（それぞれ専用の `<record>/` を持ちます）。完全にクリーンな状態にする場合は、アクティブなインテントのレコードディレクトリを `aidlc/spaces/<space>/intents/` 配下にアーカイブしてから、エンジンに新しいものを誕生させます。既存のインテントに対する 2 回目の `/aidlc` は、再初期化ではなく再開になります。

<a id="notes-3"></a>
## 注記

- 3 つのステージはすべて自動進行します。初期化フェーズに承認ゲートはありません
- すべてのステージは `aidlc-state.md` の `Current Stage` を通じてステータスラインを更新します
- すべてのステージは状態チェックボックス（`[ ]` → `[x]`）を更新し、ツールから直接監査イベントを追記します
- 初期化 → アイデア創出のフェーズ遷移にはガバナンス境界チェックがありません
