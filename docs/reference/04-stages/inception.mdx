## フェーズの概要

インセプション・フェーズは、AI-DLC 方法論を構成する 5 フェーズのうち第 3
フェーズです。アイデア創出フェーズで定めたビジネス意図とスコープを、具体的な
技術成果物へと変換します。対象には、既存プロジェクト向けのリバースエンジニアリング
によるコードベース理解、チームプラクティスと運用ルール、正式な要件、ユーザーストーリー、
詳細モックアップ、アプリケーションアーキテクチャ、作業単位への分解、そして構築フェーズを
統制するデリバリープランが含まれます。

インセプションではステージ 2.1〜2.8（全 8 ステージ）を実行します。構築フェーズへ
引き継ぐ前に、ステージ 2.8（デリバリー計画）でフェーズ境界の検証チェックを実施して
完了します。

> **パス規約。** 各ワークフローの成果物は、その **意図レコードディレクトリ** —
> `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` — の配下に置かれます（`<space>` は
> 非デフォルトのスペースを使用していない限り `default`、`<YYMMDD>` はレコードを時系列で
> ソートできる短縮 UTC 日付プレフィックス、`<label>` はリクエストの要点を表す短い
> ケバブケース名です。同日の重複には数値カウンターを付与します）。正規かつ衝突しない ID は
> `intents.json` のレジストリ行に記録された UUIDv7 であり、ディレクトリ名は人間が読める
> ラベルにすぎません。以降では `<record>/` をそのディレクトリの略記として用います。例:
> `<record>/inception/requirements-analysis/requirements.md` は
> `aidlc/spaces/default/intents/<YYMMDD>-<label>/inception/requirements-analysis/requirements.md`
> に展開されます。（意図ごとのレイアウト導入前に作成されたプロジェクトはフラットなツリーを
> 使用していました。エンジンは初回実行時にそれらを移行します。）

このフェーズには、インライン実行とサブエージェント実行の両方があります。ステージ 2.1
では 2 段階のサブエージェント委任（コードスキャンは AIDLC 開発者エージェント、統合は
AIDLC アーキテクトエージェント）を実施し、ステージ 2.2（プラクティス探索）では複数エージェントを
並列にディスパッチします。

**インセプション・フェーズの主な特徴:**

- このフェーズはサブエージェント委任を使う技術探索ステージ（2.1 リバース
  エンジニアリング）から始まり、続いて並列マルチエージェント・ディスパッチを使う
  方法論探索ステージ（2.2 プラクティス探索）、さらに 6 つのインライン分析・設計ステージを
  実行します。
- ステージ 2.1 は 2 段階のサブエージェントパターンです。AIDLC 開発者エージェント がコードを
  スキャンし、AIDLC アーキテクトエージェント がその結果を 9 種類の構造化成果物に統合します。既存
  プロジェクトには常に再実行するポリシーが適用されます。
- ステージ 2.2 は既存プロジェクトで証拠スキャンを行うために 4 エージェント
  （パイプライン・デプロイ、品質、開発者、開発セキュリティ運用）を並列にディスパッチし、その後に
  インタビューと確証ゲートを実行します。確証時には、コンテンツを
  `<record>/inception/practices-discovery/` からスペースのメモリレイヤー —
  `aidlc/spaces/<space>/memory/team.md` と `aidlc/spaces/<space>/memory/project.md` —
  へ昇格させます。この行をまたぐ昇格が、他のステージとは構造的に異なる理由です。
- ステージ 2.7 は `unit-of-work.md` を作成します。これは構築フェーズの段階的な構築フローを
  駆動する単位を定義します。
- ステージ 2.8 は、各単位で実行する構築ステージとその順序を決める実行計画を作成します。
  チームの作業方法（ブランチ戦略）、ウォーキングスケルトンの方針、デプロイメントの各節を
  得るために `aidlc/spaces/<space>/memory/team.md` を読み取ります。
- ステージ 2.8 のフェーズ境界検証は、要件、ストーリー、アーキテクチャ間の整合を検証します。

**スコープによるステージ選択:**

| スコープ | 対象ステージ |
|----------|--------------|
| エンタープライズ | 2.1〜2.8 のすべて |
| 機能追加 | 2.1〜2.8 のすべて |
| 最小実用製品 | 2.1（既存プロジェクトの場合）、2.2、2.3、2.4、2.5（UI がある場合）、2.6、2.7、2.8 |
| 概念実証 | 2.1（既存プロジェクトの場合）、2.3（最小限） |
| バグ修正 | 2.1（常に実行 — バグを特定）、2.3（最小限 — バグ記述） |
| リファクタリング | 2.1（常に実行 — 現行コードを理解）、2.3（最小限） |
| インフラ | 2.2、2.3（インフラ要件） |
| セキュリティパッチ | 2.1（脆弱性コンテキストの特定）、2.3（最小限） |
| ワークショップ | 2.1〜2.8 |

<a id="stage-21-reverse-engineering"></a>
## ステージ 2.1: リバースエンジニアリング

<a id="metadata"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.1 |
| 条件 | 条件付き — 既存プロジェクトを検出した場合。最新性のため常に再実行 |
| 主担当エージェント | AIDLC 開発者エージェント |
| 支援エージェント | AIDLC アーキテクトエージェント |
| モード | サブエージェント（2 段階: AIDLC 開発者エージェント、次に AIDLC アーキテクトエージェント） |
| 完了絵文字 | （`stage-protocol.md` の完了テンプレートを使用） |

<a id="purpose"></a>
### 目的

リバースエンジニアリングは、既存プロジェクトの既存コードベースを包括的に分析します。
2 段階のサブエージェント委任パターンを使用します。まず AIDLC 開発者エージェント がコード
ベース全体をスキャンし、次に AIDLC アーキテクトエージェント がスキャン結果を 9 種類の構造化
成果物に統合します。これらの成果物は、後続するすべてのインセプションおよび構築ステージが
依拠する技術的基盤となります。

**常時再実行ポリシー:** リバースエンジニアリングは、以前の成果物が存在していても
既存プロジェクトでは常に再実行します。これにより、成果物は古いスナップショットではなく、
コードベースの現行状態を反映します。

<a id="inputs"></a>
### 入力

- `<record>/aidlc-state.md`（プロジェクト種別の確認）

<a id="steps"></a>
### 手順

1. **条件を確認** — `<record>/aidlc-state.md` を読み、プロジェクト種別が既存
   プロジェクトであることを確認します。既存プロジェクトでない場合は、このステージを
   スキップし、スキップ理由を `aidlc-state.md` に更新します。

2. **開発者コードスキャン** — `Task` ツールを用いて AIDLC 開発者エージェント サブエージェント
   （`subagent_type="aidlc-developer-agent"`）へ委任します。委任プロンプトには
   `agents/aidlc-developer-agent.md` の AIDLC 開発者エージェント ペルソナと、
   `.claude/knowledge/aidlc-developer-agent/` のナレッジを含めます。
   `aidlc-state.md` のワークスペース状態もコンテキストとして含めます。

   開発者はコードベース全体で次をスキャンします。
   - すべてのパッケージ、モジュール、およびその目的
   - ビルドシステム、設定、依存関係
   - 外部および内部 API（エンドポイント、契約、メソッド）
   - フレームワーク、ライブラリ、およびそのバージョン
   - テストディレクトリ、テストフレームワーク、カバレッジ設定
   - コード品質の指標（リンティング、CI/CD、ドキュメント）
   - 技術的負債の兆候

   開発者は `templates/re-artifacts.md` の開発者コードスキャンテンプレートに従った
   構造化スキャン結果を返します。

3. **アーキテクトによる統合** — `Task` ツールを用いて AIDLC アーキテクトエージェント サブエージェント
   （`subagent_type="aidlc-architect-agent"`）へ委任します。委任プロンプトには
   `agents/aidlc-architect-agent.md` の AIDLC アーキテクトエージェント ペルソナと、
   `.claude/knowledge/aidlc-architect-agent/` のナレッジを含めます。開発者の完全な
   スキャン結果をコンテキストとして渡し、`aidlc-state.md` のワークスペース状態も含めます。

   アーキテクトはスキャン結果を 9 つの出力成果物（下記「出力」を参照）に統合します。

4. **状態を更新** — `<record>/aidlc-state.md` でリバースエンジニアリングを `[x]` 完了として
   マークします。現在のステージと次のステージを更新します。

5. **完了を提示して承認を求める** — 作成した 9 成果物すべての完了サマリーを表示します。
   標準の承認ゲート: Approve（要件分析へ進む）/ Request Changes。

<a id="outputs"></a>
### 出力

9 つの成果物をすべて `<record>/inception/reverse-engineering/` に書き出します。

| # | ファイル | 内容 |
|---|----------|------|
| 1 | `business-overview.md` | ビジネス領域、目的、主要機能 |
| 2 | `architecture.md` | システムアーキテクチャ、パターン、コンポーネント間の関係（Mermaid 図を含む）。ビジネストランザクションがコンポーネント間でどのように実装されるかを示す、インタラクション図の節（シーケンス図またはフロー図）を必ず含めること。 |
| 3 | `code-structure.md` | パッケージ／モジュール構成、ファイル分類、コードパターン |
| 4 | `api-documentation.md` | 外部および内部 API サーフェス、エンドポイント、契約 |
| 5 | `component-inventory.md` | 責務と依存関係を含む完全なコンポーネント一覧 |
| 6 | `technology-stack.md` | バージョンを含む言語、フレームワーク、ライブラリ |
| 7 | `dependencies.md` | 外部依存関係、パッケージ間の内部依存関係 |
| 8 | `code-quality-assessment.md` | テストカバレッジ、リンティング、CI/CD、ドキュメント品質、技術的負債 |
| 9 | `reverse-engineering-timestamp.md` | リバースエンジニアリングの実施時点（日付、利用可能ならコミットハッシュ、分析範囲） |

<a id="approval-gate"></a>
### 承認ゲート

標準の 2 選択肢ゲート: **Approve**（要件分析へ進む）/
**Request Changes**。

<a id="notes"></a>
### 注記

- **常時再実行ポリシー:** 以前の成果物が存在していても、このステージは既存プロジェクトで
  常に再実行されます。これは `SKILL.md` の「意図的な差異」節に記録された、上流リファレンス
  からの意図的な差異です。
- **2 段階のサブエージェントパターン:** AIDLC 開発者エージェント が生のコードスキャンを実行し、
  その後 AIDLC アーキテクトエージェント がスキャンを構造化成果物へ統合します。この分離により、
  スキャンの網羅性（開発者の視点）と、統合のアーキテクチャ的な妥当性（アーキテクトの視点）
  が確保されます。
- バグ修正およびリファクタリングスコープでは、既存コードの理解が不可欠なため、このステージを
  常に実行します（実質的には新規プロジェクトに近い場合も含む）。
- セキュリティパッチ スコープでは、脆弱性コンテキストを見つけるためにこのステージを実行します。
- ここで作成する 9 成果物は、要件分析（2.3）、ユーザーストーリー（2.4）、
  アプリケーション設計（2.6）、単位生成（2.7）で利用されます。
- `architecture.md` 成果物には、シーケンス図またはフロー図を用いてビジネストランザクションが
  コンポーネント間でどのように実装されるかを示すインタラクション図を必ず含めます。

<a id="stage-22-practices-discovery"></a>
## ステージ 2.2: プラクティス探索

<a id="metadata-1"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.2 |
| 条件 | 条件付き — EXECUTE スコープでは最新性のため常に再実行 |
| 主担当エージェント | AIDLC パイプライン・デプロイエージェント |
| 支援エージェント | AIDLC 品質エージェント、AIDLC 開発者エージェント、AIDLC 開発セキュリティ運用エージェント |
| モード | インライン（既存プロジェクトでは並列マルチ `Task` ディスパッチ） |
| 完了絵文字 | （`stage-protocol.md` の完了テンプレートを使用） |

<a id="purpose-1"></a>
### 目的

プラクティス探索は、2 軸の設定モデルの両方の行に書き込む、AI-DLC で唯一のステージです。
証拠からチームのブランチ戦略、ウォーキングスケルトンの方針、テスト姿勢、デプロイ頻度、
コードスタイル規則を発見します（既存プロジェクト）。または `org.md` のデフォルトを用いて
`AskUserQuestion` 経由で取得します（新規プロジェクト）。提案を
`<record>/inception/practices-discovery/` に下書きし、確証ゲートで確認済みの内容を
チーム作成のハーネス設定、すなわち `aidlc/spaces/<space>/memory/team.md` と
`aidlc/spaces/<space>/memory/project.md` に**昇格**させます。確証ゲートがあるからこそ
行をまたぐ書き込みが正当化されます。これがなければ、フレームワークが自らのハーネス設定で
チームの意図を勝手に記述することになります。

<a id="inputs-1"></a>
### 入力

- `<record>/aidlc-state.md`（プロジェクト種別）
- 既存プロジェクトのみ: リバースエンジニアリングの 9 成果物（`business-overview.md`、
  `architecture.md`、`code-structure.md`、`api-documentation.md`、`component-inventory.md`、
  `technology-stack.md`、`dependencies.md`、`code-quality-assessment.md`、
  `reverse-engineering-timestamp.md`）
- `aidlc/spaces/<space>/memory/org.md`（新規プロジェクトへのデフォルト提案）
- `.claude/knowledge/aidlc-pipeline-deploy-agent/branching-strategies.md`（主担当エージェントのナレッジベース）

<a id="outputs-1"></a>
### 出力

4 つの成果物を `<record>/inception/practices-discovery/` に書き出します。

- `team-practices.md` — チームの語り口による記述的な文章。`team.md` の見出しに対応する
  5 節: 作業方法、ウォーキングスケルトン、テスト姿勢、デプロイメント、コードスタイル。
- `discovered-rules.md` — 修正指向でエージェント向け。2 節: 必須（`ALWAYS …` 規則）と
  禁止（`NEVER …` 規則）。
- `evidence.md` — エージェントごとの調査結果サマリー。再実行時の最新性をたどる記録。
- `practices-discovery-timestamp.md` — 実行タイムスタンプとコミットハッシュ。

確証時には内容を次へ昇格させます。

- `aidlc/spaces/<space>/memory/team.md` — `replaceSection` による節の置換（再実行時は
  節の内容を蓄積せず上書きします）。
- `aidlc/spaces/<space>/memory/project.md` — `appendUnderHeading` による見出し配下への
  追記（規則は蓄積され、日付スタンプで区別されます）。

<a id="steps-1"></a>
### 手順

1. 条件を確認します。`aidlc-state.md` を読み、新規プロジェクトか既存プロジェクトかを
   判定します。既存プロジェクトは手順 2 を実行し、新規プロジェクトはスキップします。
2. **探索（既存プロジェクトのみ）** — コンダクターは 1 件のアシスタントメッセージ内で
   4 つの並列 `Task` 呼び出しを発行します。AIDLC パイプライン・デプロイエージェント（ブランチ戦略）、
   AIDLC 品質エージェント（テスト姿勢）、AIDLC 開発者エージェント（コードパターン）、
   AIDLC 開発セキュリティ運用エージェント（CI／セキュリティ）です。各エージェントは自身のナレッジベースを
   読み、証拠をスキャンして所見を返します。コンダクターは続行前に 4 つすべての所見を
   収集します。
3. **インタビュー（常に実行）** — 証拠の不足について `AskUserQuestion` を実施します。
   既存プロジェクトでは不足箇所だけを尋ねます。新規プロジェクトでは、`org.md` の
   デフォルトを推奨回答のテキストとして、5 つすべてのプラクティス領域を尋ねます。
   再実行時は以前の `team.md` の内容を事前入力します。
4. 統合します。4 つの成果物を書き出し、`PRACTICES_DISCOVERED` を発行します。
5. 確証ゲート。`AskUserQuestion` で両方の下書きを提示します。選択肢:
   Approve / 編集してから Approve / 却下して最初から書き直し。
6. 昇格（Approve 時）。`aidlc/spaces/<space>/memory/team.md` の 5 つのプラクティス節を
   節置換し、`aidlc/spaces/<space>/memory/project.md` の `## Mandated` および
   `## Forbidden` の下へ日付スタンプ付きで規則を追記します。原子性のため、まず
   `project.md`、次に `team.md` を書き込みます。失敗時は `PRACTICES_OVERRIDE` を発行し、
   PRACTICES_AFFIRMED を記録せずに中止します。
7. `PRACTICES_AFFIRMED` を発行し、状態チェックボックスを更新します。
   `Practices Affirmed Timestamp`（状態テンプレートのバージョン 7 フィールド、マイルストーン 6）も更新します。

<a id="notes-1"></a>
### 注記

- `.claude/tools/aidlc-lib.ts` の `replaceSection` ヘルパーは、`team.md` の行をまたぐ昇格を
  サポートする目的でマイルストーン 8 に追加されました（既存の `appendUnderHeading` は
  再実行をまたいで重複を蓄積します）。
- `org.md` と `team.md` は 1 組のタイトルケース見出し
  （`## Way of Working`、`## Walking Skeleton`、`## Testing Posture`、
  `## Deployment`、`## Code Style`）を共有しています。このステージは対応する
  タイトルケース見出しを使って `extractMarkdownSection` で各節を `org.md` から読み、
  `team.md` の同じ見出しの節を置換します。

<a id="stage-23-requirements-analysis"></a>
## ステージ 2.3: 要件分析

<a id="metadata-2"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.3 |
| 条件 | 常に実行 — 深さは複雑性に応じて変化 |
| 主担当エージェント | AIDLC プロダクトエージェント |
| 支援エージェント | （なし） |
| モード | インライン |
| 完了絵文字 | 🔍 |

<a id="purpose-2"></a>
### 目的

要件分析は、ユーザーの意図とリバースエンジニアリングによって得られたコードベース理解を、
正式で構造化された要件に変換します。リクエストの明確さ、種別、スコープ、複雑性を評価し、
適切な深さを決定します。既知の情報を抽出し、6 つの観点で完全性分析を実行して明確化の質問を
生成し、正式な要件文書を作成します。

このステージは常に実行しますが、プロジェクトの複雑性に応じて深さを調整します。
明確で狭いスコープには最小限、中程度のスコープには標準、大きく不確実性の高いスコープには
包括的な深さを適用します。

<a id="inputs-2"></a>
### 入力

- 既存プロジェクトの場合、ステージ 2.1 のリバースエンジニアリング成果物
  （`<record>/inception/reverse-engineering/`）
- 意図の `audit/` シャードにあるユーザーのプロジェクト説明

<a id="steps-2"></a>
### 手順

1. **エージェントペルソナを読み込む** — `agents/aidlc-product-agent.md` から
   AIDLC プロダクトエージェント のペルソナを、`.claude/knowledge/aidlc-product-agent/` から
   ナレッジを読み込みます。

2. **前提コンテキストを読み込む** — 既存プロジェクトの場合、
   `<record>/inception/reverse-engineering/` のリバースエンジニアリング成果物を読みます。
   意図の `audit/` シャードからユーザーのプロジェクト説明を読みます。

3. **ユーザーリクエストを分析する** — リクエストについて次を評価します。
   - **明確さ**: リクエストはどの程度明確に定義されているか
   - **種別**: 新機能、拡張、リファクタリング、バグ修正、移行
   - **スコープ**: 単一コンポーネント、複数コンポーネント、システム全体
   - **複雑性**: 単純、標準、複雑

4. **深さを決定する** — 複雑性の評価に基づきます。
   - **最小限**: 明確なリクエスト、狭いスコープ、十分に理解された領域
   - **標準**: 中程度のスコープ、いくつかの不確実性、複数のステークホルダー
   - **包括的**: 大きいスコープ、重大な不確実性、複雑な領域

5. **現行要件を評価する** — ユーザー入力から既知の情報、すなわち明示的な機能要件、
   暗黙の非機能要件、制約と前提、ビジネスコンテキストと目標を抽出して整理します。

6. **完全性分析** — 次の 6 観点でカバレッジを評価します。
   1. 機能要件 — 中核となる振る舞い、機能、ユースケース
   2. 非機能要件 — パフォーマンス、セキュリティ、スケーラビリティ、信頼性
   3. ユーザーシナリオ — ユーザーワークフロー、エッジケース、エラーシナリオ
   4. ビジネスコンテキスト — 目標、成功指標、ステークホルダー、制約
   5. 技術コンテキスト — 統合ポイント、プラットフォーム要件、技術的制約
   6. 品質特性 — 保守性、テスト容易性、アクセシビリティ、ユーザビリティ

   各観点の不足を特定します。

7. **明確化の質問を生成する** — **プロアクティブに**、要件が例外的に明確で 6 観点すべてに
   わたり完全でない限り、常に明確化の質問を生成します。`[Answer]:` タグ形式で
   `<record>/inception/requirements-analysis/requirements-analysis-questions.md`
   を作成します。コンテキストに合った A〜E の選択肢を持つ質問を含めます。各質問は最後の
   選択肢を必ず `X. Other (please specify)` とします。すべての `[Answer]:` タグは空欄のままにします。

   3 モードの質問フローを提示します: Guide me / I'll edit the file / Chat。

8. **回答を収集して分析する** — 質問ファイルを読み、すべての `[Answer]:` タグが埋まって
   いることを確認します。空欄があれば、未回答の質問を `AskUserQuestion` で提示して回答を
   書き戻します。部分的な回答のまま進めてはいけません。以下を実行します。
   - **必須**の曖昧性検出: すべての回答を曖昧な表現
     （「混在」「不明」「場合による」「おそらく」「たぶん」）についてスキャン
   - 回答間の矛盾チェック
   - 詳細不足の特定

9. **フォローアップ質問** — 曖昧さ、不明確さ、矛盾が**少しでも**見つかった場合、該当する
   問題を対象とするフォローアップ質問を作成します。すべての曖昧さを解決してから先へ進みます。
   「迷ったら質問する」。

10. **要件を生成する** — 次を含む
    `<record>/inception/requirements-analysis/requirements.md` を作成します。
    - 意図分析 — ユーザーが達成しようとしていること（機能だけでなく目標）
    - 機能要件 — 機能領域またはドメイン別に整理
    - 非機能要件 — パフォーマンス、セキュリティ、スケーラビリティの目標
    - 制約 — 技術、ビジネス、組織
    - 前提 — 根拠を添えて文書化
    - 対象外 — 明示的に除外する項目
    - 未解決の質問 — 後続ステージに残る不確実性

11. **状態を更新** — `<record>/aidlc-state.md` で要件分析を `[x]` 完了としてマークします。
    現在のステージと次のステージを更新します。

12. **完了を提示して承認を求める** — 🔍 絵文字とレビュー経路を含む完了メッセージを
    表示します。承認ゲートには 2 つの種類があります。

    **実行状態でユーザーストーリーが SKIP に設定されている場合:** 3 選択肢ゲート:
    Approve / Request Changes / ユーザーストーリーを追加（現在スキップされているユーザー
    ストーリーステージを含める）。「ユーザーストーリーを追加」を選択した場合は、`aidlc-state.md`
    を更新してユーザーストーリーを実行待ちとしてマークします。

    **ユーザーストーリーが SKIP に設定されていない場合:** 標準の 2 選択肢ゲート:
    Approve / Request Changes。

<a id="outputs-2"></a>
### 出力

すべての成果物を `<record>/inception/requirements-analysis/` に書き出します。

| ファイル | 内容 |
|----------|------|
| `requirements.md` | 正式な要件: 意図分析、機能／非機能要件、制約、前提、対象外、未解決の質問 |
| `requirements-analysis-questions.md` \| `[Answer]:` タグ付きの明確化質問（入力成果物） |

<a id="approval-gate-1"></a>
### 承認ゲート

条件付きゲートの形式:

- **ユーザーストーリーをスキップする場合:** 3 選択肢ゲート — **Approve** / **Request Changes** / **ユーザーストーリーを追加**
- **ユーザーストーリーをスキップしない場合:** 標準の 2 選択肢ゲート — **Approve** /
  **Request Changes**

<a id="notes-2"></a>
### 注記

- これはワークフローで最も詳細な質疑応答ステージです。必須の曖昧性検出を強制し、
  部分的または曖昧な回答では先へ進みません。
- 深さは複雑性に応じます。バグ修正／概念実証は最小限、機能追加は標準、エンタープライズは包括的です。
- バグ修正 スコープでは、このステージで最小限の深さでバグ記述を取得します。
- インフラ スコープでは、このステージでインフラ要件を取得します。
- ここで作成する要件文書は、ユーザーストーリー（2.4）、詳細モックアップ（2.5）、
  アプリケーション設計（2.6）、単位生成（2.7）、デリバリー計画（2.8）で利用されます。

## ステージ 2.4: ユーザーストーリー

<a id="metadata-3"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.4 |
| 条件 | 条件付き — ユーザー向け機能、複数ペルソナ、複雑なビジネスロジック、またはチーム横断作業で実行 |
| 主担当エージェント | AIDLC プロダクトエージェント |
| 支援エージェント | AIDLC デザインエージェント |
| モード | インライン |
| 完了絵文字 | 📚 |

<a id="purpose-3"></a>
### 目的

ユーザーストーリーは、正式な要件を、各機能の「誰が、何を、なぜ」を定義するユーザー
中心のストーリーに変換します。このステージは 2 部構成です。パート 1 で明確化の質問を含む
ストーリー計画を作成し、パート 2 で実際のストーリーとペルソナを生成します。計画とストーリーは
完了ゲートでまとめて提示し、統合レビューを行います。

AIDLC デザインエージェント はユーザーエクスペリエンスに関する支援的な視点を提供します。これは上流
リファレンスにはない意図的な追加であり、`SKILL.md` の「意図的な差異」節に記録されています。

<a id="inputs-3"></a>
### 入力

- `<record>/inception/requirements-analysis/requirements.md`
- 既存プロジェクトの場合、ステージ 2.1 のリバースエンジニアリング成果物
  （`<record>/inception/reverse-engineering/`）

<a id="steps-3"></a>
### 手順

1. **エージェントペルソナを読み込む** — `agents/aidlc-product-agent.md` と
   `.claude/knowledge/aidlc-product-agent/` から AIDLC プロダクトエージェント のペルソナと
   ナレッジを読み込みます。ユーザーエクスペリエンスに関する支援的な視点のため、
   `agents/aidlc-design-agent.md` と `.claude/knowledge/aidlc-design-agent/` から
   AIDLC デザインエージェント のペルソナとナレッジを読み込みます。

2. **ユーザーストーリーが必要か検証する** — ユーザーストーリーがこのプロジェクトに
   価値を加えるかを評価します。
   - **実行する場合**: ユーザー向け機能、複数のユーザーペルソナ、複雑なビジネスロジック、
     チーム横断の調整が必要
   - **スキップする場合**: 純粋なリファクタリング、局所的なバグ修正、インフラのみ、
     開発者ツール

   決定（実行またはスキップ）、根拠、考慮した要因、主要な価値領域（実行時）または代替
   カバレッジ（スキップ時）を記録する
   `<record>/inception/user-stories/user-stories-assessment.md` を作成します。

   スキップする場合は、`aidlc-state.md` にスキップ理由を更新して次のステージへ進みます。

3. **前提コンテキストを読み込む** — 
   `<record>/inception/requirements-analysis/requirements.md` を読みます。既存
   プロジェクトの場合は、`<record>/inception/reverse-engineering/` の関連する
   リバースエンジニアリング成果物を読みます。

**パート 1: 計画**

4. **質問付きストーリー計画を作成する** — 次を含む
   `<record>/inception/user-stories/user-stories-questions.md` を作成します。
   - ペルソナ策定アプローチ（ユーザーは誰か、その目標は何か）
   - INVEST 基準（独立性、交渉可能性、価値、見積可能性、小ささ、テスト可能性）を使う
     ストーリー形式
   - モスクワ優先度（必須／推奨／できれば／対象外）による
     ストーリーの優先順位付け。MVP 境界はデリバリー計画で正式に決めます。ストーリーの
     優先度はその決定に情報を与えます。
   - 分解アプローチの選択肢（機能、ペルソナ、ワークフロー、ドメイン領域、エピック別）
   - ペルソナとストーリー粒度についてユーザー入力を得る `[Answer]:` タグ形式の埋め込み質問

5. **回答を収集する** — `stage-protocol.md` の第 3 節の質問フロー（対話モードの選択提示、
   回答の収集、ファイルへの書き戻し）に従って回答を収集します。

6. **回答を分析する** — **必須**の曖昧性分析として、すべての回答を曖昧な表現
   （「混在」「不明」「場合による」「おそらく」）についてスキャンします。矛盾を確認し、
   詳細不足を特定します。曖昧さが**少しでも**あればフォローアップ質問を作成します。

7. **計画を提示して生成する** — ストーリー計画のサマリー（ペルソナ数、ストーリー数、
   分解アプローチ）をインラインで提示します。続けて直ちにパート 2 へ進みます。ユーザーは
   完了ゲートで統合出力（計画と生成されたストーリー）をレビューして承認します。

   生成の完了前にユーザーがフィードバックを挟んだ場合は、修正リクエストとして扱い、
   計画を更新してから生成を続けます。

**パート 2: 生成**

8. **計画を実行する — ストーリーとペルソナを生成する** — 承認済みの計画に基づき、次を
   生成します。

   `<record>/inception/user-stories/personas.md`:
   - ユーザーペルソナ定義（名前、役割、目標、ペインポイント、コンテキスト）
   - ペルソナ間の関係と優先順位

   `<record>/inception/user-stories/stories.md`:
   - 標準形式のユーザーストーリー: 「[ペルソナ] として、[目標] をしたい。なぜなら
     [便益] が得られるから」
   - 各ストーリーの受け入れ基準
   - ストーリー優先度（必須／推奨／できれば／対象外）
   - ストーリーの依存関係と関連性
   - INVEST 準拠に関する注記

9. **状態を更新** — `<record>/aidlc-state.md` でユーザーストーリーを `[x]` 完了として
   マークします。現在のステージと次のステージを更新します。

10. **完了を提示して承認を求める** — 📚 絵文字、作成したペルソナとストーリーの
    サマリー、レビュー経路を含む完了メッセージを表示します。標準の 2 選択肢承認ゲート:
    Approve（次のステージへ進む）/ Request Changes。

<a id="outputs-3"></a>
### 出力

すべての成果物を `<record>/inception/user-stories/` に書き出します。

| ファイル | 内容 |
|----------|------|
| `stories.md` | 受け入れ基準、優先度、依存関係、INVEST 注記を持つユーザーストーリー |
| `personas.md` | ユーザーペルソナ定義、関係、優先順位 |
| `user-stories-assessment.md` | 根拠と考慮要因を伴う実行／スキップの決定 |
| `user-stories-questions.md` \| `[Answer]:` タグを使う明確化質問付きストーリー計画（入力成果物） |

<a id="approval-gate-2"></a>
### 承認ゲート

標準の 2 選択肢ゲート: **Approve**（次のステージへ進む）/ **Request Changes**。

<a id="notes-3"></a>
### 注記

- スキップ条件: 純粋なリファクタリング、局所的なバグ修正、インフラのみ、開発者ツール。
- 2 部構成（計画、次に生成）により、ユーザーはストーリーを書き始める前に分解アプローチへ
  影響を与えられます。
- ユーザーストーリーの優先度（モスクワ法）は最小実用製品の境界に情報を与えますが、決定はしません。
  正式な最小実用製品の境界はデリバリー計画（ステージ 2.8）で設定されます。
- ステージをスキップする場合も、根拠を記録するため `user-stories-assessment.md` 成果物は
  常に作成します。
- ここで作成するストーリーは、詳細モックアップ（2.5）、アプリケーション設計（2.6）、
  単位生成（2.7）、デリバリー計画（2.8）で利用されます。
- AIDLC デザインエージェント の支援は UX を考慮した開発のための意図的な追加であり、`SKILL.md` の
  「意図的な差異」節に記載されています。

<a id="stage-25-refined-mockups--ux-design"></a>
## ステージ 2.5: 詳細モックアップと UX 設計

<a id="metadata-4"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.5 |
| 条件 | 条件付き — 非 UI、API のみ、インフラのみの取り組みではスキップ |
| 主担当エージェント | AIDLC デザインエージェント |
| 支援エージェント | AIDLC プロダクトエージェント（ストーリーに照らして検証） |
| モード | インライン |
| 完了絵文字 | 🎨 |

<a id="purpose-4"></a>
### 目的

詳細モックアップは、アイデア創出ステージ 1.6 の粗いコンセプトワイヤーフレームを、
正式な要件とユーザーストーリーに基づく中〜高忠実度のモックアップへ発展させます。詳細な
インタラクション仕様、デザインシステムの対応付け、レスポンシブ動作定義、アクセシビリティ
準拠チェックリストを作成します。

非 UI の取り組み（API のみ、バックエンド）では、このステージはインタラクション図を API
開発者エクスペリエンス仕様へと詳細化します。

通常、このステージはステージ 1.6（粗いモックアップ）もスキップされていた場合にスキップします。

<a id="inputs-4"></a>
### 入力

- 存在する場合、ステージ 1.6 の粗いモックアップ（`<record>/ideation/rough-mockups/`）
- ステージ 2.4 のユーザーストーリー（`<record>/inception/user-stories/`）
- ステージ 2.3 の要件
  (`<record>/inception/requirements-analysis/`)

<a id="steps-4"></a>
### 手順

1. **エージェントペルソナを読み込む** — `agents/aidlc-design-agent.md` から
   AIDLC デザインエージェント のペルソナを、`.claude/knowledge/aidlc-design-agent/` から
   ナレッジを読み込みます。

2. **前提コンテキストを読み込む** — 存在する場合は
   `<record>/ideation/rough-mockups/` の粗いモックアップを読みます。
   `<record>/inception/user-stories/` のユーザーストーリーと
   `<record>/inception/requirements-analysis/` の要件を読みます。

3. **明確化の質問を生成する** — 次を対象とする質問を含む
   `<record>/inception/refined-mockups/refined-mockups-questions.md` を作成します。
   - 各ユーザーストーリーを UI で表現する方法
   - 必要なインタラクションパターン（モーダル、インライン編集、ウィザード、段階的開示）
   - 各画面が扱うべき状態（読み込み、空、エラー、成功、部分的）
   - 既存デザインシステム／コンポーネントライブラリとの整合
   - アクセシビリティ要件（WCAG レベル）
   - 必要なレスポンシブブレークポイント

### 目的

単位生成は、アプリケーション設計を、構築フェーズの段階的な構築フローを駆動する

## ステージ 2.8: デリバリー計画

<a id="metadata-7"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | インセプション |
| ステージ番号 | 2.8 |
| 条件 | ALWAYS — インセプションの締めくくりステージ |
| 主担当エージェント | AIDLC デリバリーエージェント |
| 支援エージェント | AIDLC アーキテクトエージェント（アーキテクチャ依存に照らしてビルド順を検証） |
| モード | インライン |
| 完了絵文字 | 📅 |

<a id="purpose-7"></a>
### 目的

デリバリー計画はインセプション・フェーズの締めくくりです。実行単位の並び — ステージ 2.7
が作成した作業単位を構築で実行する順序 — を計画します。ステージ 2.7 が分析的
（依存関係 DAG）であるのに対し、ステージ 2.8 は経済的です。リスク、価値、チーム容量、
学習で重み付けした経路を DAG から選びます。

`stage-protocol.md` の正規用語集によれば、**実行単位**は
「構築内のデプロイ可能な作業単位 — ステージ 3.1〜3.5 を 1 回通すこと」です。実行単位は
1 つ以上の作業単位に対する 1 回の構築パスであり、MMF やスプリントとは異なります。
（ステージ 3.6 のビルドとテスト、および 3.7 の CI パイプラインは、実行単位ごとではなく、
すべての実行単位の終了後に一度だけ実行します。）

経済的価値は DAG から導けません — AI エージェントはトポロジカルソートはできますが、
どの実行単位が市場仮説を最速で検証するか、どの実行単位がコミットメントが積み上がる前に
最も怖い未知を表面化させるかは決められません。それはこのステージで捉える人間の価値判断です。

このステージはまた、構築へ移行する前にすべてのインセプション成果物の整合性を検証する
フェーズ境界検証チェックも実行します。

**重要な区別:** このステージは実行単位の順序付けを計画します。どの AI-DLC ステージを
どの深さで実行するかは決めません — それは `/aidlc` スキルのスコープ選択が扱います。

<a id="inputs-7"></a>
### 入力

インセプション・フェーズのすべての成果物:

- ステージ 2.3 の要件（`<record>/inception/requirements-analysis/`）
- ステージ 2.4 のユーザーストーリー（`<record>/inception/user-stories/`）
- ステージ 2.6 のアプリケーション設計
  （`<record>/inception/application-design/`）
- ステージ 2.7 の単位（`<record>/inception/units-generation/`）
- 存在する場合、ステージ 1.5 のチーム編成
  （`<record>/ideation/team-formation/`）

<a id="steps-7"></a>
### 手順

1. **エージェントペルソナを読み込む** — `agents/aidlc-delivery-agent.md` と
   `.claude/knowledge/aidlc-delivery-agent/` から AIDLC デリバリーエージェントの
   ペルソナとナレッジを読み込みます。ビルド順の検証のため AIDLC アーキテクトエージェントも
   読み込みます。

2. **前提コンテキストを読み込む** — インセプション・フェーズのすべての成果物、すなわち
   要件、ユーザーストーリー、アプリケーション設計、単位、および存在する場合はチーム編成を
   読みます。

3. **明確化の質問を生成する** — 次を扱う質問を含む
   `<record>/inception/delivery-planning/delivery-planning-questions.md`
   を作成します。
   - 順序付けヒューリスティック: リスク優先、価値優先、ワーキングスケルトン優先、
     またはハイブリッド
   - 使う場合の WSJF（重み付き最短ジョブ優先）スコアリングモデルと重み
   - 最初の実行単位: ワーキングスケルトン（コックバーン）、または拡大前に方針を証明する
     自信構築スライス
   - 作業単位の実行単位への束ね方
   - 各実行単位の完了の定義
   - 実行単位ごとの自信仮説 — 出荷すると何が証明されるか
   - モブから実行単位への割り当て（利用可能なら 1.5 のチームを参照。未実行なら AI のみ）
   - 特定の実行単位をゲートする外部依存（API、データ、承認）
   - 最も早く取り組むべき主要リスク項目

   `stage-protocol.md` の質問フローに従います。

4. **回答を収集して分析する** — 選んだ実行単位の並びが 2.7 の依存関係 DAG を尊重するかを
   （AIDLC アーキテクトエージェントの入力とともに）検証します。トポロジカル順からのずれは、
   根拠成果物で正当化できるようフラグを立てます。

5. **成果物を生成する** — `<record>/inception/delivery-planning/` に 4 つの成果物を
   作成します。
   - `bolt-plan.md` — 順序付きの実行単位並び。実行単位ごとの作業単位、
     ワーキングスケルトンマーカー、完了の定義、自信仮説、想定デモ。
   - `team-allocation.md` — 実行単位からモブへの割り当て。チーム数が 1 より大きいときの
     プログラムボード類似。
   - `risk-and-sequencing-rationale.md` — 実行単位順の理由: WSJF スコア、リスク優先の
     論拠、ワーキングスケルトン優先の論拠、または価値優先の論拠。
   - `external-dependency-map.md` — 消費する実行単位へ対応付けたゲート項目
     （完全に AI 内完結のときは軽量または空）。

6. **フェーズ境界検証** — インセプションから構築への検証チェックを実行します。
   - 要件からストーリー、アーキテクチャへの整合
   - すべてのストーリーが要件へ追跡できること
   - アーキテクチャがすべてのストーリーをカバーすること
   - 結果を `<record>/verification/phase-check-inception.md` へ書き出す

7. **状態を更新** — `<record>/aidlc-state.md` で 2.8 デリバリー計画を `[x]` 完了として
   マークします。ライフサイクルフェーズを CONSTRUCTION に更新します。

8. **完了を提示して承認を求める** — 📅 絵文字を含む完了メッセージを表示します。
   承認ゲート: Approve（構築へ進む）/ Request Changes。ユーザーはこのゲートで
   ステージの含める／除外を上書きできます。

<a id="outputs-7"></a>
### 出力

すべての成果物を `<record>/inception/delivery-planning/` に書き出します。

| ファイル | 内容 |
|----------|------|
| `bolt-plan.md` | 順序付き実行単位並び。実行単位ごとの作業単位、ワーキングスケルトンマーカー、完了の定義、自信仮説、想定デモ |
| `team-allocation.md` | 実行単位からモブへの割り当て。チーム数が 1 より大きいときのプログラムボード類似。1.5 が未実行のときは AI のみの割り当て |
| `risk-and-sequencing-rationale.md` | 実行単位順に対する WSJF／リスク優先／ワーキングスケルトン優先／価値優先の正当化 |
| `external-dependency-map.md` | ゲート項目（外部 API、データ可用性、承認リードタイム、外部チーム引き継ぎ）を消費する実行単位へ対応付け |
| `delivery-planning-questions.md` \| `[Answer]:` タグ付きの明確化質問（入力成果物） |

フェーズ境界検証の出力:

| ファイル | 内容 |
|----------|------|
| `<record>/verification/phase-check-inception.md` | インセプションから構築への追跡可能性チェック結果 |

<a id="approval-gate-6"></a>
### 承認ゲート

標準の 2 選択肢ゲート: **Approve**（構築へ進む）/ **Request Changes**。
ユーザーはこのゲートでステージの含める／除外を上書きできます。

<a id="notes-7"></a>
### 注記

- **フェーズ境界ステージ。** これは 3 つのフェーズ境界ステージのうち 2 番目です
  （1.7 の後、3.7 の前）。検証チェックは要件からストーリー、アーキテクチャへの整合を
  検証します。
- **経済的順序付け対トポロジカル順序付け。** ステージ 2.7 は依存関係 DAG を作成します
  （トポロジカル順は記述的な幾何として自然に得られます）。ステージ 2.8 は人間の価値判断で
  重み付けした経路をその DAG から選びます。リスク優先またはワーキングスケルトン優先の
  論拠が正当化するとき、実行単位の順はトポロジカル順からずれても構いません — そのずれは
  `risk-and-sequencing-rationale.md` に記録します。
- **実行単位 ≠ スプリント ≠ MMF。** 正規用語集によれば、実行単位は構築ステージ
  3.1〜3.5 を 1 回通すことです（3.6 ビルドとテストおよび 3.7 CI パイプラインはすべての
  実行単位の後に一度だけ実行）。順序付けヒューリスティック（ワーキングスケルトン、WSJF）は
  実行単位の中で適用され、実行単位そのものの定義を変えません。
- **上流からの意図的な差異。** 上流リファレンスはこのステージを「ワークフロー計画」と呼び、
  純粋なステージ選択器として扱います。本実装（「デリバリー計画」へ改名）は実行単位の
  順序付け、チーム配分、リスク根拠を追加します。
- 実行単位計画は自信構築の並びを定義します。各実行単位には定義された作業単位、完了の定義、
  自信仮説があります。
- AIDLC アーキテクトエージェントは、提案された実行単位の並びが `component-dependency.md`
  および `unit-of-work-dependency.md` 成果物で定義された依存関係を尊重することを検証します。
- チーム配分は、存在する場合ステージ 1.5 のチーム編成成果物から取ります。1.5 が SKIP
  （最小実用製品、ワークショップ）のときは、すべての実行単位を AIDLC 開発者エージェント
  （AI）が実行します。

<a id="phase-summary"></a>
## フェーズのまとめ

<a id="key-outputs"></a>
### 主な出力

インセプション・フェーズは、構築と運用へ引き継がれる次の主要出力を作成します。

1. **リバースエンジニアリング成果物**（2.1）— 既存コードベースを記録する 9 成果物:
   ビジネス概要、アーキテクチャ、コード構造、API ドキュメント、コンポーネント一覧、
   技術スタック、依存関係、コード品質評価、タイムスタンプ。（既存プロジェクトのみ。）
2. **要件文書**（2.3）— 正式な要件: 機能、非機能、制約、前提、対象外、未解決の質問。
3. **ユーザーストーリーとペルソナ**（2.4）— 受け入れ基準、優先度、依存関係を持つ
   ユーザーストーリー。ユーザーペルソナ定義。（該当する場合。）
4. **詳細モックアップ**（2.5）— 中〜高忠実度モックアップ、インタラクション仕様、
   デザインシステム対応、アクセシビリティチェックリスト。（該当する場合。）
5. **アプリケーション設計**（2.6）— コンポーネント定義、メソッドシグネチャ、
   サービス定義、依存関係マトリックス、アーキテクチャ決定記録。（該当する場合。）
6. **作業単位**（2.7）— 境界と複雑さ見積もり付きユニット定義、ビルド順付きユニット
   依存関係マトリックス、ストーリーからユニットへの対応。（該当する場合。）これは
   構築の段階的な構築フローを駆動する成果物です。
7. **デリバリープラン**（2.8）— 実行単位計画、ビルド順、依存関係マトリックス、
   チーム配分。これは構築と運用を統制する実行計画です。
8. **フェーズ境界検証**（2.8）— `<record>/verification/phase-check-inception.md`
   に書き出される、インセプションから構築への追跡可能性チェック。

<a id="handoff-to-construction"></a>
### 構築への引き継ぎ

ステージ 2.8 の承認後、フレームワークは構築フェーズへ移行します。構築はデリバリー計画の
実行計画に基づくステージレベルのタスクを作成し、段階的な構築フローを実行します。

構築は `bolt-plan.md` に従い実行単位ごとに進み、実行単位計画が許す範囲で並列バッチを
認めます。各実行単位は、1 つ以上のユニットのまとまりのあるスライスをカバーします
（`unit-of-work.md` および `unit-of-work-dependency.md` に従う）:

各実行単位について:
1. **3.1 機能設計**（実行計画に応じて条件付き）
2. **3.2 NFR 要件**（実行計画に応じて条件付き）
3. **3.3 NFR 設計**（実行計画に応じて条件付き）
4. **3.4 インフラ設計**（実行計画に応じて条件付き）
5. **3.5 コード生成**（常に、実行単位内のユニットごと）

最終実行単位の完了後:
6. **3.6 ビルドとテスト**（常に）
7. **3.7 CI パイプライン**（条件付き）

実行単位は依存関係グラフが許す範囲で並列バッチとして実行できます。ワーキングスケルトン
実行単位は常に単一実行単位バッチとして最初に走り、並列バッチ開始前にエンドツーエンドの
形を検証します。実行単位ごとの詳細な説明は
`docs/guide/04-phases-and-stages.md:263-293` を参照してください。
