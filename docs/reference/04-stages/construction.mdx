<a id="phase-overview"></a>
## フェーズの概要

構築フェーズでは、インセプションで作成した設計成果物を、動作しテスト済みの
ソフトウェアへと変換します。機能設計、非機能要件と設計、インフラストラクチャ
設計、コード生成、ビルド／テスト検証、CI パイプライン設定にわたる 7 つの
ステージ（3.1～3.7）で構成されます。

構築は AI-DLC 方法論における 5 フェーズのうち第 4 フェーズです。デリバリー
計画（ステージ 2.8）で作成された**実行計画**に基づいて進めます。この計画に
より、実行するステージ、スキップするステージ、ユニットを構築する順序が
決まります。

すべてのステージは、承認ゲート、質問形式、完了メッセージ、状態追跡について
`stage-protocol.md` に従います。

> **パス規約。** 各ワークフローの成果物は、対応する**意図レコードディレクトリ**
> `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` に格納されます（`<space>` は
> 非デフォルトのスペースを使用していない限り `default` です。
> `<YYMMDD>-<label>` は意図ディレクトリを示し、`260624` のような短い UTC 日付
> プレフィックスと、レコードを時系列に並べられる短いケバブケースのラベルで
> 構成されます）。以降の `<record>/` はこのディレクトリの短縮表記です。たとえば
> `<record>/construction/{unit-name}/functional-design/` は
> `aidlc/spaces/default/intents/<YYMMDD>-<label>/construction/{unit-name}/functional-design/`
> に展開されます。ディレクトリ名は人が読めるラベルであり、正規の識別子は
> `intents.json` レジストリ行に保存された UUIDv7 です。（意図ごとのレイアウト
> より前に作成されたプロジェクトではフラットツリーを使用しており、エンジンが
> 初回実行時に移行します。）

## 実行単位ごとの構築

構築は、ステージ 2.8 の `bolt-plan.md`（実行単位の順序とワーキングスケルトン
マーカー）およびステージ 2.7 の依存関係 DAG に基づき、**実行単位ごと**に実行
されます。[実行単位](/guide/glossary) は、1 つのユニットまたは依存関係で
結ばれた小さなユニット群に対する、ステージ 3.1～3.5 の 1 回の通過です。
ステージ 3.6（ビルドとテスト）および 3.7（CI パイプライン）は、すべての実行単位
を対象に最後に**一度だけ**実行します。

```
Bolt 1 (walking skeleton) — always gated:
  Questions (3.1–3.4 across the Bolt's Units in QUESTION-ONLY mode)
  → Answers gate (Bolt-level)
  Design artifacts (3.1–3.4 in ARTIFACT-ONLY mode)
  Code generation (3.5 per Unit via Task delegation)
  → Walking-skeleton gate
  → Ladder prompt (fires once): "autonomous" or "gated"
  → Write Construction Autonomy Mode to state

Bolt 2..N — autonomy mode governs the gate:
  (Parallel-eligible Bolts run as a batch; single batch-level gate covers
   every Bolt in it.)
  Questions → Answers gate (Bolt-level) → Design → Code-gen → Bolt/batch
  gate (skipped if autonomous). Failure always halts and asks.

After all Bolts:
  3.6 Build and Test (runs once across the full codebase)
  3.7 CI Pipeline    (runs once, conditional)
```

各設計ステージファイル（3.1～3.4）は、QUESTION-ONLY および ARTIFACT-ONLY
の実行モードをサポートします。詳細は各ステージファイルを参照してください。
コード生成の手順 3 の**計画の承認は、実行単位の実行中を含め、生成前に必ず
ハードストップ**します。通常の実行単位の実行時に**エンジンによって抑制**される
のは、その手順 7 のユニット単位の完了承認ゲートだけであり、単一の実行単位レベル
（またはバッチレベル）の完了ゲートに置き換えられます。ユニット単位の完了ゲートは
直接起動（例: `/aidlc --stage code-generation`）では残ります。

**設計ステージの反復順序（オプトイン）。** 既定では、エンジンは 4 つのインライン
設計ステージ（3.1～3.4）をステージ優先で反復します。すべてのユニットに 3.1 を
実行してから、すべてのユニットに 3.2 を実行する、といった順序です。状態ファイル
の「ランタイム状態」見出し配下に `Construction Iteration: unit-major` が記録されて
いる場合（デリバリー計画で `aidlc-state.ts set-construction-iteration unit-major`
により設定するか、人が設定します）、エンジンは代わりにユニット優先で処理します。
実行単位の構築順序にある各ユニットについて、次のユニットを開始する前に 4 つの
設計文書（3.1～3.4）を連続して作成します。ステージごとの 4 つの承認ゲートの数と
仕組みは変わりません。ユニット優先では、ユニットごとの設計グリッド全体を完了した
後に、ステージ順で遅れて発火し、ステージごとに 1 回の人による承認を行います。
`code-generation`（3.5、`mode: subagent`）はこの反復に含まれません。これを有効に
するのは厳密に `unit-major` という値だけで、未設定または `stage-major` が既定です。

**並列バッチ。** 2 つ以上の実行単位が依存関係の充足を共有し、互いに依存していない
場合、コンダクターは 1 つのアシスタントメッセージで N 回の `Task` 呼び出しを
発行し、そのコード生成ステージを並行して実行します。すべてを 1 つの
バッチレベルゲートで扱います。監査イベント（`BOLT_STARTED`、`BOLT_COMPLETED`）には
`Batch=N` フィールドが含まれるため、ログから兄弟の実行単位を復元できます。

**失敗時の処理。** 実行単位の失敗は自律モードにかかわらず、常に構築を停止します。
選択肢は再試行（失敗した実行単位だけを再実行）、スキップ（`[S]` を記録して続行。
依存する実行単位も失敗する可能性があります）、または中止です。並列バッチ内で成功
した兄弟の実行単位は、`[x]` の状態と成果物を維持します。正規仕様については
`stage-protocol.md` の §1「構築の実行単位ゲート」と、`SKILL.md` の
§CONSTRUCTION フローを参照してください。

<a id="stage-31-functional-design"></a>
## ステージ 3.1: 機能設計

<a id="metadata"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| ステージ | 3.1 |
| フェーズ | 構築 |
| 実行 | CONDITIONAL（実行計画に従う） |
| 条件 | 新しいデータモデル、複雑なビジネスロジック、またはビジネスルールの設計が必要。新しいビジネスロジックのない単純なロジック変更ならスキップ。 |
| ユニット単位 | はい |
| 主担当エージェント | AIDLC アーキテクトエージェント |
| 支援エージェント | AIDLC 開発者エージェント |
| モード | インライン |
| 入力 | 作業ユニット定義、ストーリーマップ、要件、アプリケーション設計成果物 |
| 出力 | `<record>/construction/{unit-name}/functional-design/` — ビジネスロジックモデル、ビジネスルール、ドメインエンティティ、条件付き: フロントエンドコンポーネント |

<a id="purpose"></a>
### 目的

単一の作業ユニットについて、ビジネスロジック、ドメインモデル、ルールを設計します。
AIDLC アーキテクトエージェントが主導し、AIDLC 開発者エージェントが技術的実現可能性の
入力を提供します。

<a id="inputs"></a>
### 入力

- `<record>/inception/units-generation/unit-of-work.md` からのユニット定義
- `<record>/inception/units-generation/unit-of-work-story-map.md` からの割り当て済みストーリー
- `<record>/inception/requirements-analysis/requirements.md` からの要件
- `<record>/inception/application-design/` からのアプリケーション設計成果物

<a id="steps"></a>
### 手順

1. **ペルソナを読み込む** — AIDLC アーキテクトエージェント（主担当）のペルソナと
   ナレッジを読み込みます。技術実装の入力のため AIDLC 開発者エージェントの
   ペルソナとナレッジも読み込みます。主視点として AIDLC アーキテクトエージェントを
   適用します。

2. **ユニットコンテキストを読む** — ユニット定義、割り当て済みストーリー、要件、
   アプリケーション設計成果物を読みます。

3. **機能設計計画を作成する** — ユニットのスコープを分析し、コンテキストに適した
   質問と `[Answer]:` タグを用いて、次の場所に質問ファイルを作成します。
   `<record>/construction/{unit-name}/functional-design/functional-design-questions.md`
   着目点:
   - ビジネスロジックのワークフローとアルゴリズム
   - ドメインモデルとエンティティ関係
   - ビジネスルール、制約、検証ロジック
   - データフローと変換
   - 他ユニットまたは外部システムとの統合点
   - エラー処理とエッジケース
   - フロントエンドコンポーネント（コンポーネント階層、プロパティ／状態、
     操作フロー、フォーム検証）
   - ビジネスシナリオ（エンドツーエンドのユーザージャーニー、正常／異常パス、
     並行性のエッジケース）

4. **回答を収集して分析する** — `stage-protocol.md` の質問フローに従って回答を
   収集します（対話モードの選択を提示し、回答を集め、ファイルへ書き戻す）。
   曖昧さの分析は必須です。
   - 曖昧な回答を特定する（「混在」「わからない」「場合による」「おそらく」など）
   - 回答間の矛盾を確認する
   - 成果物生成に必要な欠落詳細に印を付ける
   - 曖昧さが 1 つでもあれば、フォローアップ質問を作成し、解決してから進む

5. **成果物を生成する** — 次を
   `<record>/construction/{unit-name}/functional-design/` に生成します。
   - **ビジネスロジックモデル**: ユニットのビジネスロジック向けの詳細な
     アルゴリズム、ワークフロー、データ変換、処理順序、決定木
   - **ビジネスルール**: 決定ルール、検証ロジック、制約、ポリシー、条件付き
     振る舞い、ビジネス不変条件
   - **ドメインエンティティ**: エンティティ、関係、データ構造、属性、
     ライフサイクル状態、エンティティ相互作用パターン
   - **フロントエンドコンポーネント**（条件付き — ユニットにフロントエンド／UI が
     含まれる場合のみ）: コンポーネント階層、プロパティ／状態設計、操作フロー、
     フォーム検証ルール、API 統合点

6. **完了を準備する** — ユニットの機能設計成果物を検証します。状態は編集せず、
   ゲート結果を `aidlc-orchestrate.ts` を通じて報告します。

7. **完了** — 完了メッセージと承認ゲートを提示します。

<a id="outputs"></a>
### 出力

| 成果物 | 説明 |
|--------|------|
| ビジネスロジックモデル | アルゴリズム、ワークフロー、データ変換、処理順序、決定木 |
| ビジネスルール | 決定ルール、検証ロジック、制約、ポリシー、条件付き振る舞い |
| ドメインエンティティ | エンティティ、関係、データ構造、属性、ライフサイクル状態 |
| フロントエンドコンポーネント | （条件付き）コンポーネント階層、プロパティ／状態、操作フロー、フォーム検証、API 統合 |



<a id="approval-gate"></a>
### 承認ゲート

厳格な 2 選択肢: Approve / Request Changes。

<a id="notes"></a>
### 注記

- 質問ファイルはステージ成果物と同じ場所に置きます。
  `<record>/construction/{unit-name}/functional-design/functional-design-questions.md`
- フロントエンドコンポーネント成果物は、ユニットにフロントエンド／UI 作業が
  含まれる場合のみ生成します。
- すべての質問は 3 モード対話フロー（Guide me / I'll edit the file / Chat）を
  使用します。

<a id="stage-32-nfr-requirements"></a>
## ステージ 3.2: NFR 要件

<a id="metadata-1"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| ステージ | 3.2 |
| フェーズ | 構築 |
| 実行 | CONDITIONAL（実行計画に従う） |
| 条件 | パフォーマンス要件、セキュリティ上の考慮、スケーラビリティの懸念、または技術スタック選定が必要。NFR 要件がなく技術スタックが既に決まっている場合はスキップ。 |
| ユニット単位 | はい |
| 主担当エージェント | AIDLC アーキテクトエージェント |
| 支援エージェント | AIDLC 開発セキュリティ運用エージェント、AIDLC コンプライアンスエージェント、AIDLC 品質エージェント |
| モード | インライン |
| 入力 | 機能設計成果物、要件、リバースエンジニアリング成果物 |
| 出力 | `<record>/construction/{unit-name}/nfr-requirements/` — パフォーマンス要件、セキュリティ要件、スケーラビリティ要件、信頼性要件、技術スタック決定 |

<a id="purpose-1"></a>
### 目的

単一ユニットについて、パフォーマンス、セキュリティ、スケーラビリティ、信頼性、
技術選定にわたる非機能要件を定義します。AIDLC アーキテクトエージェントが主導し、
AIDLC 開発セキュリティ運用エージェントがセキュリティ入力、AIDLC コンプライアンス
エージェントが規制入力、AIDLC 品質エージェントがテスト容易性と測定可能性の入力を
提供します。

<a id="inputs-1"></a>
### 入力

- `<record>/construction/{unit-name}/functional-design/` からの機能設計成果物
  （存在する場合）
- `<record>/inception/requirements-analysis/requirements.md` からの要件
- `aidlc/spaces/<active-space>/codekb/<repo>/` からのリバースエンジニアリング成果物
  （存在する場合）

<a id="steps-1"></a>
### 手順

1. **ペルソナを読み込む** — AIDLC アーキテクトエージェント（主担当）のペルソナと
   ナレッジを読み込みます。支援入力として AIDLC 開発セキュリティ運用エージェント
   （セキュリティ要件）、AIDLC コンプライアンスエージェント（規制要件）、
   AIDLC 品質エージェント（テスト容易性）のペルソナとナレッジも読み込みます。

2. **先行成果物を読む** — 機能設計成果物（存在する場合）、要件、
   リバースエンジニアリング成果物を読みます。

3. **NFR カテゴリを評価する** — 次の NFR カテゴリでユニットを分析します。
   - **パフォーマンス**: 応答時間、スループット、レイテンシ目標、リソース使用率
   - **セキュリティ**: 認証、認可、データ保護、コンプライアンス要件
   - **スケーラビリティ**: 負荷処理、成長予測、スケーリング戦略
   - **信頼性**: 可用性目標、耐障害性、災害復旧、データ耐久性
   - **可観測性**: 監視、ログ、アラート、トレーシング要件

4. **質問を生成する** — 不明な NFR 領域について `[Answer]:` タグを用いて、次の場所に
   質問ファイルを作成します。
   `<record>/construction/{unit-name}/nfr-requirements/nfr-requirements-questions.md`
   定量的な目標と具体的な制約に焦点を当てます。

5. **回答を収集して分析する** — `stage-protocol.md` の質問フローに従って回答を
   収集します。曖昧さの分析は必須です。
   - 曖昧な回答を特定する（「十分速い」「高可用」「セキュア」など）
   - NFR 目標間の矛盾を確認する
   - 欠落している定量目標に印を付ける
   - 曖昧さが 1 つでもあれば、フォローアップ質問を作成し、解決してから進む

6. **成果物を生成する** — 次を
   `<record>/construction/{unit-name}/nfr-requirements/` に生成します。
   - **パフォーマンス要件**: 応答時間目標、スループット要件、レイテンシ予算、
     リソース制約、ベンチマーク
   - **セキュリティ要件**: 認証要件、認可モデル、データ保護、コンプライアンス、
     脅威の考慮
   - **スケーラビリティ要件**: 負荷予測、スケーリング契機、キャパシティ計画、
     データ成長、同時実行目標
   - **信頼性要件**: 可用性目標（SLA／SLO）、耐障害性要件、バックアップ／復旧、
     グレースフルデグラデーション
   - **技術スタック決定**: 技術選定とその根拠 — 言語、フレームワーク、
     データベース、インフラツール、各選択の正当化

7. **完了を準備する** — ユニットの NFR 要件成果物を検証します。状態は編集せず、
   ゲート結果を `aidlc-orchestrate.ts` を通じて報告します。

8. **完了** — 完了メッセージと承認ゲートを提示します。

<a id="outputs-1"></a>
### 出力

| 成果物 | 説明 |
|--------|------|
| パフォーマンス要件 | 応答時間、スループット、レイテンシ予算、リソース制約、ベンチマーク |
| セキュリティ要件 | 認証、認可、データ保護、コンプライアンス、脅威 |
| スケーラビリティ要件 | 負荷予測、スケーリング契機、キャパシティ計画、同時実行 |
| 信頼性要件 | 可用性目標（SLA／SLO）、耐障害性、バックアップ／復旧 |

   ```
   手順 6:  API テスト（手順 5 のユニット＋統合テスト）
   手順 7:  リポジトリ／データアクセス層（クエリ、ORM 設定）
   手順 8:  フロントエンドコンポーネント（該当する場合 — UI コンポーネント、ページ、状態）
   手順 9:  フロントエンドテスト（コンポーネントテスト、操作テスト）
   手順 10: 設定と環境整備（.env テンプレート、ビルド設定）
   手順 11: テスト設定（vitest.config、jest.config、または同等）
   手順 12: ドキュメント（インライン文書、API 文書、README 更新）
   ```

   この層ごとの方針により、依存先より先に依存元を構築します（データモデルの後に
   ビジネスロジック、ビジネスロジックの後に API）。アーキテクチャが求める場合は
   逸脱します（例: イベント駆動システム、独立スタックのマイクロサービス）。

   **計画へのテストファイル含有は必須です。** 計画には必ず次のステップを含めます。
   - ユニットテストファイル（主要振る舞いをカバーするコンポーネント／モジュールごと）
   - テスト設定（テストランナー設定、または同等）

   計画がテストファイルのステップを省略している場合、ユーザーへ提示する前に
   追加しなければなりません。テストはビルドとテストへ先送りしません。あの
   ステージは検証と拡張であり、ゼロからの作成ではありません。

   明確な実行順序と追跡性のため、各計画ステップに連番を付けます
   （手順 1、手順 2、など）。

3. **計画の承認** — 計画サマリーをユーザーへ提示し、承認を求めます。まず
   `<record>/construction/{unit-name}/code-generation/code-generation-questions.md`
   を **Plan Approval** の質問と空欄の `[Answer]:` で作成またはリセットし、
   構造化質問として提示してターンを停止します。
   - 「計画を承認」— コード生成へ進む
   - Request Changes — 計画を改訂する

   タグへの記入は、人間が応答した後にのみ行います。変更要求は記録され、計画が
   改訂され、再提示の前に Plan Approval タグは空欄へリセットされます。転送
   ループの継続は決して承認ではありません。

<a id="part-2----generation-steps-4-7"></a>
#### 第 2 部 — 生成（手順 4～7）

4. **コードを生成する** — 委任する前に、ユーザーへ次を表示します。
   「[N] 件の計画ステップについてコードを生成します。プロジェクトの複雑さに
   より数分かかることがあります。完了時にサマリーを示します。」

   `Task` ツールで AIDLC 開発者エージェントのサブエージェント
   （`subagent_type="aidlc-developer-agent"`）へ委任します。

   **サブエージェントへ渡すコンテキスト:**
   - `agents/aidlc-developer-agent.md` の主担当ペルソナと、
     `.claude/knowledge/aidlc-developer-agent/` のナレッジ（サブエージェントは
     会話履歴にアクセスできないためプロンプトに含める）
   - **現在のユニットのみ**の設計成果物（全ユニットではない）
   - インセプションフェーズ成果物それぞれの 1～2 行サマリーとファイルパス
     （要件サマリー、ストーリーサマリー、アプリ設計サマリー）— 全文が必要なら
     サブエージェントが個別ファイルを読めます
   - 承認済みのコード生成計画（全文）
   - プロジェクトワークスペースの詳細（言語、フレームワーク、規約を
     `aidlc-state.md` から）
   - 各計画ステップを順に実行し、完了時にチェックボックスを付ける指示

   **コンテキスト予算:** 現在のユニットの設計成果物だけを渡し、全ユニットは渡しません。
   インセプション成果物は全文埋め込みではなく、ファイルパス付きで要約します。
   サブエージェントはワークスペース内にすべてのコード、テストファイル、設定
   成果物を生成します。

5. **コードサマリーを生成する** — サブエージェント完了後、次を作成します。
   `<record>/construction/{unit-name}/code-generation/code-summary.md`
   記載内容:
   - 作成／変更したファイル
   - 主要な実装上の決定
   - テストカバレッジの要約
   - 計画からの逸脱

6. **完了を準備する** — ユニットのコードとサマリー成果物を検証します。
   状態は編集せず、ゲート結果を `aidlc-orchestrate.ts` を通じて報告します。

7. **完了** — 完了メッセージと承認ゲートを提示します。

<a id="outputs-4"></a>
### 出力

| 成果物 | 説明 |
|--------|------|
| コード生成計画 | チェックボックス、ストーリー追跡性、手順順序付きの詳細計画 |
| コード生成質問 | 永続化された Plan Approval の質問と明示的な人間の回答 |
| コードサマリー | 作成／変更ファイル、決定、テストカバレッジ、計画からの逸脱 |
| （アプリケーションコード） | ワークスペースルートへ書き出された全ソース、テスト、設定 |



<a id="approval-gate-4"></a>
### 承認ゲート

厳格な 2 選択肢: Approve / Request Changes。

<a id="notes-1"></a>
### 注記

- **二部構成**: 計画フェーズ（手順 1～3）はユーザー対話と計画承認付きでインライン
  実行します。生成フェーズ（手順 4～7）は `Task` ツール経由で AIDLC 開発者
  エージェントのサブエージェントへ委任します。構築の大半のステージが完全に
  インラインである点と異なります。
- **開発者エージェントのサブエージェント**: コード生成はインライン実行ではなく、
  `subagent_type="aidlc-developer-agent"`（`Task` ツール経由の委任）を使います。
  構築でサブエージェントを使う唯一のステージです。サブエージェントはセッションの
  ツール一式を継承します（AIDLC 開発者エージェントは `tools:` 許可リストを宣言しない）
  ため、読み取り・編集・書き込み・検索・シェル・ユーザー質問、および継承した
  MCP ツールに到達できます。
- **コンテキスト予算**: サブエージェントへ渡すのは現在のユニットの設計成果物のみです。
  インセプションフェーズ成果物は 1～2 行とファイルパスで要約し、必要なものだけを
  選択的に読ませます。
- **テストファイルの必須含有**: テストファイルはコード生成計画の一部でなければ
  なりません。ステージ 3.6（ビルドとテスト）はテストを検証・拡張しますが、
  ゼロからは作成しません。
- **ブラウンフィールドへの配慮**: ブラウンフィールドでは、サブエージェントは
  複製を作らず既存ファイルをその場で変更します。

<a id="stage-36-build-and-test"></a>
## ステージ 3.6: ビルドとテスト

<a id="metadata-5"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| ステージ | 3.6 |
| フェーズ | 構築 |
| 実行 | ALWAYS（すべてのユニット完了後） |
| 条件 | すべてのユニット単位ステージ完了後に一度だけ常に実行します。 |
| ユニット単位 | いいえ（全ユニットに対して一度実行） |
| 主担当エージェント | AIDLC 品質エージェント |
| 支援エージェント | AIDLC 開発セキュリティ運用エージェント |
| モード | インライン |
| 入力 | 全ユニットのコード生成出力すべて |
| 出力 | `<record>/construction/build-and-test/` — ビルド手順、ユニットテスト手順、統合テスト手順、パフォーマンステスト手順、セキュリティテスト手順、ビルドとテストのサマリー、テスト結果、および条件付きの追加テスト手順ファイル |

<a id="purpose-5"></a>
### 目的

すべてのテスト種別にわたるテスト手順を生成し、その後シェル経由でビルドとテストを
実際に実行します。このステージは**全ユニット横断**で動作し、ユニット単位では
ありません。AIDLC 品質エージェントが主導し、AIDLC 開発セキュリティ運用エージェントが
セキュリティテストの専門知識を提供します。

<a id="inputs-5"></a>
### 入力

- `<record>/construction/*/code-generation/code-summary.md` からの全ユニットの
  コード生成出力
- パフォーマンスおよびセキュリティテストの必要性を把握するための、ユニット横断の
  NFR 要件（存在する場合）

<a id="steps-5"></a>
### 手順

1. **ペルソナを読み込む** — AIDLC 品質エージェント（主担当）のペルソナとナレッジを
   読み込みます。セキュリティテスト入力のため AIDLC 開発セキュリティ運用エージェントの
   ペルソナとナレッジも読み込みます。

2. **テスト要件を分析する** — 全ユニットのコード生成出力を読みます。パフォーマンスと
   セキュリティのテスト必要性を特定するため、NFR 要件（存在する場合）を確認します。
   必要なテスト種別を一覧化します。

3. **ビルド手順を生成する** — 次を作成します。
   `<record>/construction/build-and-test/build-instructions.md`:
   - 依存関係のインストール手順
   - 環境整備（環境変数、設定ファイル、ローカルサービス）
   - ビルドコマンド（コンパイル、バンドル、トランスパイル）
   - ビルド検証手順
   - よくあるビルド問題のトラブルシューティング

4. **ユニットテスト手順を生成する** — 次を作成します。
   `<record>/construction/build-and-test/unit-test-instructions.md`:
   - テストフレームワークのセットアップと設定
   - ユニットテストの実行方法（コマンド、フラグ、フィルタ）
   - 期待するテストカバレッジ目標
   - モック／スタブの指針
   - テストデータ管理

5. **統合テスト手順を生成する** — 次を作成します。
   `<record>/construction/build-and-test/integration-test-instructions.md`:
   - テスト環境の前提条件（データベース、サービス、キュー）
   - 統合テストの実行方法
   - ユニット横断の相互作用テスト
   - 外部依存の扱い（スタブ、テストダブル、サンドボックス）
   - テストデータの準備と後始末

6. **パフォーマンステスト手順を生成する**（条件付き）— いずれかのユニットに
   NFR パフォーマンス要件がある場合、次を作成します。
   `performance-test-instructions.md`:
   - 負荷テストのツールと設定
   - NFR 目標に対応付けたパフォーマンステストシナリオ
   - ベース測定とベンチマーク
   - ストレステストおよびソークテストの手順
   - パフォーマンス回帰の検出

7. **セキュリティテスト手順を生成する**（条件付き）— いずれかのユニットに
   NFR セキュリティ要件がある場合、次を作成します。
   `security-test-instructions.md`:
   - セキュリティスキャンツール（SAST、DAST、依存関係監査）
   - 認証／認可のテストシナリオ
   - 入力検証とインジェクションテスト
   - コンプライアンス検証手順
   - 脆弱性評価手順

8. **追加のテスト種別を生成する**（条件付き）— プロジェクトアーキテクチャに応じて、
   次の名前のファイルを作成します。
   - **契約テスト手順**: マイクロサービス API 向け — コンシューマ駆動契約、
     スキーマ検証、API 互換性
   - **エンドツーエンドテスト手順**: UI 主導アプリケーション向け — ブラウザ自動化、
     ユーザージャーニーテスト、クロスブラウザ検証
   - **アクセシビリティテスト手順**: ユーザー向けインターフェース向け —
     WCAG 準拠、スクリーンリーダーテスト、キーボード操作

   すべてのファイルは `<record>/construction/build-and-test/` に置きます。

9. **ビルドとテストのサマリーを生成する** — 次を作成します。
   `<record>/construction/build-and-test/build-and-test-summary.md`:
   - 全体のビルド状態と前提条件
   - テスト種別インベントリ（生成したテスト種別）
   - ユニットごとのカバレッジ期待値
   - 準備状況の評価（ビルド準備、テスト準備、デプロイ準備）
   - 既知の制限や未解決事項

10. **ビルドとテストを実行する** — 手順ファイルに記載したビルドおよびテスト
    コマンドを**シェル経由で**実行を試みます。

    a. **ビルド**: ビルド手順からビルドコマンドをシェルで実行し、出力を捕捉します。
    b. **ユニットテスト**: ユニットテスト手順からユニットテストコマンドをシェルで
       実行し、合否件数を捕捉します。
    c. **統合テスト**（該当する場合）: 統合テストコマンドを実行し、結果を捕捉します。
    d. **結果を報告する**: 次を作成または更新します。
       `<record>/construction/build-and-test/test-results.md`
       - ビルド状態（成功／失敗＋出力）
       - テスト結果（合計、成功、失敗、スキップ）
       - 失敗の詳細（テスト名、アサーション、スタックトレース）
       - カバレッジレポート（テストフレームワークが対応する場合）

    **失敗診断ループ（最大 2 回）:** ビルドまたはテストが失敗した場合、問題の診断と
    修正を試みます。
    - エラー出力を読む
    - 失敗しているコードを特定する
    - 修正を適用する
    - 失敗したステップを再実行する
    - 2 回試しても修正できない場合は、テスト結果に失敗を記録し、承認ゲートで
      ユーザーへ提示する

    **成功時:** ビルドとテストのサマリーを、手順だけでなく実際の結果で更新します。

11. **完了を準備する** — ビルド／テストの証拠を検証します。ステージや
    フェーズの状態は編集せず、報告されたゲート結果が遷移を所有します。

12. **完了** — 完了メッセージと承認ゲートを提示します。

<a id="outputs-5"></a>
### 出力

| 成果物 | 説明 | 条件 |
|--------|------|------|
| ビルド手順 | 依存関係インストール、環境整備、ビルドコマンド、トラブルシューティング | 常時 |
| ユニットテスト手順 | テストフレームワーク設定、実行コマンド、カバレッジ目標、モック | 常時 |
| 統合テスト手順 | 前提条件、ユニット横断テスト、外部依存、データ準備 | 常時 |
| パフォーマンステスト手順 | 負荷テスト、NFR シナリオ、ベースライン、ストレス／ソーク | NFR パフォーマンスがある場合 |
| セキュリティテスト手順 | SAST／DAST、認証テスト、インジェクションテスト、コンプライアンス | NFR セキュリティがある場合 |
| 契約テスト手順 | コンシューマ駆動契約、スキーマ検証、API 互換 | マイクロサービスの場合 |
| エンドツーエンドテスト手順 | ブラウザ自動化、ユーザージャーニー、クロスブラウザ | UI 主導の場合 |
| アクセシビリティテスト手順 | WCAG 準拠、スクリーンリーダー、キーボード操作 | ユーザー向け UI の場合 |
| ビルドとテストのサマリー | 全体状態、テストインベントリ、カバレッジ、準備状況評価 | 常時 |
| テスト結果 | 実際のビルド／テスト実行結果、合否、カバレッジ | 常時 |



<a id="approval-gate-5"></a>
### 承認ゲート

厳格な 2 選択肢: Approve / Request Changes。

<a id="notes-2"></a>
### 注記

- **実際のシェル実行**: このステージはテスト手順を文書化するだけでなく、シェル経由で
  ビルドとテストコマンドを実際に実行し、実結果を捕捉します。コードベースに対して
  実コマンドを実行する数少ないステージのひとつです。
- **失敗診断ループ**: 最大 2 回まで、失敗の自動診断と修正を試みます。2 回失敗すると、
  失敗を記録し、承認ゲートでユーザーへ提示します。
- **条件付きテスト種別**: パフォーマンステスト、セキュリティテスト、契約テスト、
  エンドツーエンドテスト、アクセシビリティテストは、関連条件が満たされる場合のみ
  生成します（NFR 要件の存在、マイクロサービスアーキテクチャ、UI 主導アプリ、
  ユーザー向けインターフェース）。
- **ユニット横断スコープ**: ユニット単位のステージ 3.1～3.5 と異なり、ビルドとテストは
  すべてのユニットが生成したコード全体に対して一度実行します。個別ユニットではなく、
  統合されたコードベースを検証します。
- **フェーズ完了**: このステージ（該当すれば 3.7 とともに）が構築フェーズの終わりを
  示します。最終の承認済み報告により、エンジンが構築を完了としてマークし、
  運用への遷移を原子的に行います。

<a id="stage-37-ci-pipeline"></a>
## ステージ 3.7: CI パイプライン

<a id="metadata-6"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| ステージ | 3.7 |
| フェーズ | 構築 |
| 実行 | CONDITIONAL（十分な CI が既にある場合はスキップ） |
| 条件 | CI パイプラインの作成または大幅な変更が必要な場合に実行 |
| ユニット単位 | いいえ（全ユニットに対して一度実行） |
| 主担当エージェント | AIDLC パイプライン・デプロイエージェント |
| 支援エージェント | （なし） |
| モード | インライン |
| 入力 | ステージ 3.5 のコード生成出力、ステージ 3.6 のビルド／テスト結果 |
| 出力 | `<record>/construction/ci-pipeline/` — CI 設定、品質ゲート、CI パイプライン質問 |

<a id="purpose-6"></a>
### 目的

品質ゲート、成果物管理、ビルド／テスト自動化を備えた CI（継続的インテグレーション）
パイプラインを設定します。AIDLC パイプライン・デプロイエージェントが主導し、
支援エージェントはありません。

<a id="inputs-6"></a>
### 入力

- `<record>/construction/build-and-test/` からのビルド／テスト結果
- `<record>/construction/infrastructure-design/` からのインフラストラクチャ設計
  （存在する場合）
- 既存の CI 設定のためのワークスペースプロファイル

<a id="steps-6"></a>
### 手順

1. **エージェントペルソナを読み込む** — AIDLC パイプライン・デプロイエージェントの
   ペルソナとナレッジを読み込みます。

2. **先行コンテキストを読み込む** — ビルド／テスト結果、インフラストラクチャ設計
   （存在する場合）、既存 CI 設定のためのワークスペースプロファイルを読みます。

3. **明確化の質問を生成する** — 次の質問を含むファイルを作成します。
   `<record>/construction/ci-pipeline/ci-pipeline-questions.md`
   - 使用中の CI ツールは何か？
   - ブランチ戦略は何か？
   - マージ前に必要な品質ゲートは何か？
   - 使用する成果物リポジトリは何か（ECR、S3 など）？

   例示する CI ツールと成果物ストア:

   ```
   CodePipeline, CodeBuild, GitHub Actions, Jenkins
   ECR, CodeArtifact, S3
   ```

   `stage-protocol.md` の質問フローに従います。

4. **回答を収集して分析する** — 既存インフラとチーム能力に対して CI の選択を検証します。

5. **成果物を生成する** — CI パイプライン設定（ビルド仕様やワークフロー YAML、
   または同等）、品質ゲート定義、成果物リポジトリ設定を作成します。

6. **フェーズ境界検証** — 構築から運用への検証チェックを実行します。
   - アーキテクチャからコード、テストへの整合
   - すべてのコードが設計へ追跡できること
   - 受け入れ基準に対するテストカバレッジ
   - 結果を `<record>/verification/phase-check-construction.md` へ書き出す

7. **完了を準備する** — CI と境界の成果物を検証します。ステージや
   フェーズの状態は編集せず、報告されたゲート結果が遷移を所有します。

8. **完了** — 完了メッセージと承認ゲートを提示します。

|--------|------|

| 品質ゲート | マージ／昇格のための品質ゲート定義 |
| CI パイプライン質問 | 回答付きの明確化の質問 |

### 承認ゲート

厳格な 2 選択肢: Approve / Request Changes。

### 注記

- **フェーズ境界検証**: これは構築フェーズの最終ステージです。構築から運用への
  フェーズ境界検証チェック（`stage-protocol-governance.md` のセクション 13）を実行し、
  アーキテクチャがコードへ、コードがテストへ追跡できることを検証します。結果は
  `<record>/verification/phase-check-construction.md` に書き出します。
- **条件付き実行**: プロジェクトに十分な CI パイプラインが既にある場合、この
  ステージはスキップされます。デリバリー計画の実行計画が実行有無を決めます。
- **ユニット後実行**: ステージ 3.6 と同様、すべてのユニット単位作業の完了後に
  一度実行し、ユニット単位ではありません。

<a id="phase-summary"></a>
## フェーズのまとめ

構築フェーズは、段階的な構築フローにより、インセプションの設計を動作する
ソフトウェアへ変換します。

**ユニット単位ステージ（3.1～3.5）:**
- 3.1 機能設計 — ビジネスロジック、ドメインモデル、ルール（アーキテクト主導）
- 3.2 NFR 要件 — パフォーマンス、セキュリティ、スケーラビリティ、信頼性、
  技術スタック（アーキテクト主導）
- 3.3 NFR 設計 — NFR カテゴリ向けの具体パターン（アーキテクト主導）
- 3.4 インフラストラクチャ設計 — デプロイ、サービス、監視、CI／CD
  （AWS プラットフォーム主導）
- 3.5 コード生成 — 計画＋生成の二部構成をサブエージェント経由（開発者主導）

**ユニット後ステージ（3.6～3.7）:**
- 3.6 ビルドとテスト — 手順生成＋失敗診断付きの実際のシェル実行（品質主導）
- 3.7 CI パイプライン — CI 設定＋フェーズ境界検証（パイプライン・デプロイ主導）

**主な特性:**
- ステージ 3.1～3.4 は CONDITIONAL、3.5～3.6 は ALWAYS 実行、3.7 は CONDITIONAL
- 条件付きステージはすべてデリバリー計画の実行計画に従う
- ユニット単位ループにより、次のユニット開始前に 1 ユニットを完了させる
- NFR 成果物は上流リファレンスより粒度を拡張（要件 5 ファイル、設計 5 ファイル）
- インフラストラクチャ設計は監視と CI／CD の専用ファイルを含む 5 成果物へ拡張
- コード生成はコンテキスト予算制御付きで AIDLC 開発者エージェントのサブエージェントを使用
- ビルドとテストは実コマンド実行と自動失敗診断を行う
- CI パイプラインは運用へ移行する前にフェーズ境界検証を含む

**上流リファレンスからの意図的な逸脱:**
- NFR 要件: 5 ファイル（リファレンスの 2 から拡張）
- NFR 設計: 論理コンポーネントを含む 5 ファイル（リファレンスの 2 から拡張）
- インフラストラクチャ設計: 監視設計と CI／CD パイプラインを含む 5 ファイル
  （リファレンスの 2～3 から拡張）
- 計画／質問ファイルをステージ成果物と同じ場所に配置
