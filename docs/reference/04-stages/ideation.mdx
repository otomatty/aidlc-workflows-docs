<a id="phase-overview"></a>
## フェーズの概要

アイデア創出フェーズは、AI-DLC ライフサイクルを構成する 5 つのフェーズのうち 2 番目です。技術作業を開始する前に、意図を捉え、実現可能性を検証し、スコープを定義し、承認を得ることで、取り組み全体の基盤を築きます。このフェーズではステージ 1.1 から 1.7 を実行し、インセプションフェーズへの移行を制御する実行／中止ゲートで完了します。

7 つすべてのステージはインラインで実行され（サブエージェントへの委任はありません）、承認ゲート、質問形式、完了メッセージについて標準の `stage-protocol.md` に従います。オーケストレーターはそれらを順番に処理し、現在のスコープに該当しない CONDITIONAL ステージをスキップします。

**アイデア創出フェーズの主な特性:**

- すべてのステージはインライン実行モード（ユーザーとの直接対話）を使用します。
- 各ステージは、意図のレコードディレクトリにある `<record>/ideation/<stage-name>/` 配下に成果物を生成します。ここで `<record>` は `aidlc/spaces/<space>/intents/<YYMMDD>-<label>/` です（`audit/` のシャードディレクトリ、ステージごとの `memory.md`、検証レポートも同じレコードディレクトリに置かれます）。
- ステージ 1.1 以外のすべてのステージは、先行ステージの出力に依存します。
- ステージ 1.7 は、インセプションへ引き継ぐ前にフェーズ境界検証チェックを実行します。
- このフェーズは 2 つの ALWAYS ステージ（1.1 意図の収集、1.7 承認と引き継ぎ）で区切られます。中間の 5 ステージは CONDITIONAL であり、スコープに応じてスキップされる場合があります。

**スコープに応じたステージの採用:**

| スコープ | 採用するステージ |
|----------|------------------|
| エンタープライズ | 1.1〜1.7 のすべて |
| 機能追加 | 1.1〜1.7 のすべて |
| 最小実用製品 | 1.1、1.3（軽量）、1.4、1.6 |
| 概念実証 | 1.1（最小限） |
| バグ修正 | なし（アイデア創出を完全にスキップ） |
| リファクタリング | なし（アイデア創出を完全にスキップ） |
| インフラ | なし（アイデア創出を完全にスキップ） |
| セキュリティパッチ | なし（アイデア創出を完全にスキップ） |
| ワークショップ | なし（アイデア創出を完全にスキップ） |

<a id="stage-11-intent-capture--framing"></a>
## ステージ 1.1: 意図の収集とフレーミング

<a id="metadata"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | アイデア創出 |
| ステージ番号 | 1.1 |
| 条件 | ALWAYS — すべてのワークフローの最初のステージ。取り組みの基盤を確立します。 |
| 主担当エージェント | AIDLC プロダクトエージェント |
| 支援エージェント | AIDLC アーキテクトエージェント（技術的コンテキスト） |
| モード | インライン |
| 完了絵文字 | 💡 |

<a id="purpose"></a>
### 目的

意図の収集は、すべての AI-DLC ワークフローの入口です。ビジネス上の課題を収集し、ステークホルダーを特定し、成功指標を設定し、プロジェクト種別（グリーンフィールド、ブラウンフィールド、または移行）を分類します。生成される意図ステートメントとステークホルダーマップは、以降のすべてのステージがその上に構築される基盤となります。

ユーザーが `$ARGUMENTS` を通じて自由形式の意図テキストを提供した場合、そのテキストはシードコンテキストとして渡されるため、このステージで「何を作りたいですか？」と再度質問することはありません。

<a id="inputs"></a>
### 入力

- `$ARGUMENTS` または意図の `audit/` シャードから得られるユーザーのプロジェクト説明
- 以前のセッションにおける既存の `<record>/` 成果物（存在する場合）
- `aidlc/spaces/<space>/memory/` にあるガードレール

<a id="steps"></a>
### 手順

1. **エージェントペルソナを読み込む** — AIDLC プロダクトエージェントのペルソナとナレッジを読み込みます。技術的コンテキストの観点から AIDLC アーキテクトエージェントのペルソナも読み込みます。
2. **既存コンテキストを読み込む** — ユーザーのプロジェクト説明を読みます。既存の成果物を確認し、ガードレールを読み込みます。
3. **明確化の質問を生成する** — ビジネス課題、顧客、成功指標、取り組みの契機、プロジェクト種別を扱う質問を含む `<record>/ideation/intent-capture/intent-capture-questions.md` を作成します。A〜E の選択肢に加え X（その他）を持つ `[Answer]:` タグ形式を使用します。3 モードの質問フローを提供します。
4. **回答を収集して分析する** — すべてのタグが入力されたことを確認します。曖昧さ・矛盾の分析を実行します。
5. **成果物を生成する** — 意図ステートメントとステークホルダーマップを作成します。
6. **状態を更新する** — 1.1 を完了済みの `[x]` としてマークします。
7. **完了を提示して承認を求める** — 標準の 2 選択肢ゲートを提示します。

<a id="outputs"></a>
### 出力

| ファイル | 内容 |
|----------|------|
| `intent-statement.md` | 問題ステートメント、対象顧客、成功指標、取り組みの契機、プロジェクト種別、初期スコープシグナル |
| `stakeholder-map.md` | 主なステークホルダーと関心事項、意思決定者と影響者の区別、コミュニケーション要件 |
| `intent-capture-questions.md` | `[Answer]:` タグを持つ明確化の質問（入力成果物） |

<a id="notes"></a>
### 注記

- すべてのワークフローの最初のステージです。ユーザー入力以外の先行成果物はありません。
- `$ARGUMENTS` 内の自由形式の意図はシードコンテキストとして使用されます。
- 意図ステートメントは、後続するすべてのアイデア創出ステージに入力され、インセプションへも引き継がれます。

<a id="stage-17-initiative-approval--handoff"></a>
## ステージ 1.7: 取り組みの承認と引き継ぎ

<a id="metadata-6"></a>
### メタデータ

| 項目 | 値 |
|------|----|
| フェーズ | アイデア創出 |
| ステージ番号 | 1.7 |
| 条件 | ALWAYS — インセプション前の最終アイデア創出ゲート。 |
| 主担当エージェント | AIDLC デリバリーエージェント |
| 支援エージェント | AIDLC プロダクトエージェント（完全性を検証） |
| モード | インライン |
| 完了絵文字 | ✅ |

<a id="purpose-6"></a>
### 目的

すべてのアイデア創出成果物を単一の取り組み概要にまとめ、すべての決定を記録し、フェーズ境界検証を実行して、実行／中止ゲートを提示します。

<a id="inputs-6"></a>
### 入力

ステージ 1.1〜1.6 からのすべてのアイデア創出フェーズ成果物。

<a id="steps-1"></a>
### 手順

1. AIDLC デリバリーエージェントのペルソナとナレッジを読み込みます。
2. すべてのアイデア創出フェーズ成果物を読み取ります。
3. 承認質問を生成します。
4. 取り組み概要（すべての出力を統合した 1 ページ資料）を作成します。
5. フェーズ境界検証を行います（意図 -> スコープ -> 意図バックログの整合性）。
6. 状態を更新し、インセプションフェーズへ移行します。
7. 3 選択肢の承認ゲートを提示します。

<a id="outputs-6"></a>
### 出力

| ファイル | 内容 |
|----------|------|
| `initiative-brief.md` | すべてのアイデア創出出力を統合した 1 ページの要約 |
| `decision-log.md` | アイデア創出中に行われたすべての決定の記録 |
| `approval-handoff-questions.md` | `[Answer]:` タグを持つ承認質問 |

フェーズ境界検証:

| ファイル | 内容 |
|----------|------|
| `<record>/verification/phase-check-ideation.md` | アイデア創出からインセプションへのトレーサビリティチェック |

<a id="approval-gate"></a>
### 承認ゲート

特別な 3 選択肢ゲート:

- **承認** — インセプションフェーズへ進みます。
- **変更を要求** — 修正フィードバックを提供します。
- **取り組みを却下** — ワークフロー全体を終了します。

<a id="notes-6"></a>
### 注記

- フェーズ境界ステージです。ステージプロトコルのガバナンスに従って検証を実行します。
- 取り組み概要は、アイデア創出フェーズ全体のエグゼクティブサマリーとして機能します。

<a id="phase-summary"></a>
## フェーズの要約

<a id="key-outputs"></a>
### 主な出力

1. **意図ステートメント**（1.1）— 問題ステートメント、対象顧客、成功指標、プロジェクト分類。
2. **ステークホルダーマップ**（1.1）— 主なステークホルダー、意思決定者、コミュニケーション要件。
3. **競合分析**（1.2）— 市場での位置付け、内製と購入の比較（該当する場合）。
4. **実現可能性評価と RAID ログ**（1.3）— 技術的実現可能性、リスク登録簿、制約（該当する場合）。
5. **スコープ文書と意図バックログ**（1.4）— 正式なスコープ境界、優先順位を付けた作業単位の原型リスト。
6. **チーム計画**（1.5）— スキルマトリクス、モブ構成、キャパシティ配分（該当する場合）。
7. **コンセプトモックアップ**（1.6）— ワイヤーフレーム／ユーザーフロー、またはシステムコンテキスト図（該当する場合）。
8. **取り組み概要**（1.7）— すべてのアイデア創出出力を統合したエグゼクティブ向け 1 ページ資料。
9. **フェーズ境界検証**（1.7）— トレーサビリティチェックの結果。

<a id="handoff-to-inception"></a>
### インセプションへの引き継ぎ

ステージ 1.7 で承認されると、フレームワークはインセプションフェーズに移行します。インセプションは、ステージ 2.1 のリバースエンジニアリング（ブラウンフィールドプロジェクト向け）、またはステージ 2.3 の要件分析（グリーンフィールドプロジェクト向け）から始まります。
