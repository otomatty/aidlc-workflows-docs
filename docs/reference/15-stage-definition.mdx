この章では、AI-DLC ステージ定義の**ファイル形式**、すなわち YAML フロントマター
契約、本文の 3 区画モデル、およびそれらのソースを `stage-graph.json` へ変換する
コンパイル・パイプラインを説明します。実行時の振る舞い契約（承認ゲート、質問
フロー、状態追跡）を扱う[ステージ・プロトコル](/reference/stage-protocol)を補完します。
本章はステージファイルが*含むもの*について、ステージ・プロトコル章はステージが
*行うこと*についての説明です。

貢献者は形式を理解するためにこれを読みます。ステージファイルを書くまたは編集
するときは、`dist/claude/.claude/aidlc-common/protocols/stage-definition.md` にある
正規の契約を参照してください。このファイルが正式な仕様であり、この章では説明と
「いつ使うか」のガイダンスを補います。

<a id="two-audiences-one-file"></a>
## 2 つの読者、1 つのファイル

各ステージ `.md` ファイルには 2 種類の読者がいます。

- **パーサー**（`lib.ts` の `parseStageFrontmatter`、マイルストーン 7 で出荷）。YAML フロントマターを読み、構造化された `StageEntry` を生成します。本文には触れません。
- ステージを実行する **LLM エージェント**。本文を読み、散文の指示に従って成果物を生成します。フロントマターには触れません。

両方を 1 ファイルに置くことで、貢献者はグラフの辺と実行手順を並べて見られます。
ファイルを分ける（グラフ用の YAML とエージェント用の散文）と、ステージを
レビュー可能にするインラインの可視性が失われます。

---

<a id="why-variant-a3"></a>
## バリアント A3 である理由

この形式は「バリアント A3」と呼ばれます。バージョン 0.3.0 の計画時に検討した、
3 つの記述方式の 3 番目です。

- **分割ファイルより 1 ファイル。** フロントマターと散文を 1 つの `.md` に置くことで、グラフ構造と実行手順をまとめます。ステージを読むレビュアーは両方を見られます。
- **`grep` しやすい。** プレーンテキストです。バイナリー形式も、作成時の YAML と JSON 間の変換もありません。
- **差分を確認しやすい。** フィールド追加、リネーム、本文編集はいずれもコードレビューに明瞭に現れます。

採用しなかった案は、中心グラフ・ファイル（手編集の `stage-graph.json`）と散文だけの
ステージを組み合わせるものでした。散文を編集しながら、ステージがどの成果物を
生成するかを知るインライン可視性を失います。

<a id="authoring-flow"></a>
## 作成フロー

```
┌─────────┐         ┌──────────────────┐         ┌──────────────────┐         ┌──────────────────┐
│ Edit    │  ───→   │ Pre-commit hook  │  ───→   │ stage-graph.json │  ───→   │ loadStageGraph() │
│ stage   │         │ aidlc-graph      │         │ (build artifact, │         │ (runtime,        │
│ .md YAML│         │ compile          │         │  checked in)     │         │  unchanged)      │
└─────────┘         └──────────────────┘         └──────────────────┘         └──────────────────┘
     │                                                                                 ▲
     │                              ┌──────────────────┐                               │
     └────────────────────────────→ │ CI drift check   │ ──── blocks merge on drift ───┘
                                    │ compile --check  │
                                    └──────────────────┘
```

YAML が正です。JSON はビルド成果物です。継続的インテグレーションがこの関係を強制します。

`aidlc-graph compile` と `compile --check` は CLI サブコマンドとして出荷されます
（マイルストーン 9）。ステージ YAML を編集したら手動で `compile` を実行し、
継続的インテグレーションは `compile --check` を強制してドリフトを検出します。
これを自動化するコミット前フックは後のプルリクエストへ先送りします。
`stage-graph.json` はコンパイル済み成果物です。手で編集せず、YAML を編集して
再コンパイルしてください。

<a id="field-reference--when-to-use"></a>
## フィールド・リファレンス — いつ使うか

正規仕様には型と制約を含む完全なフィールド表があります。この節では、判断を
要するフィールドについて説明を加えます。

<a id="requires_stage"></a>
### `requires_stage`

依存関係の辺をコード化します。役割は 2 つです。

1. **意味上のデータ依存。** 「ステージ Y が生成する成果物 X を消費する」→ `requires_stage` に `Y` を追加します。
2. **表示順の辺。** 同じフェーズ内で意味上の依存はないが順序が固定の 2 ステージ（例: 着想フェーズの `market-research` は `feasibility` より前）。計算される `display_order` を安定させるため弱い辺を追加します。

コンパイル時の `slug` アルファベット順のタイブレークは安全網です。特定の順で
配置しなければならないステージは、アルファベット順の偶然に頼らず明示的に辺を
記述してください。

<a id="for_each"></a>
### `for_each`

インスタンスが反復を駆動する成果物を指定します。ステージはインスタンスごとに
一度実行されます。

現在のユースケース: 5 つの構築ステージ（`functional-design`、
`nfr-requirements`、`nfr-design`、`infrastructure-design`、`code-generation`）
はユニットごとに一度実行されます。それぞれが
`for_each: unit-of-work`（`units-generation` が生成する成果物）を宣言します。

将来のユースケース: 環境ごと、テナントごと、リージョンごと、コンプライアンス
管轄ごとに実行するステージです。このプリミティブはワークフロー・エンジン一般の
ものであり、たまたま構築フェーズが最初に利用します。

**集約は宣言せず推論されます。** `for_each` ステージが生成する成果物を消費し、
自身の `for_each` を宣言しないステージは、定義上の集約ステップです。
`build-and-test` が典型例です。5 つの構築 `for_each` ステージが全ユニットを
反復したあとに一度だけ実行され、集約された出力を消費します。明示的な
`fan_in` や集約フィールドは不要であり、グラフ走査が判断します。

<a id="workspace_requires"></a>
### `workspace_requires`

ブール値で、既定は `false` です。意図ごとの記録ディレクトリに計画文書を置くだけ
ではなく、**ワークスペースルートへソースコードを書かなければならない**ステージで
`true` に設定します。

存在する理由: ステージの `produces[]` 成果物は常に記録ディレクトリ内の Markdown に
解決されます（パス解決器が書き込む唯一の場所）。そのため「`produces` は存在するか？」
という検査は、`code-generation-plan.md` と `code-summary.md` を書いただけで
実際のコードを 1 行も出さない `code-generation` ステージでも満たされてしまいます
（課題 #366）。`workspace_requires: true` がその穴を閉じます。ステージ完了時の
成果物ガード（`aidlc-state.ts` の `approve` / `advance` / `finalize` /
`complete-workflow`）は、ステージ完了前に `aidlc/` ワークスペースツリーと
ハーネスディレクトリの外で実ソース作業の証拠を追加で要求します。

「ソース作業」の検出方法はワークスペースによって異なります。
- **Git ワークスペース** - ガードは Git に問い合わせるため、このセッションのコードと
  ブラウンフィールドリポジトリの既存 `src/` を区別できます。未コミットまたは未追跡の
  非ドキュメント変更（`git status --porcelain`）があるか、直近コミットが非ドキュメント
  パスに触れている（`git diff --name-only HEAD~1 HEAD`）ときに合格します。後者の条項により、
  コミットしてから承認する流れ（作業ツリーはクリーンで、コードは直近コミット内）でも
  合格し、#366 の第 3 更新でのクリーンツリー誤ブロックを閉じます。
- **非 Git ワークスペース**（または任意の Git エラー）- ガードはシェルを使わない
  ファイルシステム存在チェックにフォールバックします。`aidlc/` ワークスペースツリーと
  ハーネスディレクトリの外に、少なくとも 1 ファイルが存在する必要があります。

今日これを宣言するのは `code-generation` だけです（本文がワークスペースルートへ
アプリケーションコードを書く唯一のステージです）。コードや設定を出力する独自ステージ
（契約生成器、インフラコード実行器など）を追加するチームは、同じガードが適用されるよう
`workspace_requires: true` を設定してください。継続的インテグレーションでは
`AIDLC_SKIP_ARTIFACT_GUARD=1` でバイパスできます。

<a id="produces_kinds"></a>
### `produces_kinds`

`for_each: unit-of-work` ステージ上の任意マップです。各キーはステージの
`produces` または `optional_produces` 成果物名、各値はその成果物が適用される
ユニットの**種別**一覧です。種別は `units-generation` のエッジブロックでユニットごとに
宣言され（[ランタイムグラフ](/reference/runtime-graph) の
`bolt_dag.units[].kind` を参照）、次のいずれかです。
`service | spec | ui | packaging | library`。

```yaml
produces:
  - performance-requirements
  - security-requirements
  - scalability-requirements
produces_kinds:
  performance-requirements: [service, ui]
  scalability-requirements: [service]
```

存在する理由: 4 つの構築設計ステージは、固定の `produces` 一覧を全ユニットに
同一適用していたため、仕様ユニットにもスケーラビリティ文書が、パッケージング
ユニットにもビジネスロジックモデルが課され、人手で N／A スタブを書く必要が
ありました。`produces_kinds` によりエンジンはユニットごとに行列を刈り込めます。
ユニットが `kind` を持つとき、エンジンはその種別一覧に含まれる `produces`
エントリーだけを残します。マップに**載っていない**成果物は全種別に適用されます
（種別固有のものだけ注釈してください）。種別が**ない**ユニット、または
`produces_kinds` マップ自体がないステージは全行列を保持します。したがって既存の
すべてのワークフローに対して不活性です。

刈り込みは対称です。`run-stage` ディレクティブの `produces` パス
（指揮者が書くもの）とユニット単位のカバレッジ検査（承認パスのガードが要求するもの）
の両方をフィルタします。`optional_produces` とも合成されます。ディレクティブのパスは
両リストの種別フィルタ済み和集合であり、カバレッジは必須の `produces` だけを
キーにします（任意成果物は適用されない種別ではディレクティブから刈られ、いずれにせよ
カバレッジ免除のままです）。必須集合が**空**に刈られるユニットは定義上カバー済み
です。ステージが適用されないためです。全ユニットが空に刈られるユニット単位
ステージは、成果物ガードでデッドロックせずノーオペレーションとして承認されます。
4 ステージの既定種別行列はステージフロントマターのデータであり、エントリーごとに
レビューと差し戻しが可能です。誤ったエントリーを削除すると、その成果物の全行列が
復元されます。

信頼に関する注意: `kind:` 値は `units-generation` ゲートで列挙検査されます
（`required-sections` センサーは誤記で大きく失敗します）が、その後のコンパイル済み
ランタイムグラフは信頼されます。エンジンは `bolt_dag.units[].kind` を読むときに
種別の形状だけを検査します（コンパイル済みバッチを信頼するのと同じです）。
コンパイル済みグラフで手編集され、妥当だが誤った種別にされたユニットは、その誤った
種別の集合へ黙って刈られます。

<a id="consumesrequired"></a>
### `consumes[].required`

消費エントリーごとのブール値です。意味上は**アクティブな計画にスコープ**され、
成果物がどこかに常に存在する、という大域アサーションではありません。

> `required: true` は「生成ステージがアクティブな計画で走るなら、この消費は
> 満たされなければならない」を意味します。「生成側が常に走る」意味では
> **ありません**。スコープが生成側を除外するとき（例: `bugfix` が
> `units-generation` をスキップ）、その生成側成果物に対するすべての
> `required: true` 消費は無意味になります。要求すべきものがありません。

**スコープ付き読みの理由。** `all` 実行のスコープ（`enterprise`、`feature`、
`workshop`）以外は、意図的に上流ステージをスキップします。平坦で大域的な
`required: true` はそれらのスコープを構造的に無効にし、それは誤りです。正当な
運用モードだからです。本当の契約は条件付きです。「上流が走るなら、下流に渡せ」。
ステージ本文は既に不在ケースを穏当に扱います（「利用可能なら」などの散文指示や、
コンテキストからのフォールバック）。

**診断リントへの意味。** リントは各アクティブ・スコープを歩き、「ステージ X の
成果物 Y に対する `required: true` 消費は、このスコープで Y の生成側が `SKIP`
のため無意味」と報告します。助言であり、ブロックしません。ユーザーはスコープ選択で
既に短縮を選んでいます。

**バージョン 0.10.0 が追加するもの。** 予約済みの `when:` プリミティブ
（下記の「予約」節を参照）により、作者はより豊かな述語を書けます。
`when: producer-in-plan`、`when: mode == brownfield`、
`when: scope != poc`。今日の `required: true` と
`conditional_on: brownfield`（およびグリーンフィールド条件）の組が
バージョン 0.3.0 に必要な 2 次元をカバーし、`when:` がそれを一般化します。

<a id="consumesconditional_on"></a>
### `consumes[].conditional_on`

ブラウンフィールド／グリーンフィールドの分岐を捉えます。例:
`reverse-engineering` はブラウンフィールド・モードでのみ成果物を生成します。
それらを消費するステージは、スコープ解決器へ「ブラウンフィールドのときだけ
この消費が必要」と伝えるため、消費に `conditional_on: brownfield` を付けます。

無条件の消費では、**フィールドを完全に省略**します。`always` 値はなく、
無条件の消費は単に `conditional_on` キーを持ちません。

<a id="optional_produces"></a>
### `optional_produces`

`produces:` と並ぶ、プレーンなケバブケース文字列リストです。ステージがユニットごとに
**書いてよい**が**必須ではない**成果物を名付けます。不在はなしを意味します。
必要な 2 ステージだけが宣言するため、コンパイル済み `stage-graph.json` は
最小のままです。

存在する理由: ユニット単位の構築ステージ（`for_each: unit-of-work`）は、その
ユニットの記録ディレクトリ上にすべての `produces[]` 成果物があるときだけ、その
ユニットについてカバー済みになります（`aidlc-orchestrate.ts` 内のユニット単位
カバレッジ検査）。一部の成果物はユニットに対して本当に条件付きです。
`functional-design` はユニットに UI があるときだけ `frontend-components` を書き、
`infrastructure-design` はユニットがインフラを共有するときだけ
`shared-infrastructure` を書きます。それらを `produces:` に載せると、バックエンド
専用ユニットでもカバレッジを満たすため N／A スタブを書かされ、書くまでステージ
ゲートに到達できませんでした。`optional_produces:` へ移すと免除されます。

- **カバレッジ免除。** `optional_produces` エントリーはユニット単位カバレッジ
  ループで無視されます。ユニットは**必須**の `produces[]` 成果物が存在する時点で
  カバー済みです。任意のものは `next` の進行も `approve` のコミットも妨げません。
- **指揮者には依然として解決される。** `run-stage` ディレクティブの `produces`
  パスは `produces` + `optional_produces` の和です。ユニットが条件付き成果物を
  **実際に**書くとき、指揮者は配置先を知ったままです。ステージが
  `produces_kinds` も宣言している場合、その和は解決前に種別フィルタされます
  （上記の `produces_kinds` を参照）。任意成果物が適用されない種別にはパスが
  見えません。
- **語彙にも残る。** `artifactsRegistry()` と `producersOf()` は両リストを和集合
  にするため、成果物名とその生成ステージは登録されたままです。

**対の規約。** すべての `optional_produces` エントリーは、ステージ本文の散文
（および `outputs:` 文字列）に、いつ書くかをエージェントへ伝える
`(CONDITIONAL - ...)` マーカーを**必ず**持たなければなりません。フロントマター
キーはエンジンのカバレッジビュー、散文はエージェントへの指示です。同期を保って
ください。

**警告。** `optional_produces` と印された成果物はユニット単位カバレッジ台帳から
見えません。エンジンはユニットがそれを生成したことを証明できません。ユニットに
対して正当に条件付きの成果物にだけ使い、ステージがユニットごとに常に書くべき
成果物のカバレッジを静かに緩める用途には使わないでください。

<a id="mode"></a>
### `mode`

ディスパッチ機構で、値は 3 つです。

- `inline` — 指揮者が自身のコンテキストでステージを実行します。短いステージで、
  実行が速く、コンテキスト圧迫がありません。
- `subagent` — `Task` ツール経由で新しいサブエージェント・コンテキストへ委譲します。
  メイン指揮者のコンテキストを溢れさせる長いステージ（構築のコード生成）向けです。
- `agent-team` — **予約済み**。実験的な
  `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` プリミティブが安定し、エージェント間の
  直接メッセージングが必要になったとき用です。最初の消費者はバージョン 0.8.0 の
  反復ループ・ドライバーになる見込みです。バージョン 0.3.0 ではどのステージも
  `agent-team` を宣言しません。

注: 今日のマルチエージェント実行は `support_agents` で表現されます。指揮者は
まずリードエージェントを呼び、続けて各サポーターをリードの出力をコンテキストとして
順に呼びます（`dist/claude/.claude/aidlc-common/protocols/stage-protocol.md:611`
を参照）。エージェント同士は呼び合いません。委譲するのは指揮者だけです。
`agent-team` は直接のエージェント間メッセージングが必要な将来ケース専用であり、
一般の「複数エージェントがステージに関与する」ケースではありません。

**消費者契約。** `mode` フィールドを読むオーケストレーターコードは
`agent-team` を明示的に扱わなければなりません。最低でも
「`agent-team` モードは未実装」を送出してください。既定の実行パスへ
フォールスルーしてはいけません。列挙拡張時の黙ったフォールスルーは既知の落とし穴です。

<a id="lead_agent-and-support_agents"></a>
### `lead_agent` と `support_agents`

リードエージェントがステージを所有します。リードのペルソナ（スキル、ナレッジ、
ツール許可リスト）はステージ開始時に読み込まれます。サポートエージェントは視点を
足します。要件作業では `aidlc-product-agent` をリードにし、容量の現実確認のため
`aidlc-delivery-agent` をサポートとして読む、といった具合です。

両フィールドは `loadAgents()`（マイルストーン 3 で導入）経由で
`.claude/agents/*.md` に対して動的検証されます。`aidlc-graph.ts compile` は
発見したエージェント `slug` を `validateStageFrontmatter` へ渡すため、一致する
ファイルがないエージェント名を持つ `lead_agent` または `support_agents` は、
実行時に未登録サブエージェントの `Task` エラーとして現れるのではなく、コンパイル時に
大きく失敗します（`lead_agent "<name>" has no matching .claude/agents/*.md`）。
唯一の例外は予約済みの `orchestrator` 擬似エージェント（指揮者自身で、ブートストラップ
初期化ステージの `lead_agent` として名付けられます）。設計上エージェントファイルは
ありません。スキーマにハードコード列挙はなく、エージェント追加は必要なフロントマター付き
`.md` ファイルを `.claude/agents/` に置くことです。
[貢献: エージェントの追加](/reference/contributing#adding-an-agent)を参照してください。

<a id="reviewer-and-reviewer_max_iterations"></a>
### `reviewer` と `reviewer_max_iterations`

任意です。`reviewer` は、ステージ本文が成果物を生成したあと、承認ゲートの前に
呼ばれる品質ゲート・エージェントを名付けます（[ステージ・プロトコル](/reference/stage-protocol)
を参照）。今日出荷されるレビュアーは 2 つ —
`aidlc-product-lead-agent` と `aidlc-architecture-reviewer-agent` — であり、
コンパイルは `lead_agent` と同じく、発見済みエージェント名簿に対して値を検証します。

`reviewer_max_iterations` は、未解決の指摘を抱えたままゲートへ進む前の
レビュー／改訂ループの上限です。`reviewer` が宣言されて上限が無いとき
**既定は 2** です。コンパイラは欠落または非正の値を 2 へ強制します。
`reviewer` を宣言しないステージではフィールドを省略してください。コンパイラは
`reviewer` なしの `reviewer_max_iterations` を拒否します（スキーマエラー
`reviewer_max_iterations requires a reviewer` がグラフコンパイルを失敗させる）ので、
黙って無視されることはありません。

<a id="relationship-to-agent-frontmatter"></a>
## エージェント・フロントマターとの関係

ステージとエージェントは同じ YAML 優先の規律に従います。エージェント・フロントマター
（[エージェント・システム](/reference/agent-system#frontmatter-contract)を参照）は
*誰が* — エージェント名、許可ツール、階層 — を宣言します。ステージ・フロントマターは
*何を* — ステージが生成・消費する成果物、委譲先エージェント、実行方法 — を宣言します。

両形式とも:

- 各領域の権威あるソースです（並行するハードコード写像はありません）。
- 型付き構造を返す `loadX()` ヘルパーと共に出荷されます。
- ハードコード列挙ではなく、ファイルシステムに対して動的に検証します。

新しいステージの追加は新しいエージェントの追加と同じ形です。`.md` ファイルを置き、
必要なフロントマターを追加すれば、ヘルパーが実行時に拾います。

<a id="worked-example"></a>
## 実例

正規の例は `scope-definition` です。規範的な YAML ブロックは
`dist/claude/.claude/aidlc-common/protocols/stage-definition.md` にあります。
ここへ複製せず、そちらを参照してください。

この例は、今日の散文が述べる内容を構造化してコード化しています。

- `requires_stage: [intent-capture]` は、散文指示「意図の記録ディレクトリ配下の
  `ideation/intent-capture/` から意図表明を読む」をコード化します。パーサーは
  散文を気にせずグラフの辺だけを見ますが、人間の読者は両者を同期させておくべきです。
- `consumes: [{artifact: intent-statement, required: true}]` は、
  `intent-statement` が存在するまでこのステージがブロックされることを示します。
  スコープの解決器が `intent-statement` の生成側を見つけられない場合、診断の
  欠落生成側検査が失敗します。
- `produces: [scope-document, intent-backlog, scope-definition-questions]`
  は前方の辺です。「誰が `scope-document` を生成するか？」を探す他ステージは
  `aidlc-graph.ts producersOf()` 経由でこれを見つけます。
- `for_each` フィールドなし — `scope-definition` はワークフローごとに一度実行されます。

<a id="three-compartment-body-model"></a>
## 本文の 3 区画モデル

ステージファイルの本文には、この順で宣言される 3 区画があります。
バージョン 0.3.0 で埋まっているのは `## Steps` だけです。

| 区画 | バージョン 0.3.0 | バージョン 0.5.0 | ここに入るもの |
|-------------|--------|--------|----------------|
| `## Steps` | 必須、記入済み | 変更なし | エージェントが従う命令形の散文 |
| `## Sensors` \| 予約、不在 \| 記入済み \| 決定論的センサー・バインディング（フラットな `.claude/sensors/` レジストリの ID） |
| `## Learn` | 予約、不在 | 記入済み | ループ駆動バインディングと観測ルール |

バージョン 0.3.0 で 3 区画を事前宣言したため、バージョン 0.5.0 の追加は本文再構成ではなく
差し込み変更でした。`## Sensors` のバインディング意味論とプルインポートモデルは
[センサー・システム](/reference/sensor-system)を参照してください。

**マイルストーン 8 の移行規則:** 既存本文を `## Steps` の下に包むだけです。
大半のステージファイルは既に最初の本文見出しとして `## Steps` を使っています。

<a id="yaml-migration--shipped"></a>
## YAML 移行 — 出荷済み

マイルストーン 7 は `lib.ts` に `parseStageFrontmatter` と `emitStageFrontmatter` を
出荷しました。YAML のみで、散文の後方互換パスはありません。マイルストーン 8 は
31 の全ステージファイルを単一の原子的変更で YAML フロントマターへ移行しました。
マイルストーン 9 は `aidlc-graph.ts` を拡張し、YAML を `stage-graph.json` へ
コンパイルし、継続的インテグレーションのドリフトガードとして `compile --check` を
追加しました。クリーンなツリーで `bun` により `aidlc-graph.ts` の
`compile --check` を実行すると終了コード 0、JSON を再コンパイルせずにステージ YAML を
編集すると、明確なメッセージと共に終了コード 1 になります。

<a id="known-limits-in-v030"></a>
## バージョン 0.3.0 での既知の制限

- **`for_each` は新しい。** `**Per-Unit**: Yes` を持つ 5 つの構築ステージは
  `for_each: unit-of-work` へ移行し、他の 26 ステージはフィールドを完全に省略します。
- **`## Sensors` / `## Learn` 区画は宣言済みだが空。** パーサーは不在を許容します。バージョン 0.5.0 が
  埋めました（[センサー・システム](/reference/sensor-system)を参照）。
- **ドリフト検査を超える実行時検証はない。** パーサーは妥当な `StageEntry` を生む
  任意の YAML を受け入れます。その後の診断拡張が、助言的なルール／センサー検査を上に足します。

<a id="future-extensions--reserved-namespace"></a>
## 将来の拡張 — 予約名前空間

仕様は、後続リリースで AI-DLC が追加しそうなプリミティブの名前を予約します。
スキーマは未知キーを拒否します。ここでの予約は、将来の貢献が場当たり的な追加と
衝突するのを防ぎます。

| キー | 想定リリース | 役割 |
|-----|----------------|-----------------|
| `when` \| バージョン 0.10.0 適合コンパイラ \| 構造化条件。`condition` 散文を機械強制可能な論理へコンパイル。`consumes[].conditional_on` を置き換え、今日のスコープ認識 `consumes[].required` をより豊かな述語（`producer-in-plan`、`mode == brownfield`、`scope != poc`）で一般化 |
| `on_failure` \| バージョン 0.8.0 反復ループ \| 宣言的エラー回復 — 「このステージが失敗したら X へ戻る」または「調整した入力で再試行」。改訂意味論を `stage-protocol-recovery.md` の散文から移す |
| `blocks_on` \| バージョン 0.4.0 構築（表面化する場合） \| データ読みなしの完了依存 — 今日過負荷の `requires_stage`（「あなたの出力を消費する」と「あなたの後に走る」を混同）を分割 |
| `timeout` | バージョン 0.5.0 センサー・バインディング | 実行予算（期限）。ステージ・フロントマターではなくセンサー・バインディングに置く |
| `retry` | バージョン 0.8.0 反復ループ | 失敗時の再試行方針。ステージ・フロントマターではなくループ設定に置く |

設計根拠: Claude Code 自身のタスク・プリミティブ（`TaskCreate` 系に加え
`/loop` と定期実行）は依存、ブロック、再試行、タイムアウトを省略し、多段階
オーケストレーションはすべてクライアント側コードへ押し出します。この実装は、
実行振る舞い（再試行、タイムアウト、失敗処理）をステージ仕様ではなくループと
センサーのサブシステムに置くことで、その選択を映します。上記フィールドは、
消費者が現れたときの控えめな構造拡張であり、新しいパラダイムではありません。

予約名前空間のパターンは監査分類（[状態機械](/reference/state-machine)）に先例が
あります。イベント名を出力元セル `Reserved (v0.x PR N)` 付きで事前登録します。
名前はレジストリに存在しますが、消費者のプルリクエストが出荷されるまでコードは
出力せず、その時点の同一コミットが `Reserved` マーカーを実出力パスへ置き換えます。
