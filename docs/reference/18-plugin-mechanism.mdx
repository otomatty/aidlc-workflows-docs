<a id="1-what-a-plugin-is"></a>
## 1. プラグインとは何か

プラグインは、宣言的マニフェストとコア形状のサブツリーを持つディレクトリ（かつ Git リポジトリ）です。次のことができます。

- 新しいステージ（独自の表示番号範囲）、エージェント、スコープ、メソッド/ルール（スペースメモリのシードへ）、センサーを**追加**する。
- コントリビューションシーム（§6）を通じ、既存のコアステージを**加法的に変更**する。すなわち、ステージを編集せず、生成物、消費物、検証、指示を充実させる。

AIDLC チームが提供するファーストパーティプラグインと、それ以外のサードパーティプラグインは、**機構上は同一**です。構造、シーム、コンポーザー、ホストへのインストール経路は同じです。違いは来歴、すなわちプラグインのリポジトリ所有者とレビュー者だけです。`plugins/test-pro/` が参照フィクスチャです。

この機構の**設計原則**:

- **厳密な加法性、上書きはしない。** コントリビューションは*追加*のみを行います。構造サーフェスの集合和は可換であり、独立した作成者が安全に作業できる理由です。真の競合はプラグインの出所を含む合成エラーとなり、暗黙に後勝ちになることはありません。
- **コアの不変性。** プラグインが `core/` のファイルを編集することはなく、プラグインが提供するすべては自身のツリー内にあります。
- **可逆かつ無効時は不活性。** プラグインを無効にすると、それがないインストールと完全に同じ状態へ再合成されます。
- **スラッグが識別子。** ステージは重要なすべての場所（エッジ、ジャンプ、解決）でスラッグにより識別されます。`number` は表示/順序専用なので、プラグインステージを挿入してもコアの再採番や不安定化は起きません。

<a id="2-why-install-time-delivered-as-a-host-plugin"></a>
## 2. なぜインストール時にホストプラグインとして提供するのか

繰り返し検討された設計上の問いは、プラグインを*ビルド時*の成果物（中央で事前合成してコピーする）にするか、*インストール時*の成果物（利用者が選んだセットを利用者のマシンで合成する）にするかでした。この機構は**インストール時**を選びます。エコシステム全体（エヌピーエム、カーゴ、ヘルム、VS コード、ニックス）がすでに次の二つの構造的理由からこの形に収束しているためです。

- **組み合わせ爆発。** N 個のプラグインは最大 2ⁿ 個の有効化部分集合を生みます。中央ビルドではすべての組み合わせを事前合成できません。事前ビルドする価値がある唯一の成果物は、誰にとっても同一の**ベアコア**（空集合）です。
- **遅延解決。** 正しいプラグインセットと、同じステージへの二つのプラグインのコントリビューションをどうマージするかは、それらを選んだインストール時にしか分かりません。

配布媒体は独自の AIDLC インストーラーではなく、**ホスト自身のプラグインシステム**です。フレームワークが対象とするすべてのハーネスは、すでにマニフェスト優先かつ Git 配布のプラグインモデル（Claude の `.claude-plugin/`、Codex の `.codex-plugin/`）を備えています。AIDLC プラグインはその一つです。これは**ハイブリッド**です。ストアがあるホスト（Claude、Codex）では実プラグイン、ストアがない Kiro ではフォルダードロップを使います。結果は次のとおりです。

- **配布インフラは運用しない。** 顧客が自らのプラグインリポジトリ（Git、セマンティックバージョニングのタグ、`marketplace.json`）をホストします。一つのマーケットプレイスエントリで、混在する環境向けのプラグインを一覧できます。
- **信頼はホストネイティブ。** 組織の制限にはホストの管理対象許可リストを利用します（Claude の `strictKnownMarketplaces` はユーザーによる上書き不可、Codex はコンテンツハッシュ固定の信頼）。AIDLC 独自の信頼レイヤーは作りません。
- **コンポーザーはホストフックによってインストール時に実行される。** 組み合わせごとの事前ビルド済みツリーはなく、セッション開始フックが選択セットをローカルで合成します。

> **セキュリティに関する注記 — Kiro のフォルダードロップにはインストール時の信頼ゲートがありません。** Claude と Codex は、プラグインのフックが実行される*前に*、それぞれの信頼プロンプト（管理対象マーケットプレイス / ハッシュ固定の承認）を通してプラグインを仲介します。Kiro にはプラグインストアがないため、フォルダードロップ経路では、次のプロンプトで `compose.ts` を実行する `.kiro.hook` を含むプラグインファイルが、**同等のゲートなしに**コピーされます。ツリーをドロップすること自体が信頼の決定です。Kiro プラグインのドロップは `git clone && run` と同様に扱ってください。コードを実行してもよいソースからのみインストールし、ドロップによる差分をレビューし、追従するブランチではなくレビュー済みのタグにプラグインリポジトリを固定してください。コンポーザー自体は加法的で `core/` を編集しませんが、インストールするフックはシェルの権限で実行されます。

コントリビューションシーム（§6）が重要である理由はここにあります。これは構造的には VS コードの `contributes` とカーゴの加法的な機能集合和であり、この分野で最も合成しやすいモデルです。ファーストパーティ、サードパーティを問わず、ゲートキーピングなしに*すべての*プラグインが利用できます。

<a id="3-plugin-structure-and-manifest"></a>
## 3. プラグインの構造とマニフェスト

プラグインツリーは `core/` の形を反映するため、パッケージャーはこれをすべてのハーネスへ投影できます。一度だけ作成すればハーネスに依存しません。

ツリーは*設計上の*サーフェスとして `core/` の完全な形を反映しています。✅ はパッケージャーが現在投影するサブツリー、⏳ は設計済みながら未投影のサブツリー（§7）を表します。

```text
plugins/<name>/
  .aidlc-plugin/plugin.json              # the manifest
  stages/<phase>/<slug>.md               # ✅ NEW stages (slug identity; number is display-only)
  sensors/aidlc-<id>.md                  # ✅ NEW sensor manifests
  tools/<id>.ts                          # ✅ sensor scripts (so a sensor can run)
  contributions/<phase>/<slug>.md        # ✅ ADDITIVE modifications to core stages (§6)
  agents/<plugin>-<role>-agent.md        # ✅ NEW agents (stem == frontmatter name)
  scopes/<plugin>-<name>.md              # ✅ NEW scopes (stem == frontmatter name)
  knowledge/<agent-slug>/…               # ✅ per-agent METHODOLOGY knowledge
  tests/                                 # the plugin's own content validation (integration tier)
  memory/{org,team,project}.md           # ⏳ method/rule additions → default-space seed (§7)
  memory/phases/<phase>.md               # ⏳
```

コアのスコープファイルはファイル名接頭辞 `aidlc-` を使います。プラグインのスコープファイルはこれをプラグイン接頭辞に置き換え（`<plugin>-<name>.md`）、ファイル名の語幹はフロントマターの `name` と一致します。プラグインのエージェントファイルも同じルールに従います。すなわち `<plugin>-<role>-agent.md` で、フロントマターの `name` は語幹と等しくなります。

`.aidlc-plugin/plugin.json` は**宣言的な**マニフェストです。トップレベルは一般的なホストプラグインマニフェストの形を反映しており（マーケットプレイスまたはホストツールが一覧、バージョン、信頼を扱えるように）、AIDLC 固有の設定はネストされた `aidlc` ブロックに分離されます。

```jsonc
{
  "name": "test-pro",                 // == dir name; "core", "aidlc", and "aidlc-*" are reserved; kebab-case
  "version": "0.1.0",                 // semver; checked against dependents' constraints
  "description": "…",
  "author": { "name": "AWS AIDLC" },
  "dependencies": ["core", "compliance@^1.2.0"],  // resolved vs git tags (<plugin>--v<version>)
  "aidlc": {
    "contributes": {                  // which subtrees this plugin ships
      "stages": "stages/", "agents": "agents/", "scopes": "scopes/",
      "memory": "memory/", "sensors": "sensors/", "knowledge": "knowledge/",
      "tools": "tools/", "overlays": "contributions/"
    }
  }
}
```

コントリビューションパスはプラグイン相対であり、プラグインルートの外へ出てはいけません。トップレベルは**寛容**です（前方互換性とツール間の許容性のため、未知のキーを保持します）。`aidlc` ブロックは*設計上* **厳格**です（作成時のタイプミスを捉えるため、未知のキーを拒否します）。ただし現在のパッケージャーはディレクトリ慣例（`stages/`、`sensors/`、`tools/`、`contributions/`、`scopes/`、`agents/`、`knowledge/`）でコンテンツを発見しており、`aidlc.contributes` ブロックの読み取りも厳格性の強制もまだ行いません。ステージ番号は表示専用であるため、プラグインはマニフェスト内で番号範囲を要求しません。`overlays` はコントリビューションディレクトリ（§6）の意図した名称で、コピーではなくマージ時に消費されます。

<a id="4-composition-model"></a>
## 4. 合成モデル

コンポーザーは「ベアコア + 選択したプラグイン」に対して一度実行され、有効なインストールを書き出します。起動方法にかかわらず、**同じコンポーザー**が実行されます。

| ホスト | トリガー | 信頼 |
|------|---------|-------|
| **Claude** | セッション開始フック（セッション生成時に即時実行） | 管理対象許可リスト（`strictKnownMarketplaces`） |
| **Codex** | セッション開始フック（初回の対話時に遅延実行） | 一回限りの信頼プロンプト、コンテンツハッシュ固定 |
| **Kiro**（CLI/IDE） | フォルダードロップ後に `bun <plugin>/hooks/compose.ts` を手動実行（§9 を参照。`.kiro.hook` の自動実行と `aidlc plugin compose` CLI は未接続） | 該当なし — フォルダードロップ配布 |

手順（トリガーにかかわらず同一）:

1. **解決** — 選択したプラグインとその推移的な `dependencies` 閉包を、公開済みバージョンに対して解決します。
2. **新しいプリミティブをコピー** — 各プラグインの `stages`/`agents`/`scopes`/`knowledge`/`sensors`/`tools` サブツリーを対応するハーネスルートへコピーし、`{{HARNESS_DIR}}` トークンをハーネスの実ディレクトリへ置換します。`memory` は引き続き延期されています（§7）。
3. **コントリビューションをマージ** — ステージへの有効な各コントリビューションを対象ステージのソースへ畳み込みます（§6）。構造サーフェスは集合和を取り、プローズフラグメントはアンカー位置に挿入します。
4. **コンパイル** — `aidlc-graph compile` が `stage-graph.json` と `scope-grid.json` を再生成します。オーケストレーターは完全にこれらに基づいてルーティングするため、プラグインステージは合成された瞬間に実行対象となり、プローズやスキルの編集は不要です。

合成は独立したオーバーレイの連続ではなく一つの多方向マージなので、**同じステージにコントリビュートする二つのプラグインは真にマージされます**。構造上の追加は集合和を取り、プローズフラグメントは決定的な順序になります。一方がもう一方を黙って上書きすることはありません。合成に関してランタイムは**読み取り専用**のままです。マージはセッションごとではなく、すべて合成時に起きます。マージ対象はコンパイル済み JSON ではなく**ステージソース**であるため、その後の `aidlc-graph compile`（たとえばランタイムコンパイルフック）をまたいでも**永続的**であり、**冪等**です。各セッション開始時に再実行しても、新たに合成されるものはありません。

<a id="5-selection"></a>
## 5. 選択

プラグインは追加し、インストールが選択します。プラグインを合成すると、そのファイルがインストールへコピーされ、加法的なコントリビューションがマージされますが、そのインストールの利用者に見えるのは `<harness-dir>/tools/data/harness.json` に名前が挙がったプラグインだけです。

```json
{
  "harnessDir": ".claude",
  "rulesSubdir": "rules",
  "plugins": ["aidlc", "test-pro"]
}
```

`plugins` キーは任意です。存在しない場合はインストール済みのすべてのプラグインが有効になり、既存のインストールが維持され、出荷されるコアはバイト単位で同一に保たれます。存在する場合、そのリストが有効セットです。`aidlc` は暗黙のコアプラグインであり、これを省略するとコアのステージ/スコープ/ランナーが無効になります。ただしファイルはインストールされたまま残り、再度有効化できます。三つの Initialization ステージは例外です。ブートストラップにはプラグインの帰属がないため、これらのステージは有効なすべてのスコープで常に有効です。

選択の確認や変更には、決定的なユーティリティコマンドを使います。

```bash
bun <harness-dir>/tools/aidlc-utility.ts select-plugins
bun <harness-dir>/tools/aidlc-utility.ts select-plugins test-pro
bun <harness-dir>/tools/aidlc-utility.ts select-plugins aidlc,test-pro
```

`select-plugins` は、既知のセット（`aidlc` に加え、コンパイル済みノードとスコープファイルに見つかるプラグイン名）に対して名前を検証し、`harness.json` を書き込み、グラフを再コンパイルし、ステージ/スコープランナーを再生成し、生成される SKILL.md のスコープ/ステージ表を一つのトランザクションで更新します。`harness.json`、`stage-graph.json`、`scope-grid.json` のスナップショットを取り、後段の再生成ステップが失敗した場合は三つすべてを復元し、復元された選択に対して再生成チェーンを再実行するため、インストールが中途半端な状態のまま残ることはありません。`/aidlc --doctor` は有効なプラグインとプラグインごとの有効ステージ数を報告し、グラフの `enabled:false` フラグが `harness.json` と食い違う場合はハードフェイルします。

コンパイル済みの `stage-graph.json` は、インストール済みの全ステージセットを保持します。無効なノードは `"enabled": false` を持ち、有効なノードはこのキーを省略します。ランタイムローダーが無効なノードをフィルタリングするため、ランナー、状態行、スコープ表、オーケストレーションには選択されたグラフだけが見えます。`loadStageGraphAll()` はドクターと選択ツール専用に予約されています。ステージ番号は完全なグラフから割り当てられるため、プラグインを無効化して後で再有効化しても番号は正確に保たれます。選択フィルターが対象とするのはステージ、スコープ、ランナーです。無効なプラグインの `agents/` と `knowledge/` のファイルはディスク上に残り、かつ読み込み可能なままです（エージェント名簿とナレッジ検索は選択でフィルタリングされません）。それらのエージェントをディスパッチするはずのステージがフィルタリングされるため、何かに参照されない限り不活性です。

コンパイル済みのスコープグリッドには、有効なスコープ識別情報だけが含まれます。無効なプラグインのスコープファイルはディスク上に残りますが、プラグインが再度選択されるまでは有効なランタイムスコープではありません。コアが無効で、有効なプラグインスコープ所有者がちょうど一つの場合、フリーフォーム/デフォルトスコープのフォールバックは、そのプラグインのアルファベット順で最初のスコープを使います。複数のプラグインスコープ所有者が有効で、コアの `feature` フォールバックが使えない場合、オーケストレーターはエラーとなり、明示的な `--scope` を求めます。

プラグインを無効化すると、そのプラグイン自身のファイルだけでなく、コアステージへマージした内容も取り除かれます。合成は、実際に適用した構造上の追加（対象ステージごとの produces / sensors / consumes / required_sections）を `tools/data/plugin-contrib-<key>.json` のプラグイン別サイドカーに記録し、挿入されたプローズフラグメントは自身のセンチネルマーカーを持ちます。無効化時、`select-plugins` は同じロールバックトランザクションの中で両方をインストール済みのステージソースから取り除くため、無効化されたプラグインのコントリビューションが有効なステージを操り続けることはありません。再有効化すると、次のセッション開始時に復元されます。プラグインの合成フックが再マージし、バイト単位で同一になります。

選択はコンパイル時に閉包チェックされます。有効なステージは、そのプロデューサーステージがすべて無効化されている成果物を必須にできません。エラーは、消費側ステージ、成果物、無効化されたプロデューサーステージ、それらを提供するプラグインを名指しした上で、それらのプラグインを有効にするか消費側を無効にするよう指示します。これにより、必須入力が枯渇したままステージをルーティングしかねないプラグインのみの選択を捕捉します。無効なステージを指す `requires_stage` エッジはエラーには**なりません**（依存先が決して実行されないなら順序エッジは空虚に成立します。プラグインのみのインストールがプラグインステージをコアステージの後に順序付けるのは正当です）が、ドクターはそのようにドロップされたエッジを勧告として一覧します。

`select-plugins` はまた、アクティブなワークフローを立ち往生させる変更を拒否します。実行中のワークフローのスコープを所有するプラグイン、またはプランに保留中の EXECUTE ステージを所有するプラグインの無効化は、各依存関係を名指しして拒否されます（先にワークフローを完了または休止させるか、プラグインを有効のままにしてください）。すでに立ち往生を起こしている選択には、ドクターがハードフェイルします。

選択がすでに存在する場合、プラグインを合成しても自動では有効になりません。合成フックは引き続きプラグイン自身のファイル（ステージ、スコープ、エージェント、ナレッジ、センサー、ツール — すべてランタイムでフィルタリングされます）をコピーし、そのプラグインを公開するための `select-plugins` コマンドを名指しした勧告ドロップを記録しますが、無効の間はコアステージのソースへコントリビューションをマージ**しません**。マージ済みのコントリビューションは選択フィルターを迂回するため、マージしてしまうと無効化時の除去がセッション開始のたびに巻き戻されてしまうからです。選択キーがない場合、合成されたプラグインは直ちにアクティブになり、従来の現状が維持されます。

`bundle` は今日、意図的に使われていません。この語は将来あり得る「プラグインの集合（collection of plugins）」概念のために予約されており、プラグインの所有権は常に `plugin:` で表現します。

<a id="6-the-contribution-seam"></a>
## 6. コントリビューションシーム

**コントリビューション**とは、プラグインが名前付き既存ステージを加法的に変更するために提供する `contributions/<phase>/<slug>.md` のファイルです。対象そのものを編集することはありません。

```yaml
---
target: build-and-test        # the existing core stage being enriched
plugin: test-pro
adds:                         # STRUCTURAL — set-unioned into the stage node
  produces:
    - test-pro-regression-suite            # plugin-namespaced (§8)
  consumes:
    - artifact: test-pro-testability-requirements
      required: false
  sensors:
    - coverage-threshold
  required_sections:
    - "Branch Coverage"        # declared H2 (merged into the stage; not machine-enforced yet — see §9)
fragments:                    # PROSE — spliced into the stage body
  - anchor: after-step:9
    order: 100
---

## fragment: after-step:9

### Step 9a (test-pro): Branch + coverage enrichment
…prose the agent reads, inserted after the target stage's Step 9…
```

`bundle:` は予約済みで未使用です。プラグインの所有権には `plugin:` と書いてください。

**マージの意味論:**

- **構造サーフェス** — 対象ステージのソースフロントマターへ**集合和**を取ります。可換で順序に依存せず、調整していない作成者間でも安全です。*現在実装済み:* `produces`、`consumes`（成果物 + `required` + `conditional_on` をそれぞれ保持）、`sensors`、`required_sections`。*未マージ（延期）:* `adds.scopes` と `adds.requires_stage`。コントリビューションで宣言はできますが、合成フックはマージせずドロップログへ記録します（`--doctor` で確認可能）。よって欠落が黙殺されることはありません。これらが実装されると他と同様に集合和を取ります。
- **プローズフラグメント**（ステップ/質問プローズの `fragments`）— 宣言されたアンカー位置でステージ本文へ挿入し、`(order, plugin)` により決定的に並べます。挿入される各ブロックはコンテンツハッシュ付きセンチネルで囲まれるため、再合成は冪等であり、更新されたフラグメントは以前のブロックを置換します。フックの実行順にかかわらず、別々のプラグインのブロックは `(order, plugin)` で交互配置されます。エージェントは実行時にベース本文と順序付けられたフラグメントを読み取ります。
- **上書きは決してしない。** コントリビューションができるのは追加だけです。ステージの `lead_agent` の変更、`consumes[].required` の緩和、フィールドの削除、既存ステッププローズの置換はできません。アップストリームの挙動を*変更*する真の必要性はフレームワークレベルの判断であり、プラグイン内の静かなパッチではありません。

**フラグメントアンカー:**

| アンカー | フラグメントの挿入位置 |
|--------|------------------------|
| `after-step:<n>` | `### Step <n>` のコンテンツ直後（次の見出しの前） |
| `before-step:<n>` | `### Step <n>` の直前 |
| `after-questions` | 質問生成ステップの後 |
| `end-of-steps` | `## Steps` ブロックの末尾 |
| `in:<Compartment>` | 指定された `## <Compartment>` ブロックの末尾 |

**サーフェス別** — アップストリーム変更の種類ごとにプラグインが使う方法です。「ステータス」は、合成フックが現在マージするものと、設計済みながら延期されているものを示します。

| 変更 | 機構 | ステータス |
|--------|-----------|--------|
| ステージに新しい質問をさせる | 質問プローズの `fragments` | ✅ 実装済み |
| ステージに追加成果物を生成させる | `adds.produces` + それを出力する `fragments` ステップ | ✅ 実装済み |
| ステージに新しいセクションを要求させる | `adds.required_sections` | ✅ マージ済み（⚠️ 宣言的 — まだ機械的には強制されない、§9） |
| ステージに検証を追加する | `adds.sensors`（マニフェストと `tools/` スクリプトを提供） | ✅ 実装済み |
| 消費エッジを追加する | `adds.consumes` | ✅ 実装済み |
| `requires_stage` エッジを追加する | `adds.requires_stage` | ⏳ 延期（宣言 → ログ、未マージ） |
| 既存ステージをプラグインスコープに含める | `adds.scopes`（またはスコープ自身の `includes_*` — §7） | ⏳ 延期（宣言 → ログ、未マージ） |
| フェーズポリシー / ガードレールを注入する | `memory/phases/<p>.md` をデフォルトスペースのシードへ提供（§7） | ⏳ 延期（未投影） |

<a id="7-methodrules-agents-knowledge-scopes-and-activation"></a>
## 7. メソッド/ルール、エージェント、ナレッジ、スコープ、アクティベーション

このセクションでは、新しいステージとコントリビューションシームを超えたサーフェスを、作成者を誤解させないようステータスを付けて説明します。

**メソッド/ルール → メモリシード** *(⏳ 延期).* フレームワークのルールレイヤーは、`core/memory/` からシードされるスペースごとの**メモリ**ツリー（`aidlc/spaces/<space>/memory/{org,team,project}.md`、`phases/<phase>.md`）です。設計上は、プラグインが `memory/phases/<p>.md` を提供し、そのシードに集合和を取ります。パッケージャーはまだプラグインの `memory/` ツリーを投影しないため、現在は効果がありません。

**エージェント** *(✅ 投影・合成済み).* プラグインは新しいペルソナを `agents/<plugin>-<role>-agent.md` として提供します。フロントマターの `name` はファイル名の語幹と等しく、`plugin: <plugin>` を持ちます。合成はコアや他のプラグインを上書きすることなく `<harness>/agents/` へコピーします。同一内容のファイルは冪等にスキップされ、同じ宛先に異なる内容がある場合はドロップログへ記録されます。

Kiro CLI/IDE、Codex、OpenCode では、エンジンロスター内の Markdown ペルソナは `mode: inline` に対してのみ利用可能です。ネイティブディスパッチにはさらに、ハーネスごとのディスパッチサーフェスが必要です。Kiro では手書きの agent-v1 JSON とコンダクターの `trustedAgents` リストへの登録、Codex ではエージェント設定 TOML（出荷される `aidlc-*-agent.toml` の形）、OpenCode ではネイティブの `.opencode/agents/` サブエージェントファイルです。したがってコンポーズは、ディスパッチトポロジー（リードとサポートに対する `mob`、`pipeline`、`subagent`。モードにかかわらず、ゲート付きステージの `reviewer:` も対象）が、完全にインストールされたディスパッチサーフェスを持たないエージェントを名指しするプラグインステージを拒否し、そのステージ、エージェント、修復方法をコンポーズのドロップログへ記録します。Kiro では JSON と `trustedAgents` 登録が独立して検査され、片方しかない場合でもステージは拒否されます。OpenCode — コンポーズ自身がネイティブサーフェスを出力する唯一のハーネス — では、プラグインが同梱するペルソナがネイティブツイン出力を通過できる場合（閉じたフロントマターで、投影不能な `disallowedTools` がない）にサーフェスとして数えられます。Kiro/Codex のサーフェスは常に手書きなので、プラグイン自身のファイルがこれらの検査を満たすことはありません。不足しているサーフェスを手書きしてコンポーズを再実行すれば、ステージは受理されます。Markdown ペルソナは、それを併用する受理済みインラインステージのために引き続き合成されます。

`agent-team` はスキーマ上予約されていますがランタイムのコンシューマーが存在しないため、コンポーズはそれを選択するプラグインステージを、黙ってインライン扱いにするのではなく、すべてのハーネスで拒否します。インストール済みのステージパーサーが利用できない場合、Kiro/Codex/OpenCode のコンポーズはフェイルクローズします。明示的な `mode: inline` スカラーを持ち `reviewer:` を持たないステージだけが受理されます（引用符付きスカラー形式も認識されます）。no-clobber なアップグレードは古いフックが合成したステージを削除できないため、これらのディスパッチ検査に失敗する既存ステージはディスク上に残りますが、修復方法を名指しする劣化ヘルス行を出力します。

**ナレッジ = 方法論のみ** *(✅ 投影・合成済み).* プラグインはエージェントごとの方法論ナレッジを `knowledge/<agent-slug>/` に提供し、`<harness>/knowledge/<agent-slug>/` へ合成されます。ドメイン/スペースナレッジ（`aidlc/spaces/<space>/knowledge/`）は、ブートストラップ時に空のユーザーランタイム状態であり、プラグインは提供もシードも行いません。

**スコープ** *(✅ 投影・合成済み).* プラグインスコープの識別情報は `scopes/<plugin>-<name>.md` の一つのファイルであり、フロントマターの `name` はファイル名の語幹と等しく、`plugin: <plugin>` を持ちます。合成は上書きすることなく `<harness>/scopes/` へコピーします。プラグインが作成したステージのメンバーシップは、それらのステージの `scopes:` フロントマターを通じて機能します。既存のコアステージへ `adds.scopes` でプラグインスコープを追加することは引き続き延期中で、マージされずログ記録されます（§6）。

**アクティベーション（`when:`）** *(⚠️ 解析済み、未評価).* ステージは構造化された `when:` 述語を持てます。`{producer-in-plan: X}` はスキーマ検証・解析されますが、**まだそれを評価するエンジンコンシューマーはありません**。`aidlc-graph` が将来の実装先です。したがって、`when:` を持つステージは現在、宣言されたスコープの下で無条件に EXECUTE されます。プラグイン自身のステージが存在するのはプラグインが選択セットに含まれる場合だけなので、「このプラグインが有効か」はすでに合成時の事実です。

**`plugin.json` の `aidlc.contributes`** *(⏳ 延期).* マニフェストは `aidlc.contributes` ブロックを持てますが、パッケージャーは現在そのブロックではなくディレクトリ慣例（`stages/`、`sensors/`、`tools/`、`contributions/`、`scopes/`、`agents/`、`knowledge/`）でコンテンツを発見しており、まだ読み取りません。`aidlc.lock.json` も読み取りません。現在、コンポーザーがロックファイルから解決するものはありません。

<a id="8-multi-tenant-guards"></a>
## 8. マルチテナントガード

互いに調整を行わない独立した作成者の安全は、次によって確保されます。

- **名前空間化。** コントリビュートする成果物の論理名には `<plugin>-` 接頭辞を付けます。`core-*` は予約済みです。プラグインのステージ、エージェント、スコープ、センサーは、選択セット全体およびコアに対してスラッグが一意であるべきです。プリミティブファイルの衝突は no-clobber（上書きせず）で、出所付きでドロップログに記録されます（暗黙のシャドーイングはありません）。
- **依存関係解決。** `dependencies` は Git タグに対してセマンティックバージョニングで解決します。循環は拒否され、満たせない依存関係は要求プラグイン名を含む合成エラーになります。
- **決定的な順序。** 唯一の非可換サーフェスであるプローズフラグメントは、ロード順ではなく明示的な `(order, plugin)` で順序付けられます。
- **競合は可視化される。** 同じステージで同じフラグメントアンカー・同じ順序、満たせないプラグイン間エッジ、プリミティブパスの重複といった真に非可換な衝突は、オーバーレイ順で解決するのではなく、出所付きでドロップまたは拒否されます。

<a id="9-as-built-emission-install-and-the-worked-example"></a>
## 9. 実装済みの出力、インストール、および実例

`bun scripts/package.ts` は `plugins/<name>/`（`.aidlc-plugin/plugin.json` を持つ任意のディレクトリ）を検出し、`dist/plugins/<name>/<harness>/` にハーネスごとのホストプラグインを出力します。これは四つのハーネスツリーに加わる投影先です。各投影にはホストネイティブマニフェスト（`.claude-plugin/` / `.codex-plugin/` / `.kiro-plugin/`）、`marketplace.json`、合成フック、プラグインコンテンツが含まれます（ステージは完全な `number`/`plugin`/`when` フロントマターを持ち、スキーマがネイティブに受け入れます）。合成フックは、GNU 固有シェルを使わない Bun の単一ポータブル `compose.ts` で、**ハーネス非依存**です。プラグインルートは `CLAUDE_PLUGIN_ROOT | PLUGIN_ROOT | AIDLC_PLUGIN_ROOT`、プロジェクトディレクトリは `CLAUDE_PROJECT_DIR | AIDLC_PROJECT_DIR | PWD`（Codex はプロジェクトディレクトリ変数を設定しないため PWD がフォールバック）、ハーネスのリーフは各ホストのフックコマンドがエクスポートする `AIDLC_HARNESS_DIR` から解決します。新しいステージ / スコープ / エージェント / ナレッジ / センサー / ツールを上書きせずにコピーし、シームを冪等にマージし（コンテンツハッシュ付きセンチネル挿入、書き込み前比較）、ドロップすべきコントリビューション（対象欠落、不正なアンカー、インストール済みエンジンが受け付けないキー）はセッションを失敗させず、プラグインごとの `<hooksHealthDir>/plugin-compose-<key>.drops` ファイルに記録します。これはコアフックが書き込み `/aidlc --doctor` が走査する、同じスペース単位のヘルスディレクトリです。

**ホスト別のインストール:**

```bash
# Claude Code
/plugin marketplace add <repo-or-path>/dist/plugins/<name>/claude
/plugin install aidlc-<name>@aidlc-plugins        # SessionStart hook composes on next session

# Codex CLI (in a git repo)
codex plugin marketplace add <…>/dist/plugins/<name>/codex
codex plugin add aidlc-<name>@aidlc-plugins       # approve the one-time hook trust

# Kiro (no store — folder-drop, then run the composer explicitly).
# PLUGIN_ROOT is the emitted projection dir (it carries hooks/compose.ts + the
# plugin content); PROJECT_DIR is the install you dropped .kiro into.
PLUGIN_ROOT="$(pwd)/dist/plugins/<name>/kiro"
cp -r "$PLUGIN_ROOT"/. <project>/
AIDLC_PLUGIN_ROOT="$PLUGIN_ROOT" AIDLC_PROJECT_DIR="<project>" \
  AIDLC_HARNESS_DIR=.kiro bun "$PLUGIN_ROOT/hooks/compose.ts"
# ^ the one working invocation today. A `.kiro.hook` auto-fire and an
#   `aidlc plugin compose` wrapper CLI are designed but NOT yet wired (see Status).
```

その後、`/aidlc --doctor` は有効セットを反映します（例: `core + test-pro` では 34 ステージのグラフ、`select-plugins` で絞り込んだ場合はフィルター済みグラフ）。スコープ指定実行（`/aidlc --scope enterprise`）は、各ステージのスコープがパス上に置く場所へプラグインステージをルーティングします。

**実例 — 混在環境におけるテストプロ。** プラットフォームチームは `test-pro` を一度だけ公開します（`core/` に対して作成し、`bun scripts/package.ts` を実行し、`<plugin>--v<version>` タグをプッシュし、`marketplace.json` を配置します）。Claude チームは `/plugin install`、Codex チームは `codex plugin add`（信頼を一度承認）、Kiro チームは `git pull` の後に上記のとおりコンポーザーを明示的に実行します。いずれの場合も、コンポーザーはテストプロの二つの新しいステージ**と**、`build-and-test`/`nfr-requirements`/`nfr-design`/`performance-validation` へのコントリビューションをマージします。結果は同じ、拡張済みで 34 ステージ、診断が正常なインストールです。四つすべてのハーネス投影（Claude、Codex、Kiro CLI、Kiro IDE）で検証済みです。

**ステータス。** 実装・検証済み: `number`/`name`/`plugin`/`when` のスキーマサポート（`aidlc-stage-schema.ts`）、作成時の `plugin` 所有権をコンパイル済みステージノードへ引き継ぐコンパイル側の処理（コアはこのフィールドを省略）、`harness.json` + `select-plugins` によるインストール時選択（全グラフの永続化、フィルター済みランタイムロード、閉包チェック、ランナーの刈り込み、doctor の行、合成の助言ドロップを含む）、プラグイン名前空間付きのステージ / スコープランナー生成、パッケージャーエミッター（四つすべてのハーネス投影）、プラグインの `stages/`・`scopes/`・`agents/`・`knowledge/`・`sensors/`・`tools/` の投影と no-clobber 合成、ハーネス非依存の合成フック（`scripts/plugin-hooks-template/compose.ts`）、`produces` / `consumes` / `sensors` / `required_sections` とプローズフラグメントのコントリビューションシーム（コンテンツハッシュ付き、冪等、決定的順序）。`tests/integration/t188-plugin-compose.test.ts`（合成機構）、`tests/integration/t224-plugin-selection.test.ts`（選択）、および各プラグイン自身の `tests/`（コンテンツ、統合ティアに接続）が保護します。**延期 / 未接続:** `.kiro.hook` の自動実行と `aidlc plugin compose` ラッパー CLI（現在 Kiro は明示的な `bun compose.ts` 呼び出しで合成）、プラグインの `memory/` サブツリーの投影・マージ、`adds.scopes` / `adds.requires_stage` のマージ（宣言 → ログ）、`when:` 述語の評価（解析済み、エンジンコンシューマーなし）、マージ済み `required_sections` の機械的強制（フィールドはマージ・検証されるもののコンパイル済みノードに届かず、提供される必須セクションセンサーはテンプレートから期待値を導くため、宣言されたセクションの欠落でステージが失敗することはまだない）、`after-questions` フラグメントアンカー（`locateAnchor` にケースがなく「不明なアンカー」をドロップログに記録するため、`after-step:<n>` を使用）、`aidlc.contributes` / 任意のロックファイル / `dependencies` の読み取り、作成時の `number` をコンパイル済みノードへ引き継ぐ処理（現在のステージは再シード / 固定された番号を使用）。

<a id="9-invariants"></a>
## 9. 不変条件

- **コアは不変。** プラグインが `core/` を編集することはありません。
- **加法性のみ。** コントリビューションは追加し、上書き・削除は決してしません。
- **無効時は不活性。** すべてのプラグインを無効にすると、ベアコアとバイト単位で同一になります。
- **一つのコンポーザー、ホストが起動。** 実行場所にかかわらず同じコードが合成します。中央で事前ビルドされる唯一の成果物はベアコアです。
- **プラグインはホストプラグインそのもの。** パッケージャーは実際の `.claude-plugin/` / `.codex-plugin/` / `.kiro-plugin/` マニフェストを出力し、ホストのネイティブコマンドでインストールされます。AIDLC は配布インフラを運用しません。
- **スラッグが識別子、番号は表示専用。** プラグインステージを挿入してもコアの再採番は起きません。
- **信頼はホストネイティブ。** 組織の制限にはホストの管理対象許可リストを使い、AIDLC 独自の信頼レイヤーは作りません。
- **ゲートキーピングなし。** ファーストパーティとサードパーティのプラグインは機構上同等で、違いは来歴だけです。
