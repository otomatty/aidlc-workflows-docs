<a id="agent-structure"></a>
## エージェントの構造

各エージェントは、`.claude/agents/` に置かれるフラットな `.md` ファイルで、YAML のメタデータ部とそれに続く Markdown 本文から成ります。コンダクターは、インラインのステージ実行中に自らの視点を組み立てるため、あるいはサブエージェント委譲のためのコンテキストを構築するために、これらのファイルを読みます。

<a id="frontmatter-contract"></a>
### メタデータ部の契約

すべてのエージェントファイルには、次の YAML メタデータ部を含めなければなりません。

```yaml
---
name: aidlc-architect-agent               # Agent identifier (matches filename without .md)
description: >                      # Brief role summary (shown in Claude Code agent list)
  System architect responsible for application design,
  NFR design, and component decomposition.
disallowedTools: Task               # Agents cannot spawn subagents
tier: judgment                      # judgment | balanced | templated (see Agent Tiers)
---
```

| 項目 | 必須 | 説明 |
|-------|----------|-------------|
| `name` | はい | エージェント識別子。ファイル名と一致しなければなりません |
| `description` | はい | 役割の短い要約 |
| `tools` | いいえ | 任意の許可一覧です。省略すると完全なセッションツールセットを継承します。列挙するとエージェントは絞り込まれ、`mcp__<server>__<tool>` 識別子も明示しない限り継承済み MCP ツールは落ちます |
| `disallowedTools` | はい | `Task` を含めなければなりません。委譲を行うのはコンダクターだけです |
| `tier` | はい | `judgment`、`balanced`、`templated` のいずれかです。これは作成元側のダイヤルであり、パッケージャーが各ハーネスのネイティブなモデル／推論量キーへ射影します（後述のエージェントティアを参照）。生の `model:` / `effort:` は作成元のメタデータ部には決して現れません。これらは `dist/<harness>/` における射影結果です |

<a id="markdown-body-sections"></a>
### Markdown 本文セクション

メタデータ部の下では、Markdown 本文が次を定義します。

| 節 | 目的 |
|---------|---------|
| **中核責務** | 各自が担当するステージでそのエージェントが何を行うか |
| **担当ステージ** | 主担当および支援担当のステージ割り当て |
| **連携** | 入力元／協働相手／引き渡し先 |
| **ナレッジ読み込み** | 6 段階のロード順序（[ナレッジシステム](/reference/knowledge-system) を参照） |
| **主要原則** | そのエージェントの行動指針 |

<a id="shared-configuration"></a>
## 共有設定

14 体すべてのエージェントは、共通の設定ベースラインを共有します。どれも `tools:` の許可一覧を宣言していないため、すべてのエージェントは**完全なセッションツールセット**を継承します。つまり Claude Code のすべての組み込みツールと、セッションへプロビジョニングされた MCP ツール一式です。出荷時点で唯一の制限が `disallowedTools: Task` です。

<a id="the-session-toolset-inherited-by-every-agent"></a>
### セッションツールセット（すべてのエージェントが継承）

すべてのエージェントは、次を含む Claude Code 組み込みツールを継承します。

| ツール | 目的 |
|------|---------|
| 読み取り | ファイルシステムからファイルを読む |
| 編集 | ファイル内で正確な文字列置換を行う |
| 書き込み | ファイルシステムへファイルを書き込む |
| ファイル照合 | 高速なファイルパターン照合 |
| 内容検索 | 高速検索器を使った内容検索 |
| ユーザーへの質問 | 対話的なユーザープロンプト（メインスレッドのステージのみ） |

<a id="common-disallowed-claude-code-tools"></a>
### 共通の禁止 Claude Code ツール

| ツール | 理由 |
|------|--------|
| タスク起動 | エージェントは委譲されたワーカーとして動作します。`Task` 呼び出しを行うのは `SKILL.md` コン ダクターだけです。`disallowedTools: Task` はサブエージェント連鎖の多段化を防ぎます。 |

<a id="tools-each-persona-is-expected-to-exercise"></a>
### 各ペルソナが使うことを想定されているツール

すべてのエージェントは継承によりシェルとウェブ検索へ到達できます。この表が記録しているのは、方法論が各ペルソナに対してそのステージ作業の中で**使うことを期待している**かどうかであって、エージェント別の付与ではありません。あるペルソナを本当に制限したい場合は、任意の `tools:` の許可一覧を追加してください（その場合、`mcp__<server>__<tool>` 識別子も列挙しない限り継承済み MCP は落ちます）。この実装ではそのような制限は何も出荷していません。

| ツール | 使用を期待する担当 |
|------|---------------------|
| シェル | クラウド基盤、開発セキュリティ、開発、品質、パイプライン配備、運用の各エージェント |
| ウェブ検索 | プロダクト、デザイン、コンプライアンスの各エージェント |

<a id="agent-tiers"></a>
### エージェントティア

各エージェントにおける作成元側のダイヤルは `tier:` です。これはそのペルソナが行う仕事の**種類**を示し、パッケージャー（`bun scripts/package.ts`）がそれを各ハーネスのネイティブなモデル／推論量形式へ射影します。以前の挙動（バージョン 2.2.15 から 2.2.19。さらにそれ以前はキーが無効な `modelOverride:` でした）は、各エージェントごとに生の `model: opus|sonnet` を固定しており、その結果としてより大きなモデルで実行中のセッションまで強制的にダウングレードしていました。ティア射影はその固定を置き換えるものです。

| ティア | エージェント | 意味 |
|------|--------|---------|
| `judgment` | アーキテクト、クラウド基盤、コンプライアンス、コンポーザー、デザイン、開発、開発セキュリティ、プロダクト、品質 | 曖昧さの中で複数制約を推論し、その出力が下流へ波及する仕事です。決してダウングレードされず、セッションのモデルと推論量の両方を継承します |
| `balanced` | アーキテクチャレビュー、プロダクトリード | 明示的な基準に照らして新規入力をレビューする、レビュー形状の仕事です。中規模モデルで、推論量はセッションから継承します |
| `templated` | デリバリー、運用、パイプライン配備 | 主としてパターン追従型の出力であり、方法論自体がすでにナレッジに入っています（デリバリープラン、CI/CD YAML、運用手順書）。中規模モデルを低い推論量で使う、唯一の意図的ダウングレードです |

ハーネスごとの射影は次のとおりです（`core/tools/aidlc-tiers.ts` が唯一の正本です）。

| ティア | Claude Code（`.md` のメタデータ部） | Codex CLI（TOML） | Kiro CLI/IDE（エージェント JSON の `"model"`） | Kiro の CLI 設定にある `chat.modelDefaults` |
|------|-------------------------------|-------------------|--------------------------------------|-------------------------------------|
| `judgment` | `model: inherit`、`effort:` 行なし | `model`／`model_reasoning_effort` キーなし（設定のセッション既定値が適用される） | フィールドを省略（スキーマのフォールバック: ユーザーの既定モデル） | 項目なし（既定モデル固有の既定推論量） |
| `balanced` | `model: sonnet`、`effort:` 行なし | `model = "openai.gpt-5.4"`、推論量キーなし | `claude-sonnet-4.5` | sonnet-4.5 → `high` |
| `templated` | `model: sonnet`、`effort: medium` | `model = "openai.gpt-5.4"`、`model_reasoning_effort = "medium"` | `claude-sonnet-4.5`（`balanced` と統合） | （sonnet-4.5 の項目を共有し、`balanced` の `high` が統合時に優先する） |

この表を支える重要事実:

- **継承機構は省略です。** Claude Code では、`.md` のエージェントに `effort:` キーがないとセッションの推論量を継承し、逆に `effort:` を固定すると双方向にセッションを上書きします（固定値は下限ではなく上限です）。したがって `judgment` と `balanced` では「書かないこと」が契約です。Codex では、役割用 TOML に `model` がなければ、出荷済み設定のセッション既定値で起動します（診断ツールが強制する Codex CLI の最小バージョン 0.139.0 と 0.142.5 で実環境検証済み）。Kiro では、エージェント用バージョン 1 スキーマが `model` 不在時のフォールバックを文書化しています。すなわち「指定されなければ既定モデルを使う」です（`/model` の永続化された設定）。
- **Kiro にはエージェント単位の推論量設定面がありません。** Kiro CLI はエージェント JSON に推論量類似キーがあると閉鎖的に失敗するため、推論量はモデルを介して CLI 設定の `chat.modelDefaults[<modelId>].output_config.effort` に乗ります。異なる固定モデルごとに 1 項目です。このファイルは CLI 専用であり、Kiro IDE は CLI 設定を完全に無視して、拡張機能埋め込みのモデルごとの既定値（またはユーザーの `/effort` セッション状態）を適用します。
- **Kiro の統合ルール。** 2 つのティアが同じ Kiro モデル ID を共有すると、Kiro 上では区別できません。その共有設定項目は、より高いティアの推論量を取ります（ソネット 4.5 上では `balanced` の `high` が `templated` の `medium` に勝つ）。この統合はバグではなく、意図的であり文書化済みです。

<a id="tier-cap-cost-override"></a>
### ティア上限（コスト上書き）

プロジェクトは、どのエージェントファイルも編集せずに、パッケージ化時点ですべての射影に上限をかけられます。

- **永続ノブ:** スペースメモリ層ファイル（`core/memory/org.md` → `team.md` → `project.md`、最後の書き手が優先）の YAML メタデータ部にある `tier_cap:` キーです。プロジェクトは組織上限を下げることも上げることもできます。例: `tier_cap: balanced` は、全ハーネスの射影で `judgment` を `balanced` へ統合します。
- **呼び出し単位の上書き:** `AIDLC_TIER_CAP` 環境変数は、1 回のパッケージャー実行に限ってメモリ層に勝ちます（`AIDLC_TIER_CAP=templated bun scripts/package.ts`）。メモリ上限が有効な状態で一度だけ上限なしでビルドしたい場合は、最上位ティアである `judgment` を設定してください。これはメモリ層より優先され、何も制限しません（空値は上限なしではなく未設定を意味します）。

この 2 つのノブはスコープが異なります。メモリ上限はリポジトリと共に移動するため、書き込みモードでも `--check` モードでも適用されます（上限付きの配布物をコミットしたプロジェクトは、その状態で自己整合になります）。一方、環境変数は一回限りの書き込みノブであり、`--check` では無視されます。差分検知器は、コミット済み配布物が何を正当に元にしてビルドされたかを比較するため、CI やテストランナー環境に紛れた `AIDLC_TIER_CAP` が差分検知を失敗させても隠してもいけません（パッケージャーはそれを無視したとき通知を出力します）。またパッケージャーは、上限が有効な実行では毎回、どの上限がどこから来たかも表示します。

単一のエージェントだけを除外したい場合は、代わりにインストール済み `dist/<harness>/` コピー内の射影済み値を編集してください（たとえば Claude のエージェント `.md` 1 つに `model: opus` を設定するなど）。この編集は配布物の外殻を再コピーするまで残ります。

**所見:**
- アーキテクトは最も広くステージへ関与しています（3 フェーズにまたがる 9 ステージ）。
- 全 14 エージェントの構成では、9 体が `judgment` ティアを持ち、5 体がそれより 1 段下がります（2 体の `balanced` レビュワーと 3 体の `templated` プランナー）。段を下げたエージェントは、明示的チェックリストに対するレビュー、または主としてテンプレート化された計画／CI/CD／運用手順書作業を担います。上のマトリクスが対象としているのは 11 体のドメイン専門エージェントです。
- コンプライアンスは純粋に助言的な立場で動作します（4 つの支援ステージ、主担当ステージは 0）。
- 11 体中 6 体がシェルへアクセスでき、いずれも CLI 対話が必要な役割です。
- 3 体のエージェントが調査タスクのためにウェブ検索へアクセスできます。

<a id="how-to-add-an-agent"></a>
## エージェントを追加する方法

エージェント表示名とナレッジファイル例は、各エージェントの `.md` メタデータ部にある `display_name` と `examples` フィールドが正本です。TypeScript の編集は不要です。完全な手順（必須メタデータ部フィールド、検証手順、自動検証される部分と手動検証部分の境界）は [コントリビューション: エージェントの追加](/reference/contributing#adding-an-agent) を参照してください。以下は手順の要約です。

1. `core/agents/{name}-agent.md` を、必須メタデータ部の `name`、`display_name`、`examples`、`description`、`disallowedTools`（`Task` を含む）、`tier` とともに作成します。コアのメタデータ部に生の `model:` / `effort:` を書いてはいけません。これらは射影結果です（上のエージェントティアを参照）。任意の `tools:` の許可一覧は継承ツールセットを絞り込みます。完全なセッションツールセットを継承するなら省略してください。`core/tools/aidlc-lib.ts` の `loadAgents()` は、次回呼び出し時にそのファイルを発見します。
2. `core/knowledge/{name}-agent/` にナレッジファイルを追加します。
3. そのエージェントが参加するステージファイル `core/aidlc-common/stages/` にエージェントを追加します。各ステージのメタデータ部にある `lead_agent` / `support_agents` を設定してください。コンパイル済み `tools/data/stage-graph.json` は、そのメタデータ部から `bun scripts/package.ts` により生成されます。手編集してはいけません（`package.ts --check` の差分検知器が、手編集した配布物に対して CI を失敗させます）。
4. 配布物を再生成します。`bun scripts/package.ts` を実行し、その後 `--check` で差分がないことを確認します。
5. 手動管理のナレッジ表に、そのエージェントから例への行を追加します（スペースレベルのチームナレッジディレクトリは `aidlc/knowledge/{name}-agent/` であり、内容があるときにチームが作成します。エンジンは足場を作りません）。
6. テストを更新します。ファイル存在用のスモークテスト、ステージとエージェントの相互参照用フィーチャーテストを追加します。
7. このファイルと [エージェント技術リファレンス](https://github.com/awslabs/aidlc-workflows/blob/main/docs/reference/agents/) の文書を更新します。

<a id="how-to-modify-an-agent"></a>
## エージェントを変更する方法

- **ツールを変更する:** メタデータ部に `tools:` の許可一覧を追加または編集してエージェントを絞り込んでください。完全なセッションツールセットを継承するなら省略します。`tools:` 一覧は、`mcp__<server>__<tool>` 識別子も併記しない限り継承済み MCP ツールを落とします。
- **ティアを変更する:** `tier:` を `judgment`、`balanced`、`templated` のいずれかへ編集し、`bun scripts/package.ts` で再生成してください。代わりに、インストール済みコピー内の単一エージェントだけに特定モデルを強制したい場合は、`dist/<harness>/` のエージェントファイルにある射影済み `model:` を直接編集します（Claude Code は別名、完全識別子、`inherit` を受け入れます）。
- **振る舞いを変更する:** Markdown 本文セクション（責務、原則）を編集します。
- **ステージ割り当てを変更する:** エージェントファイル（担当ステージ節）と、対応するステージファイル（`core/aidlc-common/stages/`）の両方を編集し、その後 `bun scripts/package.ts` で再生成してください。コンパイル済みステージグラフはステージのメタデータ部由来であり、手編集はしません。
