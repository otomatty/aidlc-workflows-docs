> 対象読者: 第 2・第 3 層（チーム導入者、フレームワークコントリビューター）。

この章は、バージョン 0.5.0 のルールシステムに関するスキーマレベルのリファレンスです。ルールファイルがどこに置かれるか、スコープをどう導出するか、継承チェーンがどう解決されるか、どのメタデータフィールドが有効かを定義します。これは、解決器（`aidlc-graph.ts compile`）と診断コマンドのルールドリフト検査が参照する仕様です。ルールは制御ループにおけるフィードフォワード側であり、対になる決定論的検証側は [センサーシステム](/reference/sensor-system) が扱います。学習ループ儀式、ANZ の実例、確認済み学習がどうルールファイルへ着地するかといった利用者向け説明は、ユーザーガイドの [ルールと学習ループ](/guide/rules-and-the-learning-loop) を参照してください。

<a id="layout"></a>
## 配置

ルールは、アクティブなスペースのメモリ層 `aidlc/spaces/<active-space>/memory/` に置かれます。ここはワークスペースルートにある、人が直接編集できる単一の集合であり、各ハーネスはそれぞれのネイティブな取り込み機構（Claude の `@` による取り込み用スタブ、Kiro のリソースグロブ、Codex の `AIDLC_RULES_DIR`）を通じてこれを読みます。ファイル名は中立なスコープ名です。

```
aidlc/spaces/<active-space>/memory/
├── org.md
├── team.md
├── project.md
└── phases/
    ├── ideation.md
    ├── inception.md
    ├── construction.md
    └── operation.md
```

この配置は、チーム著述のハーネス設定（以前は別のプラクティス名前空間にあったもの）と、自己学習型ガードレール（以前は 2 階層だったもの）を、`aidlc/spaces/<active-space>/memory/` という 1 つのアクティブなスペースメモリディレクトリに統合したものです。

<a id="filename-derived-scope"></a>
## ファイル名から導出するスコープ

ルールファイルは `scope:` メタデータフィールドを**持ちません**。スコープはファイル名から導出されます。

| ファイル名パターン | スコープ |
|---|---|
| `org.md` | `org` |
| `team.md` | `team` |
| `project.md` | `project` |
| `phases/<phase>.md` | `phase`（フェーズ値 = ファイル名） |

組織・チーム・プロジェクト・フェーズのルールは、パススコープ用メタデータを持ちません。関係はステージ側からの取り込みを著述することで置かれます。組織・チーム・プロジェクトの各ルールは、ファイル名由来の共通既定値としてすべてのステージへ適用されます。対応するフェーズルールは、ステージメタデータの `phase: <name>` フィールドが `phases/<name>.md` を取り込むため、自動で付与されます。

<a id="five-layer-inheritance"></a>
## 5 層継承

ルールは、ワークフロー開始時に次の 5 層チェーンで解決されます。

```
org → team → project → phase → stage
```

組織層はフレームワーク既定値を担い、チーム層とプロジェクト層はチームが確認したプラクティスとプロジェクト固有の特殊化を追加します。フェーズ層は直交する層であり、ステージがメタデータに `phase: <name>` をすでに宣言しているため、対応する `phases/<name>.md` を付与します。これは `requires_stage` や `consumes` と同じ著述方向です。ステージルールは将来予約済みです。将来 `aidlc-stage-<slug>.md` が著述されるようになれば、各ステージの `slug:` 宣言経由で付与されます。このチェーンが実装している広い 2 軸設定モデルについては [アーキテクチャの「設定レイヤー」](/reference/architecture) を参照してください。

コンパイル出力では、`stage-graph.json` の各ステージノードにある `rules_in_context` フィールドへ、この解決済みチェーンが焼き込まれます。実行時にチェーンをたどり直すことはありません。解決はコンパイルの責務です。

<a id="strict-additive-runtime-model"></a>
## 厳密加算の実行時モデル

適用対象のルールは、すべて `rules_in_context` に現れます。組織・チーム・プロジェクトのルールは単純連結され、実行時に落とされるものはありません。フェーズルールは、そのファイル名がステージの `phase:` 宣言と一致したときに付与されます。グロブフィルターも、具体パスの合成もありません。エージェントはセッション開始時にこの完全なチェーンを読みます。

競合、つまりより狭いスコープがより広い方針へ矛盾する場合は、学習が解決器へ到達する前、すなわちメモリゲートである §13 の学習儀式における受け入れ検査によって拒否されます。この検査はセクション単位です。提案された日付付き学習エントリを書き込もうとする際、オーケストレーターは `memory/project.md`（または `memory/team.md`）への候補を `memory/org.md` の対応見出しと比較します。競合が見つかれば、ユーザーは**修正・スキップ・エスカレーション**を選びます。上書き経路はありません。プラクティス発見の確認ゲートも別の受け入れゲートですが、そちらの昇格は `aidlc-state.ts practices-promote` による決定論的なセクション置換であり、ユーザーの確認によって正当化されます。自動の組織ルール競合チェックは行いません。書き込み後に組織・チーム・プロジェクトの内容がずれた場合は、診断コマンドのルールドリフト行が別途可視化します。

この設計は、以前の `enforcement: enforced` キーワードと `overrides:` ブロックモデルを置き換えます。どちらのキーワードもスキーマから削除されました。メタデータパーサーは未知キーを許容する方針に従ってそれらを無視します。つまりエラーにはしませんが、解決器は参照しません。

診断コマンドのルールドリフト検査は、後から生じたドリフトをオンデマンドで可視化します。組織ルールが更新された後にチームやプロジェクトの内容がすでにディスク上へ存在している場合、診断コマンドは `memory/team.md` と `memory/project.md` の `##` 見出しのうち、*内容を持つ*組織見出しと重なるものを決定論的に見つけ、各重なりを助言候補として表示します。表示内容はファイル、セクション、引用された組織文です。行は `Rule drift: N team/project rule(s) overlap org policy (review for contradiction)` という形になります。診断コマンド自体は LLM を使いません。検出は見出しと文字列のバイト再現可能な処理です。矛盾かどうかの判断は、観測時点でそれを消費するオーケストレーター LLM が担う非ブロッキング判断です。人間はその後、修正・エスカレーション・許容を選びます。

<a id="pairing-field"></a>
## `pairing:` フィールド

ルールは、`.claude/sensors/aidlc-*.md` にある決定論的センサーと対にできます。

```yaml
---
pairing: aidlc-required-sections
---
# or
pairing: feedforward-only
---
```

有効値:

- `feedforward-only` — このルールに対応センサーがないことを明示します。つまりフレームワークでは決定論的に検証できません。
- `<sensor-id>` — 既存の `.claude/sensors/aidlc-<id>.md` の `id:` フィールドと一致しなければなりません。

診断コマンドの対カバレッジ行は、対付きルールとフィードフォワード専用ルールを数え、対になっていないルールを網羅率ギャップとして可視化します（[ルールドリフト検出](#rule-drift-detection) を参照）。

<a id="rule-drift-detection"></a>
## ルールドリフト検出

`/aidlc --doctor` には、ルール / センサー状態を観測する 2 つの助言行があります。どちらも読み取り専用で、常に成功扱いです。どちらもヘルスチェック終了コードを変えません。（バージョン 0.6.10 以降、`--doctor` はコールドセーフです。対カバレッジ検査が出す `GUARDRAIL_LOADED` 監査行は、アクティブな意図の `audit/` 分割記録がすでにある場合にだけ書き込まれます。意図がまだない新しいシェルでは、診断コマンドは行を表示しますが、イベントを出さず、ファイルも作りません。）

- **ルールドリフト** — 各チーム / プロジェクトのプラクティスファイル（`memory/team.md`, `memory/project.md`）について、診断コマンドは `memory/org.md` 側にも存在する `##` 見出しを探し、各重なりを候補ペアとして表示します。表示内容はファイル、セクション、そして組織側の最初の文の逐語引用です。組織側の本文が空、たとえばフレームワーク既定の `## Forbidden`, `## Mandated`, `## Corrections` のように HTML コメントしかない見出しは数えません。重なりは双方に内容がある必要があります。件数 N は、LLM が矛盾と認定した件数ではなく、*構造的* な候補ペア数です。診断コマンドは決定論的に検出するだけで、矛盾判定は観測時点のオーケストレーター LLM が行います（[厳密加算の実行時モデル](#strict-additive-runtime-model) を参照）。
- **対付きセンサーの網羅率** — `pairing: <sensor-id>` を持つ各ルールについて、診断コマンドは `aidlc-` 接頭辞を剥がし、そのセンサーが少なくとも 1 つのステージの解決済みセンサー集合（`sensors_applicable`）に存在するかを確認します。行は `Paired sensor coverage: P/(M-X) guardrails paired (X feedforward-only)` という形式です。M は `pairing:` 値を持つルール総数、X はフィードフォワード専用ルール数、P は名前付きセンサーが実際に解決されるルール数です。対になっていないルール、つまりセンサー ID は書かれているがどのステージにも束縛されていないものは、行内に列挙されます。これは**ファイル存在確認であって意味論チェックではありません**。束縛が解決されるかだけを確認し、そのセンサーがそのルールに適切かまでは見ません。この行は、初期化済みプロジェクトでは 1 実行あたり 1 回 `GUARDRAIL_LOADED` 監査イベントを出します（監査記録がない未使用のプロジェクトでは抑制されます。上のコールドセーフ注記を参照）。

<a id="forward-compat-policy"></a>
## 前方互換ポリシー

ルールメタデータは、加法的拡張に対して前方互換です。

- 新しいフィールドは加法的に追加されます。既存のルール行を書き換える必要はありません。
- コンシューマーは未知のメタデータキーを許容しなければなりません（無視して通す）。これにより、古い `enforcement:` / `overrides:` キーがユーザー拡張オーバーレイに残っていてもエラーなく読み込めます。
- ルール本文の慣習（`## Forbidden`, `## Mandated` のようなセクション）は、既存のガードレール規約に従います。
- 学習儀式（メモリゲート）は、確認済み学習を、その日付付きエントリとして直接スペースメモリファイルへ書き込みます。既定は `memory/project.md`、ワンクリック昇格先は `memory/team.md` です。組織スコープへの書き込みはありません。確認済み学習はそのままプラクティスであり、プラクティス発見が確認するのと同じファイル群へ入ります。別の `*-learnings.md` 面はありません。解決器はこれらを、`SCOPE_PRIORITY` による単純な整数チェーン（組織:0、チーム:1、プロジェクト:2、フェーズ:3）で整列します。1.5 や 2.5 のような中間ティアはありません。`aidlc-learnings.ts` のメモリゲートこそが、上の「厳密加算の実行時モデル」で説明した受け入れ時の競合検査の実装です（単一の提案日付エントリを `memory/org.md` の対応セクションと比較し、修正 / スキップ / エスカレーションを行う。上書き経路はありません）。

バージョン 0.5.0 より前のルールファイルも、改名と平坦化が入った後にそのまま読み込めます。移行はパスだけです。
