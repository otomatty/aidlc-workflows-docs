エンジンとコンダクター

v2のオーケストレーションは、ひとつの関心事を2つに分離しています。

エンジン（決定論的コード）

aidlc-orchestrate.ts。ステージ間ルーティングを所有します: スコープ解決、フラグ優先順位、ジャンプ方向計算、resume/initガード、ステージ順序、ゲート状態、ワークフロー完了判定。ルーティングは決定論の関心事なので、LLMのプロンプト散文ではなくツール（Bun CLI）として実装されています。

コンダクター（LLMセッション）

skills/aidlc/SKILL.md。エンジンが指名した1手の実行品質を所有します: ペルソナの体現、良い質問、ステージ日誌、ステージ内のKeep/Modify/Redoループ、ゲートでの人間への判断の提示。

エンジンはサブコマンドを2つだけ持ちます。

サブコマンド役割状態変更
nextワークフロー状態（aidlc-state.md）とコンパイル済みステージグラフ（stage-graph.json）を読み、型付きディレクティブをちょうど1つJSONで出力なし
reportコンダクターがディレクティブに基づき行動した後、遷移をコミット。--stage <slug> で対象ステージを固定あり

コンダクターの制御構造は次の転送ループだけです。

Loop:
  1. directive = `bun .claude/tools/aidlc-orchestrate.ts next $ARGUMENTS`
  2. directive.kind に応じて行動
  3. `... report --stage <stage> --result <outcome> [--user-input "..."]`
  4. directive.kind == done でなければ繰り返す

型付きディレクティブ契約

aidlc-directive.ts が kind をキーとする9種の判別可能ユニオンを定義します（現在7種を出力、2種は将来用プレースホルダ）。不正なディレクティブは出力前の検証で非ゼロ終了します。

kind出力コンダクターの動作
print○message の指示どおり実行（terminal型は表示して停止、run-then-continue型は実行してループ継続）
error○message をそのまま表示して停止
done○完了サマリーを提示して停止
parked○ステージ間境界でパーク中。/aidlc --resume での再開方法を案内して停止
run-stage○リードエージェントのペルソナ + support_agents をロードし、ステージ本文を実行、produces を書き、gate で分岐
ask○AskUserQuestion で質問を提示し、回答を次の report --user-input で返す
invoke-swarm○自律許可下の適格なConstructionバッチをスウォームへファンアウトし、収束ループを実行
dispatch-subagent—将来用プレースホルダ
present-gate—将来用プレースホルダ（現在ゲート判断は run-stage の gate フィールドに畳み込み）

ガード機構

2.1.x系で、エージェントの「暴走」を決定論的に防ぐガードが積み上げられています。

人間実在ゲート（HUMAN_TURN、v2.1.6）

承認・インタビューのチェックポイントは、監査台帳に記録された HUMAN_TURN イベント（実際の人間プロンプトでフックが記録）が、直近のゲート解決（GATE_APPROVED / GATE_REJECTED / QUESTION_ANSWERED）以降に存在しない限り拒否されます。自律IDEモード下のエージェントが1ターン内で承認をねつ造して複数ゲートを通過することを防ぎます。Kiro CLI/IDEでは preToolUse フックがゲート開放中のツール呼び出しをハードブロック（exit 2）。自律Construction（swarm/Bolt）ではデッドロック回避のためスコープ外。CI用バイパスは AIDLC_SKIP_HUMAN_PRESENCE_GUARD=1。

アーティファクトガード（v2.1.3）

ステージを [x]（完了）にするすべてのサブコマンド（approve / advance / finalize / complete-workflow）は、宣言された produces[] アーティファクトがディスク上に存在することを検証します。v2.2.9以降、状況次第でのみ書かれるCONDITIONALアーティファクトは optional_produces: に宣言でき、ユニット単位のカバレッジ検証から除外されます（REQUIREDの欠落は従来どおりブロック）。workspace_requires ステージ（code-generation）はさらに aidlc/ 外の実ソース作業を要求（git対応）。バイパスは AIDLC_SKIP_ARTIFACT_GUARD=1 のみ（--test-run はv2.1.4で完全撤廃）。v2.2.14で、ランタイムグラフの bolt_dag ノード欠損時にユニット単位ステージが黙って1イテレーションに縮退する問題が修正され、エンジンは unit-of-work-dependency.md からユニットDAGを自己修復的に再計算するようになりました（approve 時の全ユニットカバレッジガードも再計算リストに対して適用）。v2.2.11では、開いているゲートでの修正後に reject の記録を忘れた場合、approve が欠落した監査ペアを Recovered: true タグ付きでバックフィルします。

park / Stopフック / 自律ガード（v2.1.3）

park コマンドはワークフローをステージ間境界で清潔に一時停止し、Stopフックは parked を正当な終了として扱います。長いワークフローで「doneに到達するためのラバースタンプ」を防ぎます。逆に自律Constructionラン中は自己パークできず（自律ガード）、無人ループは止まりません。同じ理由で、ライブプランのEXECUTE/SKIP構成を作り替える recompose（v2.2.8）と scope-change（v2.3.3）は自律Construction中エンジンレベルで拒否されます — 無人ランには新しいプラン形状を承認する人間がゲートにいないため。会話カーブアウトにより、ワークフロー途中の雑談ターンはナッジなしで終了できます。

レビュアー機構（v2.0.0、v2.1.8で全ハーネス完全動作）

ステージはfrontmatterで任意の reviewer: を宣言でき、ステージ本文実行後・承認ゲート前に、レビュアーサブエージェントが助言的品質ゲートとして最大 reviewer_max_iterations 回（既定2）レビューします。あくまで助言であり、最終判断は常に人間の承認ゲートです。v2.1.8でCodexオーケストレーターへの§12aステップ追加・Kiroでの trustedAgents 登録・Kiroレビュアーの fs_write を aidlc/spaces/** に制限する修正が入り、全ハーネスで確実に動作するようになりました。v2.2.16でレビュアーの読み取りスコープが「レビュー対象 + consumes: の共有契約」に限定されました。

状態と監査

各インテントは aidlc/spaces/<space>/intents/<YYMMDD>-<label>/ 配下に自身の記録ディレクトリを持ちます（v2.1.0のワークスペースレイアウト）。

状態ファイル aidlc-state.md

ワークフロー進捗の単一情報源。ステージ進捗は6状態チェックボックスで表現されます。

表記意味
[ ]未着手
[-]進行中
[?]承認待ち（ゲート開放中）
[R]リビジョン中（ゲートで差し戻し）
[x]完了
[S]スキップ（スコープ除外・skip・ジャンプによるバイパス）

監査証跡

69種のイベントタクソノミーによる構造化ログ。クローンごとのシャード（audit/<host>-<clone>.md）としてコミットされ、並行クローンがマージ競合を起こしません。読み手はglobしてタイムスタンプでマージします。HUMAN_TURN（v2.1.6）や WORKFLOW_PARKED / WORKFLOW_UNPARKED（v2.1.3）や RECOMPOSED（v2.2.0、実行中プラン再構成）もこの台帳のイベントです。

プラグイン機構（v2.3.0）

オプトインの拡張機構。plugins/<name>/ で「新規ステージの追加」と「既存コアステージへの追加的コントリビューションシーム」（produces / consumes / sensors / required_sections のset-union、ステージ本文への散文フラグメント接合）を著述すると、パッケージャーがハーネスごとの実ホストプラグイン（dist/plugins/<name>/{claude,codex,kiro,kiro-ide}/）として出力します。composeフック（ポータブルな compose.ts、冪等・アップグレード対応・複数プラグイン間で順序決定的）がインストールへマージした後にステージグラフを再コンパイルし、オーケストレーターが新ステージをルーティングします。配布はハイブリッド（Claude/Codexはホストネイティブの /plugin、Kiroはフォルダドロップ + 明示compose）で、信頼判断はホストに委ねられます（AIDLC独自のトラスト層なし）。プラグイン非アクティブ時はコアがバイト同一に保たれ、既存インストールは無影響。リファレンス実装として test-pro プラグインを同梱。プラグインの agents / scopes / memory 投影やmarketplace・lockfileなどは明示的に将来対応（deferred）です。

consumes / consumes_absent 契約（v2.1.7）

リーンスコープは上流のプロデューサーステージを意図的にスキップします。v2.1.7から、run-stageディレクティブは宣言された入力をディスク上の存在で分割します: consumes には存在する入力のみが載り、新設の consumes_absent には欠落した必須入力が expected: true（スコープ設計によりプロデューサーがスキップされた＝設計どおりの欠落）または expected: false（経路上にあるのにファイルがない＝想定外）の注釈付きで列挙されます。コンダクターは前者ではステージ文書のフォールバックで進行し、後者はリカバリプロトコルで表面化します。スコープごとの意図的なショートカット辺集合はテストスイートに固定され、新たな dangling consume はCIで検出されます。
